# tools — 외부 도구 운영 가이드

이 폴더는 graphify·serena 같은 **외부 도구·플러그인·MCP를 운영할 때 필요한 정보**를 일정한 틀로 정리한다.

## 왜 있는가

도구의 README는 보통 **효과·유용성·설치 방법**까지만 다루고, 실전 운영 정보 — *컴퓨터당 1회 설치인지 / 프로젝트마다 무엇을 하는지 / 팀과 어떻게 공유하는지 / 소스가 바뀌면 재인덱싱은 자동인지* — 를 구조적으로 빠뜨린다. README가 "처음 한 번 돌려보기(first-run)"를 위한 장르이기 때문이다.

이 폴더의 가이드는 **README가 끝나는 곳에서 시작한다.** 도구를 *운영*할 때 매번 다시 하게 되는 고민을 한 번 정리해, 같은 고민을 반복하지 않게 하는 것이 목적이다.

> **담는 것 / 담지 않는 것**
> - 담는 것: 도구의 **일반 운영 성질**(모든 사용자에게 해당).
> - 담지 않는 것: 내 특정 머신의 **현재 설치 상태**(스냅샷). 이건 가이드 문서지 운영 반영 문서가 아니다.

## 어떻게 쓰나

- 새 도구를 정리할 때: [`_template.md`](_template.md)를 복사해 `<도구명>.md`로 채운다.
- 도구 도입을 검토할 때: 해당 가이드의 **머릿말("언제 필요한가")** 만 읽고 내 프로젝트에 맞는지 거른 뒤, 맞으면 운영 절차를 본다.

## 작성 규칙 (요약 — 상세는 `_template.md` 주석)

1. **2단 구조** — 머릿말(멘탈 모델: 언제 필요한가 → 사상 → 동작 방식) + 본문(운영 절차 9칸).
2. **머릿말 순서는 결론 먼저** — "언제 필요한가"(한 줄 결론)를 맨 앞에 둬 관련성을 먼저 거른다.
3. **출처 기반, 추측 금지** — 공식 docs·소스·신뢰 사례로만 채운다. 확인 안 되면 `⚠️ 미확인`, 해당 없으면 `해당 없음`. 직접 설치·실행은 필수가 아니다.
4. **README와 분업** — 동작 방식은 기능 자랑이 아니라 *운영에 영향 주는 메커니즘*만 간단히.

## 사상(thinking model) 어휘

도구의 "사상"은 동작 방식의 한 줄 요약 태그다. 사상이 정해지면 연결 방식·생명주기·산출물 유무가 대체로 따라온다. **아래는 현재까지 관찰된 것이며, 도구가 늘면 확장한다(고정된 분류가 아니다).**

- **생성·인덱스형** — 한 번 스캔/처리해 산출물을 만들고 그것을 반복 질의. 보통 CLI·one-shot·산출물 폴더 동반. (예: graphify)
- **질의·상주형** — 영구 산출물 없이 세션 동안 떠서 실시간 응답. 보통 MCP·상주 서버. (예: serena)
- *(후보, 아직 사례 없음)* 변환형(입력→출력 변환, 영구 인덱스 없음) / 오케스트레이션형(다른 도구를 부림) 등 — 해당 도구가 추가될 때 정의한다.

> ⚠️ 한 도구가 여러 사상을 겹쳐 가질 수 있다 (예: graphify는 그래프를 *생성*하고 MCP로 *질의*도 한다). 그럴 땐 "주 사상 + 부 모드"로 적는다.
