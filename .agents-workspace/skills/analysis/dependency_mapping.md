# 스킬: 의존성 맵 작성

> 작업 간 선후 관계와 공유 인터페이스를 정의하여 컨텍스트 단절을 방지하는 기법  
> 사용 시점: Layer 0 작업 분해, 멀티 태스크 설계

---

## 언제 쓰는가

- 전체 작업을 세부 단위로 나눈 후
- "어떤 작업을 먼저 해야 하는가"가 불명확할 때
- 여러 작업이 같은 모듈/인터페이스를 건드릴 것 같을 때

---

## 핵심 원리

작업 분해는 "무엇을 만들지"에서 끝나면 안 된다.  
"어떻게 연결되는지"까지 정의해야 컨텍스트 단절을 막을 수 있다.

```
작업 목록만 있을 때  →  각 작업이 독립 최적화 → 전체 충돌
의존성 맵이 있을 때  →  공유 인터페이스 사전 정의 → 전체 정합성 유지
```

---

## 프롬프트 템플릿

### 의존성 분석

```
아래 작업 목록을 분석해서 의존성 맵을 만들어줘.

포함할 내용:
1. 선후 관계: A가 완료돼야 B를 시작할 수 있는 것
2. 병렬 가능: 동시에 진행해도 되는 것
3. 공유 인터페이스: 여러 작업이 함께 사용하는 모듈/데이터 모델/API
4. 충돌 위험: 같은 부분을 건드릴 가능성이 있는 작업 쌍

[작업 목록 붙여넣기]
```

### 공유 인터페이스 사전 정의

```
아래 의존성 맵에서 공유 인터페이스가 필요한 부분을 찾아줘.
각 인터페이스마다:
- 입력/출력 형식
- 어떤 작업들이 사용하는가
- 먼저 정의해야 하는 이유

[의존성 맵 붙여넣기]
```

---

## 출력 형식 예시

```
[선후 관계]
작업 A → 작업 C  (A의 DB 스키마가 C에서 사용됨)
작업 B → 작업 D  (B의 API가 D에서 호출됨)

[병렬 가능]
작업 A // 작업 B  (서로 독립적)

[공유 인터페이스 — 먼저 정의 필요]
- UserModel: 작업 A, C, E가 공통 사용
- AuthToken 형식: 작업 B, D, F가 공통 사용

[충돌 위험]
작업 C ↔ 작업 D: 둘 다 UserService를 수정할 가능성 있음
```
