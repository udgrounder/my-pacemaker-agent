# 스킬: 반례 탐색

> 계획이나 코드가 실패할 수 있는 시나리오를 찾아 논리적 약점을 드러내는 기법  
> 사용 시점: 설계 검토, 계획서 검토, 결과 확인

---

## 언제 쓰는가

- 계획이나 코드가 "맞는 것 같은데" 확신이 없을 때
- Agent의 동조 경향을 깨고 진짜 문제를 찾아야 할 때
- "이게 왜 틀릴 수 있지?"를 체계적으로 생각해야 할 때

---

## 핵심 원리

"이게 왜 맞지?"가 아니라 "이게 왜 틀릴 수 있지?"를 먼저 생각한다.

```
1. 입력 반례   →  예상치 못한 입력값이 들어오면?
2. 환경 반례   →  외부 시스템이 실패하면?
3. 상태 반례   →  데이터가 예상과 다른 상태이면?
```

---

## 프롬프트 템플릿

### 계획서 반례 탐색

```
아래 계획이 실패할 수 있는 시나리오를 찾아줘.

탐색 범위:
1. 입력값이 예상과 다를 때 (빈 값, 극단값, 잘못된 형식)
2. 외부 시스템이 실패할 때 (네트워크, DB, 외부 API)
3. 데이터가 예상과 다른 상태일 때 (중복, 누락, 오염)
4. 동시에 여러 요청이 들어올 때

각 시나리오마다:
- 어떤 상황인가
- 현재 계획이 어떻게 실패하는가
- 어디서 처리가 빠져있는가

[계획서 붙여넣기]
```

### 코드 반례 탐색

```
아래 코드에서 아래 케이스들이 들어왔을 때 어떻게 동작하는지 확인해줘.

케이스:
- (구체적인 반례 1: 예. 동시 요청 2개)
- (구체적인 반례 2: 예. DB 연결 중간에 끊김)
- (구체적인 반례 3: 예. 입력값이 null)

각 케이스마다 실제로 어떻게 처리되는지 코드 흐름과 함께 답해줘.

[코드 붙여넣기]
```

---

## 출력 활용 방법

```
즉시 처리  →  태스크 계획의 "검증 체크리스트"에 추가
나중에 처리 →  다음 작업 반영 사항으로 기록
```
