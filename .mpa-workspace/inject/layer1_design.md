> **사용 시점:** 새 기능 / 버그픽스 / 리팩터링의 구현 계획을 세울 때

## 작업 전 맥락 파악

INDEX.md에서 이 태스크의 **파생 출처**를 확인한다.

- **파생 출처가 있으면**: 원래 태스크의 plan.md 중 "구현 후 발견" 섹션을 읽는다.
  - 이 태스크가 어떤 발견에서 생겼는지 파악
  - 원래 태스크에서 이미 처리된 것과 여기로 넘어온 것을 구분
  - 이 맥락을 새 plan.md의 `파생 출처` 필드에 기록한다
- **파생 출처가 없으면**: 그냥 진행한다

## 작업

오늘 작업 요구사항을 파악한 뒤 아래 형식으로 태스크 계획을 작성한다.

> **계획은 파일이 먼저다 (휘발 방지):** 계획을 채팅에만 출력하지 않는다.
> 작성하는 **즉시** `workspace/tasks/active/yyyymmdd_[태스크명]/plan.md` 에 파일로 저장한 뒤,
> 그 파일 경로를 사용자에게 알리고 검토를 요청한다.
> 검토·수정·승인은 채팅 초안이 아니라 **저장된 파일**을 대상으로 한다. (저장은 "세션 종료 시"가 아니라 지금이다.)

**실패 비용 추정 (계획 전 판단):**

아래 질문에 답하고 각 축의 레벨을 도출한다.  
불명확한 경우 위험 방향(Yes / No)으로 판단한다.

| 축 | 질문 | 판단 |
|----|------|------|
| **심각도** | 외부 사용자에게 노출되는가? | Yes → 높음 |
| | 보안·금융·법규에 닿는가? | Yes → 높음 |
| | → 둘 다 No | → 낮음 |
| **발견 가능성** | 로컬 실행으로 즉시 확인 가능한가? | Yes → 높음 |
| | 자동화 테스트가 커버하는가? | Yes → 높음 |
| | → 둘 다 No | → 낮음 |
| **가역성** | git reset으로 되돌릴 수 있는가? | Yes → 높음 |
| | 영구 저장(DB, 파일, 외부 서비스)이 없는가? | Yes → 높음 |
| | → 하나라도 No | → 낮음 |

**자율성 레벨 도출:**

| 조건 | 레벨 |
|------|------|
| 심각도 높음 OR 가역성 낮음 | 1 — 제안 (사용자가 결정) |
| 발견 가능성 낮음 | 2 — 초안 (기본값) |
| 셋 다 안전 방향 | 3 — 실행 후 보고 가능 |

### 태스크 계획 형식

```
## 태스크: [이름]
**목적:** 한 줄

### 사전 결정 필요 사항
> 구현 전에 내가 결정해야 할 것들 (비가시적 위임 후보)
- [ ] 질문 1 — [기술적 결정 / **가치 결정**]
- [ ] 질문 2 — [기술적 결정 / **가치 결정**]

> ⚠️ **가치 결정**("무엇이 더 중요한가")은 AI에게 위임 불가 — 반드시 사용자가 결정  
> 가치 결정 항목에는 선택지별 영향도를 함께 명시한다:  
> - [선택 A] → 영향: [...]  
> - [선택 B] → 영향: [...]

### 구현 단계
1. Step 1 — [무엇을] / 이유: [왜]
2. Step 2 — ...

### 예상 조용한 결정(Silent Decisions)
> 명세하지 않으면 구현자가 조용히 결정할 것들
- 항목 1: [어떤 결정인가] / 권장: [추천 방향 또는 "먼저 확인"]

### 검증 체크리스트
- [ ] 정상 경로:
- [ ] 실패 경로:
- [ ] 엣지 케이스:

### 반례 (이 계획이 실패할 수 있는 시나리오)
- 시나리오 1:
- 시나리오 2:
```

---

## 완료 기준

- [ ] 사전 결정 필요 사항이 모두 해소됨 (또는 해소 방법 명시됨)
- [ ] 각 단계에 "왜"가 있음
- [ ] 조용한 결정 후보가 명시됨
- [ ] 반례 검토 완료

---

## 세션 종료 시

작업 완료 후 다음을 확인하고 필요한 파일을 직접 업데이트한다:

1. 아키텍처 업데이트 → `workspace/memory/shared/architecture.md`
   - **추가:** 설계 중 새로 확립된 결정
   - **수정:** 기존 결정이 바뀌었다면 기존 항목을 교체 (이전 내용을 남기지 않는다)
   - **삭제:** 더 이상 유효하지 않은 규칙
2. 태스크 계획 파일 확정 → `workspace/tasks/active/yyyymmdd_[태스크명]/plan.md` (작업 중 이미 저장된 파일 — 미저장 상태면 지금 즉시 저장)
   - **상태를 `설계 완료`로 표시**
   - `에이전트 보고` 섹션을 채운다:
     - **사용자 결정 필요**: 가치 결정 또는 비즈니스 맥락 의존 항목 (결정 없이 구현 불가능한 것)
     - **에이전트 가정**: 브리핑에 없어 임의로 채운 내용 + 근거 + 틀렸을 때 영향
   - `구현 단계` 체크리스트를 실제 구현 단계로 채운다 (템플릿의 "Step 1/Step 2" 대체)
3. tasks/INDEX.md 업데이트 → 새 태스크 항목 추가, 상태: `active`
   > INDEX는 **생명주기 상태**(`active`/`done`/`hold`)만 쓴다. 세부 진행 단계(`설계 완료`/`구현 중` 등)는 plan.md의 상태에 둔다 — 두 축을 섞지 않는다.
4. MPA 시스템 개선 후보 발견 → `.mpa-workspace/upgrade-candidates/[내용].md`
5. **태스크 분리 검토** — 아래 중 하나에 해당하면 분리안을 계획서와 함께 제안한다
   - 구현 단계가 5개 이상
   - 독립적으로 완료·검증 가능한 단위가 2개 이상 존재
   - 단계 간 의존성이 낮아 병렬 진행이 가능한 경우
   - **분리 시:** 별도 폴더를 만들지 않고 같은 태스크 폴더 안에 `sub_[이름].md` 파일로 분리한다
6. **계획서를 사용자에게 제시하고 검토를 요청한다** (검토는 필수 단계 — 생략 불가)

   계획서 제시 후 다음 메시지를 함께 전달한다:

   ```
   계획서를 검토해 주세요. (검토는 필수입니다)

   검토 방법을 선택하실 수 있습니다 — 하나만 쓰거나 섞어서, 여러 번도 가능합니다:
   · 직접 검토: 직접 읽으신 후 수정 요청 또는 "승인"
   · 인터뷰 지원: 결정 항목을 함께 채워드립니다 → "인터뷰 해줘" / "같이 봐줘"
   ```

   - 사용자가 인터뷰를 요청하면 → `.mpa-workspace/skills/analysis/plan_interview.md` 지시에 따라 진행
   - 사용자가 직접 검토한다면 → 수정 요청 또는 승인을 기다린다
   - 두 방법은 배타적이지 않다 — 한 방법 후 다른 방법으로 보완하거나 반복할 수 있다
   - **검토를 거치지 않고, 또는 명시적 승인 없이 구현을 시작하지 않는다**

7. **다음 작업 예측 (승인 후):**
   - 비평 권장 기준 해당 시 → *"독립 비평(layer1_critique)을 먼저 진행할까요? 새 스레드에서 plan.md만 보고 약점을 찾아드립니다."*
   - 비평 권장 기준 미해당 시 → *"바로 구현을 진행할까요?"*

   > ⚠️ **독립 비평은 반드시 새 스레드에서 실행한다.**  
   > 이 설계 세션 스레드에서 critique를 직접 수행하지 않는다.  
   > 사용자에게 새 스레드 시작을 안내하고, 이 스레드에서는 대기한다.
