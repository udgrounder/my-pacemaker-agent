## 작업

오늘 작업 요구사항을 파악한 뒤 **실패 비용을 먼저 추정**하고, 그 결과에 따라 흐름을 분기한다.

> **minor 판단 시 경량 흐름으로 전환:**  
> 실패 비용이 `minor`로 판단되면 아래 전체 형식을 따르지 않는다.  
> `agent_rules.md` "minor 태스크 경량 처리" 절차에 따라 최소 plan.md를 작성하고 즉시 구현한다.  
> 이 세션의 나머지 내용(자기 점검, 독립 비평, 인터뷰, 반례, 검증 체크리스트, docs 업데이트 대상)은 minor에서 생략한다.

> **계획은 파일이 먼저다 (휘발 방지):** 계획을 채팅에만 출력하지 않는다.
> 작성하는 **즉시** `workspace/tasks/active/yyyymmdd_[태스크명]/plan.md` 에 파일로 저장한 뒤,
> 그 파일 경로를 사용자에게 알리고 검토를 요청한다.
> 검토·수정·승인은 채팅 초안이 아니라 **저장된 파일**을 대상으로 한다. (저장은 "세션 종료 시"가 아니라 지금이다.)

**실패 비용 추정 (계획 전 판단):**

아래 질문에 답하고 각 축의 레벨을 도출한다.  
불명확한 경우 위험 방향(Yes / No)으로 판단한다.

| 축 | 질문 | 판단 |
|----|------|------|
| **심각도** | 외부 사용자에게 노출되는가? | Yes → 높음 |
| | 보안·금융·법규에 닿는가? | Yes → 높음 |
| | → 둘 다 No | → 낮음 |
| **발견 가능성** | 로컬 실행으로 즉시 확인 가능한가? | Yes → 높음 |
| | 자동화 테스트가 커버하는가? | Yes → 높음 |
| | → 둘 다 No | → 낮음 |
| **가역성** | git reset으로 되돌릴 수 있는가? | Yes → 높음 |
| | 영구 저장(DB, 파일, 외부 서비스)이 없는가? | Yes → 높음 |
| | → 하나라도 No | → 낮음 |

**실패 비용 등급 도출:**

| 조건 | 등급 |
|------|------|
| 심각도 높음 OR 가역성 낮음 | `critical` — 구현 중 단계별 확인 필요 |
| 발견 가능성 낮음 | `major` — 기본값, 완료 후 검토 |
| 셋 다 안전 방향 | `minor` — 완료 후 간략 보고 |

### 가치 결정 vs 기술 결정 판별

"사전 결정 필요 사항"을 작성하기 전에 아래 두 질문으로 각 항목을 분류한다.

| 질문 | Yes → | No → |
|------|-------|-------|
| 이 결정이 잘못됐을 때 **코드 수정만으로** 되돌릴 수 있는가? | 기술 결정 | → 다음 질문 |
| 이 결정이 **비즈니스 목표·우선순위·사용자 경험**에 따라 달라지는가? | **가치 결정** | 기술 결정 |

**가치 결정 패턴 (예시):**
- 기능 포함/제외 여부 (범위 결정)
- 트레이드오프 선택 — 성능 vs 일관성, 단순성 vs 기능성
- 오류/엣지케이스를 사용자에게 어떻게 보여줄 것인가 (UX 방향)
- 외부 정책·법규·비즈니스 규칙에 따른 선택

**기술 결정 패턴 (예시):**
- 동일 기능의 구현 방법 선택 (라이브러리, 알고리즘)
- 코드 구조·패턴 선택 (기능 동등성이 확보된 경우)
- 성능 최적화 접근법

> 경계가 불명확하면 **가치 결정**으로 분류한다.

### 태스크 계획 형식

```
## 태스크: [이름]
**목적:** 한 줄
**실패 비용 등급:** critical / major / minor

### 사전 결정 필요 사항
> 구현 전에 내가 결정해야 할 것들 (비가시적 위임 후보)
- [ ] 질문 1 — [기술적 결정 / **가치 결정**]
- [ ] 질문 2 — [기술적 결정 / **가치 결정**]

> ⚠️ **가치 결정**("무엇이 더 중요한가")은 AI에게 위임 불가 — 반드시 사용자가 결정  
> 가치 결정 항목에는 선택지별 영향도를 함께 명시한다:  
> - [선택 A] → 영향: [...]  
> - [선택 B] → 영향: [...]

### 구현 단계
1. Step 1 — [무엇을] / 이유: [왜]
2. Step 2 — ...

### 예상 조용한 결정(Silent Decisions)
> 명세하지 않으면 구현자가 조용히 결정할 것들
- 항목 1: [어떤 결정인가] / 권장: [추천 방향 또는 "먼저 확인"]

### 검증 체크리스트
- [ ] 정상 경로:
- [ ] 실패 경로:
- [ ] 엣지 케이스:

### 반례 (이 계획이 실패할 수 있는 시나리오)
- 시나리오 1:
- 시나리오 2:

### 결정 이력
> 구현 중 내려진 결정을 시간순으로 누적 기록한다.  
> 서브에이전트·다음 세션이 "왜 이 방향인가"를 이해하는 단일 소스다.  
> 최초 설계 결정부터 구현 중 변경까지 모두 포함한다.

| 시점 | 결정 내용 | 근거 |
|------|---------|------|
| 설계 | (초기 계획 핵심 결정 요약) | |
```

## 완료 기준

- [ ] 사전 결정 필요 사항이 모두 해소됨 (또는 해소 방법 명시됨)
- [ ] 각 단계에 "왜"가 있음
- [ ] 조용한 결정 후보가 명시됨
- [ ] 반례 검토 완료
- [ ] **자기 점검 완료** — `inject/layer1_critique.md` "자기 점검" 섹션 체크리스트 통과
- [ ] **독립 비평 기준 평가** — `inject/layer1_critique.md` "독립 비평 세션이 필요한 기준" 충족 시 사용자에게 권장

> **자기 점검과 독립 비평은 별개다.** 자기 점검은 형식적 완전성만 확인한다. 숨은 가정·관점 검증은 독립 비평이 한다.

## 세션 종료 시

작업 완료 후 다음을 확인하고 필요한 파일을 직접 업데이트한다:

1. 아키텍처 업데이트 → `workspace/memory/shared/architecture.md`
   - **추가:** 설계 중 새로 확립된 결정
   - **수정:** 기존 결정이 바뀌었다면 기존 항목을 교체 (이전 내용을 남기지 않는다)
   - **삭제:** 더 이상 유효하지 않은 규칙
2. 태스크 계획 파일 확정 → `workspace/tasks/active/yyyymmdd_[태스크명]/plan.md` (작업 중 이미 저장된 파일 — 미저장 상태면 지금 즉시 저장)
   - **상태를 `설계 완료`로 표시**
   - `에이전트 보고` 섹션을 채운다:
     - **사용자 결정 필요**: 가치 결정 또는 비즈니스 맥락 의존 항목 (결정 없이 구현 불가능한 것)
     - **에이전트 가정**: 브리핑에 없어 임의로 채운 내용 + 근거 + 틀렸을 때 영향
   - `구현 단계` 체크리스트를 실제 구현 단계로 채운다 (템플릿의 "Step 1/Step 2" 대체)
   - **결정 이력 일괄 정리** — 설계 단계 중 내려진 핵심 결정을 `결정 이력` 테이블에 기록한다:
     - 초기 설계 방향과 그 근거
     - 사용자와의 대화에서 합의된 사항 (인터뷰 결과 포함)
     - 가치 결정에 대한 사용자 답변
     - 검증 완료된 가정 (가정 → 사실로 확정된 항목)
     > 매 대화마다 즉시 기록하지 않고, **plan.md 저장 시점에 누적 결정을 한 번에 정리**한다.
3. tasks/INDEX.md 업데이트 → 새 태스크 항목 추가, 상태: `active`
   > INDEX는 **생명주기 상태**(`active`/`done`/`hold`)만 쓴다. 세부 진행 단계(`설계 완료`/`구현 중` 등)는 plan.md의 상태에 둔다 — 두 축을 섞지 않는다.
4. MPA 시스템 개선 후보 발견 → `.mpa-workspace/upgrade-candidates/[내용].md`
5. **태스크 분리 검토** — 아래 중 하나에 해당하면 분리안을 계획서와 함께 제안한다
   - 구현 단계가 5개 이상
   - 독립적으로 완료·검증 가능한 단위가 2개 이상 존재
   - 단계 간 의존성이 낮아 병렬 진행이 가능한 경우
   - **분리 시:** 별도 폴더를 만들지 않고 같은 태스크 폴더 안에 `sub_[이름].md` 파일로 분리한다
6. **계획서를 사용자에게 제시한다** — "에이전트 보고" 섹션을 기반으로 핵심 항목만 꺼내서 제시한다 (Zone 2)

   **확인 필요 항목이 있는 경우 (가치 결정 OR 영향: 계획 변경인 에이전트 가정):**
   ```
   계획서가 준비됐습니다. ([파일 경로])

   확인이 필요한 항목:
   ❓ 가치 결정: [항목] — [선택 A] / [선택 B]   ← 있는 경우만
   ⚠️ 검증 필요 가정: [내용]                     ← 영향: 계획 변경인 것만

   → 위 항목에 답하시면 구현을 시작합니다.
     계획 전체를 검토하시려면 파일을 열어주세요.
     인터뷰 지원이 필요하면 "인터뷰 해줘"라고 말씀해주세요.
   ```

   **확인 필요 항목이 없는 경우:**
   ```
   계획서 준비 완료. ([파일 경로]) — 바로 구현할까요?
   ```

   - **가치 결정 답변 / 가정 확인 완료 시** → 내용을 plan.md에 반영하고 구현으로 이동
   - **인터뷰 요청 시** → `.mpa-workspace/skills/analysis/plan_interview.md` 지시에 따라 진행
   - **"승인" / "구현해줘" 등 명시적 진행 의사 시** → 구현으로 이동
   - **명시적 승인 없이 구현을 시작하지 않는다**

   **GATE 1 통과 직전 (구현 중 전환 전):** 사용자 검토·인터뷰 중 추가로 내려진 결정이 있으면 plan.md `결정 이력` 테이블에 일괄 반영한다.
   - 인터뷰 중 답변된 가치 결정
   - 추가 검증된 가정
   - 사용자와 합의된 방향 변경
   > 구현 세션은 이 결정 이력만 보고 "왜 이 방향인가"를 파악한다. 누락되면 구현 중 같은 논의가 반복된다.

   **결정 이력 반영이 끝난 직후 (상태를 `구현 중`으로 바꾸기 직전):**
   ```bash
   python3 .mpa-workspace/hooks/plan_hash.py approve workspace/tasks/active/[작업명]/plan.md
   ```
   이 명령으로 현재 plan.md 본문의 해시를 `승인해시` 필드에 기록한다. 이후 plan.md를 수정하면 code_gate.py가 자동으로 감지해 차단한다.

7. **다음 작업 예측 (승인 후):**
   - 비평 권장 기준 해당 시 → *"독립 비평(layer1_critique)을 먼저 진행할까요? 서브에이전트가 plan.md만 보고 약점을 찾습니다."*
   - 비평 권장 기준 미해당 시 → *"바로 구현을 진행할까요?"*

   > ⚠️ **독립 비평은 반드시 서브에이전트로 실행한다.**  
   > 이 설계 세션 스레드에서 critique를 직접 수행하지 않는다.  
   > 서브에이전트 도구가 없는 환경이면 위험을 고지하고 독립 비평 없이 진행할지 사용자에게 묻는다.
