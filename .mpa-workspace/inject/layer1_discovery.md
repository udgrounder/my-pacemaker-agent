# Layer 1 — 구현 후 발견 처리 세션

> **사용 시점:** 결과물을 직접 경험한 후 수정·추가하고 싶은 사항이 생겼을 때

---

## 이 세션의 목적

첫 결과물을 보고 생긴 발견 항목들을 분류하고, 각 항목을 올바른 처리 경로로 연결한다.

이것은 실패가 아니다. 구현 전 명세만으로는 동작 방식·UX·기능 간 연결을 완전히 정의하기 어렵다. 결과물을 경험해야 비로소 드러나는 요구사항이 있고, 첫 결과물은 그 요구사항을 가시화하는 도구다.

---

## 역할

`.mpa-workspace/personas/task_designer.md`를 읽고 그 역할로 작업한다.

---

## 작업 시작 전 읽을 파일

다음 파일을 순서대로 읽는다:

1. `workspace/memory/shared/project_identity.md`
2. `workspace/memory/shared/architecture.md`
3. 이번 태스크의 계획서 (`workspace/tasks/active/[태스크명]/plan.md`)

---

## 스킬 참조

`.mpa-workspace/skills/analysis/discovery_classification.md`를 읽는다.

---

## 진행 순서

### 1단계 — 발견 항목 수집

사용자에게 묻는다:

> "결과물을 보고 수정하거나 추가하고 싶은 것들을 모두 이야기해 주세요.  
> 크고 작음 상관없이, 좋고 나쁨의 판단 없이 먼저 전부 꺼내주시면 함께 정리하겠습니다."

수집된 항목을 목록으로 정리한다. 이 단계에서는 분류하지 않는다.

### 2단계 — 항목별 분류

각 항목을 `discovery_classification` 스킬로 분류한다. 항목이 어느 유형인지 불명확하면 사용자에게 확인한다.

| 유형 | 특징 | 판단 기준 |
|------|------|-----------|
| 조정 | 현재 구현의 작은 변경 | 기존 plan.md 범위 안에서 처리 가능한가? |
| 계획 확장 | 계획서에 빠진 요구사항 | plan.md를 보완하면 처리 가능한가? |
| 신규 태스크 | 이 결과물이 드러낸 새 기능 | 별도의 계획과 구현이 필요한 규모인가? |

### 3단계 — 분류 결과 요약 및 승인

분류된 항목 전체를 사용자에게 보여주고 승인을 받는다. 이후 유형별로 처리한다.

**조정:**
- 현재 plan.md의 해당 구현 단계 수정
- plan.md 상태를 `구현 중`으로 변경
- → 다음 단계: `layer1_implement.md`로 재구현

**계획 확장:**
- plan.md에 "구현 후 발견 추가 범위" 섹션 추가
- 추가된 요구사항에 대해 검토 후 사용자 승인
- plan.md 상태를 `구현 중`으로 변경
- → 다음 단계: 확장 범위 포함하여 `layer1_implement.md`로 재구현

**신규 태스크:**
- `workspace/tasks/INDEX.md`에 새 태스크로 등록 (상태: `active`)
- **반드시 포함:** `파생 출처: [현재 태스크명] — [발견 맥락]`
- **원래 태스크 plan.md의 "구현 후 발견 > 파생된 태스크"에 링크 추가**
  - `[신규 태스크명] — [발견 맥락]`
- → 다음 단계: 이후 별도 `layer1_design.md` 사이클로 plan.md 작성

### 4단계 — 원래 태스크 종료

> **이 세션이 원래 태스크 종료를 전담한다.** layer1_review에서 넘어온 경우도 여기서 처리한다.

신규 태스크 등록이 완료된 것을 확인한 뒤 원래 태스크를 처리한다.

| 발견 항목 구성 | 원래 태스크 처리 |
|--------------|----------------|
| 조정 또는 계획 확장 포함 | plan.md 상태 `구현 중` 유지 (재구현 후 재검토) |
| 신규 태스크만 | plan.md 상태 `검토 완료` → `done/`으로 이동 + INDEX.md 업데이트 |

> **순서:** 신규 태스크 등록 완료 확인 → 원래 태스크 종료. 순서를 바꾸지 않는다.

### 5단계 — 방향 메모리 업데이트

분류가 완료되면, 사용자에게 묻는다:

> "이번 결과를 경험하면서 제품 방향에 대해 새로 알게 된 것이 있으신가요?  
> (UX 원칙, 잘 작동하는/않는 패턴, 우선순위 결정 등)"

발견한 방향 학습이 있으면 `workspace/memory/shared/direction.md`에 추가한다.
- 파일이 없으면 `templates/direction_template.md`를 복사해 생성 후 기록한다.
- 업데이트 이력 테이블에 날짜와 출처 태스크, 발견 요약을 기록한다.

기록할 것이 없으면 생략한다.

---

## 완료 기준

- [ ] 발견 항목 전체 수집 완료
- [ ] 각 항목의 유형 분류 완료 (사용자 확인 포함)
- [ ] 유형별 처리 경로 결정 및 사용자 승인 완료
- [ ] 해당 파일 업데이트 완료 (plan.md 및/또는 INDEX.md)
- [ ] 신규 태스크 등록 완료 확인 (있는 경우)
- [ ] 원래 plan.md "파생된 태스크"에 링크 추가 완료 (신규 태스크가 있는 경우)
- [ ] **원래 태스크 종료 처리 완료** — 신규 태스크 등록 이후에 처리
- [ ] direction.md 업데이트 여부 확인 (발견된 방향 학습이 있을 때)

---

## 세션 종료 시

1. 조정/계획 확장 항목이 있으면 → plan.md 업데이트 완료 확인
2. 신규 태스크 항목이 있으면 → `workspace/tasks/INDEX.md` 업데이트 완료 확인 (파생 출처 포함)
3. **원래 태스크 종료 완료 확인** (4단계 처리 결과)
   - 조정/계획 확장 포함 → plan.md 상태 `구현 중` 확인
   - 신규 태스크만 → plan.md 상태 `검토 완료` + `done/` 이동 완료 확인
4. 방향 학습이 있으면 → `workspace/memory/shared/direction.md` 업데이트 완료 확인
5. **다음 작업 예측:**
   - 조정 또는 계획 확장 있음 → *"재구현을 시작할까요?"*
   - 신규 태스크만 있음 → *"[신규 태스크명] 설계를 시작할까요?"*
