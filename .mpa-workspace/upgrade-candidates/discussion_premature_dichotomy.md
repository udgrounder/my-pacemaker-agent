# 토론 모드: 범주를 성급히 이분법으로 닫는 경향 + 기반 매몰

**타입**: 방법론 개선
**발견 상황**: 멀티에이전트 효용성 토론(`workspace/exploration/discussion/multi_agent_utility.md`, 28라운드) 중 사용자가 두 번 정정.
**적용 범위**: 모든 프로젝트 (discussion_partner 페르소나 / discussion_mode)

## 현재 방식

discussion_partner 페르소나는 "steelman 후 반론", "동조 금지", "정밀성 우선"은 명시하나, 토론 중 에이전트가 자주 저지르는 아래 두 실수를 명시적으로 경계하지 않는다.

**관찰된 실수 두 가지:**

1. **범주를 성급히 이분법으로 닫음.** 이번 토론에서 반복됨:
   - R20: "협업형 멀티에이전트"를 한 덩어리로 기각 → 사용자가 "협업형에도 효용 있다"(다관점 발산형)고 정정. 협업형을 *다관점 발산형 / 자율 합의형*으로 쪼개야 했음.
   - R21: "작업 분배 = 멀티에이전트 아님"으로 단정 → 사용자가 개발/광고 예시로 보완. 작업 분배를 *동질적(단순 병렬) / 이질적(전문성 분업)*으로 쪼개야 했음.
   - 같은 실수가 두 번 — A 아니면 B로 닫고, 사용자가 "사실 세 번째가 있다"고 열어줌.

2. **기반에 매몰되어 본주제로 못 올라감.** R1~19를 격리·페르소나·절차·입도 등 *기반*에 쏟았고, 정작 본주제(멀티에이전트 효용·활용)는 R7·R8만 스침 → 사용자가 "흐름이 벗어났다, 이제 본주제로"라고 정정(R20). 추상 기반으로 깊이 파고드는 관성이 본주제 진입을 미룸.

## 개선 방안

discussion_partner 페르소나의 "LLM 행동 편향 경고"에 두 항목 추가:

- **이분법 조기 폐쇄 경계** — A/B로 가르기 전에 "C는 없는가, 이 범주를 더 쪼갤 수 있는가"를 먼저 자문한다. 특히 같은 종류의 이분법을 반복하면 경계 신호로 본다(과단순화의 한 형태, `feedback_concept_precision`과 연결).
- **본주제 고도 점검** — 다회 교환 중 주기적으로 "지금 파는 것이 사용자가 물은 본주제인가, 그 기반인가"를 점검한다. 기반이 깊어지면 본주제로 올라갈 시점을 사용자에게 제안한다.
