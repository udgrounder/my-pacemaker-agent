# Agent 행동 규칙

---

## 세션 시작 루틴

> 사용자 요청을 처리하기 전, 매 세션 시작 시 자동으로 실행한다.

### 1. 현재 상태 파악

다음 파일을 순서대로 읽는다:

1. `workspace/tasks/active/` 폴더 확인
2. `workspace/tasks/INDEX.md` 확인 (존재하는 경우)

**진행 중 태스크가 있는 경우:**

plan.md를 읽어 현재 상태를 확인하고, 상태에 따라 다음 메시지를 출력한다:

| plan.md 상태 | 출력 메시지 | 기본 다음 단계 |
|-------------|-----------|--------------|
| 검토 중 / 설계 완료 | "계획서 검토가 완료되지 않았습니다. 검토를 이어서 진행할까요?" | 계획서 검토 |
| 구현 중 | "구현이 진행 중입니다. 이어서 구현을 계속할까요?" | `layer1_implement.md` |
| 구현 완료 | "구현이 완료됐습니다. **다음 단계: 작업 결과 검토**를 진행할까요?" | `layer1_review.md` |
| 검토 완료 | "작업 결과 검토까지 완료됐습니다. 태스크를 완료 처리할까요?" | 태스크 완료 처리 |

> 사용자가 다음 단계를 건너뛰길 원하면 명시적으로 말할 수 있다. 에이전트가 먼저 선택지를 균등하게 나열하지 않는다.

**진행 중 태스크가 없는 경우:**  
상태 보고 없이 사용자 요청을 기다린다.

**upgrade-candidates 확인 (진행 중 태스크 유무 무관):**  
`.mpa-workspace/upgrade-candidates/` 폴더에 파일이 있으면 한 줄로 고지한다:
```
📦 하네스 개선 후보가 [N]개 있습니다. 검토하시겠어요? (나중에 하셔도 됩니다)
```
사용자가 거절하거나 무시하면 더 이상 언급하지 않고 요청을 기다린다.

**workspace/ 자체가 없는 경우 (하네스 첫 적용):**

```
workspace/가 없습니다. 프로젝트 초기화(Layer 0)가 필요합니다.
지금 시작할까요?
```

### 2. 요청 유형 자동 판단

사용자 요청을 받으면 아래 패턴으로 유형을 판단하고,  
**해당 inject 파일을 내부적으로 읽어 그 지시에 따라 행동한다.**

| 사용자 발화 패턴 | 판단 유형 | 자동 로딩 |
|---------------|---------|---------|
| "~기능 만들어줘", "~추가해줘", "~구현해줘" | 새 기능 설계 | `inject/layer1_design.md` + `personas/task_designer.md` |
| "~버그", "~안 돼", "~에러", "~오류" | 버그 수정 | `inject/layer1_review.md` (원인 분석 우선) + `personas/code_reviewer.md` |
| "이어서 해줘", "계속해줘", "어디까지 했어" | 작업 재개 | `plan.md` + `changelog.md` 읽어 복원 |
| "코드 봐줘", "검토해줘", "리뷰해줘" | 작업 결과 검토 | `inject/layer1_review.md` + `personas/code_reviewer.md` |
| "리팩터링", "정리해줘", "개선해줘" | 리팩터링 | `inject/layer1_design.md` + `personas/task_designer.md` |
| "전체 확인", "정합성 점검", "Layer 2" | 통합 감사 | `inject/layer2_checkpoint.md` + `personas/integration_auditor.md` |
| "초기화", "프로젝트 설정", "Layer 0" | 프로젝트 초기화 | `inject/layer0_init.md` + `personas/architect.md` |
| "하네스 업데이트", "업그레이드" | 하네스 업데이트 | `inject/layer0_update.md` + `personas/architect.md` |

**판단 후 반드시 고지한다 (투명한 라우팅):**

```
→ [새 기능 설계] | 로드: layer1_design.md · task_designer.md
```

한 줄로 짧게. 로드한 파일 목록을 명시해 실제로 읽었는지 확인 가능하게 한다.  
사용자가 다르게 처리하길 원하면 정정할 수 있어야 한다.

**판단 불가 시:**

요청 의도를 추론해 가장 가능성 높은 유형으로 먼저 처리를 시작한다.

```
→ [추론된 유형]으로 처리하겠습니다. 다르게 처리하시려면 말씀해주세요.
```

### 3. Task 필요 여부 판단

요청이 Task를 만들어야 할 규모인지 판단한다.

**Task 없이 바로 처리:**
- 단순 질문·설명 요청
- 탐색적 확인 (코드 읽기만)
- 영향 범위가 해당 파일로 국한됨 (다른 모듈 인터페이스·전역 설정 변경 없음) · 비가역적 변경 없음

**Task 만들어 진행:**
- 새 기능 추가 / 다중 파일 변경
- 가치 결정이 포함된 작업
- 비가역적 변경 (데이터 삭제, 스키마 변경, 공개 API 변경 등)
- 사용자 검토·승인이 필요한 결과물

판단 후 다음 형식으로 고지하고 기본 경로로 진행한다. 사용자가 정정할 수 있다:

**Task 없이 진행 시:**
```
→ [판단 근거] → Task 없이 바로 진행합니다. 다르게 하시려면 말씀해주세요.
```

**Task 만들어 진행 시:**
```
→ [판단 근거] → Task를 만들어 진행합니다. 바로 진행하시려면 말씀해주세요.
```

## 작업 생성

사용자가 작업 생성을 요청하거나, 세션 시작 루틴에서 Task가 필요하다고 판단되면:

1. `workspace/tasks/active/yyyymmdd_[작업명]/` 폴더 생성 (날짜는 오늘 날짜)
2. `inject/layer1_design.md` 지시에 따라 `plan.md` 초안을 작성하고 **즉시 파일로 저장한다**
   > 계획을 채팅에만 출력하고 끝내지 않는다 — 반드시 `plan.md` 파일을 먼저 만든 뒤 그 파일을 검토 대상으로 제시한다. (계획 휘발 방지)
   - `에이전트 보고` 섹션을 반드시 채운다:
     - **사용자 결정 필요**: 가치 결정 또는 비즈니스 맥락에 의존하는 항목
     - **에이전트 가정**: 브리핑에 없어 임의로 채운 내용 + 근거 + 틀렸을 때 영향
   - `구현 단계` 체크리스트를 실제 구현 단계로 채운다
3. **비평 필요 여부 자동 평가** (아래 섹션 참조)
4. 계획서를 사용자에게 제시하고 검토를 요청한다 (검토는 필수 단계 — 생략 불가):

   ```
   계획서를 검토해 주세요. (검토는 필수입니다)

   검토 방법을 선택하실 수 있습니다 — 하나만 쓰거나 섞어서, 여러 번도 가능합니다:
   · 직접 검토: 직접 읽으신 후 수정 요청 또는 "승인"
   · 인터뷰 지원: 결정 항목을 함께 채워드립니다 → "인터뷰 해줘" / "같이 봐줘"
   ```

5. 사용자 응답에 따라 분기한다 (두 방법은 배타적이지 않다 — 섞거나 반복 가능):
   - **인터뷰 요청 시** → `.mpa-workspace/skills/analysis/plan_interview.md` 지시에 따라 진행 → 완료 후 승인 요청
   - **직접 검토 시** → 수정 요청 또는 승인을 기다린다
   - **검토를 거치지 않고, 또는 명시적 승인 없이 구현을 시작하지 않는다**
   - 단, 사용자가 "바로 진행" 등 검토 생략을 **명시적으로** 요청한 경우, 다음을 먼저 고지한 뒤 구현한다:
     ```
     ⚠️ 검토 생략 시 아래 에이전트 가정이 검증되지 않습니다:
     - [plan.md "에이전트 가정" 목록]
     그래도 진행할까요?
     ```

### 태스크 분리 판단

plan.md 작성 후 아래 기준으로 판단한다:

- 구현 단계가 5개 이상
- 독립적으로 완료·검증 가능한 단위가 2개 이상 존재
- 단계 간 의존성이 낮아 병렬 진행이 가능한 경우

분리가 필요하면 분리안을 계획서와 함께 제안한다.  
분리 시 별도 폴더를 만들지 않고 같은 태스크 폴더 안에 `sub_[이름].md`로 분리한다.

## 비평 필요 여부 자동 평가

> plan.md 작성이 완료되면 자동으로 실행한다. 사용자가 요청하지 않아도 된다.

plan.md에서 아래 항목을 확인한다:

| 위험도 | 기준 | 확인 방법 |
|--------|------|---------|
| 🔴 고위험 | 비가역적 변경 포함 | 데이터 삭제, 마이그레이션, 공개 API 변경 포함 여부 |
| 🔴 고위험 | 외부 시스템 연동 | DB 스키마 변경, 외부 API, 인증·결제 포함 여부 |
| 🟡 중위험 | 가치 결정 미해소 | `사용자 결정 필요` 항목 중 아직 결정되지 않은 것이 있는가 |
| 🟡 중위험 | 타 도메인 영향 | contracts.md 변경 예상 여부 |
| 🟡 중위험 | 실패 비용 높음 | 심각도 높음 + 발견 가능성 낮음 + 가역성 낮음 중 2개 이상 해당 |

> 단계 수나 항목 수는 기준이 아니다. 4개의 고위험 단계가 6개의 저위험 단계보다 비평이 더 필요하다.

**독립 비평 권장 조건: 🔴 고위험 1개 이상 OR 🟡 중위험 2개 이상 해당 시**

```
📋 비평 권장: [해당 기준 나열]
→ 새 스레드에서 독립 비평을 진행하면 완성도가 높아집니다.
  진행할까요? (건너뛰면 구현으로 바로 이동합니다)
```

> ⚠️ **독립 비평은 반드시 새 스레드에서 실행한다.**  
> 같은 스레드에서 `layer1_critique.md`를 로딩해 비평하는 것은 금지한다.  
> 설계 세션의 컨텍스트를 이어받은 비평은 독립성이 없다 — completion bias가 작동해 약점을 찾지 못한다.

**기준 미해당 시 → 자기비평 후 구현:**

계획서 작성 후 내부적으로 "이 계획이 실패하는 시나리오는?"을 한 번 검토하고,  
발견된 것이 있으면 plan.md 반례 섹션에 추가한 뒤 사용자에게 제시한다.

## 작업 재개

세션 시작 루틴에서 진행 중 태스크를 발견했거나, 사용자가 이어가기를 요청하면:

1. 해당 태스크의 `plan.md` 읽기 → 상태·구현 단계 체크리스트 확인
2. `changelog.md` 존재 시 읽기 → 완료된 부분 파악
3. 재개 지점 보고:

```
[태스크명] 재개합니다.
- 완료된 단계: [체크된 항목들]
- 이어갈 단계: [첫 미완료 항목]
```

4. `inject/layer1_implement.md` 지시에 따라 첫 미완료 단계부터 진행

---

## 작업 진행

사용자가 계획서를 승인하면:

1. **승인 마커 생성** — 해당 태스크 폴더에 `.approved` 빈 파일을 만든다
   (`workspace/tasks/active/[작업명]/.approved`).
   > 이 마커는 코드 수정 게이트(`hooks/code_gate.py`)의 통행증이다.
   > 마커가 없으면 소스코드 수정(Edit/Write)이 차단된다.
   > `workspace/`·문서·메모리·`*.md` 작성은 마커와 무관하게 항상 허용된다.
2. `inject/layer1_implement.md` 지시에 따라 작업 진행
3. 작업 완료 후 `changelog.md` 작성:
   - 수정한 파일 경로
   - 변경된 메서드 / 함수 / 클래스 이름과 위치
   - 변경 내역 요약 (소스 전체 아님)
4. 사용자에게 검토 요청

> `changelog.md`는 작업 결과 검토 세션과 세션 재개 시 핵심 참조 파일이다.

## 작업 결과 검토

구현이 완료된 태스크의 결과를 검토하는 단계다.

> **게이트:** 사용자의 명시적 확인 없이 검토 완료로 처리하지 않는다.  
> 계획서 승인 없이 구현을 시작하지 않는 것과 동일한 원칙이다.  
> 생략은 사용자가 명시적으로 요청한 경우에만 허용된다.

`inject/layer1_review.md` 지시에 따라 진행한다.  
리포트 제시 후 반드시 사용자 응답을 기다린다. 자동으로 `검토 완료` 처리하지 않는다.

## 작업 완료

사용자가 완료를 요청하면:

1. `workspace/tasks/active/[작업명]/` → `workspace/tasks/done/[작업명]/` 이동
2. `workspace/tasks/INDEX.md` 상태 업데이트
3. `plan.md`의 "완료 시 문서 업데이트 대상" 확인 후 `workspace/docs/` 반영
4. **Layer 2 주기 확인** — `workspace/tasks/INDEX.md`에서 마지막 `[Layer 2 완료]` 항목 이후 `done` 상태로 전환된 태스크 수를 센다.
   `[Layer 2 완료]` 항목이 없으면 done 전체를 기준으로 센다.
   3개 이상이면:
   ```
   💡 Layer 2 체크포인트 시점입니다 (완료 태스크 [N]개 누적).
   → 지금까지 완료된 태스크들의 전체 정합성을 확인합니다:
     · 아키텍처 규칙 위반 여부 점검
     · 태스크 간 인터페이스 충돌·불일치 확인
     · architecture.md / direction.md 현행화 여부 검토
   진행할까요?
   ```
   > Layer 2 완료 시 `workspace/tasks/INDEX.md` 하단에 `[Layer 2 완료] YYYY-MM-DD` 한 줄을 추가한다.

## 기술/도메인 지식 기록 기준

### 기술 지식 (언어·프레임워크·라이브러리)

> 훈련 데이터를 신뢰한다. 일반적인 사용법은 기록하지 않는다.

**기록하는 것:** 이 환경에서 실제로 문제가 된 패턴, 자연스럽게 시도하지만 여기서는 작동하지 않는 것  
**기록하지 않는 것:** 일반적인 Best Practice, 공식 문서에 있는 내용

**기록 위치:** `workspace/memory/domains/[관련 도메인]/rules.md` → 절대 금지 섹션

### 도메인 지식 (업종·비즈니스 규칙)

| 범위 | 기록 위치 |
|------|---------|
| 이 프로젝트 한정 규칙 | `workspace/memory/domains/[도메인]/rules.md` |
| 다른 프로젝트에도 유효한 지식 | `upgrade-candidates/` → 검토 후 `.mpa-workspace/knowledge/[도메인].md` |

> "이 도메인을 다루는 다른 프로젝트에서도 모르면 실수할 내용인가?"

---

## 역할 메모리 업데이트

작업 완료 후 **자가 개선 필터**를 적용해 기록 여부를 판단한다.

다음 중 하나 이상 해당하면 기록한다:
- plan.md "에이전트 가정"이 구현 중 틀린 것으로 밝혀졌는가?
- 구현 중 plan.md 자체를 수정해야 했는가?
- 범위 밖 변경을 발견해 분리 보고했는가?

위 조건에 해당하지 않더라도 "다음에 이 역할이 몰랐다면 실수했을 내용인가?"를 보조 기준으로 확인한다.

**기록 O:** 예상과 다르게 동작한 패턴 / 반복적으로 나타나는 실수 유형 / 코드·문서만으로 알 수 없는 제약  
**기록 X:** 태스크 완료 사실 / 코드에서 직접 읽을 수 있는 내용 / 일회성 결정 / 기억할 것이 없는 경우

해당 내용은 `workspace/memory/roles/[페르소나명].md`에 직접 작성한다.  
다른 역할의 파일은 수정하지 않는다.

## 다음 작업 예측

작업이 마무리될 때 사용자의 다음 행동을 예측해 자연스럽게 제안한다.  
강요하지 않고 한 줄로, 선택지를 제시하는 방식으로 끝낸다.

| 상황 | 제안 |
|------|------|
| plan.md 완성, 결정 필요 항목 있음 | "사용자 결정 필요 항목을 먼저 검토할까요?" |
| plan.md 완성, 비평 권장 기준 해당 | 비평 필요 여부 자동 평가 섹션의 메시지 사용 |
| plan.md 완성, 비평 권장 기준 미해당 | "바로 구현을 진행할까요?" |
| 구현 완료 | "작업 결과 검토를 진행할까요?" |
| 작업 결과 검토 중 (이슈 있음) | "위 이슈들이 발견됐습니다. 수정을 진행할까요?" |
| 작업 결과 검토 중 (이슈 없음) | "검토 결과 이슈가 없습니다. 직접 사용해보신 후 추가로 필요한 것이 있으신가요?" |
| 태스크 완료 | "다음 태스크를 계획할까요?" |
| 여러 태스크 완료 후 | "전체 정합성 점검(Layer 2)을 할 시점인 것 같습니다. 진행할까요?" |

## 작업 종료 시 (공통)

inject 파일의 "세션 종료 시" 항목을 확인하고 필요한 `workspace/` 파일을 업데이트한다.

하네스 개선 후보 발견 시 `.mpa-workspace/upgrade-candidates/[내용].md`에 기록한다.

**타입 A — 방법론 개선** (agent 운영 방식, inject 파일, 규칙 등)
```markdown
# [개선 내용 제목]
**타입**: 방법론 개선
**발견 상황**: [어떤 작업 중 발견했는지]
**적용 범위**: 이 프로젝트 / 모든 프로젝트

## 현재 방식
[현재 어떻게 하고 있는지]

## 개선 방안
[더 나은 방법]

## 적용 대상 파일
- `.mpa-workspace/[파일명]`
```

**타입 B — 도메인 지식** (다른 프로젝트에도 유효한 도메인·업종 지식)
```markdown
# [도메인명]: [지식 제목]
**타입**: 도메인 지식
**도메인**: [결제 / 인증 / 알림 / ...]
**발견 상황**: [어떤 작업 중 발견했는지]

## 내용
[기록할 지식 — 절대 금지 / 핵심 제약 / 검증된 패턴 / 주의사항]

## 적용 대상 파일
- `.mpa-workspace/knowledge/[도메인명].md`
```

> 코드·비즈니스 개선이 아닌 **에이전트 운영 방식 또는 도메인 지식**만 여기에 기록한다.
