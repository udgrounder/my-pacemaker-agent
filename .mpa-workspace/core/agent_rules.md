## 세션 시작 루틴

> 사용자 요청을 처리하기 전, 매 세션 시작 시 자동으로 실행한다.

### 1. 현재 상태 파악

1. `workspace/project_rules.md` 확인 (존재하는 경우) — 프로젝트 고유 라우팅 힌트·행동 규칙 로드
2. `workspace/tasks/active/` 폴더 확인 — **이 폴더가 진행 중 태스크의 primary source다**
3. `workspace/tasks/INDEX.md` 확인 (존재하는 경우) — 요약·타입 참조용 cache

> **INDEX.md 동기화 원칙:** active/ 폴더에 있는데 INDEX에 없는 항목만 누락 처리. INDEX 업데이트는 태스크 생성·완료·Layer 2 체크포인트 시에만 수행.

**메모리 시스템 우선순위:**

두 개의 메모리 시스템이 병존한다:

| 시스템 | 위치 | 역할 |
|--------|------|------|
| **프로젝트 메모리** (primary) | `workspace/memory/` | 프로젝트 설계 결정, 아키텍처, 계약, 도메인 규칙 |
| **세션 메모리** (보조) | `~/.claude/projects/.../memory/` | 사용자 선호, 협업 방식, 개인 컨텍스트 |

**충돌 시 원칙:**
- `workspace/memory/`가 항상 우선한다 — 프로젝트 사실(코드·설계·결정)에 관한 한 정답은 workspace다.
- 세션 메모리의 "기억"이 workspace 내용과 다르면 → workspace를 신뢰하고 세션 메모리를 오래된 것으로 간주한다.
- 세션 메모리는 workspace에 없는 **사용자 행동 방식·선호·협업 스타일**에만 권위를 가진다.

**단계 모델 — major (기본, 7단계):**
```
설계 중 → 설계 완료 ⛔G1 → 구현 중 → 검증 중 → 테스트 중 → 검토 완료 ⛔G2 → 완료 승인 → done
```
- **GATE 1** (`설계 완료 → 구현 중`): 사용자 명시적 승인 필요. plan.md 재수정 시 재승인.
- **GATE 2** (`검토 완료 → 완료 승인`): 사용자 명시적 완료 요청 필요. `완료 승인` 없이 done 이동 차단.
- **소스 수정**: `구현 중` 상태에서만 허용. 다른 단계에서 소스 수정 필요 시 `구현 중`으로 되돌린다.
- **되돌아가기**: 어느 단계에서든 이전 단계로 자유롭게 복귀 가능 (게이트 없음).

**단계 모델 — minor (경량, 3단계):**
```
메모 [자동 승인] → 구현 중 → done
```
- **GATE 1·2 생략**: 에이전트가 최소 plan.md 작성 직후 즉시 `approve` 실행. 사용자 명시적 승인 불필요.
- **검증·테스트 단계 생략**: 구현 완료 후 에이전트가 바로 done 처리.
- **소스 수정**: 동일하게 `구현 중` 상태에서만 허용 (code_gate.py 기계 강제).
- **major 전환**: 구현 중 범위가 예상보다 커지면 사용자에게 알리고 major 흐름으로 전환 가능.

**진행 중 태스크가 있는 경우** — plan.md YAML 프론트매터 `상태` 필드 기준:

| 상태 | 출력 메시지 | 기본 다음 단계 |
|------|-----------|--------------|
| 설계 중 | "계획서를 작성 중입니다. 이어서 진행할까요?" | 계획서 작성 |
| 설계 완료 | "계획서가 준비됐습니다. 검토 후 구현을 승인해주세요." | 계획서 검토 (GATE 1) |
| 구현 중 | "구현이 진행 중입니다. 이어서 계속할까요?" | `layer1_implement.md` |
| 검증 중 | "에이전트 검증이 진행 중입니다. 이어서 계속할까요?" | `layer1_review.md` |
| 테스트 중 | "사용자 테스트 단계입니다. 테스트 완료 후 알려주세요." | 사용자 확인 대기 |
| 검토 완료 | "검토가 완료됐습니다. 완료 처리를 승인해주세요." | 완료 처리 (GATE 2) |
| 완료 승인 | "완료 승인됐습니다. done으로 이동할까요?" | 태스크 완료 처리 |

**진행 중 태스크가 없는 경우:** 상태 보고 없이 사용자 요청을 기다린다.

**upgrade-candidates 확인:**
`.mpa-workspace/upgrade-candidates/` 폴더에 파일이 있으면:
```
📦 MPA 시스템 개선 후보가 [N]개 있습니다. 검토하시겠어요? (나중에 하셔도 됩니다)
```
사용자가 거절하거나 무시하면 이번 세션에서는 더 이상 언급하지 않는다.

**처리 완료 후 — 삭제 대신 archive로 이동:**
파일 상단에 처리 결과를 한 줄 추가하고 `.mpa-workspace/upgrade-candidates/archive/`로 이동한다.
```
# [처리] 승인 — YYYY-MM-DD / 적용: [파일명]
# [처리] 거부 — 이유: [한 줄]
# [처리] 보류 — 이유: [한 줄]
```
이력을 남기면 같은 후보가 재제안되는 것을 방지하고, 거부 이유를 다음 세션에서 참조할 수 있다.

**workspace/ 자체가 없는 경우:** "workspace/가 없습니다. 프로젝트 초기화(Layer 0)가 필요합니다. 지금 시작할까요?"

### 2. 요청 유형 자동 판단

**[선행 확인] active 태스크가 있으면 → 태스크 맥락 라우팅 우선**

| plan.md 상태 | 라우팅 | inject |
|-------------|--------|--------|
| 설계 중 | 계획서 작성 이어서 | `layer1_design.md` |
| 설계 완료 | 계획서 검토 (GATE 1 대기) | — |
| 구현 중 | 구현 이어서 | `layer1_implement.md` |
| 검증 중 | 에이전트 검증 이어서 | `layer1_review.md` |
| 테스트 중 | 사용자 테스트 대기 | — |
| 검토 완료 | 완료 처리 (GATE 2 대기) | — |
| 완료 승인 | done 이동 처리 | — |

**단, 아래 조건 중 하나라도 해당하면 신규 요청으로 처리한다:**
- 현재 태스크와 무관한 파일·기능을 명시적으로 언급
- "별개로", "새로", "다른 거" 등 분리 의도가 명확한 발화
- 현재 태스크 범위 밖의 버그·기능 보고

**신규 여부가 모호한 경우:** "지금 [태스크명] 작업 중입니다. → 이 작업의 일부로 처리할까요, 새 작업으로 분리할까요?"

---

**[신규 요청] active 태스크가 없거나 명확히 새 요청인 경우 → 의도 판단**

> `workspace/project_rules.md`의 라우팅 힌트가 있으면 아래 순서보다 **우선 적용**한다.

**판단 우선순위 (순서대로 적용):**

1. **명시적 유형 선언** — "버그야", "새 기능", "리팩터링" 등 사용자가 유형을 직접 말함 → 즉시 해당 유형으로 라우팅
2. **동작 서술 vs 추가 요청** — "~하면 ~가 돼" / "~이 이상해" 형태 → 버그 수정 / "~해줘" / "~만들어줘" 형태 → 새 기능 또는 리팩터링
3. **키워드 힌트** — 아래 테이블 참조 (힌트일 뿐, 결정적 기준이 아님)
4. **불명확** → 2단계 진단

| 발화 패턴 (힌트) | 판단 유형 | 자동 로딩 |
|---------------|---------|---------|
| "~기능 만들어줘", "~추가해줘", "~구현해줘" | 새 기능 설계 | `workflows/new_feature.md` + `layer1_design.md` + `personas/task_designer.md` |
| "~버그", "~안 돼", "~에러", "~오류", "~이상해", "~이상하게", "예상이랑 달라", "안 맞아", "틀린 것 같아" | 버그 수정 | `workflows/bug_fix.md` + `layer1_review.md` + `personas/code_reviewer.md` |
| "이어서 해줘", "계속해줘", "어디까지 했어" | 작업 재개 | → "작업 재개" 섹션(`agent_rules_detail.md`) |
| "코드 봐줘", "검토해줘", "리뷰해줘" | 작업 결과 검토 | `workflows/code_review.md` + `layer1_review.md` + `personas/code_reviewer.md` |
| "리팩터링", "정리해줘", "개선해줘", "느려", "무거워", "복잡해" | 리팩터링 | `workflows/refactoring.md` + `layer1_design.md` + `personas/task_designer.md` |
| "전체 확인", "정합성 점검", "전체 점검", "충돌 확인" | 전체 정합성 점검 | `layer2_checkpoint.md` + `personas/integration_auditor.md` |
| "초기화", "프로젝트 설정", "Layer 0" | 프로젝트 초기화 | `layer0_init.md` + `personas/architect.md` |
| "MPA 시스템 업데이트", "업그레이드" | MPA 버전 업데이트 | `layer0_update.md` + `personas/architect.md` |
| "규칙 바꿔줘", "inject 수정", ".mpa-workspace 수정" | MPA 파일 수정 | → "MPA 시스템 파일 수정 규칙" 섹션 |

**판단 후 반드시 고지 — 라우팅 결과 + 정정 옵션:**

```
→ [버그 수정]
   로드: workflows/bug_fix.md · layer1_review.md · personas/code_reviewer.md
  ↪ 다른 유형이면: 새 기능 / 개선 / 단순 질문
```

- 로드한 파일 전체를 명시한다 — 사용자가 에이전트가 무엇을 읽었는지 확인할 수 있어야 한다.
- workflows 파일이 없는 유형(전체 정합성 점검, 초기화 등)은 로드 파일만 나열한다.
- 두 번째 줄은 가능한 다른 유형을 짧게 보여 사용자가 한 단어로 정정할 수 있게 한다.
- 정정 옵션은 발화 맥락에 맞는 유형 2~3개만 보여준다 (가능한 전체 유형을 다 나열하지 않는다).

**유형 불명확 시 — 2단계 진단 (라우팅 추정조차 어려운 경우):**
1. "지금 [기능]이 의도한 대로 작동하고 있나요?" → 아니오: 버그 / 예: 질문 2
2. "새 기능 추가인가요, 기존 기능 개선인가요?" → 새 기능: 설계 / 개선: 리팩터링

**라우팅 정정 발생 시:** 정정된 유형으로 즉시 재라우팅 + `.mpa-workspace/upgrade-candidates/`에 타입 A(방법론 개선) 형식으로 기록.

### 4. Task 필요 여부 판단

**Task 없이 바로 처리:** 단순 질문·탐색 / 단일 파일·비가역적 변경 없음
**Task 만들어 진행:** 새 기능·다중 파일 변경 / 가치 결정 포함 / 비가역적 변경 / 사용자 검토 필요

판단 후 고지하고 기본 경로로 진행. 사용자가 정정 가능:
```
→ [판단 근거] → Task 없이 바로 진행합니다. / Task를 만들어 진행합니다.
```

## 작업 생성

세부 절차: `inject/layer1_design.md` 참조.

**핵심 규칙:**
- plan.md를 채팅이 아닌 **파일로 먼저 저장**한다 (휘발 방지)
- **자기 점검**: 모든 plan.md에 적용 (같은 스레드, 체크리스트 기반) — `inject/layer1_critique.md` "자기 점검" 섹션
- **독립 비평**: 기준 충족 시만 (서브에이전트 전용) — `inject/layer1_critique.md` "독립 비평 세션이 필요한 기준" 섹션
- 같은 스레드 비평·새 스레드 비평·자기비평은 폴백으로 사용하지 않는다. 서브에이전트 도구가 없으면 사용자에게 위험을 고지하고 독립 비평 없이 진행할지 묻는다.

**minor 태스크 경량 처리 (GATE 1·2 생략):**

> **minor 판단 기준 — 셋 다 명확히 Yes여야 minor:**
> - 변경이 한 파일 또는 단일 관심사에 국한된다 — 코드를 열어보지 않아 불확실하면 No
> - 구현 전 설계 결정이 필요 없다 — 방법이 자명하지 않으면 No
> - git reset으로 즉시 복구된다 — 외부 상태(DB·파일시스템·외부 API) 변경이 있으면 No
>
> 하나라도 No이거나 **불확실하면 major**. 보안·외부 연동·공개 API 변경이 있으면 critical.  
> 프로젝트가 라이브러리인지 서비스인지에 따라 기준이 달라지지 않는다.

`실패비용: minor` 판단 시 전체 plan.md 형식 대신 아래 경량 흐름만 처리한다. 목적은 게이트 감사 기록을 남기되, 작업 운영 비용을 최소화하는 것이다.

1. **최소 plan.md 작성** — `templates/minor_plan_template.md` 형식 사용. 목적·요청·핵심 기능·구현 항목만 기록한다. 반례·검증 체크리스트·문서 업데이트 대상은 쓰지 않는다.
2. **계획서 제시 + 검토 요청** (Zone 3 — 명시적 승인 불필요):
   - 필수 판단 질문(언어, 기능 범위 등)이 있으면 **2개 이내**로 먼저 확인한다.
   - 아래 형식으로 계획서를 제시한다. 거부나 방향 변경이 아닌 응답이면 구현을 시작한다:
   ```
   minor 태스크: [목적].
   [계획 핵심 1-2줄]
   → 이대로 진행할까요? 다른 방향이면 말씀해 주세요.
   ```
3. **`approve` 실행** — 검토 요청 직후 또는 사용자 응답 반영 후:
   ```bash
   python3 .mpa-workspace/hooks/plan_hash.py approve workspace/tasks/active/[작업명]/plan.md
   ```
4. 구현 완료 후 **GATE 2 없이 바로 done 처리** (아래 "작업 완료" 섹션 참조)
5. minor에서는 changelog.md, review_phase*.md, 역할 메모리, docs 업데이트, Layer 2 현황 표시는 기본 생략한다. 외부 동작·아키텍처·계약이 바뀐 경우에만 해당 문서를 갱신하고, 그 순간 major 전환 여부를 재판단한다.

> **minor → major 전환**: 구현 중 예상 범위를 초과하거나 가치 결정이 생기면 즉시 사용자에게 알리고 major 흐름으로 전환한다.

**검증 격리 (major 전용):**
- **에이전트 검증**(`검증 중` 단계): 비평과 동일하게 **서브에이전트로 실행** — 구현 컨텍스트 격리. 1차(changelog 미참조) → 2차(changelog 대조) 두 단계 서브에이전트 호출. 상세: `inject/layer1_review.md` "실행 환경" 섹션.
- 서브에이전트 사용 불가 환경 → 자가 검증으로 대체 (위험 고지 후 진행).
- **명시적 승인 없이 구현을 시작하지 않는다** (GATE 1, major 전용 — minor는 자동 승인)

**승인 시 처리:**
1. **approve 명령 실행** — 상태 전환과 해시 기록을 원자적으로 처리:
   ```bash
   python3 .mpa-workspace/hooks/plan_hash.py approve workspace/tasks/active/[작업명]/plan.md
   ```
   - plan.md 상태가 `설계 완료`(또는 minor의 `설계 중`)일 때만 실행된다.
   - 자동으로 상태를 `구현 중`으로 전환하고 승인해시를 기록한다.
   - 이미 `구현 중`인 경우 approve가 거부된다 — plan.md를 직접 수정해 상태를 우회할 수 없다.
   - 이 해시가 GATE 1 재진입 검증의 기준이 된다.

**GATE 1 재진입 (해시 기반 자동 검증):**
- code_gate.py가 소스 수정 시도 시 `구현 중` 상태 태스크의 본문 해시 vs `승인해시`를 자동 비교한다.
- **불일치·승인해시 없음** → 자동 차단. 복구 절차: `agent_rules_detail.md` "GATE 1 재진입 복구" 섹션 참조.
- ⚠️ 에이전트가 단독으로 "설계 영향 없음"을 판단해 해시를 갱신하지 않는다 — 사용자 확인 필수.

## 작업 완료

**minor 태스크 — 자동 완료 (GATE 2 없음):**

구현 완료 직후 에이전트가 바로 done 처리한다. 사용자 완료 요청 불필요.
Zone 3(자동 처리)로 아래 순서만 실행한다:

1. **plan.md `상태`를 `완료 승인`으로 업데이트** — GATE 2 우회가 아니라 자동 설정 (code_gate.py가 이 상태를 확인 후 mv 허용)
2. **INDEX.md 업데이트** — mv 전에 먼저 처리한다 (누락 방지):
   - `타입`: minor
   - `상태`: done
   - `요약`: plan.md 목적 1줄
   - `생성일`: plan.md `생성일` 필드값
   - `점검`: `-`
3. `workspace/tasks/active/[작업명]/` → `workspace/tasks/done/[작업명]/` 이동
4. 완료 내용을 한 줄로 보고:
   ```
   완료: [무엇을 했는가]. done 처리했습니다.
   ```

> minor 완료에서는 changelog, 에이전트 검증, 사용자 테스트 단계, docs 업데이트, 정합성 점검 제안을 생략한다. 필요한 항목이 생기면 minor가 아니라 major로 전환한다.

---

**major 태스크 — ⛔ GATE 2: 명시적 완료 요청만 처리한다.**

> **완료로 인정:** "완료", "완료 처리해줘" 등 완료 의사가 명확한 발화
> **에이전트가 먼저 물었을 때만:** "응", "맞아", "해줘"
> **완료로 인정하지 않음:** "넘어가자", "다음으로", "계속해" 등 방향 전환 발화
> **에이전트는 완료 처리를 제안할 수 있다. 단, 사용자 승인 없이 완료 처리하지 않는다.**

사용자가 완료를 요청하면:
1. **plan.md `상태`를 `완료 승인`으로 업데이트** (GATE 2 통과 기록)
2. `workspace/tasks/active/[작업명]/` → `workspace/tasks/done/[작업명]/` 이동
3. **INDEX.md 업데이트** — 다음 컬럼 기록:
   ```
   | 태스크명 | 타입 | 상태 | 요약 | 생성일 | 점검 |
   ```
   - `타입`: `major` / `minor`
   - `상태`: `done`
   - `점검`: 신규 완료는 `-` (미점검). 전체 정합성 점검 후 `✅`로 갱신.
4. `plan.md`의 "완료 시 문서 업데이트 대상" 확인 후 `workspace/docs/` 반영
5. **전체 정합성 점검 현황 표시**
   - INDEX.md에서 `점검` 컬럼이 `-`인 done 태스크를 집계한다 (단일 패스).
   - 집계 결과를 사용자에게 보여준다:
   ```
   완료됐습니다.
   아직 전체 점검을 거치지 않은 완료 작업: major [N]개 / minor [N]개

   이 작업들이 서로 충돌하지 않는지 전체 점검할 시점입니다.
   점검 내용:
     · 태스크 간 인터페이스 충돌·불일치 확인
     · 문서(architecture.md 등)와 실제 코드의 괴리 확인
     · 누적된 안티패턴 점검
   → 지금 전체 점검할까요?
   ```
   - 사용자가 판단. 에이전트는 임계값을 갖지 않는다.
   - 전체 점검 완료 시: 점검 대상 태스크들의 `점검` 컬럼을 일괄 `✅`로 갱신.

> **점검 상태 단일 소스: INDEX.md**
> `점검` 컬럼은 INDEX.md에서만 관리한다. plan.md에는 점검 상태를 기록하지 않는다.
> INDEX.md가 손상된 경우 점검 상태는 재확인이 필요하다 — plan.md로 복구할 수 없다.

## 작업 종료 시 (공통)

inject 파일의 "세션 종료 시" 항목을 확인하고 필요한 `workspace/` 파일을 업데이트한다.

**정정 회고 (강제):**

세션 종료 직전 한 가지를 자가 회고한다 — 한 줄 짧게 답한다. 회고 자체를 건너뛰지 않는다.

> **이번 세션에서 내가 잘못 판단한 것 / 사용자가 정정한 것이 있는가?**

- **있음:** 어떤 패턴인지 식별하고 `.mpa-workspace/upgrade-candidates/[내용].md`에 기록 (방법론 개선 타입 A 형식).
- **없음:** "정정 회고: 없음"이라고 명시한다. (회고 자체를 건너뛰지 않음으로 다음 세션에도 회고 습관이 유지된다.)

> **회고의 한계 인지:** LLM은 자신이 체계적으로 잘못 판단하는 패턴은 인식하지 못한다.  
> 이 회고는 "사용자가 명시적으로 정정한 것"과 "스스로 의식한 오판"만 잡는다.  
> 완전한 검증 메커니즘이 아니다 — 일부라도 잡는 차선책이다.

MPA 시스템 개선 후보 발견 시 → `.mpa-workspace/upgrade-candidates/[내용].md` 기록.
형식: `agent_rules_detail.md` "upgrade-candidates 형식" 섹션 참조.
