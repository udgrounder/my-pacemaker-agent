# Agent 행동 규칙 — 상세

> **이 파일은 `agent_rules.md`의 상세 참조용이다.**  
> 세션마다 자동 로드되지 않는다. 필요한 시점에 해당 섹션을 읽는다.

---

## 작업 재개

사용자가 "이어서 해줘", "계속해줘", "어디까지 했어" 등을 발화하면:

1. `workspace/tasks/active/` 및 `workspace/tasks/INDEX.md`에서 `active`, `hold` 태스크 확인

**active 1개:**
```
[태스크명] 재개합니다.
- 완료된 단계: [체크된 항목들]
- 이어갈 단계: [첫 미완료 항목]
```
→ 바로 진행.

**active 0개 또는 2개 이상:**
```
진행할 작업을 선택해주세요:
[active] 태스크명1 — 구현 중
[active] 태스크명2 — 설계 완료
[hold]   태스크명3 — 보류 중
```
목록이 없으면: "진행 중인 작업이 없습니다. 새 작업을 시작할까요?"

**재개 시 공통:**
1. plan.md 읽기 → 상태·구현 단계 체크리스트 확인
2. changelog.md 존재 시 읽기 → 완료된 부분 파악
3. plan.md 상태 기준으로 컨텍스트 선택 로딩(`agent_rules.md` 섹션 3) 적용
4. 해당 inject 파일 지시에 따라 첫 미완료 단계부터 진행

## 기술/도메인 지식 기록 기준

> **단방향 흐름 원칙:** 발견 시점에는 경계를 판단하지 않는다. 항상 단일 위치(`domains/`)에 기록하고, 정기적으로 승격 후보를 평가한다.

### 1단계 — 발견 시: 항상 domains/에 기록

지식을 발견했을 때는 "이 프로젝트만의 것인가, 보편적인가"를 판단하지 않는다.  
일단 `workspace/memory/domains/[관련 도메인]/rules.md` 에 기록한다.

**기록하는 것:**
- 이 환경에서 실제로 문제가 된 기술 패턴 (언어·프레임워크·라이브러리)
- 업종·비즈니스 규칙 (이 프로젝트에서 확인된 도메인 제약)
- 자연스럽게 시도하지만 작동하지 않는 것 (절대 금지 섹션)

**기록하지 않는 것:**
- 일반적인 Best Practice, 공식 문서에 있는 내용 (훈련 데이터를 신뢰한다)
- 일회성 결정

### 2단계 — Layer 2 체크포인트 시: 승격 후보 평가

전체 정합성 점검(`layer2_checkpoint.md`)을 진행할 때, `domains/`에 기록된 지식 중 다음 질문에 답한다:

> **"이 지식이 다른 프로젝트의 의사결정도 바꾸는가?"**

답이 Yes인 항목만 `.mpa-workspace/upgrade-candidates/[내용].md`에 도메인 지식 형식으로 export한다. (`agent_rules_detail.md` "upgrade-candidates 형식" 타입 B 참조)

### 3단계 — upgrade-candidates 검토 시: knowledge/로 승격

사용자가 `upgrade-candidates`를 검토하고 승인하면 `.mpa-workspace/knowledge/[도메인명].md`로 이동한다.  
승격된 지식은 **실제로 다른 프로젝트에서 import해 사용 가능한 검증된 지식**이다.

### 위치 요약

| 위치 | 단계 | 신뢰도 |
|------|------|--------|
| `workspace/memory/domains/[도메인]/rules.md` | 발견 즉시 기록 | 이 프로젝트 한정 |
| `.mpa-workspace/upgrade-candidates/` | Layer 2 시 export 후보 | 평가 중 |
| `.mpa-workspace/knowledge/[도메인].md` | 사용자 승인 후 승격 | 검증된 범용 지식 |

> 기록 시점에 경계를 판단하지 않으므로 인지 부담이 분산된다.
> knowledge/는 진짜 검증된 지식만 모이는 곳이 되어 신뢰도가 보장된다.

## 역할 메모리 업데이트

작업 완료 후 **자가 개선 필터** 적용:

다음 중 하나 이상 해당하면 기록한다:
- plan.md "에이전트 가정"이 구현 중 틀린 것으로 밝혀졌는가?
- 구현 중 plan.md 자체를 수정해야 했는가?
- 범위 밖 변경을 발견해 분리 보고했는가?

**기록 O:** 예상과 다르게 동작한 패턴 / 반복적으로 나타나는 실수 유형 / 코드·문서만으로 알 수 없는 제약
**기록 X:** 태스크 완료 사실 / 코드에서 직접 읽을 수 있는 내용 / 일회성 결정

기록 위치: `workspace/memory/roles/[페르소나명].md`

## 다음 작업 예측

작업이 마무리될 때 사용자의 다음 행동을 자연스럽게 제안한다. 강요하지 않고 한 줄로.

| 상황 | 제안 |
|------|------|
| plan.md 완성, 결정 필요 항목 있음 | "결정 필요 항목을 먼저 검토할까요? → 결정 후 구현 / 바로 구현" |
| plan.md 완성, 비평 권장 기준 해당 | "독립 비평을 진행할까요? → 비평 진행 / 바로 구현" |
| plan.md 완성, 비평 기준 미해당 | "바로 구현을 진행할까요? → 구현 시작 / 계획서 재검토" |
| 구현 단계 종료 | 에이전트 검증(`검증 중`)으로 자동 전이 후 사용자 테스트(`테스트 중`) 안내 |
| 테스트 완료 (이슈 있음) | "위 이슈들이 발견됐습니다. → 수정 진행 / 이슈 무시하고 완료" |
| 테스트 완료 (이슈 없음) | "이슈가 없습니다. 완료 처리를 원하시면 말씀해주세요." |
| 태스크 완료 | "다음 태스크를 계획할까요? → 새 태스크 / 잠시 중단" |

## upgrade-candidates 형식

**타입 A — 방법론 개선** (agent 운영 방식, inject 파일, 규칙 등)
```markdown
# [개선 내용 제목]
**타입**: 방법론 개선
**발견 상황**: [어떤 작업 중 발견했는지]
**적용 범위**: 이 프로젝트 / 모든 프로젝트

## 현재 방식
[현재 어떻게 하고 있는지]

## 개선 방안
[더 나은 방법]

## 적용 대상 파일
- `.mpa-workspace/[파일명]`
```

**타입 B — 도메인 지식** (다른 프로젝트에도 유효한 도메인·업종 지식)
```markdown
# [도메인명]: [지식 제목]
**타입**: 도메인 지식
**도메인**: [결제 / 인증 / 알림 / ...]
**발견 상황**: [어떤 작업 중 발견했는지]

## 내용
[기록할 지식]

## 적용 대상 파일
- `.mpa-workspace/knowledge/[도메인명].md`
```

> 코드·비즈니스 개선이 아닌 **에이전트 운영 방식 또는 도메인 지식**만 여기에 기록한다.

사용자가 라우팅을 정정한 경우 `.mpa-workspace/upgrade-candidates/routing_mismatches.md`에 누적:

```markdown
## [날짜] 오라우팅 사례
- 원래 발화: "[사용자 발화]"
- 판단된 유형: [유형]
- 정정된 유형: [유형]
- 제안 패턴: "[발화 패턴]" → [유형]으로 분류
```
