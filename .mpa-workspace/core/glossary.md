# MPA 용어집

> 매 세션 자동 로드 대상이 아니다. 용어 의미가 모호하다고 판단될 때만 필요한 항목을 찾아 읽는다(`agent_rules.md` 상단 트리거 표 참조). 이 파일은 `core/`에 있지만 `agent_rules.md`와 달리 disclosed reference다 — `architecture.md`의 "파일별 역할" 표에 그 예외가 명시돼 있다.

---

## Zone

사용자 개입 강도 등급(1~3). `agent_rules.md` "사용자 개입 등급" 표가 단일 진실원 — 이 파일은 정의를 반복하지 않는다.
- Zone 1: 하드 게이트 (명시적 확인 없이 진행 불가)
- Zone 2: 강조 확인 (핵심만 짧게 확인 요청)
- Zone 3: 자동 처리 (실행 후 간략 고지)

_Avoid(혼동 표현):_ 우선순위, 권한 레벨 — Zone은 "개입 강도"를 가리키지 권한 체계가 아니다.

## Tier

`workspace/memory/`의 계층 구조(1~3). `discussion_log.md` 테제 9~10 및 `workspace/memory/` 설계 원칙이 정본.
- Tier 1: 핵심(거의 안 변함) — 불변 규칙, 프로젝트 정의
- Tier 2: 진화(프로젝트와 함께 발전) — 아키텍처 패턴, 인터페이스 계약
- Tier 3: 축적(계속 쌓임) — 결정 이력, 안티패턴

_Avoid:_ 레벨, 단계 — Tier는 "기억의 변화 속도·신뢰도"를 가리키지 작업 진행 단계(Layer 0/1/2)가 아니다. `architecture.md`의 "구성 층위" 섹션이 Layer와 Tier·구성 층위를 서로 다른 개념으로 이미 구분해뒀다.

## 계획서 vs PRD

- **계획서** = `plan.md` 파일 전체(`## PRD` + `## 구현 계획` + `## 완료 기준` 세 영역을 전부 포함).
- **PRD** = 계획서 안의 앞부분(무엇을·왜: 에이전트 보고·요청 원문·목적·요구사항). 어떻게(구현 단계)는 담지 않는다.

즉 모든 PRD는 계획서의 일부지만, 계획서가 곧 PRD는 아니다.

## plan.md / 계획서 / 플랜 / 설계서 — 통용 표현

전부 같은 대상(`workspace/tasks/active/[태스크명]/plan.md`)을 가리키는 표현이다. 사용자가 대화 중 "플랜"이나 "설계서"라고 불러도 같은 파일을 가리키는 것으로 인식한다. 정본 표기는 `plan.md`(파일 경로 지칭) 또는 "계획서"(산문·사용자 대면 문구)이고, "플랜"·"설계서"는 이 두 정본에 대한 통용 별칭이다.

## MPA의 계획서 체계 vs Claude Code 네이티브 Plan Mode

이름이 비슷하고 트리거 조건(다중 파일 변경, 아키텍처 결정 등)도 겹치지만 서로 다른 메커니즘이다.

| | MPA 계획서 체계 | Claude Code 네이티브 Plan Mode |
|--|--|--|
| 파일 | `workspace/tasks/active/[태스크명]/plan.md` (git 추적, 영구 보존) | 세션에 종속된 별도 plan 파일(`EnterPlanMode`/`ExitPlanMode`) |
| 지속성 | 여러 세션에 걸쳐 지속, `done/`으로 이동 후에도 이력 보존 | 해당 세션·작업 한정, 지속 저장 안 됨 |
| 승인 게이트 | 계획 승인·완료 승인 확인(GATE, 해시 기반 재검증 포함) 이중 게이트 | 사용자 승인 1회(구현 시작 허가) |
| 범위 | 코드 변경 외에 토론·문서·리팩터링 등 전 유형 | 코드 구현 작업 전용 |

두 메커니즘이 동시에 발동할 수 있으나 서로 대체하지 않는다 — MPA 계획서 작성이 네이티브 Plan Mode 승인을 대신하지 않고, 그 역도 마찬가지다.

## GATE — 절차(메커니즘)

"단계 전환 지점에서 승인을 처리하는 절차"를 가리키는 내부 개발 용어. **훅 스크립트(`code_gate.py`/`plan_hash.py`) 내부 주석·`MPA_GATE` 환경변수 이름·`session_start.py`의 일반 명사 사용("코드 수정 게이트")에만 남아 있다.** 사용자·에이전트가 보는 절차 설명·출력 메시지에서 "GATE 1"/"GATE 2"처럼 번호로 특정 절차를 지칭하는 자리는 아래 두 이름으로 대체됐다:

- **계획 승인** — `설계 완료 → 구현 중` 전환에 필요한 절차(과거 "GATE 1")
- **완료 승인 확인** — `검토 완료 → 완료 승인` 전환에 필요한 절차(과거 "GATE 2")

_Avoid:_ "GATE 1"/"GATE 2"를 절차 설명·사용자 대면 문구에서 사용하는 것 — 훅 스크립트 내부에서만 쓴다.

## 계획 승인 / 완료 승인 확인 ↔ Definition of Ready / Definition of Done

GATE(절차)가 무엇을 "확인"하는지는 애자일의 표준 개념과 대응한다 — GATE는 메커니즘, DoR/DoD는 그 메커니즘이 판정하는 기준(criteria)으로 서로 다른 층위다.

- **계획 승인**이 확인하는 기준 = **Definition of Ready**(PRD 완결성 — 사용자 결정 필요 항목이 모두 답변됐는가)
- **완료 승인 확인**이 확인하는 기준 = **Definition of Done**(`## 완료 기준` 섹션 — 검증 체크리스트·문서 갱신·발견 항목 처리가 끝났는가)

## 완료 승인 확인(절차) vs 완료 승인(상태) — 구분

같은 문자열처럼 보이지만 다른 것을 가리킨다.

- **완료 승인 확인** — 절차(동사적). "검토 완료 → 완료 승인"으로 넘어가기 위해 사용자 확인을 거치는 행위.
- **완료 승인** — 상태(명사). `plan.md` YAML 프론트매터 `상태` 필드가 가질 수 있는 값 중 하나. `agent_rules.md`의 단계 모델 다이어그램에 그대로 등장한다.

"완료 승인 확인 단계에서"는 절차를, "상태가 완료 승인으로 바뀌었다"는 상태를 가리킨다 — 서로 바꿔 쓰지 않는다. ("계획 승인"은 대응하는 상태 이름이 따로 없어 이 구분이 필요 없다.)
