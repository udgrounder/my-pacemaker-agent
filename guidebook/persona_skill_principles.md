# 페르소나·스킬·워크플로우·도메인 — 구성 원칙 (v2)

> 출처: `workspace/exploration/discussion/persona_skill_relationship.md`(R1~14) + 설계 세션 정밀화(M1~M11) + 독립 비평 반영.
> 성격: 라운드 기록은 *과정*, 이 문서는 *쓸 수 있는 결론*.
> 교차참조: 서브에이전트·입도·캐시는 `guidebook/multi_agent_principles.md`. 그쪽이 "복수 에이전트를 어떻게 엮나"라면, 이 문서는 "하나의 전문가를 무엇으로 합성하나 + 시스템을 어떤 층위로 나누나"다.

---

## 0. 한 문장

> 시스템은 **구성 층위(워크플로우·역할·스킬·도메인)** 의 *내용*과, 그것을 단계마다 쓰레드에 싣는 **주입(직교 메커니즘)** 으로 나뉜다. 역할과 스킬은 상속이 아니라 **합성**되며(페르소나 ⊗ 스킬), 에이전트는 매 작업마다 둘을 곱이 아닌 **합**으로 조합해 전문가를 합성한다.

---

## 1. 용어 — "구성 층위" ≠ "Layer"

- **구성 층위(composition tier):** 합성·전달되는 *내용 종류* — 워크플로우 / 역할 / 스킬·도메인.
- **Layer 0/1/2:** *작업 구조*(초기화/태스크/교차 정합성, 6장).
- 같은 "층" 단어지만 다른 개념. 내용 종류=구성 층위, 작업 구조=Layer로 구별한다.

---

## 2. 경계 기준 — "역할 종속성"이지 "사실/스타일"이 아니다

**원칙 1 — 역할 범위가 가른다.** (R11·R14)
> 관련성이 *역할에 따라* 정해지면 → 페르소나. *역할과 무관*하면 → 스킬.
- 반례 탐색은 평가 역할만 쓴다 → 역할 종속 → 페르소나.
- java는 구현·리뷰·설계가 다 쓴다 → 역할 무관 → 스킬.

**원칙 2 — "사실 vs 스타일"은 틀린 축.** (R14)
- Spring 레이어 책임("Controller는 변환만")은 *관례/스타일*인데도 역할 무관 → 도메인 지식(스킬 우산).
- **프로젝트 전역 처방(관례 포함) = 스킬 / 역할 고유 태세(시선·가치) = 페르소나.**

---

## 3. 관계 모델 — 합성, 사람 프레임 탈피

**원칙 3 — 페르소나 ⊗ 스킬.** 페르소나=사실을 *어떻게 쓰나*(역할 종속), 스킬=그 *사실·도메인*(역할 무관). (R3·R10·R11)

**원칙 4 — 분리는 "사람 프레임"의 산물.** 사람=고정 존재+도구 집어듦(3층 분리). 에이전트=매 디스패치 **합성**. analysis를 "집어 드는 도구"로 보는 건 사람 흉내. (R10·R13)

**원칙 5 — 도메인별 역할을 원시 페르소나로 분리 금지(조합 폭발).** `금융/코드 분석가`를 별도 페르소나로 두면 R역할 × D도메인 폭발. 합성은 R+D(`분석가 ⊗ 금융`). 자주+이질+고실패 조합만 *캐시된 페르소나*로 둔다(파생이지 원시 아님). (R12)

---

## 4. 스킬 = 도메인 — 두 종류, 세 저장소, 역할-면

**원칙 6 — "스킬"은 how-to, "지식"은 what-is — 단 기능상 분리 불가.** "스킬"의 본뜻 = **규칙 기반 how-to**(기법·절차; 인간 "인터뷰 스킬" + Claude Code skill의 핵심). "지식"은 *무엇인가*(참조 사실). `analysis`=기법(how-to·도메인 일반), `programming`=도메인 지식(what-is·도메인 특정). **그러나 둘은 기능상 한 쌍이다** — 기법은 지식이 채워져야 작동하고(반례 탐색 *방법*만으론 공허), 지식은 기법이 있어야 활용된다. → **저장은 분리(재사용·DRY), 기능은 합성.** 별도 저장소로 쪼개지 않고 `skills/`(도메인 자원=기법+지식) 우산으로 둔다. "도메인 스킬"이라는 합성어는 피한다. (R10~14, 용법 정밀화)
- **역량은 3-way 합성:** `역량 = 역할(persona) ⊗ 기법(method) ⊗ 지식(domain)`. 셋 다 자원으로 따로 저장되나 능력은 합쳐질 때 창발한다. 진짜 "스킬"은 이 합성 결과다.

**원칙 7 — 평가 역할은 방법 도메인을 *구성적으로 요구*한다.** analysis는 페르소나에 접어 넣지 않는다 — 역할 무관 방법 도메인이고, code_reviewer·plan_critic·architect·integration_auditor가 "요구 도메인"으로 *선언*한다. "역할 종속"의 정체는 *내용*이 아니라 *필요*가 역할마다 다른 것. (R10·R14, 정밀화)

**원칙 8 — 도메인 세 저장소(역할 구분).**

| 저장소 | 내용 |
|--|--|
| `.mpa-workspace/skills/<도메인>/` | 도메인 **방법·패턴**(MPA 배포, 재사용) |
| `.mpa-workspace/knowledge/[도메인].md` | 승격된 **검증 사실**(크로스 프로젝트) |
| `workspace/memory/domains/<도메인>/` | 이 프로젝트의 **규칙·레지스트리**(=기억) |

skills=방법 / knowledge=검증 사실 / memory/domains=프로젝트 기억. 프로젝트 고유 주제 도메인이 생기면 `.mpa-workspace/skills/`(재사용)와 분리. 가용 도메인 목록은 `project_identity.md`에 선언.

**원칙 9 — 역할-면(role-facet), 기본은 단일 면.** 스킬은 역할 무관이라 전 역할이 합성한다. 기본은 단일 면(공통 사실). 같은 사실을 *쓰는 방식*이 역할별로 갈릴 때만 `[설계자/구현자/리뷰어 면]`으로 표시. (예: springboot.md "모르면 물어볼 것"이 이미 암묵적 설계/구현 면.) (R12~14)

**원칙 10 — 스킬은 2층: 가용 집합 / 활성 부분집합.** 차이는 가용 집합 출처뿐(주제 도메인=프로젝트, 방법 도메인=MPA). 활성 부분집합 선택 메커니즘은 동일. (R5)

## 5. 개념적 소유 ≠ 물리적 저장 (핵심 메타 원칙)

**원칙 11.** "무엇에 속하나"(소유=역할 종속성=*언제 활성*)와 "어디 두나"(저장=응집·DRY·이식성)는 별개. (R10·R13)
- 반례 탐색: 소유=페르소나(역할 종속) / 저장=`skills/analysis/` 공유(DRY).
- 금융 분석법: 소유=분석가 / 저장=금융 도메인(사실과 응집).
- 저장 위치로 소유를 판단하면 분류가 엉킨다.

## 6. 구성 층위 최종형 + 주입(메커니즘)

**원칙 12 — 워크플로우 3수준 + 메타 워크플로우.** (R8~R11, M8·M10)

```
메타 워크플로우 (core/agent_rules.md)        ← 진입 쓰레드 무조건 로드. 라우팅·게이트·상태·정합성 통치
   └─ 작업 워크플로우 (workflows/)            ← 작업유형 단계 시퀀스 + 자원 바인딩(정본)
        └─ 단계 워크플로우 (inject/layer1_*)   ← 재사용 단계 절차
             └─ 역할(personas) + 스킬·도메인(skills)  ← 단계에 합성
```
- 워크플로우가 주(主). `workflows/`가 단계↔자원 바인딩 정본(중복 제거 대상은 후속).
- 메타 워크플로우는 진입(PM) 쓰레드만 무조건 로드. 워커는 작업 워크플로우만.

**원칙 13 — 주입(inject)은 층위가 아니라 직교 메커니즘.** 워크플로우·역할·스킬은 *내용*, 주입은 그것을 단계에 따라 (새) 쓰레드에 *싣는 행위*. `inject/` 폴더의 정체는 "주입 레이어"가 아니라 **단계 워크플로우(내용)**. (M1 정정)

**원칙 14 — 절차 ≠ 스킬.** 시퀀싱·오케스트레이션=절차(inject) / 한 지점 적용 방법·지식=스킬. `plan_interview`는 절차라 inject에 둔다(분석 스킬 아님). (M3)

**원칙 15 — 작업 도메인 ≠ 주제 도메인.** 작업 도메인(개발·토론)은 *워크플로우+역할*을 가진 일의 종류. 주제/방법 도메인은 *지식*. 실제 작업 = 작업 도메인 워크플로우 ⊗ 주제 도메인 지식. MPA는 현재 개발 작업 도메인 시스템(+토론). (M10·M11)

## 7. 실행 모델 + 타이밍

**원칙 16 — 주입 타이밍은 단계 정의의 종속 결과.** 각 단위를 *그것이 좌우하는 결정의 직전*에. 페르소나=국면 단위 / 스킬=미시 단계 단위. 페르소나·스킬을 단계 정의에 묶으면 "일찍=낭비/늦게=재작업" 딜레마가 증발(최소이자 완전). (R7·R8)

**원칙 17 — 실행 모델.** 요청 → 작업 워크플로우 로드 → 단계마다 주입으로 페르소나·스킬 적재 → 결과물(파일) 생성 → 결과물+공유 캐시를 (새 스레드) 다음 단계로 → 단계 경계엔 사람 노드 → 반복. (M9)
- 핸드오프=파일(새 스레드가 읽음, 격리) + 공유 캐시(memory). 구현(write)은 단일 순차.

## 8. 관리 함의

- **skills/에는 역할 무관 도메인만.** 구성 기법(analysis)은 소유는 페르소나(요구 선언), 저장은 공유 DRY(원칙 11).
- **경계 기준은 절단 도구:** 한 파일이 역할-종속+무관을 섞으면 벤다(예: "보안 코딩" = OWASP 지식[스킬] + 사냥하는 눈[페르소나]).
- **단계→자원 선언의 단일 소스 = `workflows/`** (배선 정본화는 후속 태스크).
