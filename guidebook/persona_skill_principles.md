## 0. 한 문장

> 페르소나(역할에 종속된 *적용 방식*)와 스킬(역할과 무관한 *도메인 지식*)은 상속이 아니라 **합성**된다. 에이전트는 사람처럼 "고정된 존재에 도구를 쥐여주는" 것이 아니라, 매 작업마다 둘을 **곱이 아닌 합**으로 조합해 그때그때 전문가를 합성한다.

---

## 1. 경계 기준 — "역할 종속성"이지 "사실/스타일"이 아니다

**원칙 1.** 무엇이 페르소나에 속하고 무엇이 스킬에 속하는지는 **역할 범위**가 가른다. (R11·R14)

> **그 내용의 관련성이 *역할에 따라* 정해지면 → 페르소나. *역할과 무관*한 지식·도메인이면 → 스킬.**

- 반례 탐색은 평가하는 역할(설계자·비평가·리뷰어)만 쓰고 문서 작성자는 안 쓴다 → 역할 종속 → **페르소나**.
- java는 구현자·리뷰어·설계자가 모두 쓴다(역할이 무엇이든 java는 java) → 역할 무관 → **스킬**.

**원칙 2 — "사실 vs 스타일"은 틀린 축이다.** 처방적·관례적(스타일)이라는 것이 페르소나로 만들지 않는다. (R14)

- Spring의 레이어 책임("Controller는 변환만")은 명백히 *관례/스타일*인데도 *역할 무관*(모든 역할이 전제)이므로 **도메인 스킬**이다.
- 정렬:
  - **프로젝트 전역 처방**(관례·스타일 포함, 모든 역할이 따름) → 도메인 스킬.
  - **역할 고유 태세**(그 역할의 시선·가치, 예: "리뷰어는 결함을 사냥한다") → 페르소나.
- 스타일이든 사실이든 — *전역이면 스킬, 역할 고유면 페르소나.*

## 2. 관계 모델 — 합성(composition), 사람 프레임의 탈피

**원칙 3 — 페르소나 ⊗ 스킬.** 페르소나는 사실을 *어떻게 쓰는가*(적용 방식), 스킬은 그 *사실/도메인*이다. 같은 java 사실을 구현자는 쓰기에, 리뷰어는 버그 잡기에, 설계자는 제약 파악에 쓴다 — **사실은 역할 무관(스킬), 적용은 역할 종속(페르소나).** (R3·R10·R11)

**원칙 4 — 분리 자체가 "사람 프레임"의 산물이다.** (R10·R13)

| | 사람 프레임 | 에이전트 프레임 |
|--|------------|------------------|
| 존재 | 고정 → 역할 부여 → 도구 집어듦 (3층 분리) | 매 디스패치 **합성** |
| 도구/지식 | 사람이 *집어 드는* 별개 객체 | 전문가에 *내장해 합성* |

- 사람은 도메인 교체 비용이 커서(수년 학습) "분석 스킬을 가진 분석가"를 *직업*으로 캐시한다. 에이전트는 교체가 ~0이라 그 캐시가 불필요하다. analysis를 "분석가가 집어 드는 도구"로 보는 것은 사람 흉내의 잔재다.

**원칙 5 — 도메인별 역할을 *원시 페르소나*로 분리하지 마라 (조합 폭발).** (R12)

- `금융 분석가`·`코드 분석가`·`마케팅 분석가`를 별도 페르소나로 두면, 일관성을 위해 `금융 개발자`·`코드 개발자`…도 분리해야 한다 → **역할 R × 도메인 D 개**의 페르소나.
- 합성은 **R + D**: `분석가 ⊗ 금융`, `개발자 ⊗ 금융`… 곱이 아니라 합.
- "분리가 현실적"이라는 직감은 인간 직업의 현실(사람 프레임)이지 에이전트의 현실이 아니다.
- **예외 — 캐시:** 그 조합이 *자주 쓰이고* + *조준점이 이질적이며* + *실패비용이 높으면*, 합성을 미리 만든 **캐시된 페르소나**로 둘 수 있다(`multi_agent_principles` 원칙 12의 입도 조건). 단 이는 `분석가 ⊗ 금융`의 *파생·캐시*지 원시 페르소나가 아니다 — 밑단은 여전히 합성이다.

## 3. 스킬의 내부 구조 — 도메인 + 역할-면(面)

**원칙 6 — 도메인-특수 적용 방식(교차항)은 도메인 스킬에 역할-면으로 담는다.** (R12·R13·R14)

"금융을 *분석하는 방식*"은 역할(분석가)에도 도메인(금융)에도 걸리는 **교차항**이라 `페르소나 ⊗ 스킬`이 깔끔하게 안 떨어진다. 해소: 도메인 스킬을 **역할-면으로 구조화**한다.

```
금융 도메인 스킬
├─ 사실 (파생상품·회계)        ← 모든 역할 공통
├─ [분석가 면] 금융 분석법      ← 분석가 ⊗ 금융 일 때 활성
├─ [개발자 면] 금융 구현 관례    ← 개발자 ⊗ 금융 일 때 활성
└─ ...
```

- `분석가 ⊗ 금융` → 페르소나가 금융 도메인의 "분석가 면"을 골라 활성화한다. 도메인 하나·페르소나 하나, 합성이 맞는 면을 켠다.
- 분석법을 *페르소나*에 넣으면 그 페르소나가 D개 도메인의 분석법을 다 짊어진다(페르소나 쪽 폭발). 도메인에 넣어야 사실과 응집하고 페르소나는 일반 태세만 유지한다.
- **실물 증거:** 기존 `skills/tech/spring.md`의 "모르면 물어볼 것"(예외 노출?·로딩 전략?)은 *침묵 결정을 표면화하라*는 태세(페르소나 소유)가 spring 도메인에 적용된 교차항 — 즉 역할-면이 이미 암묵적으로 존재한다. 이 모델은 발명이 아니라 현 구조의 명시화다. (R14)

**원칙 7 — 스킬은 2층: 가용 집합 / 활성 부분집합.** (R5)

| | 가용 집합 (무엇을 아는가) | 활성 부분집합 (이번에 꺼내는 것) |
|--|---------------------------|----------------------------------|
| 도메인/tech 스킬 | 프로젝트가 정의(예: java·spring·js·vue) — 프로젝트마다 다름 | 작업이 닿는 것만 (프론트 작업 → js·vue) |
| 구성 기법(분석류) | MPA가 고정 보유 | 단계가 필요로 하는 것 |

- 차이는 **가용 집합의 출처뿐**(도메인=프로젝트, 구성 기법=MPA). 활성 부분집합을 고르는 메커니즘은 같다 — 작업/단계가 필요로 하는 것을 꺼낸다.

## 4. 개념적 소유 ≠ 물리적 저장 (핵심 메타 원칙)

**원칙 8.** "무엇이 페르소나/스킬에 속하나"(소유)와 "어느 파일에 두나"(저장)는 다른 문제다. (R10·R13)

- **개념적 소유** = 원칙 1의 역할 종속성 기준 = *언제 활성화되는가*를 정한다.
- **물리적 저장** = 응집도·중복 제거(DRY)로 *따로* 최적화한다.

| 내용 | 개념적 소유 (역할 종속?) | 물리적 저장 (왜) |
|------|--------------------------|-------------------|
| 반례 탐색 등 구성 기법 | 페르소나(역할 종속) | 공유 파일(여러 페르소나가 쓰므로 DRY) |
| 금융 분석법 | 분석가(역할 종속) | 금융 도메인 스킬(금융 사실과 응집) |

- 두 경우 모두 **개념상 페르소나 소유인데 물리적으로는 다른 곳에 저장**된다. 이 둘을 섞으면(저장 위치를 보고 소유를 판단하면) 분류가 엉킨다.

## 5. 주입 — 타이밍은 단계 정의의 종속 결과

**원칙 9 — 각 단위는 "그것이 좌우하는 결정의 직전"에 주입한다.** 너무 이르면 컨텍스트 낭비, 너무 늦으면 결정을 이미 잘못 내린 뒤다. (R7)

| 단위 | 좌우하는 결정 | 주입 타이밍 |
|------|----------------|--------------|
| 페르소나 | 그 국면의 모든 행동(태도) | 국면 진입 시 |
| 구성 기법 | 특정 미시 단계의 사고 | 그 단계에서 |
| 도메인 스킬 | 어느 스택/도메인을 건드리나 | 작업 범위가 그것을 드러낸 직후, 부분집합만 |

**원칙 10 — 주입의 원자 단위는 "단계"다.** 작업 = 단계 정의. 각 단계 = {할 일 + 페르소나 + 스킬}을 쥐여주고 진행 지시. 페르소나·스킬을 단계 정의에 묶으면 단계 진입 시 *선언된 것만* 들어오므로 **컨텍스트 최소이면서 동시에 완전**하다 — "일찍 부르면 낭비 / 늦게 부르면 재작업"의 딜레마가 증발한다. (R8)

- 페르소나는 **국면(phase)** 단위로 붙어 지속하고(설계·구현·검증), 스킬은 국면 안의 **미시 단계** 단위로 들고난다 — "단계"는 두 해상도를 가진다.
- 부담은 "언제 주입하나"에서 **"각 단계가 자기 페르소나·스킬을 정확히 선언했는가"** 로 옮겨간다.

**원칙 11 — PM/서브스레드 구조는 이 모델의 자연스러운 실현이다.** 메인 스레드가 PM으로 흐름·게이트를 쥐고, 국면마다 {맥락·문서·페르소나·스킬}을 서브스레드에 디스패치해 결과를 받는다. 단계=묶음 선언이 곧 디스패치 페이로드가 된다. (R9)

- 단, `multi_agent_principles`의 제약을 동반한다: **구현(write)은 단일 순차**(공유 가변 상태), 입도는 **국면 단위**(미시 스텝마다 서브스레드는 맥락 재주입 비용으로 손해).
- 디스패치 시 **도메인 스킬(관점-중립 사실)은 캐시/문서로 싸게 실어 보내고, 구성 기법(관점-특수 시선)은 페르소나와 한 몸으로** 주입한다(`multi_agent_principles` 원칙 11·15의 캐시 경계).

## 6. 관리 함의

**원칙 12 — 폴더 구조는 "역할 무관 지식"만 스킬에 둔다.** (R10·R11·R14)

- `skills/` (또는 도메인) = **역할 무관 도메인/기반 지식만** — 사실 + 역할-면 구조.
- 구성 기법(현 `skills/analysis/`)은 개념상 **페르소나 소유** — 페르소나가 선언하고, 공유는 DRY 팩토링 파일로 유지(저장≠소유, 원칙 8).
- 기준은 분류가 아니라 **절단 도구**다: 한 파일이 역할-종속 부분과 역할-무관 부분을 몰래 섞고 있으면 칼선을 따라 벤다(예: "보안 코딩" = OWASP 지식[스킬] + 사냥하는 눈[페르소나]).

**원칙 13 — 단계→(페르소나·스킬) 선언을 한곳에 모은다.** 현재 이 선언은 라우팅 표(요청→페르소나)와 inject 산문(단계→스킬)에 흩어져 있다. 단계 정의가 자기 페르소나·스킬을 선언하도록 통합하는 것이 "정리"의 실천적 핵심이다. (R8·R10)

## 부록 — 남은 미결 (구조 착지에서 다룰 것)

> 이 문서는 *원칙*이다. 아래는 실제 `.mpa-workspace/` 구조에 반영할 때 결정할 실무 항목.

1. **`skills/analysis/`의 재배치** — 페르소나가 선언하는 구성 기법으로 옮기되, 공유 파일(DRY)을 어떤 형태로 유지할지.
2. **`skills/tech/` → 도메인 + 역할-면 구조** — 기존 파일에 역할-면 태그를 명시(현재는 암묵).
3. **가용 도메인 집합의 집** — `project_identity`(사실, 코드 작업 시 로드) vs `project_rules`(항상 로드) 중 어디에 선언할지. (토론 보류)
4. **`multi_agent_principles`와의 경계** — PM/서브스레드·입도·캐시 영역이 겹친다. 교차참조로 둘지 한쪽으로 흡수할지.
5. **고위험 구성 기법의 "일찍 보장"** — pull이 단계 도달에 의존하므로, miss 비용이 큰 기법(침묵 결정 추출 등)은 원칙 9보다 일찍 보장할 기준이 필요.
