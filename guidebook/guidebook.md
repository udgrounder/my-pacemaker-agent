# my pacemaker agent 가이드북

> **이 문서의 성격:** AI agent와의 협업 구조에 관심이 있는 사람이 "왜 이런 구조인가"부터 "어떻게 쓰는가"까지 이해할 수 있도록 작성된 실용 안내서.  
> 설계 사고의 과정은 `think-more/discussion/` 참조.

---

## 들어가며: 이 가이드북이 답하는 질문들

이 가이드북은 다음 질문들에 순서대로 답한다.

- AI agent와 함께 일할 때 왜 계속 엉뚱한 결과가 나오는가?
- 왜 프롬프트를 잘 쓰는 것만으로는 부족한가?
- 체계는 무엇이고, 왜 이런 구조를 갖게 됐는가?
- 체계를 설치하고 나서 어떻게 작업을 진행하는가?
- 체계가 해결하지 못하는 문제는 무엇인가?

> **읽는 순서:** 순서대로 읽도록 설계되어 있다.  
> AI agent와 일하면서 막연한 불편함을 느껴봤다면 1부부터, 철학보다 구조와 실용이 궁금하다면 3부부터 시작해도 된다.  
> 단, 3부 이후가 "왜 이런 구조인가"로 이어지는 흐름은 1~2부에서 나온다.

---

## 1부: 왜 필요한가 — 문제 인식

> AI agent와 일할 때 느끼는 불편함의 정체를 먼저 이해한다.

---

### 1장. AI agent와 함께 일한다는 것

#### 1.1 기존 협업과 무엇이 다른가

소프트웨어를 만드는 방식은 크게 두 가지였다. 컴파일러나 린터 같은 **도구를 사용하거나**, 팀원과 **협업하거나**. AI agent는 이 두 프레임 어디에도 깔끔하게 들어맞지 않는다.

도구는 동일한 입력에 동일한 출력을 낸다. 규칙이 명확하고, 오류가 나면 즉시 알 수 있다. 사람은 다르게 작동한다. 맥락을 이해하고, 모르면 되묻고, 실수를 하면 어색함이 생긴다. 그런데 AI agent는 어떤가.

| 협업 대상 | 오류 발생 시 |
|-----------|-------------|
| 인간 동료 | 어색함이 생기고 되묻는다 |
| 컴파일러 | 에러가 나고 즉시 알 수 있다 |
| AI agent | **자신감 있게 틀리고, 오류가 보이지 않는다** |

AI agent는 컴파일러처럼 정확하지도 않고, 인간 동료처럼 스스로 멈추지도 않는다. 형식은 완벽하고, 어조는 자신감 있고, 코드는 실행된다. 그런데 원하는 것과 다르다. 이것이 AI agent 협업의 핵심 난제다.

왜 이런 일이 생기는가? 그것은 LLM이 어떤 존재인지를 이해해야 설명된다.

#### 1.2 LLM은 어떤 존재인가

LLM에 대해 "그냥 다음 단어 예측하는 것 아닌가"라고 생각하면 과소평가고, "이렇게 유창하니 이해하고 있겠지"라고 생각하면 과대평가다. 이 두 방향의 오해가 AI agent와 일할 때 예측 못 한 문제를 만드는 원인이다.

**학습 파이프라인**

LLM은 대체로 다음 세 단계를 거쳐 만들어진다. (세부 기법은 모델마다 다르지만, 큰 골격은 비슷하다.)

```
1단계: Pretraining    → 방대한 텍스트로 next-token prediction 학습
           ↓
2단계: Instruction Fine-tuning → 질문-답변 패턴 학습
           ↓
3단계: 인간 선호 정렬 (RLHF 등) → 인간이 좋아하는 답변 방향으로 조정
```

여기서 주목할 것은 3단계다. 인간 평가자들은 "자신의 의견에 동의하는 답변"에 높은 점수를 주는 경향이 있고, 정렬 과정은 그 선호를 모델에 반영한다. **동조(Sycophancy) 경향은 이 인간 피드백 기반 학습에서 비롯됐다.** 최신 모델들은 이를 줄이는 훈련(Constitutional AI, DPO 등)을 받고 있어 과거보다 완화됐지만, 사용자가 강하게 주장하거나 결정이 복잡한 상황에서는 여전히 작동한다 — 완전히 사라진 특성이 아니다. (이 성향은 정렬 단계에서 두드러지지만, 그 단계만의 산물은 아니다.)

**확률 분포와 샘플링**

항상 가장 확률 높은 토큰을 고르면(greedy decoding) 답변이 반복되고 뻔해진다. 실제로는 **확률 분포에서 샘플링**한다. temperature, top-p가 이 예측 불가능성의 수준을 조절한다. 같은 입력에 매번 다른 출력이 나오는 이유다.

**빈칸을 채우는 것이 기본 동작**

LLM은 주어진 문맥 다음에 어떤 텍스트가 그럴듯한지를 학습했다. 명세되지 않은 공간이 있으면 논리적 추론이 아니라 **학습된 패턴으로 채운다**. "합리적인 판단"이 아니라 "통계적으로 자주 등장하는 패턴"이다.

> LLM은 인간 선호 정렬을 거친 확률 분포에서 샘플링하고, 그 분포 안에는 인간 평가자의 동조 편향이 녹아 있으며, 명세되지 않은 공간은 논리가 아닌 학습된 패턴으로 채워진다.

#### 1.3 세 가지 불편한 사실

LLM의 본질에서 피할 수 없이 나오는 특성들이다. 이것들을 "나쁜 특성"으로 보면 싸우게 된다. "구조적 특성"으로 보면 대응 방법이 보인다.

**자신감 ≠ 정확도**  
확신하는 어조는 정확도를 보장하지 않는다. 틀린 답을 아주 자신 있게 말할 수 있다. 이것은 훈련 방식의 결과이지 고칠 수 있는 버그가 아니다. → 검증을 설계하지 않으면 오류를 발견할 수 없다.

**매 세션은 새로 시작이다**  
인간 동료에게는 관계가 쌓인다. 그 사람이 무엇을 잘하는지, 어떤 실수를 자주 하는지 알게 된다. AI agent는 다르다. 매 세션이 새 시작이다. 어제 잘 됐다고 오늘도 잘 된다는 보장이 없다. → "이 agent를 신뢰한다"는 말이 위험한 이유다.

**동조는 결함이 아니라 설계다**  
AI agent는 동의하는 방향으로 훈련되어 있다. "이 계획 어때?"라고 물으면 "좋아 보이는데 이런 점 고려하면 좋겠어요" 수준의 답이 나온다. 진짜 논리 검증이 안 되고 있는 것이다. → 동조를 차단하는 것이 아니라, 동조가 유용한 방향으로 흐르도록 질문 구조를 바꾸는 것이 핵심이다.

## 2부: 어떻게 생각해야 하는가 — 협업 철학

> LLM의 본질을 이해했다면, 이제 어떤 사고방식으로 협업해야 하는지를 정립한다.  
> 10개의 관점은 추상적 원칙이 아니라 LLM의 구조적 특성에서 나오는 필연적 귀결이다.

---

### 2장. 10개의 관점: 핵심 사고방식 전환

먼저 전체를 한눈에 보고, 각 관점의 상세 설명을 읽는다.

| 관점 | 한 줄 요약 |
|------|-----------|
| 1 | AI는 자신감 있게 틀린다 — 검증 구조를 설계하지 않으면 오류를 발견할 수 없다 |
| 2 | 인간의 역할은 결과를 만드는 것이 아니라 환경을 설계하는 것이다 |
| 3 | 신뢰는 에이전트에 붙이는 것이 아니라 검증 구조에 붙이는 것이다 |
| 4 | 동조는 차단 대상이 아니라, 유용한 방향으로 흐르도록 질문 구조를 바꾸는 것이다 |
| 5 | 위험은 빈칸 자체가 아니라, 어떤 빈칸이 채워지는지 모르는 것이다 |
| 6 | 위임 전에 실패 비용(심각도·발견 가능성·가역성)을 먼저 추정한다 |
| 7 | AI 역량은 실패 빈도를 줄이고, 위임 구조는 실패 비용을 결정한다 |
| 8 | 기술적 결정은 위임 가능하지만, 가치 결정의 위임은 책임의 방기다 |
| 9 | 프로젝트 기억은 저절로 전달되지 않는다 — 설계하고 관리해야 한다 |
| 10 | 완전한 문서보다 정확하고 선별적으로 전달되는 문서가 낫다 |

#### 관점 1–3: 문제의 구조 이해하기

**관점 1: AI는 보이지 않게 틀린다**

AI agent는 내 의도를 잘못 이해해도 자신감 있게, 형식도 맞게, 그럴듯하게 실행한다. 컴파일러는 틀리면 즉시 에러를 낸다. 인간 동료는 어색함이 생기고 되묻는다. AI agent는 다르다. 틀렸다는 신호 없이 틀린 결과물을 만든다. 코드가 돌아가기 때문에 나중에서야 발견된다. 이것이 "실행 가능한 오해"다.

이것은 1.2에서 배운 LLM의 본질에서 직접 나온다. LLM은 "학습된 패턴으로 빈칸을 채우고" "자신감 있게 출력"한다. 둘을 합치면 — 틀렸다는 신호 없이 틀린 결과물이 나온다.

> 도출되는 원칙: 검증을 설계하지 않으면 오류를 발견할 수 없다.

**관점 2: 인간은 "결과 생산자"에서 "환경 설계자"로**

```
전통: 인간이 생각한다 → 인간이 코드를 짠다 → 코드가 실행된다
협업: 인간이 생각한다 → 인간이 환경을 설계한다 → 에이전트가 실행한다
```

"환경"은 파일 구조를 말하는 게 아니다. 네 가지 구성 요소로 이루어진다.
- **컨텍스트**: 에이전트에게 어떤 정보를 제공하는가
- **위임 경계**: 어떤 결정은 에이전트에게, 어떤 결정은 내가 하는가
- **불확실성 프로토콜**: 에이전트가 모를 때 어떻게 행동해야 하는가
- **검증 기준**: 어떤 기준으로 결과를 판단하는가

좋은 환경을 설계한 사람은 에이전트가 바뀌거나 모델이 업그레이드되어도 안정적인 결과를 얻는다. 환경 없이 프롬프트만 잘 쓰는 사람은 매번 새로 시작하는 느낌을 받는다.

환경 설계자 역할의 핵심은 "파일을 잘 구성하는 것"이 아니다.

> **에이전트가 인식하지 못하는 가정을 인간이 먼저 인식하고, 그것을 드러낼 수 있도록 환경을 설계하는 것.**

이 역량은 체계가 대신해줄 수 없다. 체계는 인식된 가정을 구조화할 뿐, 인식 자체는 도메인 역량에서 나온다. 이 한계는 7부(13장)에서 다시 다룬다.

**관점 3: 신뢰는 관계에서 오지 않고 구조에서 온다**

인간 동료에 대한 신뢰는 관계에서 쌓인다. 그 사람이 어떤 사람인지, 어떤 상황에서 어떻게 행동하는지 알게 되면서 신뢰가 생긴다. AI agent는 다르다. 매 세션 새로 시작하고, 자신감과 정확도가 분리되어 있다. 어제 잘 됐다고 오늘도 잘 된다는 보장이 없다.

그래서 "이 에이전트를 신뢰한다"는 말은 위험하다. 에이전트에 신뢰를 붙이면, 검증을 생략하는 근거가 된다.

> "이 에이전트를 신뢰한다" → "이 유형의 작업, 이 검증 구조 안에서 이 결과를 신뢰한다"

신뢰는 에이전트에 붙이는 것이 아니라 작업 유형과 검증 구조의 조합에 붙이는 것이다. 같은 에이전트라도 단순한 코드 변환은 신뢰하고, 비즈니스 규칙 판단은 신뢰하지 않는 것이 올바른 태도다.

#### 관점 4–5: 빈칸과 위임의 이해

**관점 4: 동조는 차단이 아니라 활용 대상이다**

AI agent는 동의하는 방향으로 훈련되어 있다. 이것은 1.2에서 설명한 RLHF의 결과다. 인간 평가자들이 "자신의 의견에 동의하는 답변"에 높은 점수를 줬고, 모델은 그 패턴을 학습했다. 이걸 "나쁜 특성"으로 보면 싸우게 된다. "설계된 특성"으로 보면 활용하게 된다.

에이전트는 내가 옳다고 믿는 방향으로 나를 도와줄 준비가 되어 있다. 핵심은 그 동조가 "강화"로 흐르느냐 "검증"으로 흐르느냐다. 질문 구조를 바꾸면 동조의 방향이 바뀐다.

| 질문 방식 | 결과 |
|-----------|------|
| "이 계획 어때?" | 동조 → 강화만 됨 |
| "이 계획이 실패하는 시나리오는?" | 동조 → 검증 방향으로 흐름 |

동조를 차단하려 하지 말고, 동조가 유용한 방향으로 흐르도록 질문 구조를 바꾸는 것이 핵심이다.

단, 질문 구조를 바꾸면 동조의 **방향**이 바뀌는 것이지 동조 자체가 사라지는 것은 아니다. "실패 시나리오를 찾아줘"에도 그럴듯하지만 실제로 덜 위험한 시나리오를 생성할 수 있다. 이것이 같은 스레드의 질문 구조만으로는 부족하고, 컨텍스트가 격리된 독립 비평(8.1 참조)이 필요한 이유다.

**관점 5: 위험은 빈칸이 아니라 비가시적 위임이다**

빈칸을 채워주길 바라서 LLM을 쓰는 것이다. 빈칸 자체가 문제가 아니다. 문제는 빈칸의 종류를 구분하지 못하는 것이다.

| 종류 | 결과 |
|------|------|
| **빈칸 A — 의도적 위임** | "이 부분은 에이전트가 알아서 해줬으면 해서 비워뒀다" → 생산적 |
| **빈칸 B — 비가시적 위임** | "여기에 결정이 필요한 공간이 있었는지 몰랐다" → 위험 |

"조용한 결정(Silent Decision)"이라는 현상이 있다. 인간이라면 "이건 잘 모르겠는데요"라고 멈추는 지점에서, 에이전트는 가장 그럴듯한 선택을 하고 말하지 않은 채 진행한다. 예를 들어 이미지 업로드 기능을 구현할 때, 기존 이미지를 삭제할지 여부나 URL을 public으로 할지 presigned로 할지를 에이전트가 조용히 결정해버린다. 코드가 돌아가기 때문에 나중에서야 발견된다.

> 위험은 빈칸이 있는 것이 아니라, **어떤 빈칸이 채워지고 있는지 모르는 것**이다.
>
> 실천적 질문: "나는 지금 어떤 빈칸을 의도적으로 위임하고 있고, 어떤 빈칸이 나도 모르게 채워지고 있는가?"

#### 관점 6–8: 구조 설계의 원칙

**관점 6: 위임 범위보다 실패 비용이 먼저다**

"얼마나 위임할지"보다 먼저 물어야 할 것이 있다. 이 작업에서 실패했을 때 얼마나 나쁜가? 실패 비용은 세 축으로 추정한다: **심각도**(잘못됐을 때 얼마나 나쁜가), **발견 가능성**(잘못됐다는 걸 알 수 있는가), **가역성**(되돌릴 수 있는가).

불편한 역설이 있다. AI로 효율이 가장 큰 작업 — 복잡한 로직 구현, 아키텍처 결정 — 이 실패 비용도 가장 높다. 공짜 점심은 없다.

실패 비용이 높다고 판단했을 때 두 가지 선택이 있다.
- **선택 1**: 위임 범위를 줄인다 (AI를 덜 쓴다) — 대부분이 여기로 간다
- **선택 2**: 위임 구조를 바꾼다 — AI는 쓰되 실패가 보이고 되돌릴 수 있도록 구조를 바꾼다

선택 2의 세 가지 방법:
- **결정 가시화**: "구현 전에 결정해야 할 것들을 먼저 목록으로 보여줘" → 빈칸 B를 빈칸 A로 변환
- **결정 경계에서 분할**: 임의 크기가 아닌 결정이 일어나는 지점에서 작업을 쪼갬 → 심각도↓, 발견 가능성↑, 가역성↑
- **실패 범위 제한**: 피처 플래그, 스테이징, 점진적 배포 → 가역성과 심각도를 구조적으로 조정

선택 2가 환경 설계자의 실질적 역할이다. 실패 비용 추정 프레임 → [부록 D](#d-분석-도구-모음)

**관점 7: 위임 구조의 품질이 실패 비용을 결정한다**

> AI 역량은 실패의 빈도를 줄인다.  
> **위임 구조의 품질은 실패의 비용을 결정한다.**

모델이 강력해질수록 실패 빈도는 줄어든다. 그러나 실패가 "보이지 않는다"는 문제는 모델 역량으로 해결되지 않는다. 뛰어난 AI도 나쁜 구조 안에서 틀리면 — 코드가 돌아가기 때문에 뒤늦게 발견되고, 되돌리기 어렵고, 범위가 퍼진다.

구조의 역할은 실패를 없애는 것이 아니라, 실패가 발생했을 때 그 피해를 통제 가능하게 만드는 것이다.

좋은 위임 구조의 세 조건 — AI가 틀렸을 때:
1. **적시에 발견된다** — "코드가 돌아가면 넘어가는" 수준의 검토로는 부족하다
2. **되돌릴 수 있다** — Git이 단순한 실수 복구 도구가 아니라 가역성을 구조적으로 높이는 장치인 이유
3. **범위가 제한된다** — 한 결정이 전체에 영향을 주지 않도록 경계를 명확히 한다

이 세 조건이 충족되지 않는 작업은 위임 전에 구조를 먼저 바꾸거나, 위임하지 않아야 한다. "프롬프트 엔지니어링"보다 "협업 구조 설계"가 더 근본적인 역량인 이유가 여기에 있다.

**관점 8: 자율성과 책임 — 가치 결정은 위임할 수 없다**

자율성은 단일한 다이얼이 아니다. 결정 자율성, 실행 자율성, 범위 자율성, 시간적 자율성이 각각 다른 기준으로 조정된다. 그런데 실패 비용으로 설명되지 않는 영역이 있다. AI가 완벽히 신뢰할 수 있어도 인간이 결정해야 하는 것이 있다.

결정에는 두 종류가 있다.
- **기술적 결정** ("어떻게 할 것인가") — AI 역량이 충분하면 위임 가능
- **가치 결정** ("무엇이 더 중요한가") — AI 역량과 무관하게 인간이 결정

가치 결정을 AI에게 위임하는 것은 리스크 문제가 아니라 **책임의 방기**다. AI는 결과에 책임을 지지 않는다.

주의할 것은 숨은 가치 결정이다. 기술적으로 보이는 결정(소프트/하드 삭제, 캐시 여부, 에러 메시지 상세도) 안에 가치 판단이 숨어 있는 경우가 많다. 소프트 삭제냐 하드 삭제냐는 데이터 복구 가능성에 대한 비즈니스 정책 결정이다. 이것이 AI가 조용히 결정하는 가장 위험한 패턴이다.

자율성 수준과 적용 기준 → [부록 D](#d-분석-도구-모음)

#### 관점 9–10: 프로젝트 연속성

**관점 9: 프로젝트 기억은 설계하고 관리해야 한다**

단일 작업에서 잘 작동하는 것과 프로젝트 전체가 일관성을 유지하는 것은 다른 문제다. 각 세션의 에이전트는 현재 주어진 컨텍스트만으로 작업한다. "이전에 이렇게 결정했어"라는 것을 모른다. 결과적으로 각 작업은 로컬 최적이지만 전체가 서로 충돌하는 현상이 생긴다.

원인은 단순하다: **인간 머릿속에 있는 것은 AI에게 존재하지 않는 것과 같다.** 팀원에게는 자연스럽게 전파되는 암묵지가 AI에게는 전달되지 않는다.

프로젝트 기억에는 다섯 가지가 담겨야 한다: 불변 규칙(절대 건드리면 안 되는 것), 아키텍처 패턴(반복 문제를 푸는 방식), 결정 이력(무엇을 왜 결정했는가), 인터페이스 계약(모듈 간 약속), 안티패턴(이미 실패한 것).

관리하면서 부딪히는 세 가지 도전이 있다. 첫째, 암묵지 명시화 — 처음에 무엇을 기록해야 하는지 모른다. 둘째, 문서 부패 — 코드는 바뀌었는데 문서가 따라오지 못하면 낡은 문서가 된다. 낡은 문서는 없는 것보다 위험하다. 셋째, 컨텍스트 한계 — 모든 것을 다 넣을 수 없으므로 무엇을 버릴지도 설계 대상이다.

> 프로젝트의 일관성은 명시적으로 관리된 **프로젝트 기억** 없이는 유지될 수 없다.  
> 프로젝트 기억의 관리는 AI가 아닌 인간의 핵심 책임이다.

구성 요소 상세 → [부록 D](#d-분석-도구-모음)

**관점 10: 완전한 문서보다 정확하고 선별적인 문서가 낫다**

관점 9에서 프로젝트 기억이 필요하다는 것을 확립했다. 그렇다면 얼마나 완전해야 하는가? 완전한 문서를 유지하려는 시도는 대부분 실패한다. 문서가 코드와 함께 진화하기 어렵고, 결국 낡은 문서가 쌓인다.

프로젝트 기억의 가치는 얼마나 완전한가가 아니라, **얼마나 현재 유효하고 작업에 맞게 전달되는가**로 결정된다. 완전하지만 낡은 문서보다 불완전하지만 정확하고 선별적으로 주입되는 문서가 낫다.

AI를 위한 문서를 쓸 때 세 가지 원칙이 있다.
- **고수준 원칙보다 구체적 규칙**: "코드 품질을 높여라" 대신 "에러는 서비스 레이어에서만 처리한다" — AI가 따랐는지 확인 가능해야 의미가 있다
- **부정 제약이 핵심**: "해도 되는 것"만 알려주면 나머지 전부를 할 수 있다고 가정한다. "하면 안 되는 것"을 명시해야 경계가 생긴다
- **결정 이력에는 "Why"가 필요**: 무엇을 결정했는지만 쓰면 엣지 케이스에서 판단을 못 한다. 왜 그 결정을 했는지가 있어야 유사한 상황에서 올바르게 적용할 수 있다

### 3장. 상위 통합 원칙

10개의 관점을 조망하면 하나의 축이 보인다: "옳고 그름"이 아니라 "구조".

아래는 번호 순이 아니라 **논리적 인과 흐름** 순으로 재배치한 것이다.  
논의된 순서(번호)와 이해하는 순서(흐름)는 다르다.

```
관점 1  AI는 보이지 않게 틀린다          ← 문제 정의
관점 4  동조 경향은 설계다               ← 문제의 구조적 원인
관점 5  위험은 비가시적 위임이다         ← 문제가 발생하는 지점
관점 2  인간은 환경 설계자다             ← 해법의 방향
관점 3  신뢰는 구조에서 나온다           ← 해법의 근거
관점 6  실패 비용을 먼저 추정하라        ← 해법의 시작점
관점 7  구조가 실패 비용을 결정한다      ← 해법의 본질
관점 8  가치 결정은 위임할 수 없다       ← 해법의 경계
관점 9  프로젝트 기억은 설계해야 한다    ← 해법을 프로젝트 전체로 확장
관점 10 정확하고 선별적인 문서가 낫다    ← 해법의 지속 가능성
```

> **LLM의 한계는 주어진 것이다. 구조는 설계하는 것이다.**
>
> *전제 조건: 인간이 최소한의 검증 능력을 갖춘 영역에서*  
> *적용 방식: 실패 비용에 비례하여 — 비용이 낮으면 구조를 최소화, 높으면 강화*

세 가지 함의:

**함의 1: "AI를 더 잘 쓰는 것"이 목표가 아니다**  
프롬프트 최적화, 모델 선택 — 이것들은 실패 빈도를 줄이는 접근이다. 구조 설계가 더 근본적이다.

**함의 2: 모델 버전에 독립적인 원칙이다**  
더 뛰어난 AI가 나와도 구조 없이는 드문 실패도 보이지 않고 되돌릴 수 없다.

**함의 3: AI 협업 역량의 실체**  
AI를 잘 쓰는 사람과 못 쓰는 사람의 차이는 프롬프트 실력이 아니다. **위임 구조를 설계하는 능력이다.** 이건 AI가 아닌 인간이 개발해야 하는 역량이다.

그런데 "구조 설계 역량 자체가 없으면 어떻게 하는가?" → 이것이 체계 필요성이 도출되는 지점이다.

## 3부: 체계란 무엇인가

> 철학을 이해했다면, 이제 철학을 구조로 인코딩한 체계가 무엇인지 설명할 수 있다.

### 4장. 체계의 정의와 목적

#### 4.1 철학에서 구조로

2부에서 다음이 확립됐다: AI 협업의 핵심 역량은 협업 구조를 설계하는 능력이다.

그런데 이 능력은 어디서 오는가? 경험에서 온다. 어떤 구조가 실패하고, 어떤 구조가 작동하는지를 직접 겪어야 안다.

문제는 모든 프로젝트에서 처음부터 다시 배울 수 없다는 것이다. 그리고 팀이 있다면 각자 다른 구조를 쓰고 있을 수 있다.

해법: **검증된 구조 패턴을 재사용 가능하게 만드는 것.**

> 체계 = 협업 철학을 파일 시스템으로 인코딩한, 재사용 가능한 협업 구조 패키지.

체계는 확률적 자연어 프로그래밍의 설계도다. LLM의 해석 공간(확률 분포)에서 우리가 원하는 방향으로 수렴하도록 엔트로피를 줄이는 것이 체계의 역할이다. 완전한 결정론은 불가능하다. 체계의 목표는 확률적 실행을 충분히 예측 가능한 범위 안으로 가져오는 것이다.

체계가 모호성을 다루는 방식을 정밀하게 표현하면:

> 자연어의 모든 모호성을 제거하는 것이 아니라, **의도한 모호성(빈칸 A)은 열어두고, 의도하지 않은 모호성(빈칸 B)은 닫는 것.**

형식 언어(Python, Java)는 모든 모호성을 닫는다 — 컴파일러가 허용하지 않는다. 그래서 LLM의 창의적 gap-filling이 불가능하다. 체계는 선택적으로 닫는다. LLM을 쓰는 이유(관점 5의 빈칸 A)는 살리면서, 위험한 비가시적 위임(빈칸 B)만 차단하는 것이 목표다.

체계 구성 요소는 익숙한 프로그래밍 개념에 빗대 보면 역할이 좀 더 직관적으로 잡힌다. 어디까지나 이해를 돕는 비유이지, 일대일로 들어맞는 대응은 아니다.

| 체계 | 비슷한 프로그래밍 개념 | 닮은 점 |
|--------|----------|--------|
| inject 파일 | 함수 정의 | 입력을 받아 정해진 동작을 수행하는 재사용 단위 |
| 페르소나 | 클래스 / 인터페이스 | 역할과 행동 규약을 묶어 정의 |
| agent_rules.md | 전역 상수 / 제약 | 모든 세션에 공통으로 적용되는 불변 규칙 |
| workspace/memory/ | 프로그램 상태 | 작업 간에 유지·갱신되는 데이터 |
| plan.md "에이전트 보고" | 컴파일러 경고 | 진행 전에 위험 신호를 드러냄 |
| layer1_critique / layer2_checkpoint | 테스트 (설계·통합 단계) | 결과물을 독립적으로 검증 |

#### 4.2 체계가 담당하는 것, 사용자가 담당하는 것

역할 경계를 명확히 하는 것이 중요하다.

**체계가 담당하는 것:**
- 알려진 위험 관리
- 구조적 실패 예방
- 컨텍스트 연속성 (workspace/ 파일)
- 에이전트 행동 규칙 (inject)

**사용자가 담당하는 것:**
- 검토 결과 판단 및 다음 행동 결정
- 가치 결정 ("무엇이 더 중요한가")
- 비즈니스 맥락 판단 (외부 제약, 이해관계자 의도)

> "사용자"는 역할이다. 실제 사람일 수도 있고, 오케스트레이터 에이전트일 수도 있다.  
> 기본값: 사용자가 판단. 사용자가 명시적으로 위임한 경우: 에이전트가 판단.

#### 4.3 메타 원칙

> **신뢰할 수 없는 것에 의존하지 않고, 통제할 수 있는 것만으로 구조를 만든다.**

이것이 모든 설계 결정의 공통 근거다.

- CLI/IDE 내장 기억은 블랙박스다 → 의존하지 않는다. 체계 파일에 "CLI 메모리를 읽거나 수정하라"는 지시를 넣지 않는다.
- 관계적 신뢰는 세션 간에 유지되지 않는다 → 구조로 관리한다.
- 모델이 업데이트되면 같은 파일이 다르게 해석될 수 있다 → 특정 모델의 해석 방식에 의존하지 않는다.

체계의 기준선: **파일을 읽고 쓸 수 있는 tool-using 에이전트** (Claude Code, Cursor 등). Raw LLM API는 파일 접근이 불가하므로 지원 범위 밖이다.

## 4부: 어떻게 일하는가 — 작업 워크플로우

> 체계가 무엇인지 이해했다면, 이제 실제로 어떻게 작동하는지 배운다.  
> 이 파트에서는 작업 흐름을 설명하면서, 그 흐름에서 어떤 파일이 어떤 역할을 하는지 함께 소개한다.

---

### 5장. 작업을 실행하는 파일들

#### 5.1 전체 구조 한눈에 보기

체계는 세 종류의 파일로 이루어진다. 각각 다른 시점에, 다른 주체가 읽는다.

| 디렉토리 | 한 줄 역할 | 읽는 주체 | 시점 |
|---------|-----------|---------|-----|
| `.mpa-workspace/` | 에이전트에게 어떻게 일하라고 알려주는 파일 | 에이전트 | 매 세션 |
| `workspace/` | 이 프로젝트가 무엇인지 기억하는 파일 | 에이전트 | 필요 시 |
| `agent-specs/` | CLI/IDE와 체계를 연결하는 파일 (agent별) | install.py | 설치 시 1회 |

이 세 가지가 분리된 이유는 변경 주기가 다르기 때문이다.

| | `.mpa-workspace/` | `workspace/` |
|--|-------------------|--------------|
| 내용 | 어떻게 일할 것인가 (방법, 절차) | 이 프로젝트가 무엇인가 (데이터) |
| 바뀌는 시점 | 체계를 개선할 때 | 프로젝트가 진행될 때 |
| 재사용 범위 | 모든 프로젝트 공통 | 이 프로젝트만 |

섞어두면 체계를 개선할 때 프로젝트 데이터가 영향을 받거나, 다음 프로젝트에 이번 프로젝트 데이터가 섞여 들어간다.

#### 5.2 .mpa-workspace/ 안에 무엇이 있는가

`.mpa-workspace/`는 에이전트의 행동 규칙과 작업 도구가 들어있다. Layer 0/1/2 작업을 실행할 때 에이전트가 여기서 필요한 파일을 읽는다.

```
.mpa-workspace/
├── core/
│   ├── principles.md        ← 10개 관점의 실전 요약
│   ├── agent_rules.md       ← 에이전트 행동 규칙 (항상 로딩)
│   └── session_protocol.md  ← 세션 유형 결정 트리 + 재개 프로세스
├── personas/                ← 에이전트 역할 정의 (8장에서 설명)
├── skills/
│   ├── analysis/            ← 분석 스킬 (조용한 결정 추출 등)
│   └── tech/                ← 기술 스킬 (spring, react 등)
├── workflows/               ← 세션 시퀀스 정의
├── inject/                  ← 세션 레시피 파일 (7장에서 설명)
│   ├── layer0_init.md
│   ├── layer0_update.md
│   ├── layer1_design.md
│   ├── layer1_critique.md   ← 계획 독립 비평 (반드시 새 스레드)
│   ├── layer1_implement.md
│   ├── layer1_review.md
│   └── layer2_checkpoint.md
├── knowledge/               ← 크로스 프로젝트 도메인 지식
├── templates/               ← 파일 생성 시 복사해 쓰는 템플릿
└── upgrade-candidates/      ← 체계 개선 후보 임시 저장 (12장에서 설명)
```

각 파일이 언제 쓰이는지는 이후 각 장에서 작업 흐름과 함께 설명한다.

#### 5.2-1 workspace/ 안에 무엇이 있는가

`workspace/`는 이 프로젝트 고유의 데이터가 쌓이는 공간이다. 설치 직후에는 비어있고, 프로젝트가 진행되면서 채워진다.

```
workspace/
├── memory/                      ← 프로젝트를 이해하는 데 필요한 지식
│   ├── shared/                  ← 모든 작업에 항상 주입되는 핵심
│   │   ├── project_identity.md  ← 이 프로젝트가 무엇인가 (1-2 문장)
│   │   ├── architecture.md      ← 아키텍처 결정과 이유 (기술 결정)
│   │   ├── contracts.md         ← 모듈 간 인터페이스 계약
│   │   └── direction.md         ← 제품/UX 방향 원칙 (결과 경험 후 발견·축적)
│   ├── domains/                 ← 도메인별 규칙 (프로젝트 구조에 맞게 구성)
│   │   └── [도메인명]/
│   │       ├── rules.md         ← 이 도메인의 규칙과 제약
│   │       └── registry.md      ← 컴포넌트/패턴 레지스트리
│   └── roles/                   ← 페르소나별 프로젝트 누적 학습 (10.2 참조)
│       └── [페르소나명].md
│
├── tasks/                       ← 작업 단위 관리
│   ├── active/                  ← 현재 진행 중인 작업
│   │   └── yyyymmdd_[태스크명]/
│   │       ├── plan.md          ← 설계·진행 상태·에이전트 보고 (Layer 1의 핵심)
│   │       └── changelog.md     ← 구현 중 작성, 세션 재개·검토 시 사용 (10.1 참조)
│   └── done/                    ← 완료된 작업 아카이브
│       └── yyyymmdd_[태스크명]/ ← 승인 상태는 plan.md YAML 프론트매터로 관리 (구버전의 .approved 파일 방식은 더 이상 사용하지 않음)
│
└── docs/                        ← 구현된 기능의 결과 문서
    └── INDEX.md
```

`memory/shared/`가 가장 중요하다. 모든 Layer 1 세션에 항상 주입되는 핵심 컨텍스트다. 여기에 없는 것은 에이전트에게 존재하지 않는 것과 같다.

**shared/ 파일은 로그가 아니라 현재 상태의 스냅샷이다.** 결정이 바뀌면 기존 항목을 교체하고, 폐기된 규칙은 삭제한다. 과거 결정을 누적 보존하면 모순이 생긴다. `architecture.md`는 기술 결정을, `direction.md`는 제품/UX 방향 결정을 각각 현재 상태로 유지한다.

`tasks/active/yyyymmdd_[태스크명]/plan.md`는 현재 진행 중인 작업의 상태 파일이다. 에이전트가 계획서를 작성하고, 사용자가 검토·승인하고, 에이전트가 구현 결과를 기록하는 공간이다. 세션이 끊겨도 이 파일을 읽으면 어디까지 왔는지 알 수 있다.

`changelog.md`는 구현 세션에서 에이전트가 작성하는 진행 기록이다. 수정한 파일 경로, 변경된 메서드·함수·클래스, 변경 내역 요약을 담는다. 세션이 중단됐을 때 재개 지점을 파악하고, 작업 결과 검토 세션에서 검토 범위를 확인하는 데 사용된다.

#### 5.3 CLI/IDE와 체계는 어떻게 연결되는가

체계 파일은 텍스트다. Claude Code든 Cursor든 어디서든 읽을 수 있어야 한다. 그래서 의존성 방향을 한 방향으로만 유지한다.

```
CLI/IDE  →  체계 파일을 읽는다  ✅
체계     →  CLI/IDE 내부를 읽는다  ✗
```

CLI/IDE별 연결은 체계 저장소의 `agent-specs/`에서 처리한다. agent마다 진입점 파일명과 주입 방식이 다르기 때문에, agent별로 디렉토리를 나눠 둔다.

```
agent-specs/
  claude/      → CLAUDE.md   (Claude Code)
  codex/       → AGENTS.md   (Codex 등)
  antigravity/ → GEMINI.md   (Antigravity)
  openagent/   → spec.md 질의로 결정
    각 디렉토리: inject/ (진입점 파일), files/ (프로젝트에 복사할 파일), spec.md
```

`install.py`는 선택된 agent의 `agent-specs/{agent}/inject/` 진입점 내용을 프로젝트 루트의 해당 설정 파일(`CLAUDE.md` / `AGENTS.md` / `GEMINI.md`)에 덧붙인다. 이 진입점이 `@.mpa-workspace/core/agent_rules.md`를 참조하므로 에이전트 규칙이 매 세션 자동 로딩된다. (저장소 루트의 `agent-configs/`는 구버전 호환용 fallback이며, 신규 설치에서는 `agent-specs/`가 우선한다.)

`session_protocol.md`는 체계를 처음 접하는 사람이 "지금 어떤 inject 파일을 써야 하는가"를 빠르게 파악할 수 있는 진입점 가이드다. 에이전트가 아닌 사용자가 읽는 파일이다.

Claude Code의 `@` 참조 문법 덕분에 규칙을 수동으로 붙여넣지 않아도 된다. (agent에 따라 자동 로딩 방식은 다를 수 있다.)

### 6장. 3 Layer 작업 구조

#### 6.1 왜 Layer로 나누는가

개별 작업을 잘 해도 전체 프로젝트가 무너지는 현상이 있다.

```
작업 A  →  A 기준으로 최적 구현
작업 B  →  B 기준으로 최적 구현
작업 C  →  C 기준으로 최적 구현

전체   →  각자 최적이지만 서로 충돌, 규칙 위반
```

근본 원인: 아키텍처 규칙이 암묵적으로만 존재하기 때문. **에이전트에게 전달되지 않은 규칙은 존재하지 않는 것과 같다.**

해결 구조:
- **Layer 0**: 암묵적 규칙을 명시적으로 꺼내 문서화 (1회)
- **Layer 1**: 매 작업마다 규칙을 컨텍스트로 주입 (반복)
- **Layer 2**: 주기적으로 전체 정합성 확인 (N개 작업마다)

```
┌────────────────────────────────────────────┐
│  Layer 0: 프로젝트 초기화 (1회)             │
│  아키텍처 규칙 문서화 + 작업 분해           │
└────────────────────────────────────────────┘
        ↓ 아키텍처 규칙을 매 작업에 주입
┌────────────────────────────────────────────┐
│  Layer 1: 개별 작업 반복                   │
│  설계 → 검토 → 승인 → 구현 → 결과 확인    │
└────────────────────────────────────────────┘
        ↓ N개 작업마다
┌────────────────────────────────────────────┐
│  Layer 2: 통합 체크포인트                  │
│  전체 정합성 확인 + 규칙 위반 여부 검토    │
└────────────────────────────────────────────┘
```

#### 6.2 Layer 0 — 프로젝트 초기화 (1회)

Layer 0는 이후 모든 작업의 기준이 되는 문서와 구조를 만드는 단계다.

**신규 프로젝트와 기존 프로젝트는 성격이 다르다:**

| 구분 | 신규 프로젝트 | 기존 프로젝트 |
|------|-------------|---------------|
| 성격 | 설계 (Design) | 발굴 (Discovery) |
| 규칙 출처 | 새로 만든다 | 코드에서 추출한다 |
| 첫 번째 할 일 | 확실한 사실 수집 | 현황 분석 |

**신규 프로젝트 흐름 (3단계):**

1. **확실한 사실 확인** — 목적·기술 스택·제약·도메인 구성을 사용자에게 확인한다. 추정은 기록하지 않는다.

2. **memory 초안 작성** — `workspace/memory/shared/`와 `domains/` 파일을 작성한다. 아직 경험하지 않은 패턴·규칙은 빈칸으로 남긴다. 추정으로 채우면 낡은 문서가 된다.

3. **작업 분해 + 의존성 정의** — 공유 인터페이스는 개별 작업 전에 먼저 확정한다. 각자 만들면 충돌이 발생한다. 결과를 `workspace/tasks/INDEX.md` 초안에 기록한다.

**기존 프로젝트 흐름 (5단계):**

기존 프로젝트에서 가장 흔한 실수: 이상적인 규칙을 먼저 문서화하고 코드에 적용하려 하는 것. 현재 코드에 없는 규칙을 강제하면 충돌이 발생한다.

```
1단계: 현황 분석 — 에이전트가 코드베이스를 탐색해 구조 파악
2단계: 암묵적 규칙 추출 — 일관된 패턴 / 부분적 패턴 / 위반·불일치 분류
3단계: 현재 규칙 문서화 + 목표 규칙 정의 — 코드의 현재 상태에서 출발
4단계: memory 초안 작성 — 현재 규칙과 목표 규칙을 분리해서 기록
5단계: 작업 분해 — 신규 기능 + 불일치 해소 작업을 분리해서 INDEX.md에 기록
```

불일치 해소 작업과 신규 기능 작업을 섞지 않는다 — 목적과 검증 기준이 다르다.

**Layer 0 완료 기준:**

> "이 컨텍스트 없이 시작하면 어떤 잘못된 결정이 내려지는가?"에 답할 수 있을 때.

체크리스트:
- `workspace/memory/shared/` 파일 작성됨
- 절대 금지 항목이 추상 원칙이 아닌 조작 가능한 규칙으로 기록됨
- `workspace/tasks/INDEX.md` 초안 작성됨 (작업 목록 + 의존성)
- 기존 프로젝트의 경우: 현재 규칙과 목표 규칙이 분리됨

완벽하지 않아도 된다. 처음부터 모든 규칙이 갖춰질 수 없다. Layer 2 체크포인트를 거치며 계속 업데이트된다. 프로젝트가 진행될수록 규칙이 강화되는 구조다.

**Layer 0에서 넣어야 할 것과 넣지 말아야 할 것:**

Layer 0 기준은 타이밍이 아니라 **검증 여부**다.

| 지식 출처 | 검증 여부 | Layer 0에 넣어야 하는가 |
|---------|---------|-------------------|
| 사용자의 도메인 전문 지식 | 검증됨 | ✅ 넣어야 함 |
| 다른 프로젝트에서 검증된 패턴 | 검증됨 | ✅ 넣어야 함 |
| 이미 명확한 제약 (변경 불가) | 검증됨 | ✅ 넣어야 함 |
| "아마 이런 패턴이 필요할 것" | 추정 | ❌ 넣지 말아야 함 |

경험 전에 채운 추정이 틀린 채 남아있으면 낡은 문서가 된다. 낡은 문서는 없는 것보다 위험하다.

#### 6.3 Layer 1 — 개별 작업 반복

**Task 기준: 언제 Task를 만드는가**

이 워크플로우는 Task를 만들어야 할 규모의 작업에만 적용한다.

| 적용 O | 적용 X |
|-------|-------|
| Task가 필요한 규모의 작업 | 단순 질문 |
| 개발, 리서치, 기획, 검토 등 | 탐색적 확인 |
| | 즉시 처리 가능한 소규모 요청 |

적용 범위 밖의 요청은 체계 절차 없이 자연스럽게 응답한다.

**기본 흐름**

```
계획서 작성
    → 사용자 검토 ← 필수 단계 (생략 불가)
       두 가지 방법 중 하나 이상을 사용해 처리 (한 번 또는 여러 번, 섞어서 가능):
         · 방법 1 — 직접 검토: 사용자가 직접 읽고 피드백·수정 요청
         · 방법 2 — 인터뷰 지원: 에이전트가 결정 항목을 함께 채움 (plan_interview 스킬)
    → [선택] 독립 비평 (layer1_critique — 새 스레드)
    → 사용자 승인 ← 진행의 공식 게이트
    → 구현
    → 결과 확인
    → 피드백 루프 → 다음 작업의 설계 단계로 연결
```

**검토와 승인은 별개 단계다:**
- **검토**: 필수 단계다. 생략할 수 없다. 직접 검토와 인터뷰 지원 중 하나 이상을 사용해 처리하며, 반복하거나 섞어서 진행할 수 있다.
- **승인**: 확정된 계획에 대한 진행 허락. 구현 시작의 공식 게이트.

> 검토 자체는 필수다. "직접 검토할지 / 인터뷰 지원을 받을지"는 검토를 **어떻게** 수행할지의 선택일 뿐, 검토를 할지 말지의 선택이 아니다.

승인 단계를 건너뛰는 것은 에이전트의 판단이 아니라 **사용자의 명시적 허락**으로만 가능하다.

승인은 "전체 계획의 최종 서명"이 아니다. "지금 진행할 구간에 대한 허락"이다. 진행 중 새로운 발견이 생기면 다음 구간에 대해 검토/승인 사이클이 다시 시작된다.

**plan.md = 살아있는 항법 도구**

plan.md는 처음부터 완성되어야 하는 계약서가 아니다. 사용자와 에이전트가 최선의 길을 함께 찾아가도록 돕는 현재 인식 공유 도구다.

- 지금 알 수 있는 것에서 시작하고, 진행하면서 채워진다
- 발견이 생기거나 가정이 틀렸다는 걸 알게 되면 업데이트한다
- `구현 진행 상태` 테이블 = 현재 어디까지 왔는지 사용자와 에이전트가 함께 보는 위치 표시

plan.md는 체계의 세션 메모리이기도 하다. 세션이 끊겨도 plan.md를 읽으면 어디까지 왔는지 알 수 있다. CLI/IDE 대화 압축에 의존하지 않는다.

**에이전트 보고 섹션**

이 섹션이 생긴 배경에는 두 가지 구조적 약점이 있다.

**약점 1 — 브리핑 단계:** 자연어 브리핑을 받으면 에이전트는 plan.md를 만들기 전에 빈칸을 먼저 채워야 한다. 사용자가 실패 비용을 생각하기 전에 에이전트의 가정이 plan.md에 이미 들어간다.

**약점 2 — 검토 단계의 역동조:** plan.md가 잘 구조화돼 있으면 검토자는 자연스럽게 "잘 됐네"라고 반응한다. 관점 4의 동조는 에이전트 → 사용자 방향이었다. 여기서는 반대 방향이 일어난다. **사용자가 에이전트의 잘 만들어진 결과물에 동조**하는 역전이다. 검토에서 잡아야 할 것들 — 조용한 결정, 가치 결정 숨김 — 이 그대로 통과된다.

해법은 별도 체크리스트가 아니라, plan.md 자체가 에이전트에게 보고를 강제하도록 구조를 바꾸는 것이다.

plan.md 최상단에 에이전트 보고 섹션이 있다. 이것이 사용자-에이전트 소통 인터페이스다.

```markdown
## 에이전트 보고

### 사용자 결정 필요
AI가 결정해서는 안 되는 항목 — 가치 결정 또는 비즈니스 맥락 의존

### 에이전트 가정
브리핑에 없어 임의로 채운 내용 | 근거 | 틀렸다면 어떻게 달라지는가
```

왜 이 섹션이 있는가: 에이전트가 스스로 채운 빈칸을 명시하면, 사용자가 그것이 빈칸 A(의도적 위임)인지 빈칸 B(비가시적 위임)인지 판단할 수 있다. 가정이 드러나는 순간에 실패 비용을 판단하는 것이 실패 비용 추정의 가장 구체적인 형태다.

**계획서 검토를 수행하는 두 가지 방법**

검토는 필수 단계다. 그 검토를 수행하는 방법이 두 가지 있고, 하나만 쓰거나 둘을 섞어서, 한 번 또는 여러 번 사용할 수 있다.

검토할 때 동조 경향을 조심해야 한다. plan.md가 잘 구조화돼 있으면 "잘 됐네"라고 넘어가기 쉽다. 아래 두 방법 모두 이 동조를 깨기 위한 장치다.

**방법 1 — 직접 검토 (동조를 깨는 질문 활용):**

| 약한 질문 | 강한 질문 |
|-----------|-----------|
| "이 계획 괜찮아?" | "이 계획이 실패할 수 있는 시나리오를 찾아줘" |
| "검토해줘" | "내가 놓친 가정이 뭐야?" |
| "문제 없어?" | "이 계획대로 구현하다가 내가 예상 못 한 결정이 필요한 지점이 어디야?" |

**방법 2 — 인터뷰 지원 (`plan_interview` 스킬):**

"인터뷰 해줘" / "같이 봐줘"라고 요청하면 에이전트가 다음 순서로 함께 채운다:

1. **명시적 결정 항목** — 에이전트 보고 > 사용자 결정 필요 항목을 하나씩 논의
2. **숨은 결정 발굴** — `silent_decision_extraction` 스킬로 비가시적 위임 후보를 찾아 함께 처리
3. **실패 시나리오** — `counterexample_finding` 스킬로 반례를 탐색하고 대응 방향 결정

각 단계의 결과는 즉시 plan.md에 반영된다. 언제든 "여기까지"라고 하면 멈춘다.

**두 방법은 배타적이지 않다.** 직접 검토로 한 번 보고 인터뷰로 보완하거나, 인터뷰로 채운 뒤 다시 직접 확인할 수 있다. 어떤 방법을 몇 번 쓰든, 검토를 마친 뒤 **사용자의 명시적 승인**이 있어야 구현이 시작된다.

**루프 진입과 탈출 조건**

| 발견 내용 | 돌아갈 단계 |
|-----------|-------------|
| 구현 방식이 의도와 다름 | 2단계로 (재구현) |
| 실행 중 예상 못 한 오류 | 2단계 또는 3단계로 |
| 계획 자체가 잘못됨 | 1단계로 (계획 수정) |

루프 탈출 조건 — 아래 모두 충족 시:
```
✓ 계획서 검토 체크리스트 항목 전부 확인
✓ 정상 경로 추적 완료
✓ 실패 경로 추적 완료
```

**피드백 루프**

피드백 루프 없이 매번 결과 심문만 반복하면 비용이 줄지 않는다. 이번 작업에서 반복된 의도 간격 패턴을 기록하고, 다음 계획서에 반영하는 것이 목표다. **반복되는 오류는 설계 단계로 끌어올려서 처음부터 막는 구조.**

**구현 후 발견**

첫 결과물은 계획서를 완성시키는 도구다.

구현 전 명세만으로 요구사항을 완전히 정의하는 것은 불가능에 가깝다. 특히 동작 방식, UX, 기능 간 연결은 직접 경험하기 전에는 "내가 실제로 무엇을 원하는가"가 드러나지 않는다. 결과물을 보는 순간 비로소 가시화되는 이 발견을 **구현 후 발견(Post-Implementation Discovery)**이라 한다.

이것은 실패가 아니다. 계획이 잘못된 것도 아니다. 결과물이 계획서를 완성시켜주는 정보를 제공한 것이다.

**전진 증류 원칙**

태스크가 늘어날수록 "부모 태스크의 plan.md를 읽는다"는 방식은 무너진다. 계보가 깊어지고 기준이 계속 바뀌면, 읽어야 할 과거 파일이 기하급수적으로 늘어난다.

올바른 방향은 반대다:

> **과거로 거슬러 읽지 않는다. 배운 것을 현재의 shared/ 파일로 증류한다.**

`architecture.md`가 이미 이 방식으로 작동한다 — 기술 결정이 생기면 그 자리에서 기록하기 때문에, 이후 태스크는 과거 계획서를 읽지 않아도 된다. `direction.md`도 동일하게 작동해야 한다. 태스크가 끝날 때마다 배운 제품/UX 방향을 즉시 `direction.md`에 기록하는 것이 의무다. 선택이 아니다.

Layer 2 체크포인트는 이 증류가 제대로 됐는지 검증한다 — 기술 정합성뿐 아니라 방향 정합성까지. "direction.md가 지금 프로젝트의 실제 방향을 정확히 담고 있는가?"

계보 정보(파생 출처)는 사람이 히스토리를 이해하기 위한 메타데이터로만 기록한다. 에이전트가 설계 시 과거 plan.md를 읽도록 강제하는 메커니즘은 사용하지 않는다.

구현 후 발견을 처리하는 절차:

**흐름 — 작업 결과 검토 세션에서 시작**

작업 결과 검토(`layer1_review.md`) 세션이 끝날 때 에이전트가 묻는다:

> "추가로 수정하거나 필요한 작업이 있으신가요?"

| 상황 | 처리 |
|------|------|
| 추가 작업 없음 | layer1_review 세션에서 바로 종료 (검토 완료 + done/ 이동) |
| 추가 작업 있음 | layer1_discovery.md 세션으로 넘김 — 종료 처리는 거기서 전담 |

**1단계 — 발견 항목 수집 (즉시 구현 요청을 던지지 않는다)**

결과물을 보고 수정하거나 추가하고 싶은 것들을 먼저 목록으로 만든다. 항목별로 세 가지 유형 중 하나로 분류한다.

| 유형 | 특징 | 판단 질문 |
|------|------|-----------|
| 조정 | 현재 구현의 작은 변경 | "같은 계획 범위 안에서 처리되는가?" |
| 계획 확장 | 계획서에 빠진 요구사항 | "계획서를 더 완성하면 처리되는가?" |
| 신규 태스크 | 이 결과물이 드러낸 새 기능 필요 | "별도 계획과 구현이 필요한 규모인가?" |

**2단계 — 유형별 처리**

- **조정**: 현재 plan.md 범위 내에서 수정 후 재구현 → Layer 1 루프 재진입
- **계획 확장**: plan.md에 "구현 후 발견 추가 범위" 섹션을 만들어 기록 → 검토 + 승인 → 재구현
- **신규 태스크**: `workspace/tasks/INDEX.md`에 새 태스크로 등록 → 이후 별도 Layer 1 사이클

신규 태스크 등록 시 **양방향 링크**를 반드시 기록한다:
- 새 태스크 plan.md: `파생 출처: [원래 태스크명] — [발견 맥락]`
- 원래 태스크 plan.md "구현 후 발견 > 파생된 태스크": `[신규 태스크명] — [발견 맥락]`

**3단계 — 원래 태스크 종료**

> `layer1_discovery.md` 세션이 원래 태스크 종료를 전담한다.

| 발견 항목 구성 | 원래 태스크 처리 |
|--------------|----------------|
| 조정 또는 계획 확장 포함 | plan.md 상태 `구현 중` — 재구현 후 재검토 |
| 신규 태스크만 | 신규 태스크 등록 완료 후 → plan.md `검토 완료` + `done/` 이동 |

**순서 원칙: 신규 태스크 등록 먼저 → 원래 태스크 종료.** 순서를 바꾸면 파생 맥락이 끊긴다.

**4단계 — 발견의 맥락 기록**

발견 항목을 추가할 때 "왜 이것이 결과를 보기 전에는 보이지 않았는가"를 함께 기록한다. 이 맥락이 다음 계획서를 쓸 때 빈칸 B를 줄이는 재료가 된다.

> **주의:** 발견 항목을 즉시 구현 요청으로 던지면 안 된다. plan.md에 반영되지 않은 변경은 다음 세션에서 맥락을 잃는다. 반드시 분류 → 기록 → 승인 순서를 거친다.

#### 6.4 Layer 2 — 통합 체크포인트

Layer 2는 개별 작업 검증(Layer 1 루프)과 별개로, 전체 정합성을 확인하는 단계다.

| 구분 | Layer 1 루프 | Layer 2 체크포인트 |
|------|------------|-------------------|
| 범위 | 단일 작업 내 | 여러 작업 전체 |
| 시점 | 작업 중 문제 발견 시 | 주기적으로 |
| 목적 | 이 작업이 맞는가 | 전체가 정합한가 |

**실행 시점:**
- N개 작업 완료 시 (3~5개 작업마다 권장)
- 주요 기능 단위 완성 시
- Layer 1 피드백에서 "Layer 2 확인 필요" 항목이 누적될 때

> **주기 추적 방법:** Layer 2 완료 시 `workspace/tasks/INDEX.md` 하단에 `[Layer 2 완료] YYYY-MM-DD` 한 줄을 추가한다. 에이전트는 태스크 완료 시 이 마커 이후 done 태스크 수를 세어 3개 이상이면 Layer 2를 제안한다.

**확인할 것:**
- 지금까지 구현된 코드 중 아키텍처 규칙을 위반한 부분
- 작업 간 인터페이스 충돌·불일치
- 의도하지 않은 패턴
- 규칙 문서 중 현실과 맞지 않아 업데이트가 필요한 것

**발견 후 처리 흐름:**
```
규칙 위반 / 인터페이스 충돌 발견
    → 해당 작업으로 돌아가 수정 → Layer 1 루프 재진입

규칙 문서 업데이트 필요
    → workspace/memory/shared/architecture.md 업데이트 → 이후 작업에 반영

문제 없음
    → 다음 작업 묶음 시작
```

Layer 2를 거치며 Layer 0 문서가 점점 정교해진다. 프로젝트가 진행될수록 규칙이 강화되는 구조다.

### 7장. inject 파일과 세션 레시피

#### 7.1 inject 파일이란

inject 파일 = **에이전트에게 전달하는 세션 레시피**.

어느 스레드에서 사용하는지는 작업 성격에 따라 다르다 — 7.2 표 참조. 평가·비평처럼 컨텍스트 격리가 필요한 세션은 새 스레드에서, 구현처럼 설계 맥락이 이어져야 하는 세션은 같은 스레드에서 사용한다.

inject 파일의 구성:
```
[페르소나 선언]          ← 이 세션에서 어떤 역할로 작동하는가
[읽을 파일 목록]         ← 세션 시작 전 에이전트가 직접 읽어야 할 workspace 파일
[작업 지시]              ← 이 세션에서 무엇을 해야 하는가
[완료 기준]              ← 세션이 끝났다고 판단하는 조건
[세션 종료 시 보고 형식] ← 세션 종료 후 어떤 파일을 업데이트하는가
```

에이전트는 inject 파일을 받으면 "읽을 파일 목록"에 있는 `workspace/` 파일들을 먼저 읽어 컨텍스트를 구성한 뒤 작업을 시작한다. 컨텍스트 구성을 에이전트가 직접 수행하기 때문에, 사용자가 파일 내용을 수동으로 붙여넣지 않아도 된다.

**기본 동작 — 자동 라우팅:**

체계가 설치된 프로젝트에서는 에이전트가 요청 유형을 판단해 적절한 inject 파일을 자동으로 로드한다. 사용자는 작업 요청만 하면 된다.

**직접 지정 — 수동 @참조:**

에이전트가 잘못 라우팅했거나 특정 inject 파일로 세션을 강제로 시작하고 싶을 때는 직접 지정할 수 있다:
```
@.mpa-workspace/inject/layer1_design.md
[작업 요구사항 설명]
```

#### 7.2 각 inject 파일의 역할

| inject 파일 | 사용 시점 | 핵심 목적 | 스레드 |
|------------|-----------|-----------|--------|
| `layer0_init.md` | 프로젝트 초기화 | 아키텍처 규칙 문서화 또는 현황 분석 | 새 스레드 |
| `layer0_update.md` | 체계 업데이트 | upgrade-candidates 이전 + 인바운드 반영 | 새 스레드 |
| `layer1_design.md` | 개별 작업 설계 | 계획서(plan.md) 작성 | 새 스레드 |
| `layer1_critique.md` | 계획 독립 비평 | plan.md의 약점·숨은 가정·실패 시나리오 발굴 | **반드시 새 스레드** |
| `layer1_implement.md` | 구현 | 계획서 충실 이행 | 같은 스레드 |
| `layer1_review.md` | 작업 결과 검토 | 코드·UI/UX·기능 동작 전반을 독립적으로 검토 | 새 스레드 |
| `layer1_discovery.md` | 구현 후 발견 처리 | 결과 경험 후 수정·추가 사항 분류 및 라우팅 | 새 스레드 |
| `layer2_checkpoint.md` | 통합 체크포인트 | 전체 정합성 확인 | 새 스레드 |

`layer1_critique.md`는 반드시 새 스레드에서 시작한다. 설계 세션의 컨텍스트를 이어받으면 비평의 독립성이 사라진다 (8.1절 컨텍스트 오염 참조). 비평 에이전트는 plan.md 결과물만 보고 판단해야 하며, "설계자가 왜 그 결정을 내렸는가"를 알면 completion bias가 작동한다.

#### 7.3 컨텍스트 주입 원칙

**최소 유효 컨텍스트 원칙:**
> "이 컨텍스트가 없으면 AI가 어떤 잘못된 결정을 내리는가?"

주입 구조:

```
A. 항상 주입 (Session Baseline)
   프로젝트 정체성 + 절대 금지 + 핵심 아키텍처
   → 작고 항상 유효, 없으면 항상 문제

B. 작업별 추가 주입
   영향 범위의 인터페이스 계약
   관련 결정 이력 / 안티패턴

C. 부정 컨텍스트 — "건드리지 말아야 할 것" 명시
   해도 되는 것만 알려주면 나머지 전부를 할 수 있다고 가정함
```

부정 컨텍스트가 중요한 이유: "이건 해도 돼"라고만 알려주면, AI는 명시되지 않은 나머지 전부를 할 수 있다고 가정한다. "이건 하면 안 돼"를 명시해야 AI가 경계를 인식한다. 관점 8(범위 자율성 제한)의 컨텍스트 구현이다.

### 8장. 페르소나와 스레드 전략

#### 8.1 왜 새 스레드를 만드는가 — 컨텍스트 오염

같은 스레드에서 "만든 코드를 비판해줘"를 요청하면 어떤 일이 벌어지는가:
- 이전 대화에서 에이전트가 어떤 결정을 내렸는지가 컨텍스트에 남아 있다
- 자기가 만든 코드를 비판하기 어렵다 → 방어적으로 검토한다
- 잘 만들어진 결과물에 사용자가 동조하는 역전이 일어날 수도 있다

| 작업 성격 | 스레드 | 이유 |
|-----------|--------|------|
| 생성 — 맥락이 많을수록 좋음 | 같은 스레드 | 계획/규칙 맥락이 필요 |
| 평가 — 맥락이 없을수록 객관적 | 새 스레드 | 컨텍스트 오염 방지 |

새 스레드에서 컨텍스트 주입 방법:
```
[해당 페르소나 선언]
[필요한 컨텍스트만 압축해서 제공]
[평가 대상]
[요청]
```

핵심: 새 스레드에는 평가에 꼭 필요한 것만 넣는다. 이전 대화, 구현 과정, 맥락 설명을 넣을수록 객관성이 떨어진다.

**반대 방향의 원칙 — 너무 일찍 열지 않는다:**

새 스레드가 필요한 시점은 "평가가 시작될 때"다. 이전 단계가 끝나기 전에 미리 새 스레드를 열면 컨텍스트가 낭비된다. 설계가 완료되지 않은 상태에서 비평 세션을 열거나, 검토가 끝나기 전에 구현 세션을 준비하는 것은 이 원칙을 위반한다.

> 설계 → 사용자 검토 → 구현의 순서를 지킨다. 이전 단계가 끝나지 않은 상태에서 다음 단계의 에이전트를 준비하지 않는다.

요약하면 스레드 전략은 양방향으로 작동한다: 평가는 새 스레드로 **분리**하고, 단계 전환은 완료 시점에만 **진입**한다.

#### 8.2 페르소나별 사용 시점

페르소나 파일은 `.mpa-workspace/personas/`에 있고, 프로젝트별 누적 학습은 `workspace/memory/roles/`에 별도 저장된다. 이 분리가 중요하다: 페르소나 정의는 체계 파일(모든 프로젝트 공통)이고, 학습 기록은 프로젝트 데이터다.

| 페르소나 | 사용 시점 | 스레드 | 핵심 태도 |
|----------|-----------|--------|-----------|
| `architect` | Layer 0 초기화 + 기존 프로젝트 분석 | 새 스레드 | 전체 일관성, 결정 이유 명시, 현재 상태 사실 기록 |
| `task_designer` | Layer 1 계획 작성 | 새 스레드 | 빈칸을 드러내고 계획에 "왜"를 붙임 |
| `plan_critic` | Layer 1 계획 독립 비평 | **반드시 새 스레드** | 약점만 찾음, 동의·칭찬 없음, plan.md만 보고 판단 |
| `implementer` | Layer 1 구현 | 같은 스레드 | 계획 충실 이행, 범위 이탈 금지 |
| `code_reviewer` | Layer 1 작업 결과 검토 | 새 스레드 | 결과물 전반 검토, 논리 검증, 조용한 결정 탐지, UI/UX 동작 확인 |
| `integration_auditor` | Layer 2 통합 감사 | 새 스레드 | 규칙 준수 여부만 판단 |
| `memory_curator` | Layer 2 이후 또는 스프린트 종료 시 | 새 스레드 | memory 무결성 전담, 개발 작업 없음 |

**`task_designer`와 `plan_critic`을 왜 분리하는가**

설계자가 자신이 만든 계획을 비평하면 completion bias가 작동한다. 이것은 인간의 자아 보호가 아니라 LLM의 구조적 특성이다 — 컨텍스트에 완성된 계획이 있으면, 그것을 정당화하는 방향으로 읽으려는 경향이 생긴다. 새 스레드의 plan_critic은 설계 과정을 모른다. plan.md 결과물만 보고 약점을 찾는다. 이것이 진짜 독립 검토다.

페르소나 사용 흐름 (Layer 1 기준):
```
[같은 스레드 — task_designer]  계획서 작성 (plan.md)
        ↓
[같은 스레드]                  사용자 검토
                               ├── 직접 검토
                               └── "인터뷰 해줘" → plan_interview 스킬 (같은 스레드)
        ↓ 승인
[새 스레드 — plan_critic]      독립 비평 (plan.md만 읽음) ← 권장 기준 해당 시
        ↓ 비평 반영
[같은 스레드 — implementer]    구현 (파생 출처·파생된 태스크 참조 후 시작)
        ↓ 구현 완료
[새 스레드 — code_reviewer]    작업 결과 검토
        ↓ "추가 작업이 있으신가요?"
        ├── 없음 → 검토 완료 + done/ 이동 (여기서 종료)
        └── 있음
              ↓
[새 스레드 — task_designer]    구현 후 발견 처리 (layer1_discovery.md)
                               ├── 조정 → plan.md 수정 후 재구현
                               ├── 계획 확장 → plan.md 보완 후 재구현
                               └── 신규 태스크 → 양방향 링크 기록 + INDEX.md 등록
                                               → 원래 태스크 종료 (discovery 전담)
                                               → 새 Layer 1 사이클
```

인터뷰는 **같은 스레드**에서 진행된다 — 설계 맥락이 필요하기 때문이다. 독립 비평은 **반드시 새 스레드**다 — 설계 맥락이 없어야 객관성이 보장되기 때문이다. 두 단계의 스레드 전략이 정반대인 이유가 여기에 있다.

비평 단계는 권장 기준 해당 시 진행한다. "귀찮아서" 생략하는 것과 "복잡도가 낮아서" 생략하는 것은 다르다.

#### 8.3 스킬: "어떻게 하는가"를 정의한다

페르소나가 "누가 하는가"를 정의한다면, 스킬은 "어떻게 하는가"를 정의한다.

스킬 없이: "이 코드 검토해줘"  
스킬 있이: "정상 경로에서 입력부터 출력까지 각 단계를 추적하고, 각 단계에서 어떤 값이 전달되고 어떤 결정이 내려지는지 보여줘"

구체적인 탐색 방향이 있으면 에이전트의 동조 경향을 차단하고 진짜 문제를 찾아낼 수 있다.

| 스킬 | 목적 | 사용 시점 |
|------|------|-----------|
| `silent_decision_extraction` | 명시되지 않은 결정 지점 추출 | Layer 0 요구사항, Layer 1 계획서 |
| `path_tracing` | 정상/실패 경로 논리 추적 | Layer 1 결과 확인, Layer 2 |
| `counterexample_finding` | 계획·코드의 실패 반례 탐색 | Layer 0 설계 검토, Layer 1 계획 검토 |
| `dependency_mapping` | 작업 간 선후 관계와 공유 인터페이스 정의 | Layer 0 작업 분해 |
| `plan_interview` | 검토 인터뷰 오케스트레이션 — 기존 스킬을 순서대로 활용해 결정 공백을 채움 | Layer 1 계획서 검토 (사용자 요청 시) |
| `discovery_classification` | 구현 후 발견 항목을 조정·계획 확장·신규 태스크로 분류 | Layer 1 구현 후 발견 처리 세션 |

## 5부: 어떻게 시작하는가 — 설치와 운영

> 작업 방식을 이해했다면, 이제 실제로 체계를 설치하고 운영하는 방법을 배운다.

---

### 9장. 설치와 업데이트

#### 9.1 설치 방법

**설치의 진입점은 `install.md`다.**

`install.py`를 직접 실행하지 않는다. 대신 에이전트에게 `install.md`를 읽게 하고, 거기 정의된 질의응답 절차를 따라 파라미터를 결정한 뒤 에이전트가 `install.py`를 실행한다. 설치 자체가 "에이전트와 대화하며 진행하는 절차"인 셈이다.

```
1. 체계 저장소 클론 (최초 1회)
   git clone https://github.com/udgrounder/my-pacemaker-agent.git

2. 에이전트에게 install.md를 읽히고 설치를 요청
   "@install.md 를 읽고 /path/to/my-project 에 설치해줘"

3. 에이전트가 install.md의 질의응답 절차(Q1~Q5)를 진행
   Q1. 프로젝트 경로 확인
   Q2. 사용할 agent 결정 (프로젝트 폴더를 보고 자동 감지 → 사용자 확인)
   Q3. agent-specs/{agent}/spec.md를 읽어 설치 내용 파악·안내
   Q4. .mpa-workspace/ 존재 여부로 신규/업그레이드 자동 판단
   Q5. 수집한 정보를 요약해 알린 뒤 install.py 실행
```

에이전트가 최종적으로 실행하는 명령:
```bash
# 신규 설치
python3 install.py --project <경로> --agents <agent>

# 업그레이드
python3 install.py --project <경로> --agents <agent> --upgrade
```

`--project`와 `--agents`는 **둘 다 필수**다. 신규/업그레이드는 `.mpa-workspace/` 존재 여부로 자동 판단되며, `--upgrade`로 강제할 수도 있다.

**설치 후 프로젝트 구조:**
```
my-project/
├── .mpa-workspace/              ← 체계 파일 (에이전트 행동 규칙)
├── workspace/                   ← 프로젝트 데이터 (템플릿 골격만 있는 상태)
│   ├── memory/ tasks/ docs/
├── CLAUDE.md / AGENTS.md / GEMINI.md  ← 선택한 agent의 진입점 (기존 파일이면 섹션 추가)
├── .claude/agents/mpa_pacemaker.md    ← claude 포함 시 (native 폴더에 직접 배치)
└── .agents/rules/mpa_pacemaker.md     ← codex/antigravity 포함 시
```

진입점 파일(`CLAUDE.md` 등)에는 Agents Workspace 섹션이 추가되고, agent별 전용 파일(`mpa_pacemaker.md`)은 해당 agent의 native 폴더에 복사된다(없는 경우만). `workspace/`는 어떤 agent를 쓰든 동일하므로 프로젝트 루트에 한 번만 설치된다.

지원 agent와 감지 조건:

| Agent | 진입점 | 감지 조건 |
|-------|--------|---------|
| `claude` | `CLAUDE.md` | `CLAUDE.md` 또는 `.claude/` 존재 |
| `codex` | `AGENTS.md` | `AGENTS.md` 존재 |
| `antigravity` | `GEMINI.md` | `GEMINI.md` 또는 `.gemini/` 존재 |
| `openagent` | spec.md 질의로 결정 | 감지 불가 → 사용자 확인 |

**업그레이드 시 추가 동작:**

업그레이드도 같은 절차(install.md)로 진행되며, 신규 설치와 다른 점은 다음과 같다.
```
1. 프로젝트의 upgrade-candidates/ → 체계 저장소로 이전 (발견한 것 반영)
2. 체계 최신 .mpa-workspace/ → 프로젝트에 교체 복사 (템플릿 최신화)
3. workspace/ 템플릿의 누락 항목만 추가 (기존 프로젝트 데이터는 건드리지 않음)
```

복사되는 것과 복사 안 되는 것:
- **복사됨**: 에이전트 행동 규칙, inject 파일, 페르소나, 스킬 등 체계 운영 파일
- **복사 안 됨**: workspace/ 내 프로젝트 데이터 (누락 골격만 보충하고 기존 내용은 보존)

#### 9.2 설치가 "지식 전달의 트리거"인 이유

설치와 업데이트 단계가 프로젝트의 발견을 체계로 올리는 **공식 경로**다.

```
프로젝트 A에서 검증된 패턴
    ↓ upgrade-candidates → 체계 반영
프로젝트 B의 Layer 0에서 활용
```

이 경로가 없으면, 각 프로젝트에서 발견한 것들이 체계에 반영되지 않고 사라진다.

`upgrade-candidates/` 내용은 설치/업데이트 시 비워진다. 체계 자체 후보는 프로젝트로 넘기지 않는다.

---

### 10장. 세션 연속성

#### 10.1 컨텍스트 손실에 탄력적인 구조

이 논의 세션에서 실제로 경험한 일이다. 긴 대화가 압축되면 이전 내용이 요약본으로만 남고, 세션 재개 시 파일을 직접 다시 읽어야 컨텍스트를 복원할 수 있었다.

컨텍스트 손실 자체를 막을 수는 없다. 하지만 손실이 작업에 미치는 영향을 줄일 수 있다.

| 연속성 수단 | 형태 | 신뢰도 |
|-----------|-----|-------|
| CLI 대화 압축 | 자연어 요약, 자동 | 요약 품질에 의존, 불확실 |
| plan.md | 구조화된 상태 파일, 명시적 | 파일이 있는 한 정확 |

세션이 끊겨도 이것들이 있으면 작업을 재개할 수 있다:
- `plan.md` → 어디까지 했는지 상태 체크포인트 (구현 단계 체크리스트)
- `changelog.md` → 구현 중 완료된 부분 파악 (없으면 처음부터, 있으면 이어가기)
- `workspace/memory/` → 프로젝트 핵심 컨텍스트
- inject 파일 → 역할과 규칙을 세션마다 재로딩

**재개 프로세스:**
```
plan.md 열기 → "구현 단계" 체크리스트 확인
    ├── 상태: "구현 중" → changelog.md 읽기 → 첫 미완료 단계부터 재개
    └── 상태: "설계 완료" → layer1_implement.md로 구현 세션 새로 시작
```

**체계는 컨텍스트 손실에 탄력적인 작업 구조다.**

#### 10.2 역할 메모리 업데이트

역할별 누적 학습은 `workspace/memory/roles/[페르소나명].md`에 저장된다. `.mpa-workspace/personas/`의 파일은 역할 정의(체계 공통)이고, `workspace/memory/roles/`는 이 프로젝트에서 해당 역할이 배운 것(프로젝트 데이터)이다. 이 두 가지를 섞지 않는다.

처음부터 완성된 체계를 만들려 하지 않는다. 체계는 학습 시스템이다.

```
에이전트가 훈련 데이터 가정으로 실수
    ↓
사람이 발견
    ↓
workspace/memory/roles/[페르소나명].md에 기록 (자가 개선 필터 적용)
    ↓
다음에는 같은 실수를 안 함
```

이것은 예방이 아니라 학습이다. 첫 실수는 막을 수 없다. 두 번째부터 막는다.

학습이 쌓이는 레이어:

| 레이어 | 학습 메커니즘 |
|--------|------------|
| 역할 레벨 | `workspace/memory/roles/[페르소나명].md` — 자가 개선 필터로 기록 |
| 도메인 레벨 | `workspace/memory/domains/[도메인]/rules.md` (부정 정의 포함) |
| 시스템 레벨 | `workspace/memory/shared/project_identity.md` (부정 정의 포함) |
| 체계 레벨 | `.mpa-workspace/upgrade-candidates/` — 방법론 개선 |

체계의 성숙도는 사용 시간에 비례한다.

- 처음 적용: 보호막이 얇음
- 오래 운영: 누적된 학습으로 보호막이 두꺼워짐

---

## 6부: 어떻게 발전시키는가 — 체계 개선

> 체계를 운영하다 보면 개선할 것들이 쌓인다. 이것을 어떻게 체계에 반영하는가.

---

### 11장. 기록 원칙

#### 11.1 검증 여부가 기준이다

무엇을 기록하고, 무엇을 기록하지 않는가? 판단 기준은 하나다: **검증됐는가?**

기록 판단 기준은 "지금 vs 나중"이 아니라 "검증됐는가 vs 추정인가":

- **기록해야 하는 것**: 검증된 지식 (사용자 경험, 다른 프로젝트, 기존 코드)
- **기록하지 않는 것**: 추정 (경험 전에 "아마 이럴 것"으로 채운 내용)

추정이 틀린 채 남아있으면 관점 10에서 설명한 "낡은 문서 > 없는 것보다 위험" 상태가 된다.

#### 11.2 자가 개선 필터

기록 여부를 판단하는 사실 기반 조건 — 다음 중 하나 이상 해당하면 기록한다:
- plan.md "에이전트 가정"이 구현 중 틀린 것으로 밝혀졌는가?
- 구현 중 plan.md 자체를 수정해야 했는가?
- 범위 밖 변경을 발견해 분리 보고했는가?

위 조건에 해당하지 않더라도 보조 기준으로 자문한다: **"다음에 이 역할이 몰랐다면 실수했을 내용인가?"**

**기록 O:**
- 예상과 다르게 동작한 패턴
- 반복적으로 나타나는 실수 유형
- 코드/문서만 봐서는 알 수 없는 제약

**기록 X:**
- 태스크 완료 사실
- 코드나 문서에서 직접 읽을 수 있는 내용
- 일회성 결정

#### 11.3 부정 정의 우선

일반적 패턴과 **다른 것**만 기록한다. 일반적인 것은 기록하지 않아도 에이전트가 안다.

왜 부정 정의가 중요한가: 에이전트는 훈련 데이터에서 형성된 자신의 "기본 가정"으로 프로젝트를 해석한다. "사용자", "권한", "결제" 같은 개념을 이 시스템의 정의가 아닌 훈련 데이터의 가장 일반적인 정의로 채운다.

```
# 부정 정의 없을 때
에이전트: "사용자" = 훈련 데이터의 일반적인 웹 사용자

# 부정 정의 있을 때
에이전트: "사용자" = 캠핑 예약 맥락에서 운영자와 일반 이용자 두 종류가 있고,
          권한 구조가 다름
```

부정 정의가 유지보수 가능한 이유:
- **트리거 명확**: 에이전트가 일반적 가정으로 실수했을 때, 새 개념이 추가될 때
- **짧음**: 다른 것만 기록
- **발견 가능성 높음**: 틀렸을 때 에이전트 오류로 나타남

---

### 12장. 체계 개선 파이프라인

#### 12.1 upgrade-candidates 기록 방법

운영 중 발견한 체계 개선 후보를 기록하는 위치: `.mpa-workspace/upgrade-candidates/`

기록 전 판단 기준:
> "이 발견이 다른 프로젝트에도 적용될 가치가 있는가?"

- 이 프로젝트에만 해당하는 특수 케이스 → workspace/memory/에 기록
- 다른 프로젝트에도 일반적으로 적용될 패턴 → upgrade-candidates/에 기록

#### 12.2 프로젝트 → 체계 이전 흐름

```
운영 중 발견
    ↓
upgrade-candidates/ 기록
    ↓
(다음 업데이트 시점)
    ↓
전체 후보 목록 검토
    ↓
하나씩 확인 후 반영
    ↓
upgrade-candidates/ 비워짐 → 다음 프로젝트에 전파
```

#### 12.3 체계 시스템 업데이트 원칙

체계를 업데이트하는 것은 모든 프로젝트에 전파되는 일이다. 신중하게 진행해야 한다.

- **사용자가 명시적으로 요청했을 때만 시작**
- 전체 후보 목록을 먼저 제시한다
- **후보 하나씩, 사용자 확인 후 진행** — 일괄 자동 적용 금지
- 에이전트는 체계 변경을 사용자 없이 주도할 수 없다

왜 자동 적용이 위험한가: 하나의 변경이 다른 프로젝트에 예상치 못한 영향을 줄 수 있다. 변경 내용을 사용자가 이해하고 있어야 나중에 문제가 생겼을 때 원인을 추적할 수 있다.

## 7부: 무엇을 조심해야 하는가 — 한계와 주의사항

> 체계는 알려진 위험만 관리할 수 있다. 알 수 없는 위험은 구조 밖의 영역이다.

---

### 13장. 체계의 구조적 한계

#### 13.1 네 역설

체계를 운영하다 보면 해결되지 않는 네 가지 역설을 만나게 된다. 앞의 세 역설은 인식론적 한계, 네 번째는 실행 한계다. 공통 구조:

> **진짜 위험한 것은 보이지 않는다. 보이는 것은 덜 위험하다.**

**탐지 역설**

에이전트가 잘못 이해한 것은 잘못 이해했다는 신호를 보내지 않는다.

```
발견할 수 있는 빈칸은 이미 위험하지 않고,
위험한 빈칸은 발견할 수 없다.
→ 인식론적 문제: unknown unknowns
```

두 방향으로 탐지 범위를 넓힐 수 있다:
- **방향 A**: AI 관점에서 내 요청 읽기 → 내가 아는 도메인에서 효과적, 모르는 도메인에서는 실패
- **방향 B**: AI를 탐지 도구로 쓰기 → 훈련 데이터에 패턴이 있는 도메인에서 효과적, AI가 본 적 없는 맥락에서는 실패

두 방향을 합쳐도 나도 AI도 모르는 것은 탐지 불가능하다.

올바른 목표:
```
틀린 목표: 모든 빈칸을 발견한다
맞는 목표: 파급효과가 큰 빈칸을 발견할 확률을 높인다
```

**측정 역설**

```
측정할 수 있는 실패 비용은 이미 아는 영역이고,
모르는 영역의 실패 비용은 측정할 수 없다.
```

정밀하게 측정할 수 있다면 — 이미 충분히 아는 영역이라 실제 위험이 낮다. 측정이 안 된다면 — 그게 진짜 위험 구간이다.

도출되는 원칙: **실패 비용을 측정하기 어려울수록, 높게 잡는다.** 측정 불확실성 자체가 높은 실패 비용의 신호다.

**보고 역설**

에이전트가 놓친 것은 보고되지 않는다.

plan.md의 "에이전트 보고" 섹션은 에이전트가 인식한 가정만 보고한다. 훈련 데이터에서 "당연한 것"으로 굳어진 가정은 에이전트 입장에서 "사실"이기 때문에 보고되지 않는다.

```
구조로 해결 가능:  인식된 가정을 가시화한다
구조로 해결 불가:  인식되지 않은 가정의 존재를 드러낸다
```

**준수 역설**

체계의 기계적 강제는 소스 수정 게이트(code_gate.py)에 한정된다. 그 외의 모든 것 — inject 파일 로드, 세션 종료 절차, architecture.md 업데이트 — 은 에이전트가 agent_rules.md를 실제로 읽고 따를 것이라는 가정에 달려 있다.

이것은 앞의 세 역설과 다른 종류의 한계다. 세 역설은 "인식론적 문제"(알 수 없는 것)이고, 준수 역설은 "실행 문제"(알고 있지만 따르지 않을 수 있는 것)다. 세션 종료 절차를 빠트리거나 inject 파일을 로드하지 않는 것은 탐지도 어렵고, 체계 스스로 감지할 수 없다.

```
구조로 해결 가능:  소스 수정 게이트 (code_gate.py)
구조로 해결 불가:  에이전트가 절차를 빠트리는 것
```

**네 역설의 함의:**

네 역설을 알고 운영하는 것과 모르고 운영하는 것은 다르다. 모르면 "체계가 잘 작동하고 있다"는 착각에 빠질 수 있다. 알면 "체계가 커버하지 못하는 영역이 있고, 그 영역은 나의 도메인 역량과 주의 깊은 관찰에 달려 있다"는 것을 인식하고 운영할 수 있다.

**상위 원칙 전제 조건과의 연결:**

3장에서 "인간이 최소한의 검증 능력을 갖춘 영역에서"라는 조건을 달았다. 세 역설을 이해하면 이 조건이 왜 필요한지가 설명된다. 체계가 인식된 가정만 다룰 수 있다면, 인식되지 않은 가정을 걸러낼 최소한의 도메인 역량이 인간에게 있어야 한다. 그 역량이 없으면 체계도 작동하지 않는다.

#### 13.2 삭제 역설

체계 유지보수에서 가장 어려운 문제가 삭제다.

삭제가 필요한 규칙은 어떻게 식별하는가?

```
규칙이 자주 위반된다 → 문제가 보임 → 수정 가능
규칙이 전혀 위반되지 않는다 → ??? 잘 지켜지는 것인가, 불필요한 것인가?
```

"침묵하는 규칙"은 두 가지 중 하나다:
- **A. 잘 지켜지는 규칙**: 필요하고 효과적으로 작동 중
- **B. 불필요한 규칙**: 아무도 신경 쓰지 않아도 문제가 생기지 않음

A와 B는 외부에서 관찰 불가능하다. 둘 다 침묵한다.

삭제 역설을 극복할 수는 없지만, 줄일 수 있는 방법:

1. **빈도 신호**: 이 규칙이 실제로 몇 번 발동되었는가? Layer 2 체크포인트에서 확인
2. **기원 추적**: 규칙이 어떤 실수에서 나왔는가? 그 실수가 더 이상 발생하지 않는 환경이 됐는가?

최종 판단은 사용자가 한다. 체계가 자동으로 규칙을 삭제해서는 안 된다.

#### 13.3 체계 실패는 인간의 몫

체계 자체가 실패했을 때 그것을 감지하는 주체는 시스템 외부에 있어야 한다. 이 역할은 자동화될 수 없다.

> 깨진 시스템은 자기 자신이 깨졌다는 것을 알 수 없다.  
> 관찰자는 관찰 대상 밖에 있어야 한다.

"체계 실패는 인간의 몫"이라는 말은 체계가 나쁘다는 게 아니다. 어떤 시스템도 자기 자신의 실패를 감지할 수 없다는 근본적 한계의 인정이다.

체계와 사용자의 역할 분담:

```
체계:   결과물을 만들고, 판단할 수 있도록 가시화한다
사용자:   결과물을 쓸지 말지 판단하고, 그 결과에 책임진다
```

결과물이 읽히지 않으면 사용자가 판단할 수 없고, 판단할 수 없으면 책임을 질 수도 없다.

체계의 성공/실패 기준은 결과물의 사용 여부가 아니라 **과정의 정합성**이다.

- **과정 실패**: 잘못된 이해, 미기록 가정, 아키텍처 규칙 위반
- **실패 아님**: 결과물이 사용되지 않는 것, 방향이 바뀌어 필요 없어진 것

**Step 1 — 설치** (9장 참조)

저장소를 클론한 뒤, `install.py`를 직접 실행하지 말고 에이전트에게 `install.md`를 읽혀 설치를 맡긴다.

```bash
git clone https://github.com/udgrounder/my-pacemaker-agent.git
```
```
@install.md 를 읽고 /path/to/my-project 에 설치해줘
```

에이전트가 install.md의 질의응답 절차로 경로와 agent를 확정하고(폴더를 보고 자동 감지 → 확인), 신규/업그레이드를 판단한 뒤 `install.py`를 실행한다.  
설치 후 프로젝트에 `.mpa-workspace/`, `workspace/`(템플릿 골격)가 생기고, 선택한 agent의 진입점이 루트 설정 파일(`CLAUDE.md` 등)에 추가된다.

---

**Step 2 — Layer 0: 프로젝트 초기화** (6.2 참조)

새 채팅 스레드에서 아래와 같이 실행한다:
```
@.mpa-workspace/inject/layer0_init.md
[새 프로젝트 / 기존 프로젝트 여부와 간략한 배경 설명]
```

에이전트가 신규/기존을 판단하고 해당 흐름을 실행한다.  
에이전트와 대화를 통해 만들어지는 것:
- `workspace/memory/shared/project_identity.md` — 프로젝트 정의
- `workspace/memory/shared/architecture.md` — 아키텍처 규칙
- `workspace/tasks/INDEX.md` — 작업 목록과 의존성

Layer 0가 완료되면 "이 컨텍스트 없이 시작하면 어떤 잘못된 결정이 내려지는가?"에 답할 수 있어야 한다.

---

**Step 3 — Layer 1: 첫 번째 작업** (6.3 참조)

작업 요청을 하면 에이전트가 유형을 판단해 자동으로 설계 세션을 시작한다. 직접 지정하고 싶다면:
```
@.mpa-workspace/inject/layer1_design.md
[작업 요구사항]
```

에이전트가 `workspace/tasks/active/yyyymmdd_[태스크명]/plan.md`를 작성한다.  
계획서에서 **에이전트 보고 섹션**을 먼저 확인한다 — 에이전트가 임의로 채운 가정이 여기 나온다.

**계획서 비평 (권장 기준 해당 시):**  
계획서의 위험도를 먼저 판단한다. 🔴 고위험 항목이 하나라도 해당하거나, 🟡 중위험 항목이 두 개 이상 해당하면 독립 비평을 권장한다.

🔴 고위험: 비가역적 변경 / 외부 시스템 연동 / 보안 관련 변경 (인증·권한·암호화) / 공유 인프라 변경 (CI/CD·배포 설정)  
🟡 중위험: 가치 결정 항목 다수 / 타 도메인 인터페이스 영향 / 실패 비용 높음 / 복잡한 다단계 구현

정확한 기준은 `layer1_critique.md` "독립 비평 세션이 필요한 기준" 섹션을 참조한다.

새 스레드에서 아래를 실행한다:
```
@.mpa-workspace/inject/layer1_critique.md
```
plan_critic이 설계 과정 없이 plan.md만 보고 독립 비평을 수행한다.

비평 반영 → 승인 후, 같은 스레드에서 `layer1_implement.md`로 구현을 시작한다.

---

**Step 4 — 작업이 익숙해지면**

- Layer 2 체크포인트 (6.4): 3~5개 작업마다 전체 정합성 확인
- 체계 개선 발견 시 (12장): `upgrade-candidates/`에 기록
- 세션이 끊기면 (10장): `plan.md` + `workspace/memory/` 파일을 새 스레드에 주입해 재개

### A. 자주 발생하는 실패 패턴

**"AI가 기억을 못 해서"라고 생각할 때**  
실제 원인은 대부분 "처음부터 컨텍스트에 없었다"이다. AI의 기억 문제가 아니라 구조 문제다. 증상 치료(매번 다시 알려주기)가 아니라 원인 치료(레지스트리 + 자동 주입)가 필요하다.

**계획서 검토를 동의 구하기로 쓸 때**  
"이 계획 어때?"라고 물으면 동조 → 강화만 된다. 논리 검증이 필요하면 "이 계획이 실패할 수 있는 시나리오를 찾아줘"처럼 구조를 바꿔야 한다.

**Layer 0 없이 Layer 1을 시작할 때**  
아키텍처 규칙이 암묵적으로만 존재하는 상태에서 작업을 시작하면, 각 작업은 옳지만 전체가 무너진다. 나중에 고치는 비용이 처음에 Layer 0를 하는 비용보다 훨씬 크다.

**승인 단계를 건너뛸 때**  
에이전트가 스스로 승인을 생략하는 것은 허용되지 않는다. 검토와 승인은 별개 단계다. 승인이 없으면 구현이 시작되지 않아야 한다.

**"처음부터 완벽한 체계"를 목표로 할 때**  
체계는 처음부터 완성되는 게 아니라 운영하면서 자라는 것이다. Layer 0에서 추정을 넣으면 낡은 문서가 된다. 지금 알 수 있는 것만 넣고, 나머지는 경험으로 채우는 것이 맞다.

**구현 후 발견을 즉시 구현 요청으로 처리할 때**  
결과물을 보고 수정하고 싶은 것이 생기면 바로 "이 부분 바꿔줘"를 던지기 쉽다. 에이전트는 처리하지만 plan.md에 반영되지 않은 변경은 다음 세션에서 맥락을 잃는다. 발견 항목은 먼저 분류하고 plan.md에 기록한 뒤 승인을 거쳐 재구현해야 한다.

**구현 후 발견을 실패로 해석할 때**  
"계획을 잘못 짰다"고 자책하는 경우가 있다. 구현 전 명세만으로 동작 방식·UX·기능 간 연결을 완전히 정의하는 것은 불가능에 가깝다. 결과물을 보기 전에는 알 수 없었던 것이 가시화된 것이다. 이것은 실패가 아니라 정보 획득이다.

### B. 체크리스트 모음

**Layer 0 완료 기준 체크리스트**
```
□ "이 컨텍스트 없이 시작하면 어떤 잘못된 결정이 내려지는가?"에 답할 수 있다
□ 프로젝트 정체성 (무엇을, 왜)이 1-2문장으로 명시되어 있다
□ 절대 규칙 (변경 금지 사항, 부정 제약)이 조작 가능한 규칙으로 기록되어 있다
□ 아키텍처 레이어 구조가 명시되어 있다
□ workspace/tasks/INDEX.md 초안이 작성되어 있다 (작업 목록 + 의존성)
□ 공유 인터페이스가 사전 정의되어 있다
□ [기존 프로젝트] 현재 규칙과 목표 규칙이 분리되어 기록되어 있다
□ [기존 프로젝트] 불일치 해소 작업이 신규 기능과 별도로 분류되어 있다
```

**계획서 검토 질문 목록**
```
□ "빠진 부분이나 부족한 부분이 없어?"
□ "논리적 모순이 없어?"
□ "이 계획이 실패할 수 있는 시나리오를 찾아줘"
□ "내가 놓친 가정이 뭐야?"
□ "이 계획대로 구현하다가 내가 예상 못 한 복잡도나 결정이 필요한 지점이 어디야?"
□ "이 계획이 아키텍처 규칙을 위반하는 부분이 있어?"
```

**Layer 2 체크포인트 질문 목록**
```
□ "지금까지 구현된 코드 중 아키텍처 규칙을 위반한 부분이 있어?"
□ "작업 간 인터페이스가 충돌하거나 불일치하는 부분이 있어?"
□ "의도하지 않은 패턴이 생긴 부분이 있어?"
□ "규칙 문서 중 현실과 맞지 않아 업데이트가 필요한 것이 있어?"
```

### C. 용어 정의

**체계**: 협업 철학을 파일 시스템으로 인코딩한 재사용 가능한 협업 구조 패키지.

**inject 파일**: 에이전트에게 전달하는 세션 레시피. 페르소나 + 읽을 파일 목록 + 작업 지시 + 완료 기준 + 종료 보고로 구성된다. 새 스레드와 같은 스레드 모두에서 사용할 수 있으며, 스레드 선택 기준은 7.2 표를 따른다.

**workspace**: 프로젝트별 데이터가 쌓이는 공간. `memory/`(아키텍처 결정·도메인 규칙·역할 학습), `tasks/`(태스크 상태), `docs/`(결과 문서)로 구성된다.

**changelog.md**: 구현 세션에서 에이전트가 작성하는 진행 기록. 수정 파일·변경 내역을 담으며, 세션 재개 시 이어가기 지점을 파악하고 코드 검토 세션에서 검토 범위를 확인하는 데 쓰인다.

**completion bias**: 컨텍스트에 완성된 결과물이 있을 때 LLM이 그것을 정당화하는 방향으로 읽으려는 경향. 인간의 자아 보호와 다르다 — 심리가 아닌 확률 분포의 문제다. layer1_critique를 새 스레드로 분리하는 이유다.

**upgrade-candidates**: 운영 중 발견한 체계 개선 후보를 임시 저장하는 공간. 업데이트 시 체계에 반영되고 비워진다.

**조용한 결정(Silent Decision)**: 에이전트가 명세되지 않은 공간을 스스로 채우면서 그 결정을 명시하지 않는 현상.

**비가시적 위임(빈칸 B)**: 결정이 필요한 공간이 있었는지 몰랐던 빈칸. 에이전트가 조용히 채운다.

**의도적 위임(빈칸 A)**: 에이전트가 알아서 해주길 바라서 비워둔 빈칸. LLM을 쓰는 이유다.

**컨텍스트 오염**: 같은 스레드에서 생성과 평가를 함께 하면, 이전 대화 맥락이 평가의 객관성을 훼손하는 현상.

**문서 부패(Documentation Drift)**: 코드는 바뀌었는데 문서는 업데이트되지 않아 현실과 괴리가 생기는 현상. 낡은 문서는 없는 것보다 위험하다.

**탐지 역설**: 발견할 수 있는 빈칸은 이미 위험하지 않고, 위험한 빈칸은 발견할 수 없다는 구조적 한계.

**삭제 역설**: 침묵하는 규칙이 "잘 지켜지는 것"인지 "불필요한 것"인지 외부에서 구분할 수 없는 문제.

**구현 후 발견(Post-Implementation Discovery)**: 첫 결과물을 경험한 후 비로소 드러나는 요구사항. 동작 방식, UX, 기능 간 연결은 구현 전 명세만으로 완전히 정의하기 어렵고, 결과물이 계획서를 완성시키는 정보를 제공한다. 실패가 아니라 정보 획득이다.

**상상 간격(Imagination Gap)**: 요구사항을 명세할 때의 머릿속 이미지와 실제 구현물 사이의 불가피한 차이. 구현 전에는 좁힐 수 없고, 결과물을 본 후에야 좁혀진다. 구현 후 발견의 근본 원인이다.

**태스크 계보(Task Lineage)**: 구현 후 발견으로 생성된 태스크와 원래 태스크 사이의 양방향 연결. 두 방향 모두 기록한다:
- 새 태스크 plan.md `파생 출처` — "나는 어디서 왔는가"
- 원래 태스크 plan.md `파생된 태스크` — "거기서 무엇이 나왔는가"

설계·구현 세션에서 파생 출처와 파생된 태스크를 확인하면 이미 다른 곳에서 처리 중인 항목을 재검토하지 않을 수 있다. 에이전트가 조상 plan.md를 재귀적으로 읽는 메커니즘으로 사용하지 않는다 — 방향 연속성은 `direction.md`의 전진 증류로 유지한다.

**방향 메모리(Direction Memory)**: `workspace/memory/shared/direction.md`에 축적되는 제품/UX 방향 학습. `architecture.md`가 기술 결정(어떻게 만드는가)을 기록하듯, 이 파일은 결과를 경험하면서 발견된 "무엇을, 어떻게 느껴져야 하는가"를 기록한다. 모든 새 태스크 설계 세션에서 읽힌다.

### D. 분석 도구 모음

> 2부 각 관점의 상세 분석 도구. 본문은 핵심 개념만 담고, 판단이 필요할 때 여기서 참조한다.

#### 빈칸 품질 판단 (관점 5)

AI가 잘 채우는 빈칸: 훈련 데이터에 정답이 많고, 정답이 하나이며, 외부 맥락이 필요 없는 것.

| 범주 | 예시 |
|------|------|
| 형식·구조 | 변수명, 함수 구조, 파일 포맷 |
| 범용 지식 구현 | 정렬 알고리즘, HTTP 상태코드 |
| 형식 변환 | JSON→YAML, 언어 A→B |
| 완전 명세된 구현 | "이진 탐색 구현해줘" |

AI가 잘 못 채우는 빈칸: 정답이 비즈니스·시스템 맥락에 의존하는 것.

| 범주 | 예시 |
|------|------|
| 비즈니스 규칙 | "소프트/하드 삭제 중 뭘로?" |
| 트레이드오프 결정 | "동기 vs 비동기?" |
| 전체 시스템 일관성 | "에러 포맷을 어떻게?" |
| 보안·컴플라이언스 | "이 URL을 public으로?" |

**빈칸 품질 2×2 프레임**

```
                    컨텍스트 의존성
                    낮음            높음
                ┌───────────────┬───────────────┐
    정답의   낮음 │  위험 구간     │  최위험 구간   │
    명확성       │  (주의 필요)   │  (반드시 명세) │
                ├───────────────┼───────────────┤
              높음 │  안전 위임     │  주의 위임    │
                │  (AI에게 맡겨) │  (검증 필수)  │
                └───────────────┴───────────────┘
```

최위험 구간 패턴: 사용자가 모름 → AI도 맥락 없음 → AI가 일반적 패턴으로 채움 → 코드가 돌아감 → 나중에 요구사항과 충돌

#### 실패 비용 추정 프레임 (관점 6)

**실패 비용 2×2 (심각도 × 발견 가능성)**

```
                    발견 가능성
                    높음              낮음
              ┌─────────────────┬─────────────────┐
    심각도  높음 │  관리 가능       │  최위험          │
              │  (잡힐 테니까)   │  (잡히지도 않음) │
              ├─────────────────┼─────────────────┤
           낮음 │  무시 가능       │  수용 가능       │
              └─────────────────┴─────────────────┘
```

관점 1(실행 가능한 오해)은 왜 위험한가: 발견 가능성 낮음(코드가 돌아감) + 심각도 높음(핵심 로직) = 최위험 구간.

**자율성 4차원**

| 차원 | 기본 원칙 |
|------|-----------|
| 결정 자율성 | 실패 비용에 따라 조정 |
| 실행 자율성 | 가역적 작업에서만 허용 |
| 범위 자율성 | 거의 항상 낮게 유지 |
| 시간적 자율성 | 실패 비용 낮은 묶음에서만 |

**자율성 4단계**

```
레벨 1 — 제안:          AI가 옵션 제시, 인간이 결정 (가치 결정, 처음 하는 유형)
레벨 2 — 초안:          AI가 결과물 생성, 인간이 검토 후 적용 (기본값 권장)
레벨 3 — 실행 후 보고:  AI 실행 → 인간 사후 확인 (실패 비용 낮고 가역적)
레벨 4 — 완전 자율:     완전 명세 + 완전 가역 + 반복 검증된 경우만 (실제로 매우 드묾)
```
