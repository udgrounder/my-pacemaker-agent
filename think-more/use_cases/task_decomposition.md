# 작업 분해 전략
> 큰 프로젝트를 레이어 1 작업 단위로 나누는 방법
> 너무 크면 복잡도가 폭발하고, 너무 작으면 컨텍스트 단절이 생긴다

---

## 올바른 작업 단위란

```
좋은 작업 단위:
- "완료됐음"을 명확하게 확인할 수 있다
- 하나의 관심사(파일/모듈/기능)에 집중한다
- 독립적으로 테스트할 수 있다
- 하루 안에 구현 + 검증이 가능하다

나쁜 작업 단위:
- "사용자 관련 기능 전체 구현"
- 여러 레이어를 동시에 건드린다
- 완료 기준이 모호하다
```

---

## 분해 기준 3가지

### 1. 관심사 기준 (가장 기본)

하나의 작업 = 하나의 관심사

```
잘못된 분해:
작업 1: "로그인 기능 구현" (DB 스키마 + API + 프론트엔드 + 세션 관리 포함)

올바른 분해:
작업 1: "users 테이블 스키마 정의"
작업 2: "POST /auth/login API 구현"
작업 3: "세션 토큰 발급 로직 구현"
작업 4: "로그인 폼 UI 구현"
```

### 2. 의존성 기준

의존되는 것(인터페이스)을 먼저 정의한다. 구체적인 구현은 나중에.

```
잘못된 순서:
작업 1: 프론트엔드 API 호출 코드 작성
작업 2: 백엔드 API 설계
→ 백엔드가 확정되기 전에 프론트가 만들어져서 나중에 충돌

올바른 순서:
작업 0: API 인터페이스 정의 (입출력 형식 확정)
작업 1: 백엔드 API 구현 (정의된 인터페이스 기준)
작업 2: 프론트엔드 연동 (확정된 인터페이스 기준)
```

skill_dependency_mapping.md를 사용하면 의존성 순서를 체계적으로 파악할 수 있다.

### 3. 완료 기준 기준

"어떻게 하면 이 작업이 완료됐다고 할 수 있는가"를 먼저 정의한다.

```
모호한 완료 기준:
"로그인 기능이 동작한다"

명확한 완료 기준:
- POST /auth/login에 유효한 자격증명을 보내면 200 + 세션 토큰 반환
- 잘못된 자격증명을 보내면 401 반환
- 만료된 토큰으로 요청하면 403 반환
- 기존 로그아웃 기능이 그대로 동작한다
```

## 분해 프로세스

```
1. 전체 기능을 자연어로 나열
2. skill_dependency_mapping.md 프롬프트로 의존성 분석
3. 의존성 순서에 따라 정렬
4. 각 작업에 완료 기준 정의
5. 너무 큰 작업은 추가 분해
```

---

## 작업 크기 판단 기준

| 신호 | 대응 |
|------|------|
| task_plan.md의 구현 항목이 5개를 넘는다 | 추가 분해 필요 |
| "A를 구현하고 B도 수정한다"는 표현이 있다 | 두 작업으로 분리 |
| 완료 기준이 AND로 연결된다 | 각 조건마다 하나씩 분리 고려 |
| 3개 이상의 파일을 동시에 수정한다 | 공유 인터페이스 먼저 정의 후 분리 |

---

## 공유 인터페이스 먼저 정의하기

여러 작업이 같은 모듈/인터페이스를 사용할 때, 인터페이스를 먼저 별도 작업으로 확정한다.

```
공유 인터페이스가 필요한 신호:
- 두 작업이 같은 데이터 모델을 사용한다
- 두 작업이 같은 API를 호출/제공한다
- 두 작업이 같은 유틸리티 함수를 사용한다

처리 방법:
1. 인터페이스 정의 작업을 먼저 분리
2. 인터페이스 확정 → architecture_rules.md에 추가
3. 이후 작업들은 확정된 인터페이스를 기준으로 구현
```

## 레이어 0과의 관계

```
레이어 0 완료 후:
architecture_rules.md에 전체 구조가 정의돼 있다
        ↓
이 구조를 기준으로 작업 목록을 도출한다
skill_dependency_mapping.md로 의존성 순서 정렬
        ↓
레이어 1 작업 목록 확정
각 작업마다 task_plan.md 작성 후 진행
```

## 자주 묻는 질문

**Q: 분해를 너무 잘게 하면 오히려 비효율적이지 않나?**

각 작업마다 스레드가 필요하고 컨텍스트를 다시 주입해야 한다. 분해가 너무 잘으면 overhead가 쌓인다. 기준은 "독립적으로 검증할 수 있는 최소 단위" — 이보다 작게 자르면 오히려 손해다.

**Q: 예상보다 작업이 커지면 어떻게 하나?**

구현 중에 "구현 현실 질문"으로 조기에 감지한다:
"이 계획대로 구현하다가 내가 예상 못 한 복잡도나 결정이 필요한 지점이 어디야?"
작업이 커진다고 느끼면 task_plan.md를 수정하거나 작업을 추가 분해한다.

**Q: 작업 목록을 처음부터 완벽하게 만들어야 하나?**

아니다. 처음에는 큰 단위로 나누고, 각 작업에 가까워질 때 추가 분해한다. 나중에 진행할 작업까지 지금 완벽하게 분해할 필요 없다.
