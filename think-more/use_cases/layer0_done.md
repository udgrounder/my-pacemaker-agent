# 레이어 0 완료 기준
> 언제 레이어 0이 충분히 완료됐고 레이어 1을 시작해도 되는가
> "완벽한 설계"를 기다리다 시작을 못 하는 것도 실패다

---

## 핵심 원칙

레이어 0은 완벽할 필요 없다.
"첫 번째 작업을 시작할 수 있을 만큼" 완성되면 충분하다.

레이어 0은 고정된 문서가 아니라 살아있는 문서다.
레이어 1 작업과 레이어 2 체크포인트를 거치며 점점 정교해진다.

---

## 완료 판단 체크리스트

아래 항목이 모두 YES이면 레이어 1을 시작해도 된다.

### 필수 항목

- [ ] **기술 스택이 확정됐는가**
  - 언어, 프레임워크, 주요 라이브러리
  - 나중에 바꾸기 어려운 것들만 확정하면 충분

- [ ] **전체 구조가 그림으로 그려지는가**
  - 주요 모듈/레이어가 몇 개인지
  - 어떤 순서로 데이터가 흐르는지
  - 코드로 옮기기 전에 말로 설명할 수 있어야 함

- [ ] **첫 번째 작업의 완료 기준이 명확한가**
  - "이게 완료되면 다음 작업으로 넘어간다"를 정의할 수 있어야 함

- [ ] **건드리면 안 되는 것이 정의됐는가**
  - 레거시 인터페이스, 외부 시스템 의존, 공유 모듈 등

### 권장 항목 (없어도 시작 가능)

- [ ] 전체 작업 목록이 초안으로 있는가
- [ ] 주요 데이터 모델 구조가 정의됐는가
- [ ] 외부 시스템 인터페이스가 파악됐는가

---

## "아직 모르는 것"과 "나중에 알아도 되는 것" 구분

| 지금 확정해야 하는 것 | 나중에 확정해도 되는 것 |
|---------------------|----------------------|
| 공유 인터페이스 (여러 작업이 함께 쓰는 것) | 특정 작업 내부 구현 방식 |
| 외부 시스템 연동 방식 | UI 디테일 |
| 데이터 모델 핵심 구조 | 성능 최적화 방법 |
| 인증/권한 방식 | 에러 메시지 문구 |

"지금 모른다"고 레이어 1을 미루지 마라. "지금 안 확정하면 첫 작업을 시작할 수 없다"는 것만 확정하면 된다.

---

## 기존 프로젝트의 특수 케이스

기존 코드베이스에 적용할 때는 "완벽한 이해"를 기다리지 않는다.

```
최소 레이어 0 기준 (기존 프로젝트):
1. 수정할 파일과 영향받는 파일 파악 완료
2. 건드리면 안 되는 인터페이스/공유 모듈 파악 완료
3. 기존 패턴 (코딩 컨벤션, 에러 처리 방식) 파악 완료
```

기존 코드 전체를 이해하려는 것은 비현실적이다.
작업에 필요한 범위만 파악하면 시작할 수 있다.

---

## 레이어 0이 불충분할 때의 신호

레이어 1을 진행하다 아래 신호가 보이면 레이어 0으로 돌아가야 한다.

| 신호 | 의미 |
|------|------|
| 작업 간 인터페이스가 계속 바뀐다 | 공유 인터페이스가 미확정 |
| 앞 작업과 뒤 작업이 충돌한다 | 전체 구조 정의 부족 |
| "이 부분은 어떻게 해야 할지 모르겠다"가 반복된다 | architecture_rules.md 보완 필요 |
| 레이어 2 체크포인트에서 같은 문제가 반복된다 | 아키텍처 규칙 재정의 필요 |

---

## 레이어 0 → 레이어 1 전환 흐름

```
레이어 0 완료 판단
        ↓
architecture_rules.md 초안 완성
        ↓
첫 번째 작업 선택
(의존성이 없는 것, 또는 다른 작업들이 의존하는 것)
        ↓
레이어 1 시작
task_plan.md 작성 → workflow_new_feature.md 따라가기
        ↓
레이어 1 진행 중 레이어 0 보완 필요 발견 시
→ architecture_rules.md 업데이트 후 계속
```
