# 실패 패턴 모음
> AI agent와 작업할 때 자주 발생하는 실패 패턴과 방지 방법
> 이 패턴들은 "겪고 나서야 아는" 것들 — 미리 읽으면 반복을 피할 수 있다

---

## 패턴 1: 컨텍스트 오염 검토

**증상:** 구현이 끝난 스레드에서 "검토해줘"라고 요청했는데, agent가 문제를 잘 찾지 못하거나 자기가 만든 코드를 옹호한다.

**원인:** agent가 구현 과정에서 내린 결정들이 컨텍스트에 쌓여 있어서 객관적 평가가 어렵다.

**방지:**
```
구현 완료 후 → 반드시 새 스레드 열기
새 스레드에 → 코드만 붙여넣기 (대화 맥락 제외)
페르소나 주입 → persona_code_reviewer.md
검토 방법 → skill_path_tracing + skill_counterexample_finding 프롬프트 사용
```

---

## 패턴 2: 조용한 결정 누적

**증상:** 구현 완료 후 결과물을 보니 예상과 다른 부분이 있다. agent가 물어보지 않고 임의로 결정한 것들이 쌓여있다.

**원인:** 계획서에 명시되지 않은 부분에서 agent가 "합리적으로 보이는" 결정을 혼자 내린다.

**방지:**
```
계획서 작성 후 → skill_silent_decision_extraction.md 프롬프트 실행
"이 계획에서 내가 명시하지 않아서 agent가 임의로 결정할 수 있는 지점이 어디야?"
→ 나온 항목들을 계획서에 명시적으로 추가
```

---

## 패턴 3: 범위 이탈 구현

**증상:** 요청한 기능 외에 추가 기능이 붙어있거나, 건드리지 말아야 할 파일이 수정됐다.

**원인:** 구현 요청에 "건드리면 안 되는 것"이 명시되지 않았거나, "더 좋게" 만들려는 agent의 경향.

**방지:**
```
task_plan.md 섹션 3 (건드리면 안 되는 것) 반드시 작성:
- 수정 대상 파일 명시 (목록 외 파일 수정 금지)
- 인터페이스(입출력) 변경 금지
- 기존 동작 변경 금지
```

---

## 패턴 4: 레이어 0 없이 구현 시작

**증상:** 작업이 진행될수록 앞선 작업과 충돌이 발생한다. 인터페이스가 맞지 않거나, 같은 부분을 여러 작업이 다르게 구현한다.

**원인:** 전체 설계 기준(architecture_rules.md)이 없는 상태에서 개별 작업이 각자 최적화됐다.

**방지:**
```
첫 번째 작업 시작 전 → 반드시 레이어 0 완료
architecture_rules.md 작성 → 이후 모든 구현 요청에 포함
"아래 규칙을 반드시 따라서 구현해줘: [architecture_rules.md 내용]"
```

---

## 패턴 5: "돌아가면 맞다" 착각

**증상:** 코드가 실행되고 기본 케이스는 통과하는데, 특정 조건에서 의도와 다른 동작이 발생한다.

**원인:** 구조적 검증(코드가 실행된다)과 논리적 검증(의도대로 동작한다)을 혼동한다.

**방지:**
```
결과 확인 시 반드시 경로 추적:
정상 경로: "이 시나리오가 실행됐을 때 각 단계에서 어떤 값이 전달되는가"
실패 경로: "이 에러가 발생했을 때 어떻게 처리되는가"
→ skill_path_tracing.md 프롬프트 사용
```

---

## 패턴 6: 동조적 계획 검토

**증상:** 계획서를 검토받았는데 대부분 "좋습니다, 잘 작성됐네요"로 끝난다. 실제 구현에서 예상치 못한 문제가 발생한다.

**원인:** 같은 스레드에서 계획을 세운 agent에게 검토를 요청하거나, 검토 요청이 너무 열려있다.

**방지:**
```
계획 검토는 반드시 새 스레드
페르소나: persona_plan_critic.md (약점만 찾는 역할)
요청 방식: "이 계획이 실패할 수 있는 시나리오를 찾아줘"
             (좋은 점을 말해줘 X, 칭찬 X)
```

---

## 패턴 7: 피드백 기록 생략

**증상:** 같은 종류의 실수가 반복된다. 이전 작업에서 발견한 문제를 다음 작업에서도 다시 발견한다.

**원인:** 작업이 끝나고 나서 피드백 기록(feedback_log.md)을 쓰지 않거나, architecture_rules.md를 업데이트하지 않는다.

**방지:**
```
모든 레이어 1 작업 마지막에:
1. feedback_log.md 작성 (template_Layer1_Step5_feedback_log.md)
2. "다음 작업에 반영할 내용"이 있으면 architecture_rules.md 업데이트
3. 패턴이 보이면 이 파일(failure_patterns.md)에 추가
```

---

## 패턴 8: 컨텍스트 길이 초과 후 계속 진행

**증상:** 대화가 길어질수록 앞에서 정한 규칙이나 계획서 내용을 agent가 잊어버린다. 뒤늦게 모순된 구현이 발견된다.

**원인:** 컨텍스트 창 한계로 인해 초반에 주입된 정보가 희석되거나 잘린다.

**방지:**
```
구현 스레드가 길어질 때:
→ 새 스레드 열기
→ architecture_rules.md + task_plan.md 다시 주입
→ "지금까지 구현된 코드 요약" + 이어서 구현 요청

레이어 2 체크포인트가 이 역할을 주기적으로 수행
```

## 패턴 9: 리팩토링 중 기능 변경

**증상:** 리팩토링 후 기존 테스트가 일부 실패한다. 코드 구조는 개선됐지만 엣지 케이스 처리가 달라졌다.

**원인:** 추상화 과정에서 엣지 케이스나 에러 처리 코드가 누락되거나 다르게 처리된다.

**방지:**
```
리팩토링은 workflow_refactoring.md 따라가기
핵심: 리팩토링 전후 경로 추적 비교
"정상 경로, 실패 경로, 엣지 케이스 경로가 리팩토링 전후에 동일한가?"
기존 테스트 변경 금지 (추가는 허용)
```

---

## 패턴 10: 너무 큰 작업 단위

**증상:** task_plan.md 하나에 기능이 너무 많다. 구현 중에 복잡도가 예상보다 크게 늘어난다. 어디서 막혔는지 파악하기 어렵다.

**원인:** 작업을 충분히 분해하지 않고 한 번에 큰 덩어리로 요청한다.

**방지:**
```
작업 분해 기준 (task_decomposition.md 참고):
→ 하나의 task_plan.md는 하루 안에 구현 가능한 크기
→ "이 작업이 완료됐음"을 명확히 확인할 수 있는 크기
→ 하나의 관심사(파일/모듈/기능)에 집중하는 크기
```
