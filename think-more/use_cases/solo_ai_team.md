# 혼자서 AI 팀과 개발하기
> 1인 개발자가 여러 AI agent를 팀처럼 운영하는 방법
> 핵심: 사람은 "오케스트레이터(조율자)" — agent들은 각각 전문가 역할

---

## 이 방식이 의미하는 것

```
전통적인 1인 개발:
  나 → 설계 → 구현 → 검토 → 반복

AI 팀과의 1인 개발:
  나(오케스트레이터)
    ↓ 지시
  설계 agent (persona_architect)
    ↓ 결과 받아서
  나 → 판단/결정
    ↓ 지시
  구현 agent (persona_implementer)
    ↓ 결과 받아서
  검토 agent (persona_code_reviewer)
    ↓ 결과 받아서
  나 → 최종 판단
```

agent를 "AI 팀원"으로 대하는 것의 핵심은:
각 agent가 **다른 관점**을 갖도록 스레드와 페르소나를 분리하는 것이다.

---

## 나의 역할: 오케스트레이터

AI 팀과 일할 때 내가 해야 하는 일과 하지 않아도 되는 일이 달라진다.

| 내가 해야 하는 것 | agent에게 맡기는 것 |
|-----------------|-------------------|
| 무엇을 만들지 결정 | 어떻게 구현할지 |
| 완료 기준 정의 | 구체적인 코드 작성 |
| agent 결과물 판단/수용 여부 | 경로 추적, 반례 탐색 |
| 작업 순서 결정 | 계획의 약점 찾기 |
| 최종 책임 | 검토 리포트 작성 |

**오케스트레이터의 핵심 역량: agent 결과물을 판단하는 능력**
"agent가 뭘 만들었는가"보다 "agent가 만든 게 맞는가"를 판단하는 것이 내 주요 역할이다.

```

─────────────────────────────────────────────────

구현자        persona_implementer        ♻️ 같은 스레드
코드 검토자   persona_code_reviewer      🆕 새 스레드
```

### 확장 팀 (레이어 0 + 레이어 2 포함)

```
역할              페르소나                       스레드
────────────────────────────────────────────────────────
아키텍트          persona_architect              🆕 새 스레드
코드베이스 분석가  persona_codebase_analyst       🆕 새 스레드
설계 비평가       persona_plan_critic            🆕 새 스레드
구현자            persona_implementer            ♻️ 같은 스레드
코드 검토자       persona_code_reviewer          🆕 새 스레드
통합 감사자       persona_integration_auditor    🆕 새 스레드
```

## 작업 흐름: 새 기능 한 사이클

### 1단계 — 나: 계획서 작성

혼자 또는 구현 스레드에서 작성한다.

```
작성 도구: template_Layer1_Step1_task_plan.md
핵심 섹션:
- 무엇을 만드는가
- 완료 기준
- 건드리면 안 되는 것 (범위 제한)
- 조용한 결정 목록 (skill_silent_decision_extraction 사용)
```

### 2단계 — plan_critic에게: 계획 검토

```
[새 스레드 열기]
(persona_plan_critic.md 내용 붙여넣기)

[컨텍스트]
(architecture_rules.md 붙여넣기)

[검토 요청]
(task_plan.md 붙여넣기)
"이 계획이 실패할 수 있는 시나리오를 찾아줘.
좋은 점은 말하지 말고 약점과 빠진 것만 알려줘."
```

**내 역할:** 비평 결과를 받아서 계획서에 반영할지 판단한다.  
전부 반영할 필요 없다. "이건 지금 범위 밖" 판단도 내가 한다.

### 3단계 — implementer에게: 구현

```
[새 스레드 열기 또는 계획 스레드 계속]
(persona_implementer.md 내용 붙여넣기)

[컨텍스트]
(architecture_rules.md 붙여넣기)
(수정된 task_plan.md 붙여넣기)

[구현 요청]
"위 계획대로 구현해줘.
확신이 없는 결정이 생기면 구현하지 말고 물어봐."
```

**내 역할:** 구현 중 agent가 물어오는 것에 답한다. 이것이 "조용한 결정"을 방지하는 포인트다.

### 4단계 — code_reviewer에게: 결과 검토

```
[새 스레드 열기]
(persona_code_reviewer.md 내용 붙여넣기)

[검토 대상]
(구현된 코드 붙여넣기)

[검토 요청 - skill_path_tracing 사용]
"아래 경로들을 추적해줘:
- 정상 경로: [핵심 시나리오]
- 실패 경로: [에러 시나리오]
의도한 동작과 다른 부분이 있으면 알려줘."
```

**내 역할:** 검토 결과를 받아서 수정 여부를 판단한다. 검토자의 모든 지적을 수정할 필요 없다.

### 5단계 — 나: 실행 + 피드백 기록

직접 실행해서 확인하고 `template_Layer1_Step5_feedback_log.md` 작성.

## 스레드 관리 실전

혼자 여러 agent를 운영할 때 스레드가 많아진다. 관리 방법:

### 스레드 명명 규칙 (권장)

```
[날짜]-[작업명]-[역할]
예시:
0522-login-api-plan_critic
0522-login-api-implementer
0522-login-api-code_reviewer
```

### 언제 스레드를 닫는가

```
계획 검토 스레드 → 피드백 받고 계획서 수정 완료 시
구현 스레드      → 구현 완료 + code_reviewer 검토 완료 시
검토 스레드      → 검토 결과 정리 완료 시
```

검토 스레드는 재사용하지 않는다. 항상 새로 만든다.

## 판단 기준: 언제 다시 시작하는가

### agent 결과를 수용할지 판단하는 기준

```
수용:
- 계획서의 완료 기준을 충족한다
- 건드리면 안 되는 것을 건드리지 않았다
- 내가 경로를 추적했을 때 의도와 일치한다

재요청:
- 완료 기준 중 일부가 빠졌다
- 범위를 벗어난 변경이 있다
- 에러 처리가 빠진 경로가 있다

거부 + 재설계:
- 전체 구조가 계획과 다르다
- 공유 인터페이스를 변경했다
- 무엇이 잘못됐는지 파악하기 어렵다 → 더 작게 분해
```

### agent가 막히는 신호

```
같은 요청을 반복해서 비슷한 틀린 결과가 나온다
→ 계획서 자체가 불명확한 것 → 내가 더 명확하게 재작성

agent가 계속 물어본다
→ 조용한 결정 포인트가 있는 것 → 명시적으로 결정 후 전달

결과가 매번 다르다
→ 범위가 너무 크거나 제약이 불명확한 것 → 작업 분해
```

## 효율 극대화 팁

### 팁 1: 컨텍스트 재사용 패턴

자주 쓰는 컨텍스트 조합을 미리 준비해둔다.

```
[구현 요청 스타터 팩]
- persona_implementer.md
- architecture_rules.md
- task_plan.md 템플릿

[검토 요청 스타터 팩]
- persona_code_reviewer.md
- skill_path_tracing.md의 프롬프트
- skill_counterexample_finding.md의 프롬프트
```

### 팁 2: 한 사이클에서 배운 것을 다음에 활용

```
작업 완료 후:
1. feedback_log.md 작성
2. "다음 작업에서 이 agent에게 처음부터 알려줘야 할 것"을 정리
3. architecture_rules.md 업데이트
```

### 팁 3: 구현 현실 질문을 먼저 던지기

계획서 작성 후, 구현 전에:

```
구현 스레드에서:
"이 계획대로 구현하다가 내가 예상 못 한 복잡도나
결정이 필요한 지점이 어디야?"

→ 나온 항목들을 계획서에 미리 명시
→ 구현 중 막히는 빈도가 크게 줄어든다
```

### 팁 4: 검토는 짧고 구체적으로

```
❌ "이 코드 전체를 검토해줘"
✅ "이 함수에서 입력값이 null일 때 어떤 경로로 처리되는가"

❌ "버그가 있을 수 있어?"
✅ "동시 요청 2개가 들어왔을 때 이 코드가 어떻게 동작하는가"
```

구체적인 시나리오를 주면 agent의 동조 경향을 차단하고 실제 문제를 찾아낸다.

## 자주 빠지는 함정

| 함정 | 증상 | 방지 |
|------|------|------|
| agent 과신 | 결과를 대충 보고 넘어간다 | 완료 기준 하나씩 직접 확인 |
| 스레드 재활용 | 검토를 구현 스레드에서 한다 | 검토는 항상 새 스레드 |
| 컨텍스트 과부하 | 스레드에 모든 걸 다 넣는다 | 평가 스레드엔 최소한만 |
| 결정 위임 | "어떻게 할까요?"를 agent에게 묻는다 | 내가 결정하고 구현만 맡긴다 |
| 완벽주의 | 모든 비평을 반영하려 한다 | 우선순위 판단은 내가 한다 |

## 하루 작업 흐름 예시

```
오전:
1. 어제 feedback_log.md 복기
2. 오늘 작업 task_plan.md 작성 (30분)
3. plan_critic 스레드에서 계획 검토 (15분)
4. 피드백 반영해서 계획서 수정 (15분)

오후:
5. implementer 스레드에서 구현 (1~2시간)
   - 막히면 바로 질문 → 기다리지 말고 물어본다
6. code_reviewer 스레드에서 경로 추적 검토 (30분)
7. 직접 실행 확인 (30분)

마무리:
8. feedback_log.md 작성 (15분)
9. architecture_rules.md 업데이트 필요 여부 확인
```
