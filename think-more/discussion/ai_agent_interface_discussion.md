# AI Agent 인터페이스 & 소통 방식 — 방향성 논의

> 작성일: 2026-05-22

---

## 1. 본질적 목표

AI agent를 사용하는 사람의 핵심 목적:
> **원하는 작업을 정확하고 저렴하게**

멀티 에이전트, 페르소나 주입, RAG 등 다양한 기법이 나오고 있지만, 이것들은 모두 **수단의 레이어**다. 빠르게 나왔다 사라지는 이유는 이 수단들이 아직 "정확하고 저렴하게"를 완전히 해결하지 못하고 있기 때문. 사용자 입장에서 이 레이어를 쫓아다닐 필요는 없다.

---

## 2. 사용자가 가져야 할 핵심 관점

### 관점 1. "무엇을 원하는가"보다 "어떻게 검증할 것인가"
- 결과가 맞는지 확인하는 것이 실제로 어려운 부분
- 검증 기준이 명확하지 않으면 정확도 측정도, 비용 최적화도 불가능

### 관점 2. 사람이 개입해야 하는 지점을 설계하는 것
- Agent에게 전부 맡기거나, 너무 세세하게 개입하는 것 둘 다 비효율
- "어느 단계에서 내가 판단해야 하는가"를 먼저 정하는 것이 인터페이스 설계의 본질

### 핵심 프레임
> Agent를 도구로 보지 말고, **위임 구조를 설계하는 행위**로 볼 것
> 내가 어떤 업무를 어떤 기준으로 넘길 수 있는지를 명확히 하는 역량이 핵심

## 3. 검증 설계

### 검증의 딜레마
> "결과를 완전히 검증할 수 있다면, 처음부터 내가 할 수 있었던 것 아닌가?"

검증 설계는 "전부 확인"이 아니라 **"어디서 어느 정도 확인할 것인가"를 정하는 일**

### 검증의 3 레이어

| 레이어 | 내용 | 특징 |
|--------|------|------|
| **구조 검증** | 형식이 맞는가, 항목이 빠지지 않았는가 | 자동화 가능 |
| **논리 검증** | 의도한 대로 동작하는가 | 사람 개입 필수, 비용 높음 |
| **예외 검증** | 엣지 케이스, 의도치 않은 부작용 | Agent가 "그럴듯하게" 틀리는 영역 |

### 실용적 접근
- **작업을 검증 가능한 단위로 쪼갠다** — 블랙박스가 아닌 체크포인트 구조
- **기준을 시작 전에 정한다** — "좋은 결과물"의 정의를 작업 지시 시점에 명시
- **위험도 기반으로 검증 깊이를 다르게 한다** — 파급효과가 큰 것만 깊이 검증

---

## 4. 논리 검증 심화

### 핵심 문제: "그럴듯하게 틀리는" 현상
Agent 출력물은 형식도 맞고, 문장도 자연스럽고, 자신감도 있다. 틀렸을 때 티가 잘 안 난다.

### 두 가지 검증 관점
- **"무엇을 했는가" 검증** → 결과물 확인 (불충분)
- **"왜 그렇게 했는가" 검증** → 이유, 전제, 비교 대안 확인 (핵심)

### 실용적인 논리 검증 방법

1. **반례 제시** — "이런 경우는 어떻게 되는가?" → Agent 스스로 구멍을 발견하게 유도
2. **대안 요청** — "다른 방법은 없는가?" → 비교 과정을 통해 논리 견고성 확인
3. **요약 역전** — 출력물을 다시 agent에게 요약시켜 의도와 실제의 간격 확인

### 핵심 태도
> 논리 검증은 결과를 **읽는** 것이 아니라 **심문하는** 것
> "이게 왜 맞지?"가 아니라 "이게 왜 틀릴 수 있지?"를 먼저 생각하는 습관

---

## 5. 설계 시 검증 vs 결과 검증

### 개발자 관점의 핵심 인사이트
설계 시 검증이 **더 레버리지가 높다.** 앞에서 잘 정의하면 뒤에서 심문할 일이 줄어든다.

### 그러나 설계만으로 안 되는 이유: 의도 간격(Intent Gap)

```
내가 명세한 것
    ↓  (간격 발생 가능)
Agent가 이해한 것
    ↓  (간격 발생 가능)
Agent가 실제로 한 것
```

이 세 단계가 일치하지 않는 경우가 생각보다 많으며, 설계만으로는 이 간격을 미리 알 수 없다.

### 실용적 순서

```
설계 시 → 기준을 명확히 (주 작업, 레버리지 높음)
결과 시 → 의도 간격이 있는지 확인 (보조 작업)
```

### 피드백 루프
결과 심문에서 반복되는 오류 패턴이 발견되면 → **설계 단계로 끌어올려서 처음부터 막는 구조**가 이상적

## 6. 개발 특화: 코딩 Assistant 워크플로우

### 사용자 현재 워크플로우
```
계획서 작성 및 검토 → 구현 요청 → 결과 확인 → 실행 → 완료
```
설계 시 검증을 이미 포함하고 있는 좋은 구조.

### 코딩 Assistant 특유의 검증 문제
코드는 **"돌아간다"와 "맞다"가 분리된다.**
구조 검증(컴파일, 실행)은 통과하지만 논리 검증은 실패하는 전형적인 패턴.

### 의도 간격이 주로 발생하는 두 지점

**취약 지점 1: 계획서 → 구현 요청 사이**
계획서에 없는 내용을 agent가 조용히 채우는 순간 의도 간격이 생긴다.
- 어떤 패턴을 쓸지
- 예외 처리를 어디까지 할지
- 어떤 가정을 깔고 갈지

**취약 지점 2: 결과 확인 단계**
"코드가 돌아가면 넘어가는" 수준이면 논리 검증이 빠진 것.

## 7. 계획서 검토: Agent 동조 경향(Sycophancy) 문제

### 문제
Agent는 기본적으로 계획을 **검증하려는 방향**으로 읽는다.
"이 계획 어때?" → "좋아 보이는데 이런 점 고려하면 좋겠어요" 수준의 답이 나옴.
진짜 논리 검증이 안 되고 있는 것.

### 질문 방식 바꾸기

| 약한 질문 | 강한 질문 |
|-----------|-----------|
| "이 계획 괜찮아?" | "이 계획이 실패할 수 있는 시나리오를 찾아줘" |
| "검토해줘" | "내가 놓친 가정이 뭐야?" |
| "문제 없어?" | "이 계획에 반박해봐" |

### 혼자 검토 + Agent 검토의 이상적인 역할 분리
```
혼자 검토   → 내 의도가 명확한가
Agent 검토  → 내가 못 보는 구멍이 있는가 (동의가 아닌 반례를 구하는 것)
```

## 8. 구현 현실 질문 (핵심 추가 기법)

### 기존 계획서 검토 질문 (사용자 현재 방식)
- "부족한 부분이나 빠진 부분이 없는지"
- "논리적 모순이 없는지"

→ 계획서 **내부의 논리**를 검토하는 것. 완성도, 일관성 중심.

### 추가할 질문
> "이 계획대로 구현하다가 내가 예상 못 한 복잡도나 결정이 필요한 지점이 어디야?"

→ 계획서에는 있지만 **agent가 조용히 결정해버릴 것**을 미리 끄집어내는 질문.

### 차이점
```
기존 질문  →  계획서에 없는 것을 찾아줌
새 질문    →  계획서에 있지만 agent가 임의로 채워버릴 것을 찾아줌
```

### 구체적 사례: 프로필 이미지 업로드 기능

**계획서:**
```
- 사용자가 이미지 파일을 선택한다
- 서버에서 S3에 업로드한다
- 업로드된 URL을 DB에 저장한다
- 프로필에 이미지를 표시한다
```

**기존 질문으로 잡히는 것:**
- 파일 크기 제한 없음
- 이미지 형식 검증 없음
- 에러 처리 없음

**구현 현실 질문으로 추가로 잡히는 것:**
- S3 업로드 성공 + DB 저장 실패 시 → 트랜잭션 처리 방식 결정 필요
- 기존 프로필 이미지 S3 삭제 여부 → 결정 없으면 agent가 조용히 선택함
- 이미지 리사이징 위치 (클라이언트/서버/S3 Lambda) → agent가 임의로 선택할 수 있음
- URL 방식 (public vs presigned) → 보안 정책인데 agent가 그냥 public으로 만들 수 있음

이것들이 무서운 이유: **결과 코드가 돌아가기 때문에 나중에야 발견된다.**

## 9. 결과 확인 단계 논리 검증 구체화

### 가장 흔한 함정
```
코드 받음 → 훑어봄 → 실행해봄 → 됨 → 완료
```
이건 구조 검증만 한 것. **"돌아간다"와 "맞다"는 다르다.**

### 방법 1: 계획서 검토에서 뽑은 "조용한 결정 목록"을 체크리스트로 사용
계획서 단계에서 "구현 현실 질문"으로 추출한 항목들을 결과 확인 시 직접 확인한다.
```
계획서 검토 시 발견          →   결과 확인 시 체크
S3/DB 트랜잭션 처리          →   코드에서 어떻게 처리됐어?
이미지 삭제 여부             →   기존 이미지 삭제 로직 있어?
URL 방식 (public/presigned)  →   뭘 골랐어? 왜?
```

### 방법 2: Agent에게 자기 코드 심문하기

| 질문 | 목적 |
|------|------|
| "이 코드가 실패할 수 있는 경우를 모두 말해줘" | 예외 경로 확인 |
| "이 코드가 가정하고 있는 것이 뭐야?" | 숨겨진 전제 드러내기 |
| "이 부분을 왜 이렇게 구현했어?" | 조용한 결정 확인 |
| "이 코드의 테스트 케이스를 작성해줘" | Agent가 뭘 검증하는지 드러냄 |

테스트 케이스 요청이 특히 유용: **agent가 테스트 케이스를 뭘 쓰는지가 agent가 실제로 뭘 검증하고 있는지를 드러낸다.**

### 방법 3: 경로 추적 (전체 읽기 대신)
```
1. 정상 경로 하나를 머릿속으로 따라간다
2. 실패 경로 하나를 따라간다 (네트워크 오류, DB 오류 등)
3. 그 경로에서 의도한 대로 동작하는지만 확인한다
```
전체를 검토하는 것보다 하나의 경로를 깊이 보는 것이 논리 오류를 더 잘 잡는다.

### 계획서-결과 피드백 루프
```
계획서 검토 → "조용한 결정 목록" 추출
     ↓
결과 확인   → 그 목록 체크 + 경로 추적 + agent 심문
     ↓
반복 패턴   → 다음 계획서에 명시적으로 포함
```

## 10. 프로젝트 전체 절차: 3 레이어 구조

### 문제: 컨텍스트 단절

대규모 프로젝트에서 작업을 세분화할 때 발생하는 현상:

> 각 작업은 올바르게 완료됐는데, 전체를 합치면 구조가 무너진다

Agent는 각 작업을 **지금 주어진 문맥**으로만 풀기 때문에, 전체 아키텍처 사상·기존 규칙·다른 작업과의 관계를 모른 채 "로컬 최적화"를 해버린다.

```
작업 A  →  A 기준으로 최적 구현
작업 B  →  B 기준으로 최적 구현
작업 C  →  C 기준으로 최적 구현

전체   →  각자 최적이지만 서로 충돌, 규칙 위반
```

**근본 원인:** 아키텍처 규칙이 암묵적으로만 존재하기 때문.
Agent에게 전달되지 않은 규칙은 존재하지 않는 것과 같다.

### 해결 방향

- **레이어 0**: 암묵적 규칙을 명시적으로 꺼내 문서화 (1회)
- **레이어 1**: 매 작업마다 규칙을 컨텍스트로 주입 (반복)
- **레이어 2**: 주기적으로 전체 정합성 확인 (N개 작업마다)

```
┌─────────────────────────────────────────────┐
│  레이어 0: 프로젝트 초기화 (1회)              │
│  아키텍처 규칙 문서화 + 작업 분해             │
└─────────────────────────────────────────────┘
        ↓ 아키텍처 규칙을 매 작업에 주입
┌─────────────────────────────────────────────┐
│  레이어 1: 개별 작업 반복                    │
│  기존 1~5단계 + 규칙 컨텍스트 주입           │
└─────────────────────────────────────────────┘
        ↓ N개 작업마다
┌─────────────────────────────────────────────┐
│  레이어 2: 통합 체크포인트                   │
│  전체 정합성 확인 + 규칙 위반 여부 검토       │
└─────────────────────────────────────────────┘
```

---

### 레이어 0: 프로젝트 초기화

프로젝트 시작 시 1회 수행. 이후 모든 작업의 기준이 되는 문서와 구조를 만드는 단계.

#### 0-1. 아키텍처 규칙 문서 작성

암묵적으로만 존재하는 규칙을 명시적으로 꺼내 문서화한다.
이 문서는 매 작업 시 agent에게 컨텍스트로 주입된다.

**문서에 담아야 할 내용**

| 항목 | 내용 |
|------|------|
| 레이어 구조 | 어떤 레이어가 어떤 레이어를 호출할 수 있는가 |
| 네이밍 규칙 | 변수, 함수, 파일, 클래스 명명 방식 |
| 에러 처리 방식 | 에러를 어디서 어떻게 처리하는가 |
| 공통 패턴 | 반복되는 문제를 어떻게 푸는가 |
| 절대 바꾸면 안 되는 것 | 변경 시 전체에 영향을 주는 핵심 계약 |

**해야 할 질문**

| 구분 | 질문 |
|------|------|
| 규칙 발굴 | "이 프로젝트에서 모든 작업이 따라야 할 규칙이 뭐야?" |
| 경계 정의 | "어떤 결정은 개별 작업에서 하면 안 되고 프로젝트 레벨에서 해야 해?" |
| 불변 항목 | "나중에 바꾸면 전체가 흔들리는 부분이 어디야?" |

**놓치면 안 되는 것**
- 규칙이 머릿속에만 있으면 agent에게 전달할 수 없다 — 반드시 문서화
- 완벽하지 않아도 된다. 시작점이 있는 것이 없는 것보다 낫다
- 이 문서는 레이어 2 통합 체크포인트를 거치며 계속 업데이트된다

#### 새 프로젝트의 레이어 0 진입 방식

새 프로젝트는 요구사항이 머릿속에서 파일로 정리되는 것에서 시작된다.
이 파일이 agent와의 레이어 0 대화의 입력이 되고, 대화 결과로 아키텍처 규칙 문서가 만들어진다.

**레이어 0에서 생성되는 파일**

```
[파일 1] 요구사항 파일 (사용자가 작성)
    → Agent와 아키텍처 설계 대화의 입력
    → 내용: 무엇을, 왜, 제약 조건, 하지 말아야 할 것

[파일 2] 아키텍처 규칙 문서 (대화 결과로 생성)
    → 레이어 1에서 매 작업마다 agent에게 주입
    → 내용: 설계 결정 + 이유 + 규칙 목록
```

**요구사항 파일 포맷 (agent 친화적)**

대부분의 요구사항 파일은 "무엇을"만 담는데, agent에게는 "왜"와 "하지 말아야 할 것"이 더 중요하다.

| 섹션 | 내용 | agent에게 필요한 이유 |
|------|------|----------------------|
| 프로젝트 한 줄 정의 | 무엇을 만드는가 한 문장 | 전체 맥락 파악 |
| 해결하려는 문제 | 왜 만드는가 | 설계 방향 판단 기준 |
| 핵심 기능 | Must / Nice to have 구분 | 범위 명확화 |
| 제약 조건 | 기술 스택, 환경, 비즈니스 규칙 | 조용한 결정 방지 |
| 결정이 필요한 사항 | 아직 정하지 못한 것 | 설계 대화 시작점 |
| 하지 말아야 할 것 | 범위 제외, 절대 금지 | 범위 이탈 방지 |

**요구사항 파일로 레이어 0 대화 시작하는 방법**

> "이 요구사항을 기반으로 아키텍처를 설계해줘.
> 각 결정마다 이유를 포함하고,
> 모든 작업이 따라야 할 규칙 목록도 같이 만들어줘."

> 참고 템플릿: `template_Layer0_Step1_requirements.md`, `template_Layer0_Step2_architecture_rules.md`

---

#### 0-2. 전체 작업 분해 + 의존성 정의

전체 작업을 나누되, "무엇을 만들지"뿐 아니라 "어떻게 연결되는지"를 함께 정의한다.

**해야 할 것**
- 전체 작업 목록 작성
- 작업 간 의존성 표시 (A가 완료돼야 B를 시작할 수 있는가)
- 공유 인터페이스가 필요한 작업 식별

**해야 할 질문**

| 구분 | 질문 |
|------|------|
| 의존성 | "이 작업들 중 서로 영향을 주는 것이 어디야?" |
| 공유 계약 | "여러 작업이 함께 사용하는 인터페이스/데이터 모델이 뭐야?" |
| 선후 관계 | "반드시 먼저 끝내야 하는 작업이 뭐야?" |

**놓치면 안 되는 것**
- 공유 인터페이스는 개별 작업 전에 먼저 정의한다 — 각자 만들면 충돌 발생
- 병렬로 진행 가능한 작업과 순차적으로 해야 하는 작업을 구분한다

---

### 레이어 1: 개별 작업 반복

기존 1~5단계에 아래 항목을 추가한다.

#### 추가되는 것: 작업 시작 전 컨텍스트 주입

매 작업 시작 시 agent에게 다음을 함께 제공한다.

```
[아키텍처 규칙 문서] + [이 작업의 계획서] + [건드리면 안 되는 것]
```

#### 추가되는 것: 작업 지시 구조 변경

기존: "이걸 해줘"
변경: "이걸 해줘 + 이건 건드리지 마 + 모르면 먼저 물어봐"

#### 추가되는 질문: 1단계 계획서 검토 시

| 구분 | 질문 |
|------|------|
| 규칙 위반 | "이 계획이 아키텍처 규칙을 위반하는 부분이 있어?" |
| 영향 범위 | "이 작업이 다른 작업에 영향을 줄 수 있는 부분이 어디야?" |

#### 추가되는 질문: 3단계 결과 확인 시

| 구분 | 질문 |
|------|------|
| 규칙 준수 | "이 코드가 아키텍처 규칙 중 위반한 것이 있어?" |
| 범위 이탈 | "지시하지 않은 부분을 건드린 곳이 있어?" |

---

### 레이어 2: 통합 체크포인트

**N개 작업마다** 또는 **주요 기능 단위 완성 시** 실행한다.
개별 작업 검증과 별개로, 전체 정합성을 확인하는 레이어.

#### 해야 할 질문

| 구분 | 질문 |
|------|------|
| 규칙 위반 | "지금까지 구현된 코드 중 아키텍처 규칙을 위반한 부분이 있어?" |
| 인터페이스 충돌 | "작업 간 인터페이스가 충돌하거나 불일치하는 부분이 있어?" |
| 예상 외 패턴 | "의도하지 않은 패턴이 생긴 부분이 있어?" |
| 규칙 업데이트 | "레이어 0 규칙 문서 중 현실과 맞지 않아 업데이트가 필요한 것이 있어?" |

#### 놓치면 안 되는 것
- 통합 체크포인트는 개별 작업 루프(2~4단계)와 **별개**로 실행한다
- 여기서 발견된 문제는 해당 작업으로 돌아가 수정하거나, 레이어 0 문서를 업데이트한다
- 레이어 0 규칙 문서는 이 체크포인트를 거치며 점점 정교해진다

#### 통합 체크포인트 후 처리

```
위반 발견 → 해당 작업으로 돌아가 수정 (레이어 1 루프 재진입)
규칙 변경 필요 → 레이어 0 문서 업데이트 → 이후 작업에 반영
정합성 확인 → 다음 작업 묶음으로 진행
```

[레이어 0: 1회]
아키텍처 규칙 문서 작성
전체 작업 분해 + 의존성 정의
공유 인터페이스 사전 정의
        ↓
[레이어 1: 작업마다 반복]
아키텍처 규칙 컨텍스트 주입
        ↓

   │  1단계: 계획서 작성/검토  │

   │  3단계: 결과 확인         │ ← 작업 내 반복 루프 (2~4)
   │  4단계: 실행              │

```

## 11. 개발 단계별 실전 가이드 (레이어 1 상세)

### 전체 흐름

```
1단계: 계획서 작성 및 검토
      ↓
   ┌─────────────────────────┐
   │  2단계: 구현 요청        │
   │        ↓                │
   │  3단계: 결과 확인        │  ← 작업 내 반복 루프 (2~4)
   │        ↓                │
   │  4단계: 실행            │
   │        ↓                │
   │  문제 발견 시 ───────────┘
   └─────────────────────────┘
      ↓ (통과 시)
5단계: 피드백 루프 → 다음 작업의 1단계로 연결
```

**루프 진입 조건**

| 발견 내용 | 돌아갈 단계 |
|-----------|-------------|
| 구현 방식이 의도와 다름 | 2단계로 (재구현) |
| 실행 중 예상 못 한 오류 | 2단계 또는 3단계로 |
| 계획 자체가 잘못됨 | 1단계로 (계획 수정) |
| 사소한 수정 | 그 자리에서 처리 |

**루프 탈출 조건** — 아래 모두 충족 시 4단계 → 5단계로 이동
```
✓ 1단계 체크리스트 항목 전부 확인
✓ 정상 경로 추적 완료
✓ 실패 경로 추적 완료
```

**루프 횟수를 줄이는 원칙**
- 1단계 계획서 품질이 높을수록 루프 횟수 감소
- 2단계 구현 요청 시 조용한 결정 포인트를 명시할수록 감소
- 재구현 요청 시 범위를 좁게 — "다시 해줘"가 아니라 "이 부분에서 X가 발생할 때 Y로 처리해야 해"

**5단계의 성격**
- 2~4단계 루프의 **내부**가 아니라 루프 **간 연결**
- 이번 작업의 반복 패턴 → 다음 작업의 1단계(계획서)에 반영하는 역할

---

### 1단계: 계획서 작성 및 검토

**해야 할 질문**

| 구분 | 질문 |
|------|------|
| 완성도 | "빠진 부분이나 부족한 부분이 없어?" |
| 일관성 | "논리적 모순이 없어?" |
| 실패 시나리오 | "이 계획이 실패할 수 있는 시나리오를 찾아줘" |
| 숨겨진 가정 | "내가 놓친 가정이 뭐야?" |
| **구현 현실** | **"이 계획대로 구현하다가 내가 예상 못 한 복잡도나 결정이 필요한 지점이 어디야?"** |

**놓치면 안 되는 것**
- Agent 동조 경향 주의 — "동의를 구하는 것"이 아니라 "반례를 구하는 것"
- "구현 현실 질문"으로 나온 항목들을 **목록으로 보존** → 3단계 체크리스트가 됨
- 혼자 검토(내 의도 명확성) + Agent 검토(내가 못 보는 구멍) 역할 분리

---

### 2단계: 구현 요청

**해야 할 질문/지시**

| 구분 | 내용 |
|------|------|
| 명시화 | 1단계에서 발견한 "조용한 결정 포인트"를 구현 요청에 직접 포함 |
| 불확실성 처리 | "모르거나 결정이 필요한 부분이 있으면 구현하지 말고 먼저 물어봐" |
| 범위 제한 | 계획서에 없는 것은 임의로 추가하지 말 것 명시 |

**놓치면 안 되는 것**
- 계획서에서 발견한 의도 간격 포인트가 구현 요청에 **반영됐는지** 확인
- Agent가 조용히 채울 여백을 최대한 줄이는 것이 목표

---

### 3단계: 결과 확인

**해야 할 질문**

| 구분 | 질문 |
|------|------|
| 조용한 결정 확인 | 1단계 체크리스트 항목별 — "이 부분 어떻게 처리했어? 왜?" |
| 실패 경로 | "이 코드가 실패할 수 있는 경우를 모두 말해줘" |
| 숨겨진 전제 | "이 코드가 가정하고 있는 것이 뭐야?" |
| 검증 범위 확인 | "이 코드의 테스트 케이스를 작성해줘" |

**놓치면 안 되는 것**
- "코드가 돌아간다" ≠ 논리 검증 완료
- 경로 추적: 정상 경로 1개 + 실패 경로 1개를 끝까지 따라가기
- 코드 전체를 읽으려 하지 말 것 — 핵심 경로 하나를 깊이 보는 것이 효과적

---

### 4단계: 실행

**해야 할 것**
- 정상 케이스 + 경계값 + 실패 케이스 직접 실행
- 1단계에서 발견한 "조용한 결정 포인트"에 해당하는 시나리오 직접 재현

**놓치면 안 되는 것**
- 실행 성공이 최종 검증이 아님 — 3단계 논리 검증이 선행돼야 의미 있음

---

### 5단계: 피드백 루프

**해야 할 것**
- 이번 작업에서 반복된 의도 간격 패턴 기록
- 다음 계획서 작성 시 해당 패턴을 명시적으로 포함

**놓치면 안 되는 것**
- 피드백 루프 없이 매번 결과 심문만 반복하면 비용이 줄지 않음
- **목표: 반복되는 오류는 설계 단계로 끌어올려서 처음부터 막는 구조**

---

# 부록 1: 기존 프로젝트에서 레이어 0 적용하기

## 새 프로젝트와의 결정적 차이

| 구분 | 새 프로젝트 | 기존 프로젝트 |
|------|-------------|---------------|
| 레이어 0 성격 | 설계 (Design) | 발굴 (Discovery) |
| 규칙 출처 | 새로 만든다 | 코드에서 추출한다 |
| 위험 | 방향 설정 실패 | 기존 코드 파괴 |
| 첫 번째 할 일 | 요구사항 작성 | 현황 분석 |

기존 프로젝트에서 가장 흔한 실수:
> "이상적인 규칙"을 먼저 문서화하고 코드에 적용하려 하는 것

현재 코드에 없는 규칙을 강제하면 기존 코드와 충돌이 발생한다.
**현재 상태를 먼저 정확히 파악하고, 그 위에서 방향을 잡아야 한다.**

---

## 기존 프로젝트 레이어 0 순서

```
1단계: 현황 분석 (agent가 코드 탐색)
        ↓
2단계: 암묵적 규칙 추출 + 불일치 목록 작성
        ↓
3단계: 현재 규칙 문서화 + 목표 규칙 정의
        ↓
4단계: 작업 분해 (신규 기능 + 불일치 해소 작업 구분)
        ↓
레이어 1 시작
```

---

## 1단계: 현황 분석

### [요청 A] 현황 분석 요청

```
이 코드베이스를 분석해서 다음을 찾아줘.

1. 현재 사용 중인 레이어 구조 (어떤 레이어가 어떤 레이어를 호출하는가)
2. 에러 처리 패턴 (어디서 어떻게 처리하는가)
3. 네이밍 규칙 (파일, 함수, 변수, DB 컬럼)
4. 반복되는 공통 패턴
5. 규칙이 일관되지 않은 부분 (같은 문제를 다르게 푼 곳)

각 항목에 실제 코드 예시를 포함해줘.
```

### 놓치면 안 되는 것
- 분석 범위를 전체 코드베이스로 넓게 잡을 것
- 불일치가 발견되면 덮어쓰지 말고 반드시 기록할 것

---

## 2단계: 암묵적 규칙 추출 + 불일치 목록 작성

### [요청 B] 규칙 추출 요청

```
분석 결과를 바탕으로 architecture_rules.md 초안을 작성해줘.

조건:
- 현재 코드에 실제로 존재하는 규칙만 기록할 것 (이상적인 규칙 X)
- 일관되지 않은 부분은 '불일치' 섹션에 따로 기록할 것
- 앞으로 따라야 할 방향이 현재 상태와 다른 경우 명확히 구분해서 표시할 것
```

---

## 3단계: 현재 규칙 문서화 + 목표 규칙 정의

### 불일치 항목 처리 방식

| 불일치 유형 | 처리 방향 |
|-------------|-----------|
| 사소한 스타일 차이 | 하나로 통일, 신규 작업부터 적용 |
| 레이어 구조 위반 | 별도 리팩토링 작업으로 분리 |
| 핵심 계약 충돌 | 반드시 먼저 해결 후 신규 작업 진행 |

### 놓치면 안 되는 것
- 불일치를 한꺼번에 고치려 하지 말 것 — 기존 코드 파괴 위험
- 신규 작업과 리팩토링 작업을 반드시 분리할 것

---

## 4단계: 작업 분해

```
신규 기능 작업   → 레이어 1 일반 절차 적용
불일치 해소 작업 → 기존 코드 수정, 범위 제한 엄격히
```

불일치 해소 작업 시 추가 지시:
```
- 이 작업은 X 파일만 수정한다
- 기능 변경 없이 구조만 바꾼다
- 인터페이스(입출력)는 절대 바꾸지 않는다
```

## 레이어 2: 통합 체크포인트

### 실행 시점
- N개 작업 완료 시 (프로젝트 규모에 따라 3~5개 작업마다 권장)

| 구분 | 질문 |

| 규칙 위반 | "지금까지 구현된 코드 중 아키텍처 규칙을 위반한 부분이 있어?" |

```

### 놓치면 안 되는 것
- 레이어 2는 레이어 1 루프가 모두 통과된 후에 실행한다
- 여기서 발견된 문제를 한꺼번에 수정하려 하지 말 것 — 작업 단위로 분리해서 처리

# 부록 4: 페르소나와 스레드 전략

## 왜 페르소나가 필요한가

Agent에게 역할만 바꿔달라고 요청하는 것(같은 스레드 내 페르소나 주입)은 효과가 떨어진다.

**문제: 컨텍스트 오염**
```
agent이 코드를 직접 작성함
    ↓
같은 스레드에서 "비판적으로 검토해줘"
    ↓
자기가 만든 코드를 비판하기 어려움 → 방어적으로 검토
```

특히 "자기가 한 작업을 평가"하는 상황에서 오염이 가장 심하다.  
이전 대화 맥락, 이미 내린 결정들이 평가의 객관성을 훼손한다.

---

## 스레드 분리 원칙

| 작업 성격 | 스레드 | 이유 |
|-----------|--------|------|
| **생성** — 맥락이 많을수록 좋음 | 같은 스레드 ♻️ | 계획/규칙 맥락이 필요 |
| **평가** — 맥락이 없을수록 객관적 | 새 스레드 🆕 | 컨텍스트 오염 방지 |

---

## 페르소나별 사용 시점

| 페르소나 | 사용 시점 | 스레드 | 핵심 태도 |
|----------|-----------|--------|-----------|
| `persona_architect.md` | Layer 0 아키텍처 설계 (요청 2) | 🆕 새 스레드 | 전체 일관성, 결정 이유 명시 |
| `persona_codebase_analyst.md` | Layer 0 기존 프로젝트 분석 (요청 A) | 🆕 새 스레드 | 현재 상태 사실 기록, 평가 없음 |
| `persona_plan_critic.md` | Layer 1 계획서 검토 (요청 T1) | 🆕 새 스레드 | 약점만 찾음, 동의/칭찬 없음 |
| `persona_implementer.md` | Layer 1 구현 (요청 T2) | ♻️ 같은 스레드 | 계획 충실 이행, 모르면 물어봄 |
| `persona_code_reviewer.md` | Layer 1 결과 확인 (요청 T3) | 🆕 새 스레드 | 심문, 논리 검증, 수정 제안 없음 |
| `persona_integration_auditor.md` | Layer 2 통합 감사 (요청 L2-1) | 🆕 새 스레드 | 규칙 준수 여부만 판단, 감사 리포트 |

---

## 새 스레드에서 컨텍스트 주입 방법

평가 요청을 새 스레드로 보낼 때, 필요한 컨텍스트만 압축해서 제공한다.

```
[새 스레드 시작]

(해당 페르소나 선언 붙여넣기)

[필요한 컨텍스트]
(architecture_rules.md 또는 task_plan.md 등 해당 문서만 붙여넣기)

[평가 대상]
(계획서 또는 코드 붙여넣기)

[요청]
(해당 요청 템플릿 붙여넣기)
```

**핵심 원칙:** 새 스레드에는 평가에 꼭 필요한 것만 넣는다.  
이전 대화, 구현 과정, 맥락 설명을 넣을수록 객관성이 떨어진다.

---

## 페르소나 사용 흐름 (레이어 1 기준)

```
[새 스레드 - plan_critic]      계획서 검토 (요청 T1)
        ↓ 피드백 반영
[같은 스레드 - implementer]    구현 (요청 T2)
        ↓ 구현 완료
[새 스레드 - code_reviewer]    결과 확인 (요청 T3)
        ↓ 검증 통과
[같은 스레드]                  실행 (4단계)
        ↓
[같은 스레드]                  피드백 기록 (5단계)
```

# 부록 5: 스킬과 워크플로우

## 왜 스킬/워크플로우가 필요한가

페르소나가 "누가 하는가"를 정의한다면, 스킬은 "어떻게 하는가"를 정의한다.
워크플로우는 이 둘을 묶어 "무엇을 만들 때 어떤 순서로 누가 무엇을 하는가"를 정의한다.

```
페르소나  →  역할 (누가)
스킬      →  방법 (어떻게)
워크플로우 →  흐름 (어떤 순서로)
```

이 세 가지를 분리해두면, 새로운 작업 유형이 생겼을 때 기존 블록을 조합해서 새 워크플로우를 빠르게 구성할 수 있다.

---

## 스킬 목록

| 스킬 | 목적 | 사용 시점 |
|------|------|-----------|
| `skill_silent_decision_extraction.md` | 계획에서 명시되지 않은 결정 지점 추출 | Layer 0 요구사항 작성, Layer 1 계획서 작성 |
| `skill_path_tracing.md` | 정상/실패 경로를 끝까지 추적하여 논리 검증 | Layer 1 결과 확인, Layer 2 통합 체크 |
| `skill_counterexample_finding.md` | 계획/코드가 실패할 수 있는 반례 탐색 | Layer 0 설계 검토, Layer 1 계획 검토/결과 확인 |
| `skill_dependency_mapping.md` | 작업 간 선후 관계와 공유 인터페이스 정의 | Layer 0 작업 분해 |

### 스킬 사용 원리

스킬은 프롬프트 템플릿이다. "이 역할을 맡은 agent에게 이런 방식으로 물어봐라"를 정의한다.

```
스킬 없이:  "이 코드 검토해줘"
스킬 있이:  "정상 경로에서 입력부터 출력까지 각 단계를 추적하고,
             각 단계에서 어떤 값이 전달되고 어떤 결정이 내려지는지 보여줘"
```

구체적인 탐색 방향이 있으면 agent의 동조 경향을 차단하고 진짜 문제를 찾아낼 수 있다.

---

## 워크플로우 목록

| 워크플로우 | 목적 | 핵심 차이 |
|-----------|------|-----------|
| `workflow_new_feature.md` | 새 기능 처음부터 구현 | 계획서 → 계획 검토 → 구현 → 결과 확인 표준 흐름 |
| `workflow_bug_fix.md` | 발견된 버그 재현 및 최소 수정 | 버그 재현 먼저, 수정 범위 최소화 엄격 |
| `workflow_refactoring.md` | 기능 유지하며 구조 개선 | 동작 일치 검증이 핵심, 외부 인터페이스 변경 금지 |
| `workflow_code_review.md` | 완성된 코드 체계적 검토 | 반드시 새 스레드, 경로 추적 + 반례 탐색 |

### 워크플로우 선택 기준

```
"무언가 새로 만드는가?" → workflow_new_feature
"버그가 발생했는가?"    → workflow_bug_fix
"기능은 그대로인데 코드를 개선하고 싶은가?" → workflow_refactoring
"PR/코드를 검토하고 싶은가?" → workflow_code_review
```
