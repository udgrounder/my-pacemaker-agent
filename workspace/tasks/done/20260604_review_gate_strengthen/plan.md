# 태스크: 작업 결과 검토 체계 강화

**요청일:** 2026-06-04
**상태:** 검토 완료
**파생 출처:** 없음 — mpaWork3 테스트 관찰에서 발견

---

## 목적

작업 결과 검토의 범위를 명확히 하고, 에이전트가 사용자 확인 없이 자동 완료 처리하는 문제를 방지하는 게이트를 추가한다.

---

## 배경

세 가지 문제가 관찰됐다:
1. "코드 검토"라는 명칭이 범위를 코드에만 한정 — UI/UX, 기능 동작도 포함해야 함
2. 구현 완료 후 재개 시 에이전트가 세 가지 선택지를 균등하게 제시 — 작업 결과 검토가 명확한 다음 단계임에도
3. 에이전트가 검토 리포트 생성 후 사용자 확인 없이 "검토 완료" 처리

---

## 구현된 내용

**명칭 변경 (전체 파일):**
- "코드 검토" → "작업 결과 검토" 전체 교체
- `personas/code_reviewer.md` — "Result Reviewer (작업 결과 검토자)"로 역할 재정의
- `inject/layer1_review.md` — UI/UX·기능 동작 검토 항목 추가

**상태 관리 강화:**
- `templates/plan_template.md` — "검토 완료" 상태 추가
- `inject/layer1_implement.md` — 구현 완료 시 plan.md 상태 `구현 완료` 명시적 기록
- `inject/layer1_review.md` — 검토 완료는 사용자 확인 후에만 기록

**세션 재개 상태 인식:**
- `core/agent_rules.md` — 상태별 다른 메시지 출력 (구현 완료 → 작업 결과 검토 안내, 균등 선택지 제거)

**게이트 추가:**
- `core/agent_rules.md` — "작업 결과 검토" 섹션 신규: 사용자 명시적 확인 없이 검토 완료 처리 불가
- `inject/layer1_review.md` — 추가 작업 확인 → 신규 태스크 등록 → 기존 태스크 종료 순서 명시

---

## 핵심 결정

**게이트 대칭 원칙:**
- 계획서 승인 없이 구현 시작 불가 ↔ 사용자 확인 없이 검토 완료 처리 불가
- 두 게이트는 동등하다. 생략은 사용자가 명시적으로 요청할 때만 허용

**종료 순서 원칙:**
- 추가 작업 있음 → 신규 태스크 먼저 등록 → 원래 태스크 종료
- 원래 태스크를 먼저 닫으면 파생 맥락이 끊긴다
