# 태스크: 라우팅 개선 (문제 1)
**상태:** 구현 완료  
**목적:** 키워드 패턴 매칭 기반 라우팅의 3가지 결함을 수정하고, 버그/설계 미스 판별 흐름을 명확히 한다

---

## 에이전트 보고

### 사용자 결정 필요
- [x] **판단 불가 시 동작** — 케이스별로 분리 처리
  - 케이스 α (새 요청, 분류 불가) → 선택지 2~3개 제시 후 대기
  - 케이스 β (맥락 있는 짧은 발화: "응", "그냥 해줘") → 맥락 추론 후 진행 + "다르면 정정하세요"

- [x] **"이어서 해줘" 처리** — active/hold 목록 전체 먼저 표시, 사용자가 선택 (0개여도 표시 후 "새 작업인가요?" 처리)

- [x] **설계 미스 판별 주체**
  - Agent 검토 시: 설계 ≠ 구현 → 구현을 설계에 맞게 수정 (자동, 사용자 질문 없음)
  - 사용자 보고 시: 증상 수집 → 에이전트가 진단 → "설계를 다시 검토하시겠어요?" 또는 "구현을 수정할까요?" 제시

- [x] **반복 오라우팅 기록** — 사용자가 라우팅 정정 시 해당 패턴을 upgrade-candidates에 자동 기록

### 에이전트 가정
| 가정 | 근거 | 틀렸을 때 영향 |
|------|------|--------------|
| Agent 자가 검토 시 코드 vs plan.md 기계적 비교로 불일치 판별 가능 | plan.md에 구현 단계가 체크리스트로 명시됨 | 모호한 구현 단계가 있으면 판별 실패 가능 |
| 사용자 보고 버그의 증상 수집은 자유 발화로 충분 | 별도 입력 폼 불필요 | 증상이 불명확하면 추가 질문 필요 |

## 구현 단계

- [ ] **Step 1** — `agent_rules.md` 라우팅 테이블 수정
  - 버그 행에 조건 명시: "구현 ≠ 설계인 경우 (증상 보고)" + 사용자는 증상만 보고, 진단은 에이전트 담당 명시
  - "이어서/계속해줘" 행 → active/hold 목록 표시 후 선택 처리로 교체
  - 판단 불가 처리를 케이스 α/β로 분리해 각각 다르게 기술
  - "하네스 업데이트/하네스 개선 후보" 용어 → "체계 업데이트/체계 개선 후보"로 변경

- [ ] **Step 2** — `agent_rules.md` 오라우팅 정정 시 기록 규칙 추가
  - 라우팅 고지 후 사용자가 정정한 경우: 원래 발화 패턴 + 정정된 유형을 upgrade-candidates에 기록하는 규칙 추가

- [ ] **Step 3** — `layer1_review.md` 검토 주체 구분 추가
  - 세션 진입 시 "에이전트 자가 검토 vs 사용자 보고" 구분 명시
  - Agent 자가 검토: 코드 vs plan.md 체크리스트 기계적 비교 → 불일치 발견 시 구현 수정 (사용자 질문 없음)
  - 사용자 보고: 증상 수집 → 진단 → 버그 or 설계 미스 판별 → "구현을 수정할까요?" / "설계를 다시 검토하시겠어요?" 제시

## 예상 조용한 결정

- **hold 태스크 표시 방식**: "이어서 해줘" 시 active와 hold를 함께 표시. 시각적 구분 방법(별도 섹션 or 상태 태그)은 구현 시 결정
- **오라우팅 upgrade-candidates 파일명**: 패턴별로 파일을 만들지, 하나에 누적할지 → 누적 방식으로 처리 (`routing_mismatches.md`)
- **사용자 보고 버그 증상 수집**: 별도 질문 없이 발화 내용에서 자동 추출. 불명확하면 추가 질문

- [ ] 케이스 β: "응", "그냥 해줘" → 맥락 추론 후 진행

## 반례

- 에이전트 자가 검토 시 plan.md 체크리스트 자체가 모호하면 기계적 비교 실패 가능 → 모호한 단계는 구현 시 사용자 확인 요청 (layer1_implement의 기존 룰로 커버)
- 사용자 보고 버그인데 증상 설명이 너무 짧으면 ("그냥 이상해") 진단 불가 → 추가 질문 1회 허용
- active/hold 목록이 많을 경우(5개 이상) 선택이 번거로울 수 있음 → 현재 범위 밖, 추후 개선 후보
