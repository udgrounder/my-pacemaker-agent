## 핵심 변경 내용

### `agent_rules.md` — "요청 유형 자동 판단" 섹션 재구성

현재 구조 문제: 섹션 2가 모든 요청에 발화 패턴 매칭을 적용. 기존 태스크 맥락을 무시.

변경 후 구조:

```
섹션 2. 요청 유형 자동 판단
  ├── [선행 확인] active 태스크 있으면 → 태스크 맥락 라우팅 (plan.md 상태 기반)
  │     ├── 설계 완료 / 구현 중  → layer1_implement.md
  │     ├── 구현 완료            → layer1_review.md
  │     ├── 검토 완료            → 완료 처리
  │     └── 신규 요청으로 판단되면 → 아래 패턴 매칭으로 내려옴
  └── [신규 요청] active 태스크 없음 OR 명확히 새 요청 → 발화 패턴 매칭 (기존 테이블)
```

**신규 요청 감지 기준 (active 태스크가 있어도 새 요청으로 처리하는 경우):**
- 현재 태스크와 무관한 파일·기능을 명시적으로 언급
- "별개로", "새로", "다른 거" 등 분리 의도가 명확한 발화
- 현재 태스크 범위 밖의 버그·기능 보고

**모호한 경우 (active 태스크 있는데 새 요청인지 불명확):**
```
지금 [태스크명] 작업 중입니다.
→ 이 작업의 일부로 처리할까요, 새 작업으로 분리할까요?
```

### C: 신규 요청 모호 케이스 — 2단계 진단 질문으로 강화

현재 "판단 불가 시" 케이스 α는 3개 선택지를 나열한다. 문제: 사용자가 선택지를 고르는 것이 아니라 에이전트가 먼저 진단해야 한다.

변경 후: 에이전트가 2개의 진단 질문으로 유형을 좁힌다.

**질문 1 — 현재 동작 판단:**
```
지금 [기능/동작]이 의도한 대로 작동하고 있나요?
  → 아니오 (뭔가 잘못됨) : 버그 수정으로 처리합니다
  → 예 (더 나은 방식 원함) : 다음 질문으로
```

**질문 2 — 범위 판단 (질문 1이 "예"인 경우):**
```
새 기능을 추가하고 싶은 건가요, 기존 기능을 개선하고 싶은 건가요?
  → 새 기능 : 새 기능 설계로 처리합니다
  → 개선    : 리팩터링으로 처리합니다
```

기존 케이스 α·α-성능을 이 2단계 구조로 통합한다.
