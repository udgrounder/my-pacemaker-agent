# Docs 레이어 가이드

> **목적:** 구현이 완료된 기능/설계의 결과물을 누적·관리한다.  
> **원칙:** 요청이 완료될 때마다 해당 요청의 "완료 시 문서 업데이트 대상"을 반드시 처리한다.

---

## 파일 구조

```
workspace/docs/
├── README.md               ← 이 파일
├── docs_template.md        ← 새 문서 추가 시 복사해서 사용
├── INDEX.md                ← 전체 문서 색인 (agent가 최초 세션에 생성)
├── features/               ← 기능 문서 (예시 폴더명, 자유롭게 변경 가능)
│   └── [기능명].md
└── design/                 ← 설계 문서 (예시 폴더명, 자유롭게 변경 가능)
    └── [설계명].md
```

**폴더 구성 예시:**

| 프로젝트 유형 | docs/ 구조 예시 |
|------------|----------------|
| 풀스택 웹 | `features/`, `api/`, `infra/` |
| 모바일 앱 | `features/`, `screens/`, `flows/` |
| CLI 도구 | `commands/`, `config/`, `architecture/` |
| 단일 도메인 | 폴더 없이 docs/ 루트에 바로 |

---

## tasks/와의 차이

| | tasks/ | docs/ |
|--|-----------|-------|
| 내용 | 무엇을 만들어야 하나 | 만든 것이 어떻게 동작하나 |
| 시점 | 구현 전 작성 | 구현 완료 후 반영 |
| 수명 | 완료 후 done/으로 이동 | 계속 유지·갱신 |

---

## 새 문서 추가 절차

1. `.mpa-workspace/templates/docs_template.md`를 복사해서 적절한 경로에 저장
2. 프론트매터 채우기 (`관련태스크`는 `workspace/tasks/INDEX.md`의 태스크명)
3. `INDEX.md`에 한 줄 추가

---

## 문서 업데이트 트리거

| 트리거 | 업데이트 대상 |
|--------|-------------|
| 요청 완료 (active → done) | 해당 요청의 "완료 시 문서 업데이트 대상" 목록 |
| 기존 기능 동작 변경 | 해당 기능 문서 |
| 설계 결정 변경 | 관련 설계 문서 |

**트리거 없는 것:** 내부 리팩토링, 버그 수정 (외부 동작 변화 없는 경우)

---

## 세션별 활용

| 세션 | docs/ 활용 방식 |
|------|---------------|
| `layer1_design` | 관련 기존 문서를 참고해 설계 맥락 파악 |
| `layer1_implement` | 구현 완료 후 업데이트 대상 docs 확인 |
| `layer2_checkpoint` | INDEX.md 기준으로 미반영 요청 점검 |
