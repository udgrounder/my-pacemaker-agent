# 컨텍스트 길이 관리
> 대화가 길어질 때 agent의 초반 지시 망각을 방지하는 방법
> 컨텍스트 창은 유한하다 — 이를 설계에 반영해야 한다

---

## 왜 문제가 되는가

```
스레드 초반: architecture_rules.md + task_plan.md 주입
        ↓
구현이 이어지면서 대화가 길어짐
        ↓
컨텍스트 창 한계에 가까워질수록
초반에 주입한 규칙/계획이 희석되거나 잘림
        ↓
agent가 이전에 정의한 제약을 무시하고 구현
```

---

## 언제 새 스레드가 필요한가

| 신호 | 대응 |
|------|------|
| 구현 요청이 20회를 넘어간다 | 새 스레드로 전환 |
| agent가 이전에 정의한 내용을 다시 물어본다 | 즉시 새 스레드로 전환 |
| agent가 "앞에서 정한 대로"와 다른 방향으로 구현한다 | 즉시 새 스레드로 전환 |
| 작업 단위가 하나 완료됐다 | 다음 작업은 새 스레드로 시작 권장 |

정확한 기준은 없다. 하지만 "스레드가 길어지고 있다"고 느낄 때 미리 전환하는 것이 낫다.

---

## 새 스레드로 전환하는 방법

### 구현 중간에 전환할 때

```
[새 스레드 시작]

[컨텍스트 압축 주입]
- architecture_rules.md
- task_plan.md
- 지금까지 구현된 파일 목록과 현재 상태 요약:
  - [파일명]: [완료/진행 중/미시작]
  - 현재 막힌 지점: [설명]

[이어서 구현 요청]
"위 맥락에서 이어서 [다음 구현할 내용]을 진행해줘."
```

### 작업 단위 간 전환할 때

```
작업 1 완료 → feedback_log.md 작성
        ↓
새 스레드 열기
        ↓
[새 스레드 시작]
- architecture_rules.md (업데이트된 버전)
- 작업 2의 task_plan.md
- 작업 1에서 만들어진 인터페이스/모듈 요약

"작업 2를 시작해줘."
```

---

## 스레드별 역할 재정리

```
생성 스레드 (같은 스레드 ♻️):
- 계획 → 구현 흐름을 유지
- 길어지면 새 스레드로 전환하되 맥락 재주입

평가 스레드 (새 스레드 🆕):
- 항상 새 스레드 (컨텍스트 오염 방지)
- 코드/계획만 주입, 이전 대화 제외
```

---

## 재주입 우선순위

새 스레드에 모든 것을 다시 주입할 필요는 없다. 우선순위에 따라 선택한다.

| 우선순위 | 내용 | 이유 |
|----------|------|------|
| 1 | `architecture_rules.md` | 모든 구현의 기준 |
| 2 | 현재 `task_plan.md` | 현재 작업의 범위 |
| 3 | 공유 인터페이스 정의 | 다른 모듈과 연결되는 부분 |
| 4 | 직전에 구현된 파일 | 연속성 유지 |

---

## 컨텍스트 오염 vs 컨텍스트 부족

| 상황 | 문제 | 해결 |
|------|------|------|
| 구현 스레드가 너무 길다 | 초반 규칙이 희석됨 | 핵심 문서 재주입 또는 새 스레드 |
| 평가를 같은 스레드에서 한다 | 자기 결정을 옹호함 | 반드시 새 스레드 |
| 새 스레드에 맥락을 너무 많이 넣는다 | 평가 객관성 저하 | 평가에 필요한 최소만 주입 |

## 레이어 2와의 관계

레이어 2 체크포인트는 컨텍스트 단절을 보완하는 역할도 한다.

```
레이어 1 작업 N개 진행
→ 각 작업이 다른 스레드에서 진행돼 전체 일관성이 흐트러질 수 있음
        ↓
레이어 2 체크포인트 (새 스레드)
→ architecture_rules.md 기준으로 전체 정합성 확인
→ 스레드 단절로 생긴 불일치 발견 및 수정
```
