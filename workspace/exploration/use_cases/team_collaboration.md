# 팀 협업 가이드
> 팀 전체가 AI agent를 함께 사용할 때의 원칙과 실전 운영 방법
> workflow_team_collaboration.md가 "절차"라면, 이 파일은 "왜 그렇게 하는가"와 "어떻게 잘 할 수 있는가"

---

## 핵심 문제

팀에서 AI agent를 사용할 때 혼자 사용할 때와 다른 문제가 생긴다.

```
문제 1: 각자 다른 스레드에서 작업 → agent가 내린 암묵적 결정이 다르다
문제 2: architecture_rules.md가 각자 버전으로 diverge된다
문제 3: 한 사람이 발견한 실패 패턴을 팀 전체가 모른다
문제 4: "AI가 만든 코드"의 리뷰 책임이 모호해진다
```

이 가이드는 위 4가지 문제를 다룬다.

---

## 원칙 1: architecture_rules.md는 팀 공유 문서

가장 중요한 원칙이다.

```
❌ 각자 자기 작업에 맞게 로컬 수정
✅ 변경 = PR/MR + 팀 합의 → 공유 저장소에 반영
```

### 왜 중요한가

architecture_rules.md 없이 구현 요청을 하면 agent가 "합리적으로 보이는" 결정을 혼자 내린다.  
팀원마다 다른 버전의 규칙으로 요청하면, 각자의 구현이 서로 다른 방향으로 최적화된다.  
나중에 통합하면 충돌이 생기고, 원인을 찾기 어렵다.

### 운영 방법

```
초기:
1. 팀 레이어 0에서 공동 작성 (persona_architect.md 사용)
2. Git 저장소에 저장
3. main 브랜치 보호 (직접 push 금지)

작업 중 규칙 추가가 필요할 때:
1. 팀 채널에 "이 규칙을 추가하려는데 어떻게 생각해?" 공유
2. 팀 합의 후 PR로 업데이트
3. 규칙이 기존 작업에 영향을 주면 → 팀 레이어 2 체크포인트 즉시 실행
```

---

## 원칙 2: AI가 만든 코드도 사람이 책임진다

```
❌ "AI가 만든 코드니까 내가 다 이해 안 해도 돼"
✅ agent는 도구, 코드 품질 책임은 담당자에게 있다
```

### 리뷰 책임

AI agent를 사용했어도 코드 리뷰 절차는 동일하다.

```
코드 작성자:
- agent가 내린 암묵적 결정(조용한 결정)을 파악하고 있어야 한다
- skill_silent_decision_extraction.md로 확인한 결정 목록을 PR에 포함

코드 리뷰어:
- "AI가 만든 코드라 잘 모르겠다"는 핑계가 없다
- workflow_code_review.md 절차로 리뷰
- 특히 경로 추적 + 반례 탐색은 AI 생성 코드에 더 엄격하게 적용
```

### PR 설명에 포함할 내용

```
AI agent 사용 시 PR 설명에 추가:
- 사용한 페르소나/워크플로우
- agent가 내린 주요 결정 중 검토가 필요한 것
- 계획서(task_plan.md)와 달라진 부분이 있다면 이유
```

---

## 원칙 3: 실패 패턴은 팀 공유 자산

한 사람이 발견한 실패 패턴이 팀 전체의 자산이 돼야 한다.

### 공유 채널 운영

```
팀 채널 또는 문서에 기록:
- 발생한 문제 (어떤 상황에서)
- 원인 (왜 발생했는가)
- 해결/방지 방법

형식 자유, 기록이 중요
```

### failure_patterns.md를 팀 공유 문서로

```
개인이 발견한 새로운 실패 패턴 → PR로 failure_patterns.md에 추가
```

이 파일이 팀의 집단 학습 결과물이 된다.

---

## 원칙 4: 의존성 기준으로 작업 배분

```
❌ 담당 영역 기준으로 분리 (프론트/백엔드 등)
✅ 의존성 맵 기준으로 분리 (무엇이 먼저 되어야 하는가)
```

### 작업 배분 프로세스

```
1. 팀 레이어 0 완료 후
2. skill_dependency_mapping.md로 전체 작업 목록 + 의존성 맵 도출
3. 의존성이 없는 작업(병렬 가능) 파악
4. 팀원별 담당 작업 배분

배분 기준:
- 먼저 해야 하는 것 → 우선 배분
- 공유 인터페이스 정의 → 팀 전체 합의 후 진행
- 병렬 가능한 것 → 각자 동시 진행
```

### 공유 인터페이스 작업은 혼자 하지 않는다

여러 팀원이 함께 사용할 인터페이스(API, 데이터 모델, 공통 모듈)는 팀 합의 후 확정한다.

```
공유 인터페이스 확정 절차:
1. 담당자가 초안 작성
2. 팀 리뷰
3. 합의 후 architecture_rules.md에 추가
4. 이후 각자 구현
```

## 팀 레이어 2 체크포인트 운영

### 실행 시점

```
- N개 작업 완료마다 (3~5개 권장)
- 공유 인터페이스 변경이 생겼을 때
- 팀원 간 "이상한데?"가 공유됐을 때
```

### 누가 실행하는가

팀 전체가 함께 하거나, 담당자가 실행 후 결과를 공유한다.

```
옵션 A: 팀 전원이 모여서 새 스레드에서 함께 진행
옵션 B: 담당자 1명이 실행 → checkpoint_report.md → 팀 공유 → 이슈 논의
```

## 지식 이전 (온보딩)

새 팀원이 합류했을 때 AI agent 협업을 빠르게 시작할 수 있도록.

### 최소 온보딩 순서

```
1. README.md 읽기
2. architecture_rules.md 읽기 (현재 프로젝트 기준)
3. 기존 task_plan.md 1~2개 읽기 (형식 파악)
4. 기존 feedback_log.md 읽기 (팀이 이미 겪은 문제 파악)
5. 작은 작업 1개로 실습
```

### 신규 팀원이 흔히 하는 실수

```
- 구현 스레드에서 바로 검토 요청 → workflow_code_review.md 참고
- architecture_rules.md 없이 구현 시작 → 반드시 최신 버전 확인 후 시작
- 팀 합의 없이 규칙 로컬 수정 → 변경은 PR로
```

## 팀 컨벤션 권장 사항

팀마다 다를 수 있지만, 시작점으로 참고할 수 있다.

| 항목 | 권장 방식 |
|------|-----------|
| architecture_rules.md 관리 | Git + PR 리뷰 |
| 실패 패턴 공유 | failure_patterns.md PR |
| 팀 레이어 2 주기 | 3~5개 작업마다 |
| PR 설명 | 사용 페르소나 + 주요 결정 포함 |
| 공유 인터페이스 변경 | 팀 채널 공지 필수 |
