# 📈 연구 03 – 세션 기반 에이전트 브레인의 산업 구현 사례

> "많은 선도 기술 기업이 이미 프라이버시, 보안, 크로스 디바이스 연속성을 균형 있게 유지하기 위해 세션 기반, 선택적 메모리 에이전트를 프로토타입하거나 배포하고 있다."

*날짜*: 2026‑05‑22

---

## 1. Google – Gemini Nano + Gemini Pro 하이브리드
- **아키텍처**: 온‑디바이스 Gemini Nano(≤ 2 GB RAM)가 지연에 민감한 작업을 수행하고, 무거운 추론과 장기 메모리는 클라우드‑호스팅된 Gemini Pro에 **세션 토큰**을 통해 위임합니다.
- **세션 흐름**:
  1. 사용자는 Chrome‑OS, Android, 데스크톱 등 어느 기기에서든 로그인합니다.
  2. 클라이언트는 **수화(hydration) 페이로드**를 요청합니다 – 최근 작업 컨텍스트(지난 24 시간 프롬프트, 단기 임베딩)를 압축한 벡터입니다.
  3. Nano가 로컬에서 실행되고, 세션 종료 시 클라이언트는 **고가치 인사이트**(예: 새로운 API 스키마, 디자인 가이드)만 클라우드에 **푸시**합니다.
- **파쇄 메커니즘**: Soft purge는 로컬 캐시를 제거하고, hard purge는 Android `shred`와 같은 보안 삭제 방식으로 온‑디바이스 모델 전체를 삭제합니다.
- **참고 문서**: Google AI Blog (2025‑12) – *"Privacy‑First AI on the Edge"*; 내부 논문 *Gemini Session Architecture* (2025).

## 2. Microsoft – Copilot‑Studio와 AutoGen
- **아키텍처**: Copilot‑Studio는 **AutoGen**을 이용해 다중 에이전트 오케스트레이션을 수행합니다. 각 개발자 워크스테이션은 세션 상태를 보관하는 샌드박스된 **AutoGen 런타임**을 실행합니다.
- **세션 흐름**:
  1. `copilot init` 명령이 프로젝트‑스코프 브레인(SQLite + 벡터 DB)을 리포에 바인딩합니다.
  2. 코딩 중 브레인은 **실행 가능한 스니펫**(함수 시그니처, doc‑string)만 저장합니다.
  3. `copilot close` 시 런타임은 사용자가 **어떤 내용을 클라우드 지식 베이스에 동기화할지** 검토하도록 요청합니다.
- **파쇄**: `copilot purge --hard`는 로컬 DB와 모든 생성 artefacts를 삭제하고,`--soft`는 소스 파일은 유지하면서 브레인만 지웁니다.
- **참고 문서**: Microsoft Build 2025 – *"AutoGen for Enterprise"*; GitHub 레포 `microsoft/AutoGen` (v0.9.3).

## 3. Amazon – Bedrock Edge & Titan Hybrid
- **아키텍처**: Bedrock Edge는 사용자의 기기(Kindle, Fire Tablet 등)에서 축소된 Titan 모델을 실행합니다. 장기 메모리는 AWS Bedrock에 **세분화된 IAM 스코프**로 보관됩니다.
- **세션 흐름**:
  1. 디바이스는 MFA 후 **임시 세션 자격 증명**을 획득합니다.
  2. Edge 모델이 사용자 프롬프트를 스트리밍하고, Bedrock이 세션용 **컨텍스트 벡터**를 반환합니다.
  3. 세션 종료 시, 사용자가 승인하면 디바이스는 압축된 **지식 아티팩트**(예: 새로운 제품 분류 체계)를 업로드합니다.
- **파쇄**: Soft purge는 로컬 추론 캐시를 비우고, hard purge는 AWS KMS를 이용해 암호화된 세션 키를 삭제해 남은 블롭을 복구 불가능하게 합니다.
- **참고 문서**: AWS re:Invent 2025 – *"Secure Edge AI with Bedrock"*.

## 4. 오픈‑소스 – LangChain ↔ StateGraph
- **아키텍처**: LangChain `StateGraph` 추상화는 로컬 벡터 DB(예: Chroma)에 **상태 스냅샷**을 지속하도록 지원합니다. 세션‑기반 워크플로는 최신 그래프 스냅샷만 저장함으로써 구현됩니다.
- **세션 흐름**:
  1. `state.save()`가 로그아웃 전 스냅샷을 기록합니다.
  2. 사용자가 복귀하면 `state.load()`가 그래프를 복원합니다.
  3. 개발자는 **소프트/하드 파쇄** 훅을 구현해 스냅샷을 조건부로 삭제할 수 있습니다.
- **참고 문서**: LangChain Docs (v0.3) – *"StateGraph Persistence"*; 커뮤니티 레포 `langchain‑examples/session‑memory`.

## 5. 핵심 회고 (Honest Feedback)
1. **프라이버시 vs. 유용성 트레이드‑오프** – 컨텍스트를 많이 수화할수록 에이전트는 더 좋은 지원을 제공하지만, 프라이버시 노출 위험도 커집니다. **인간‑인‑루프 선택 필터**가 필수적입니다.
2. **벤더 락인** – 대부분 구현이 특정 클라우드 제공업체(Google, Microsoft, AWS)에 장기 브레인을 묶어두고 있습니다. 오픈‑소스 접근법(LangChain)은 이식성을 제공하지만, 대규모 스케일링에서는 제한적입니다.
3. **파쇄 안전성** – Hard purge는 OS 수준의 보안 삭제에 의존하는 경우가 많아 SSD에서는 신뢰성이 떨어집니다. 앞으로는 **암호화 후 삭제**(cryptographic shredding) 방식을 결합해 흔적을 완전히 제거해야 합니다.
4. **디바이스 이질성** – 저사양 디바이스는 전체 암호화 스냅샷을 저장할 수 없어 **소프트 파쇄**만 지원할 수 있습니다. 이는 모델의 보편성을 제한합니다.

## WithAi에 대한 시사점
- **이중‑브레인** 설계를 채택합니다: 즉시 작업을 위한 가벼운 온‑디바이스 브레인과 장기 지식을 위한 클라우드‑동기화 브레인을 구분합니다.
- **인간‑인‑루프** 기반 **큐레이션**을 구현해 중앙 브레인에 업로드하기 전 사용자가 반드시 검토하도록 합니다.
- **암호화 기반 파쇄**(cryptographic erasure)를 사용해 SSD에서도 데이터 잔존을 완전히 차단합니다.
- **플러그‑인 어댑터**를 제공해 전통적인 클라우드(Google, AWS)와 오픈‑소스(LangChain) 백엔드를 모두 연결할 수 있게 하여, 시스템을 **벤더‑중립**적으로 유지합니다.
