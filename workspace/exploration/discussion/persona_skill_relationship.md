# 페르소나와 스킬 — 관계·주입·관리

> 상태: 진행 중
> 최초: 2026-06-16 · 최종: 2026-06-16

## 테제 / 물음

MPA의 `personas/`와 `skills/`(analysis + tech)는 (1) 서로 어떤 관계인가,
(2) 어떻게 주입되는가, (3) 어떻게 관리할 것인가.
→ 첫 축으로 **개념 경계(관계)**부터 파고든다.

## 확립된 것 (사실 배선)

- **페르소나(9개)** = "누구로서". 역할·태도·LLM 편향 경고.
- **스킬** = `analysis/`(사고 기법 6) + `tech/`(스택 지식 5).
- **주입 비대칭:**
  - 페르소나 = `agent_rules.md` §2 라우팅 표가 **민다**(push, eager, 세션당 1개).
  - 스킬 = 라우팅 표에 없음. inject 파일 "스킬 참조"에서 단계 도달 시 **당긴다**(pull, lazy, 단계별 N개). tech는 더 조건적.
- **상호 참조는 삼각형:** 페르소나→inject, 스킬→페르소나, inject→둘 다. 계층이 아니라 웹.

## 오간 의견

### 라운드 1 — 에이전트 개시 입장

- ① "skills" 폴더가 이질적인 둘(analysis=방법론 / tech=레퍼런스)을 한 이름에 묶었다.
- ② 주입 비대칭은 버그가 아니라 합리적 분업(정체성 1 vs 도구 N).
- ③ Claude Code 자체 "skill"과 이름 충돌.

### 라운드 2 — 에이전트: 2×2 (태세/역량 × 이식성/종속)

- 셋은 진짜 셋이나 서로 다른 두 축에서 갈린다.
- 폴더는 "역량인가?" 축으로 잘렸는데, 관리상 아픈 축은 "이식성 vs 프로젝트종속".
- persona+analysis는 같은 생명주기(MPA 업글로 함께 따라옴)인데 따로, analysis+tech는 다른 생명주기인데 한 폴더.

### 라운드 3 — 사용자: 합성 모델 `개발자(자바)`, `개발자(파이썬)`

- **페르소나 = 역할(안정 베이스), 스킬 = 끼워 넣는 능력.** 베이스 클래스 + 믹스인.
- 이 모델이 에이전트 ①(analysis≠tech 이름 문제)을 교정:
  - 둘 다 "역할이 꺼내 쓰는 능력" → 'skill' 이름은 맞다.
  - **진짜 솔기는 "방법론 vs 레퍼런스"가 아니라 "치환적(tech, 보통 1개 선택) vs 누적적(analysis, N개 적층)".**
- **배선 구멍 발견:** 합성이 맞다면 tech 스킬은 reviewer·designer도 써야 하는데, 현재 `layer1_implement`에서만 참조 → 합성이 절반만 구현됨.

### 라운드 4~5 — 다중 스킬·다중 스택 → 2층 모델

- 사용자: `개발자(c#+유니티)`, `개발자(java, spring, js, vue)` — 한 페르소나가 여러 스킬·여러 스택.
- 에이전트 이분법이 3연속 깨짐(과단순화 신호): "치환/누적"(R3), "선택 시점"(R4) 모두 폐기.
- **살아남은 구조 = 2개 층:**

| | 가용 집합 (무엇을 아나) | 활성 부분집합 (이번에 꺼내는 것) |
|--|--|--|
| tech | 프로젝트 정의(Layer 0 선언) — 프로젝트마다 다름 | 작업이 닿는 것만 (프론트→js+vue) |
| analysis | MPA 고정 6개 | 단계가 필요로 하는 것 |
| persona | 9개 | 라우터가 ~1개 |

## 확립된 것 (추가)

- 관계 모델 = **합성(composition)**: 페르소나(역할) ⊗ 스킬(능력). 계층/상속 아님. **완전 합성**(임의 페르소나 ⊗ 임의 스킬)으로 합의.
- 스킬 = **2층**(가용 집합 / 활성 부분집합).
- tech와 analysis의 **활성 부분집합 선택 메커니즘은 동일**(작업 필요 → inject가 런타임 pull). 차이는 **가용 집합의 출처뿐**(tech=프로젝트, analysis=MPA). → 라운드 2 "이식성 vs 종속" 축이 진짜 축.
- 스킬 의존(유니티→c#)은 별도 명시 불필요 — **프로젝트 가용 집합 통째 선언**이 흡수.
- tech가 구현에만 묶인 현재 배선 = **미완성**(완전 합성과 모순).

### 라운드 6 — 주입 메커니즘 + "환경 기술의 집"

- push/pull 비대칭은 **정당**: 페르소나는 결정 시점이 세션 시작(작업 유형) → router push, 스킬은 결정 시점이 단계 중 → inject pull.
- 사용자: "project_rules에 기본 환경 기술, 작업 시 필요에 따라 페르소나·스킬 주입" — 2층 모델의 시스템 레벨 재진술. 뼈대 합의.
- **쟁점: 가용 tech 집합의 집 = project_rules vs project_identity**
  - project_rules = 행동 지시, 항상 로드. project_identity = 사실, 메모리 필요 유형일 때만 로드.
  - tech 스택은 "사실"이고 필요 시점이 곧 코드 작업 시점 → 에이전트 입장 **project_identity**.
- "필요에 따라 주입"의 결이 둘: 페르소나=**작업 단위**(router), 스킬=**단계 단위**(inject). 트리거가 두 곳에 산다.

## 확립된 것 (추가)

- push(페르소나)/pull(스킬) 비대칭 = 결함 아님. **결정 시점**(세션시작 vs 단계중)의 반영.
- 주입 트리거는 두 곳: router(페르소나) + inject(스킬). 관리 시 "어디를 고치면 주입이 바뀌나"의 기준.

### 라운드 7 — 본주제 복귀: "가장 효율적인 주입 타이밍"

> 사용자가 본주제로 끌어올림(스택 저장 위치는 기반 곁가지였음).

- 효율을 둘로 가름: **(a) 컨텍스트 효율**(안 쓸 걸 안 부름) vs **(b) 결과 효율**(결정 순간에 있어 재작업 없음). 둘은 충돌.
- **단일 원리:** 각 단위를 *그것이 좌우하는 결정의 바로 직전 — 신뢰 가능한 가장 늦은 시점*에 주입.

| 단위 | 좌우하는 결정 | 주입 타이밍 |
|--|--|--|
| 페르소나 | 세션 전체 행동(태도) | 세션 시작 (router push) |
| analysis 스킬 | 특정 단계 사고 | 그 단계 진입 (inject pull) |
| tech 스킬 | 어느 스택 코드 | 작업 범위가 스택 드러낸 직후, 부분집합만 |

- push/pull 비대칭 = 이 원리 하나의 결과. **타이밍은 단위 성격이 아니라 "결정 시점"이 정함.**
- 함정: pull은 에이전트가 트리거에 *실제 도달*해야 작동. 고위험 스킬(침묵결정추출 등)은 miss 비용 커서 **원리보다 살짝 일찍** 보장하는 게 나음.
- 에이전트 입장: **(b) 결과 효율 우선** — 빠진 스킬의 재작업 비용 > 토큰 절약.

### 라운드 8 — 단위는 "단계", (a)/(b) 딜레마 해소

- 사용자 모델: **작업 = 단계 정의. 각 단계 = {할 일 + 페르소나 + 스킬} 쥐여주고 지시.** 주입 원자 단위 = 단계.
- 에이전트 라운드 7 정정: "페르소나=세션당 1개" 틀림. MPA는 이미 단계마다 페르소나 교체(설계=task_designer, 비평=plan_critic, 검증=code_reviewer). **push/pull 비대칭은 개념상 없음** — 둘 다 단계에 붙음. 비대칭은 "첫 단계 페르소나 vs 나중 단계 스킬" 착시.
- **(a)/(b) 딜레마 증발:** 페르소나·스킬을 단계 정의에 묶으면, 단계 진입 시 *선언한 것만* 들어옴 → 최소(a)이자 완전(b). 딜레마는 타이밍을 단계 정의와 분리해 볼 때만 존재.
- **해상도 차이:** 페르소나=국면(phase, 설계/구현/검증)에 붙어 지속 / 스킬=국면 내 미시 스텝에 붙어 들고남. "단계"는 두 결.

## 확립된 것 (최종 수렴)

- **주입 타이밍 답:** 단계 진입 시, 그 단계가 선언한 페르소나·스킬을. **타이밍은 독립 변수가 아니라 단계 정의의 종속 결과.**
- 부담 이동: "언제 주입하나" → **"각 단계가 자기 페르소나·스킬을 정확히 선언했는가."**
- 단계 정의 출처: 알려진 유형 = `workflows/`+`inject/`에 기정의 / 새 작업 = `task_designer`의 plan.md 산출.

### 라운드 9 — PM/서브스레드 = 기존 결론의 실현, 새 변수 = 스킬

> 사용자: 메인 스레드=PM, 필요 시 맥락·문서·페르소나·스킬을 서브스레드에 주고 결과 받아 진행.
> 이 주제는 [multi_agent_utility.md] R1~16에서 대부분 정착됨 — 중복 금지, 보탤 것만.

- PM 모델 = 그 문서 결론의 실현형: **R4(위임=1차 효용) + R10(격리=틀,페르소나=부품) + R12(절차가 단계별 페르소나 호출)**. "맞냐"가 아니라 "이미 맞다고 닫힘".
- **반드시 들고 올 제약 둘:**
  1. **구현(write)은 단일 순차** (R7~8) — 코드베이스=공유 가변상태, 의미충돌. PM은 read 단계만 펼치고 write 직렬화.
  2. **입도 = 국면(phase)마다, 미시 스텝마다 아님** (R14) — 맥락 재주입 비용이 협소화 이득 잠식.
- **이 토론 고유 기여:** 사용자가 디스패치 묶음을 4개로 호명 — **맥락·문서·페르소나·스킬**. multi_agent는 스킬을 별도 주입물로 안 다룸. PM 모델이 R8 "단계=묶음"을 **문자 그대로 서브스레드 프롬프트**로 만듦.
- **R16 캐시 경계를 스킬에 적용 → 스킬 두 종류가 디스패치에서 갈림:**

| | 정체 | 디스패치 |
|--|--|--|
| tech 스킬 | 관점-중립 사실 | 캐시/문서로 싸게 주입 |
| analysis 스킬 | 관점-특수 시선("~의 눈") | R16 "보안의 눈"과 동형 — 캐시 불가, 매번 지불, **페르소나와 한 몸** |

## 확립된 것 (추가)

- PM/서브스레드는 새 발명 아니라 multi_agent_utility 결론의 실현. 제약(write 단일순차 · 국면 입도) 동반 필수.
- 디스패치 페이로드 = {맥락(캐시), 문서, 페르소나, 스킬}. = R8 단계묶음의 문자적 실현.
- **tech 스킬 = 캐시 가능(사실) / analysis 스킬 = 캐시 불가(시선), 페르소나에 밀착.**

### 라운드 10 — 사람 프레임 vs 에이전트 프레임 (핵심 전환)

> 사용자: 우리가 사람 기준으로 봐서 분리함. 사람=고정 존재+역할 부여+도구. 그러나 **분석 스킬은 분석가가 가질 수밖에 없는 내용**(구성적). 분리되는 건 **도메인**(프로그램 분석가 vs 광고 분석가)=기반 지식=스킬 역할.

- **라운드 1 흐림 해소:** analysis = 페르소나의 **구성요소(constitutive)**, 독립 부품 아님. "반례 보는 눈 없는 분석가"는 분석가가 아님.
- **진짜 주입물 = 도메인**(프로그램/광고; java/spring) = 같은 역할을 *이 맥락의* 전문가로 특화하는 갈아끼움 변수.
- 우리가 "스킬"로 뭉친 것 = **두 입력**: ① 역할 키트(페르소나+구성 기법=analysis) ② 도메인(특화 변수).
- **프레임 전환:** 에이전트는 고정 존재+도구가 아니라 **매 디스패치 전문가 합성**(R13 합치). 페르소나/스킬 분리 자체가 사람 흉내. analysis를 "집어 드는 도구"로 본 게 사람 프레임 잔재.
- **시험:** counterexample이 designer·critic·reviewer 셋에서 쓰임 → 모순 아니라 **여러 페르소나에 공통 구성적** → DRY 팩토링. `skills/analysis/` = 제3 범주 아니라 **재사용 메커니즘**.

## 확립된 것 (핵심 수렴)

- **analysis = 페르소나 구성요소** (별도 파일은 DRY 팩토링일 뿐, 개념상 페르소나의 일부).
- **진짜 주입 "스킬" = 도메인/기반지식** (tech 포함). 갈아끼움 변수.
- 에이전트 = 전문가 **합성**. 페르소나-스킬 분리는 사람 프레임 산물.
- → 라운드 2 "이식성(역할키트) vs 프로젝트종속(도메인)" 축이 **가장 근본 축**으로 정당화됨(구성 vs 특화).

### 라운드 11 — 기준 확정: "역할 종속성" + 절단

> 사용자: **역할에 종속되는 건 페르소나, 역할 아닌 지식·도메인은 스킬.**

- **단일 시험:** 내용의 관련성이 *역할에 따라* 정해지면 페르소나 / *역할 무관*이면 스킬. (R10 "구성적 vs 도메인"보다 단순 — 두 흐린 범주 대신 한 시험.)
  - 반례탐색 = 평가 역할만 씀(문서작성자 X) → 역할 종속 → 페르소나.
  - java = 구현·리뷰·설계 다 씀 → 역할 무관 → 스킬.
- **합성 구조 설명:** 사실(java)=역할무관=스킬 / 그 사실을 *어떻게 쓰나*=역할종속=페르소나. → `페르소나(적용) ⊗ 스킬(사실)`.
- **분류 아니라 절단:** 섞인 파일은 둘로 벰. "보안코딩" = OWASP 지식(스킬) + 사냥하는 눈(페르소나). 칼선 따라 분리.
- **귀결:** `skills/analysis/`는 역할종속 → **스킬 아님**, 페르소나로 이동.

## 확립된 것 (경계 기준 — 토론 핵심 결론)

> **역할 종속 = 페르소나 / 역할 무관(지식·도메인) = 스킬.**
> - 페르소나 = 역할 + 그 역할에 종속된 기법(현 analysis 흡수).
> - 스킬 = 역할 무관 도메인/기반 지식(tech + domains).
> - 기준은 절단 도구: 섞인 파일을 역할종속/무관으로 벤다.
> - 합성: 페르소나(적용 방식) ⊗ 스킬(사실). 에이전트는 매 디스패치 이 둘로 전문가 합성.

### 라운드 12 — 도메인별 분석가: 분리 vs 합성

> 사용자: 금융/코드/마케팅 분석가 — 페르소나 분리가 현실적이지 않나?

- **결정적 반론 = 조합 폭발:** 분리하면 R역할 × D도메인 = R×D 페르소나. 합성은 R+D(페르소나 ⊗ 도메인스킬). 곱 vs 합.
- **"현실적" 느낌 = 사람 프레임:** 인간은 도메인 교체 비용 高 → R×D가 *직업*으로 굳음(R13 사회적 캐시). 에이전트는 교체 ~0 → 캐시 불요. 직감은 인간 비용구조의 현실.
- **기준의 한계 드러남 — 교차항:** 셋으로 갈림.
  - 일반 분석가의 눈(도메인무관·역할종속) → 페르소나 코어.
  - 금융 *사실* → 스킬(깨끗).
  - **금융 분석 *방식*(역할×도메인 교차항)** → 회색지대. `페르소나 ⊗ 스킬` 곱이 안 떨어짐.
- **해소(둘 다 곱 유지):** ① 교차항을 도메인 스킬에 흡수(사실+도메인 분석법) → 페르소나 1개. ② 자주+이질+고실패면 합성을 **캐시한 페르소나**(R15 세 조건) — 단 파생·캐시지 원시 아님.
- **에이전트 입장:** 원시 분리 말 것(폭발+사람프레임). 기본 합성, 정당할 때만 캐시.

## 확립된 것 (추가)

- 도메인별 역할은 **원시 페르소나로 분리하지 않음** — R×D 폭발. `역할 ⊗ 도메인` 합성(R+D)이 기본.
- 인간 직업식 분리 = 사람 프레임 산물(R13). 에이전트는 합성, 필요 시 캐시.
- 교차항(도메인-특수 분석 방식) 존재 → 도메인 스킬에 흡수 또는 R15 조건부 캐시 페르소나.

### 라운드 13 — 도메인의 역할-면 + 저장 ≠ 소유 (교차항 해소)

> 사용자: 금융 도메인에 금융분석스킬을 넣는 게 맞다. (R12 해소 ① 채택)

- **왜 옳나:** 분석법을 페르소나에 넣으면 페르소나가 D도메인 분석법 다 짊어짐(페르소나 쪽 폭발). 도메인에 넣으면 사실과 응집, 페르소나는 일반 태세만. R12("역할×도메인 곱 금지")의 반대 방향("도메인×역할 곱 금지").
- **기준과의 외견상 충돌:** 금융 분석법 = 역할 종속(분석가의 일) → 기준대론 페르소나 쪽인데 도메인에 저장? → **도메인을 역할-면으로 구조화**하면 풀림:
  - 금융 도메인 = {사실(공통) / [분석가 면] 분석법 / [개발자 면] 구현관례 / …}
  - `분석가 ⊗ 금융` → 금융 도메인의 "분석가 면"을 활성화.
- **메타 원칙 (10~13 관통):** **개념적 소유(R11 기준=언제 활성) ≠ 물리적 저장(응집·DRY).**
  - analysis 기법: 소유=페르소나(역할종속), 저장=`skills/analysis/` 공유(DRY).
  - 금융 분석법: 소유=분석가(역할종속), 저장=금융 도메인(응집).
  - 기준은 *활성 시점*을 정하고, 저장은 *응집·중복제거*로 별도 최적화. **둘을 섞지 말 것.**

## 확립된 것 (최종 결론)

> **관계 모델 (확정):**
> - **역할 종속 = 페르소나 소유 / 역할 무관 = 스킬(도메인) 소유** (R11 기준).
> - **페르소나(적용 방식) ⊗ 스킬(도메인 사실)** — 곱이 아니라 합(R+D). 도메인별 역할을 원시 페르소나로 분리 금지(R12).
> - 도메인 스킬은 **역할-면으로 구조화** — 합성이 맞는 면을 활성화(R13).
> - **개념적 소유 ≠ 물리적 저장** — 소유는 R11 기준(활성 시점), 저장은 응집·DRY로 따로 최적화(R13).
> - 에이전트는 사람과 달리 **매 디스패치 전문가 합성** — 페르소나/스킬 분리·도메인 캐시는 사람 프레임 잔재, 곱 대신 합으로 폭발 회피(R10·R12).
>
> **주입 (확정):**
> - 단계 진입 시 그 단계가 선언한 페르소나·(도메인)스킬 주입 — 타이밍은 단계 정의의 종속 결과(R7~8).
> - PM/서브스레드는 [multi_agent_utility] 결론의 실현 — write 단일순차·국면 입도 제약 동반(R9).

### 라운드 14 — spring.md 해부: "사실 vs 스타일" 기각, 역할-범위 확증

> 시험: spring 관례 = 사실인가 스타일(역할종속)인가? → 실제 `skills/tech/spring.md`에 칼.

- **"사실 vs 스타일"은 틀린 질문.** 레이어 책임("Controller는 변환만")은 *스타일/관례*인데 *역할 무관*(구현·리뷰·설계 다 전제) → 도메인 스킬. 처방적이라는 게 페르소나로 안 만듦.
- **확증된 기준 = 역할 범위:**
  - 프로젝트 전역 처방(모든 역할 따름) → 도메인 스킬.
  - 역할 고유 태세(그 역할의 시선·가치) → 페르소나.
  - *스타일이든 사실이든 전역=스킬, 역할고유=페르소나.*
- **R13 역할-면 실물 확인:** "모르면 물어볼 것"(예외 노출?·로딩 전략?) = `silent_decision_extraction` 태세(페르소나 소유)가 spring에 적용된 **교차항**. 분해: "가정 말고 물어라"=태세(페르소나) / "spring에선 *이것들*을 물어라"=도메인 특수 결정점(도메인 저장, `[설계자/구현자 면]` 활성).
