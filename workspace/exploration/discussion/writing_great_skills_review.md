# Matt Pocock "The Missing Manual: How to Write Great Skills" 검토 — MPA와의 접점

> 상태: 종료
> 최초: 2026-07-16 · 최종: 2026-07-16

## 테제 / 물음

Matt Pocock의 AI Engineer 강연 "The Missing Manual: How to Write Great Skills"(및 이를 해설한 AgentOS 채널 영상, https://youtu.be/YLq04CDeOTE)가 제시하는 "좋은 스킬(에이전트 지침) 작성법" 프레임워크를 검토한다. 이 프레임워크가 my-pacemaker-agent(MPA)가 이미 확립한 철학(`discussion_log.md`의 테제 1~10)·구조(`.mpa-workspace/`의 persona/skill/inject 체계)와 어디서 겹치고, 어디서 MPA가 놓친 것을 보완하며, 어디서 실제로 충돌하는지 평가한다.

**자료 확보 경위(근거):**
- 원 영상(9분짜리 한국어 해설, 자동 더빙)은 YouTube 스크립트 패널이 로드되지 않아(스피너 무한 로딩) 전사(transcript)를 직접 뽑지 못함 — 대신 영상 설명란(4단계 체크리스트 요약)과 화면 슬라이드 일부(예: "Leading Word: Vertical Slice", SKILL.md/TEMPLATE.md 2계층 다이어그램)를 스크린샷으로 확인.
- 원 강연 소스: [`mattpocock/skills`](https://github.com/mattpocock/skills) 저장소의 `README.md`, 그리고 강연 내용을 그대로 응축한 메타 스킬 파일 `skills/productivity/writing-great-skills/SKILL.md` + `GLOSSARY.md`를 raw.githubusercontent.com에서 직접 받아 전문을 읽음. **이게 해설 영상보다 더 1차적이고 정확한 소스** — 영상은 이 문서의 요약본이다.

## 오간 의견

### 라운드 1 — 에이전트 (2026-07-15)

**프레임워크 골자:** 스킬의 최상위 가치는 **Predictability**(예측가능성) — "매번 같은 결과물"이 아니라 "매번 같은 프로세스"를 내는 것. 이걸 4개 축으로 조작화한다:

| 축 | 핵심 레버 | 실패 모드 |
|---|---|---|
| **Invocation(트리거)** | model-invoked(description 상주, context load) vs user-invoked(설명 없음, cognitive load) | — |
| **Information Hierarchy(구조)** | Steps → in-file Reference → disclosed Reference(별도 파일), progressive disclosure | Sprawl |
| **Steering(유도)** | Leading Word(사전학습된 개념 재활용), Completion Criterion(checkable + exhaustive) | Premature Completion, Negation |
| **Pruning(가지치기)** | Single Source of Truth, No-Op test("삭제해도 행동이 안 바뀌면 지운다") | Duplication, Sediment |

**내 입장 — 이 프레임워크가 MPA에 주는 것:**

1. **Predictability 재정의는 우리 테제 3·7보다 더 조작가능하다.** 우리는 "신뢰는 구조에서 온다"(테제 3), "위임 구조의 품질이 실패 비용을 결정한다"(테제 7)까지는 갔지만, "구조가 좋다"를 무엇으로 검증하는지는 사후적(실패가 적시에 발견됐는가)이었다. Pocock의 정의는 사전적 검증 기준을 준다 — "같은 입력에 같은 프로세스를 타는가"를 스킬 자체에 대해 직접 관찰할 수 있다. 이건 동일한 결론을 다른 각도에서 재발견한 것이라기보다, **우리 테제들이 아직 갖추지 못한 측정 가능성(operationalizability)을 메운다**고 본다.

2. **Context Load / Cognitive Load 이분법은 MPA에 없는 감사 도구이고, 이 렌즈로 보면 `agent_rules.md` 자체가 위반 사례일 가능성이 있다.** 이 파일은 세션마다 항상 상주(항상 로드)하는데, 분량이 상당히 길다. Pocock 프레임으로 재질문하면: "이 규칙 중 삭제해도 내 행동이 달라지지 않는 줄이 있는가?"(No-Op test), "라우팅 표의 각 행이 실제로 서로 다른 branch인가, 아니면 동의어 반복(duplication)인가?" — 이건 검증 안 된 가설이고, 실제로 `agent_rules.md`를 이 두 테스트로 감사해본 적은 없다.

3. **Negation 경고는 MPA 문서군의 문체와 정면으로 부딪힌다.** 방금 읽은 `agent_rules.md`·`discussion_mode.md`·`discussion_partner.md`는 "~하지 않는다", "~을 생략한다", "동조하지 않는다", "완료로 인정하지 않음" 같은 부정형 서술이 매우 많다. Pocock의 주장(negation은 금지 대상을 오히려 컨텍스트에 끌어들여 활성화시킨다)이 맞다면, 이 문서들의 핵심 규칙 다수가 스스로 약화되고 있을 수 있다. 다만 이 주장 자체도 "하드 가드레일은 부정형으로 남기고 긍정형과 병기하라"는 예외를 두고 있어, MPA의 GATE 1/2 같은 하드 게이트는 원래 예외 범주에 들어갈 수도 있다 — 이 구분을 어디서 그을지가 열린 질문이다.

4. **Leading Word 개념은 채택할 가치가 있는데, MPA는 이미 암묵적으로 쓰면서 원칙화는 안 했다.** GATE 1/2, Zone 1/2/3, Tier 1/2/3, "실행 가능한 오해" 같은 용어가 사실상 leading word 역할을 하고 있다(반복될 때마다 같은 행동을 앵커링). 하지만 Pocock처럼 "이 단어를 왜 골랐는가, 사전학습 prior를 어떻게 재활용하는가"를 의도적으로 설계하지는 않았다.

5. **Router Skill 패턴은 `agent_rules.md`의 라우팅 표와 정확히 같은 구조다.** "user-invoked 스킬이 늘어나면 cognitive load를 줄이기 위해 하나의 라우터 스킬을 둔다"는 원칙을, MPA는 §2의 라우팅 표(발화 패턴 → 유형 → 로딩 파일)로 이미 구현하고 있다. 서로 다른 두 방법론이 독립적으로 같은 해법에 도달했다는 점에서, 이 패턴 자체의 타당성에 대한 교차 검증으로 읽을 수 있다.

6. **No-Op test는 강력하지만 "model-relative, not reader-relative"라는 단서가 중요하다** — 어떤 줄이 no-op인지는 논쟁이 아니라 실제로 스킬을 돌려봐야 판정된다는 것. MPA에는 "정정 회고"(세션 종료 시 자가 회고)가 있지만, 이건 실패를 잡아내는 장치이지 **평시에 규칙 자체의 사문화(死文化)를 정기적으로 감사하는 장치는 아니다.** Sediment(가지치기 없이 방치되면 쌓이는 낡은 층) 개념이 MPA의 `agent_rules_detail.md`·`upgrade-candidates/` 누적 구조에 그대로 적용될 위험이 있다고 본다.

**아직 검증 안 된 것 — 내가 확신을 갖고 주장하지 않는 부분:**
- 원본 20분 강연의 실제 발화 내용(사례, 청중 반응, 강조점)은 스크립트를 못 뽑아 못 봤다. 위 요약은 강연의 응축본인 `SKILL.md`/`GLOSSARY.md`와 해설 영상 설명란에 근거한 것이지, 강연 전체를 직접 본 것은 아니다.
- Pocock README가 제시하는 더 넓은 철학(grilling/공유 언어/CONTEXT.md, TDD 피드백 루프, ball-of-mud 방지)은 스킬 작성법보다 더 큰 그림인데, 아직 MPA의 메모리 계층(Tier 1~3, `project_identity.md` 등)과 정밀 대조는 안 했다.

## 확립된 것

- **Predictability 재정의**(같은 프로세스 ≠ 같은 결과물)는 이견 없이 유효.
- **"라우터니까 항상 상주"는 Pocock의 실제 정의(Router Skill = user-invoked, zero context load)와 반대**라는 점, 그리고 `agent_rules.md`는 실제 라우팅 표(§2, ~30~40줄)만 상주를 정당화할 수 있고 나머지 ~340줄(GATE 절차·완료 처리·실패비용 추정 등)은 해당 트리거 시점에 온디맨드로 분리되어야 한다는 데 **사용자 동의**.
- **Leading Word 원칙**(기성 용어 재사용 > 자체 조어) 적용에 **사용자 동의**. 단, `GATE`/`Zone 1·2·3`/`Tier 1·2·3` 중 구체적으로 무엇을 어떤 기성 용어로 바꿀지는 아직 정해지지 않음.
- **Negation 경고는 영상 자체의 주장이 아니라 에이전트가 GLOSSARY.md에서 추가로 가져온 것**임을 에이전트가 자인 — 이 주장 자체를 MPA에 적용할지는 이번 토론 범위에서 제외(별도 열린 질문으로 이월).
- **"작업"이라는 단어가 `agent_rules.md` 안에서 최소 3가지 의미(태스크 동의어/비-태스크 활동/산출물)로 충돌**하고 있음을 실증(grep) — 리딩워드(강한 prior 재활용)와 구별되는 "일상어의 은밀한 협소화" 실패 사례로 확정.
- **"하네스"→"MPA 시스템" 교체(2026-06-04 완료 태스크)는 리딩워드 원칙 위반이 아니라, 애초에 서로 다른 두 개념의 정확한 재명명**이었음을 사용자가 확인 — 이 두 용어는 앞으로도 각자의 자리에서 계속 구분해서 쓴다.
- **용어집은 MPA(방법론 공통)와 프로젝트(고유) 양쪽에 각각 필요**하다는 데 사용자 동의. 다만 실제 구조는 2단(MPA/프로젝트)이 아니라 이미 존재하는 3단(탐구 언어 `workspace/exploration/` / 배포되는 MPA 운영 언어 `.mpa-workspace/` / 프로젝트 고유 도메인 언어 `workspace/memory/domains/`)에 대응시키는 쪽으로 정리됨.
