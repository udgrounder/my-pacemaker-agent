# 외부 도구·플러그인 큐레이션

> 상태: 완료 (개발 태스크 `20260625_tools_guide_system` 완료로 종결, 2026-07-03)
> 최초: 2026-06-25 · 최종: 2026-06-25 (Round 12)

## 테제 / 물음
유용한 외부 도구·플러그인·MCP를 어떻게 큐레이션해서, 에이전트가 (로컬 환경 전반에서) 적시에 쓰게 만들 것인가?

관련 토론: [[mcp_server_and_operations]] — 그쪽은 "고른 도구를 어떤 운영 모델(CLI/MCP/연결상태)로 붙일지", 이쪽은 "애초에 무엇이 가치 있고 어떻게 고를지".

## 오간 의견

### Round 1
- 사용자: graphify, serena 같은 유용한 외부 도구·플러그인을 정리해서 에이전트가 쓰게 하고 싶다. 범위는 로컬 환경 전반, 결과물은 "유용한 도구 큐레이션(논의)".
- 에이전트(사실 확인):
  - **serena** = LSP 기반 심볼 단위 코드 탐색·편집 MCP. 좁고 깊음, 실시간 정확 질의. (이미 `.serena`로 설치됨)
  - **graphify** = 코드+문서+PDF+미디어를 "쿼리 가능한 지식 그래프"로 변환(Tree-sitter AST + LLM). 넓고 관계 중심, 사전 인덱싱. CLI/MCP/스킬/라이브러리로 제공.
- 에이전트(입장): 사용자가 든 두 예시는 **우연히 같은 범주**(대규모 코드베이스 이해·탐색)다. 이건 큐레이션을 시작하기 전에 "축"을 정해야 한다는 신호다.

### Round 2 (에이전트)
- "유용함은 도구 속성이 아니라 상황의 함수" — graphify는 대규모 다언어 repo엔 강하지만 마크다운 repo(MPA)엔 무용.
- "존재 ≠ 사용" — 이 환경은 도구 과잉. 목록 추가로는 적시 선택 간극이 안 메워짐.
- 도구는 단일 덩어리가 아니라 라이프사이클 단계(설치·등록·인덱싱·활용·유지)별로 단위가 다름. graphify 예: 바이너리=머신1회, 인덱스=프로젝트마다, 갱신=반복.
- 가이드는 두 층: (a) 도구별 사실(전역 재사용) + (b) 프로젝트별 켬/끔 판단(`project_rules.md`).

### Round 3
- 사용자: 강제 설치·플로우 주입은 안 한다. **의사결정 지원 설명서**가 목적 — "툴 설치 시 드는 고민을 정리해 다음에 같은 고민을 줄인다." 설명서 다 읽고 실행 검증하기엔 시간이 너무 든다.
- 사용자: Round 2의 단계 표가 "많은 정보를 한꺼번에 주고 의문을 해소"하는 바로 그 형태다.
- 사용자가 추가로 든 고민(=스키마 항목): 컴퓨터당 1회 설치 / 프로젝트별 설정 / 스킬 연결 / **팀원 공유 방법** / 소스 변경 시 재인덱싱 수동인지 자동인지.
- 에이전트(긴장 제기): "검증할 시간이 없어 만드는 가이드"인데 "검증 안 한 가이드는 틀려서 오히려 혼란을 키운다" — 이 모순을 풀어야 함.

### Round 4
- 사용자(정정): README를 안 읽는 게 아니다. README는 보통 **효과·유용성·설치 방법**은 다루지만 **실전 운영**(컴퓨터당 1회+프로젝트별 설정, 팀 공유, 재인덱싱 등)을 빼먹어서, 그 빠진 부분을 정리하는 게 목적이다.
- 사용자: 항목 추가 — **결과물(산출물)이 어디에 저장되는지.**
- 에이전트: 이 정정은 긴장을 약화가 아니라 강화한다 — README가 빼먹는 영역은 정의상 "실제 운영해봐야 아는 것"이라 실측 검증으로만 채워진다. 가이드의 정체성 = **제작자 관점 README와 운영자 관점 사이의 간극을 메우는 문서.**

## 확립된 것
- 토론 모드. 태스크·게이트 미적용.
- serena/graphify 정체 확정 (Round 1).
- 산출물 성격: **강제 X, 의사결정 지원 설명서.** (Round 3)
- 가이드는 두 층 — (a) 전역 도구 사실집 + (b) 프로젝트별 켬/끔 판단. (Round 2)
- `workspace/project_rules.md`가 이미 (b) 자리로 존재·사용 중.
- 도구 하나가 라이프사이클 단계별로 다른 설치 단위를 가짐. (Round 2)

### 가이드 항목 스키마 (Round 4 — 9개로 안정화 중)
> 정체성: README가 다루는 "효과·유용성·설치"의 **뒤를 잇는, 운영자 관점 항목**만 모은다.
1. 무엇인가 / **언제 유용한가**(맞는 케이스) — 켤지 말지의 1차 판단
2. 설치(바이너리) — 머신 전역 1회?
3. 연결 방식 — 스킬 / MCP / CLI 중 무엇으로?
4. 프로젝트별 설정 — 무엇을 프로젝트마다 해야 하나
5. 팀원 공유 — 동료가 같이 쓰려면? (git 커밋 vs 각자 설치 등)
6. 유지보수 — 소스 변경 시 재인덱싱 수동/자동?
7. 비용·주의·보안 — LLM 호출 비용/시간, 인덱스 stale, **코드가 외부 LLM에 전송되는지**
8. 되돌리기 — 제거·비활성화
9. **산출물 저장 위치** — 결과물 생성형 도구만. 어디 저장되나 + git에 넣나/gitignore하나 (5·8과 연동)

### graphify 검증 노트 (Round 5 — README 정밀 독해, 실행 검증 전)
> 출처: README 출력 구조 섹션. ⚠️ 실제 설치·실행으로 100% 확정 전.
- **4 프로젝트별 설정**: 전통적 "config 파일"은 **없음.** 대신 프로젝트 루트에서 명령을 돌리면 거기에 산출물이 격리 생성되는 모델. 제외 규칙은 `.graphifyignore`(프로젝트 루트, `.gitignore` 유사, 선택).
  - → 사용자가 상상한 "root에 설정 파일이 생기나?"의 답: 설정이 아니라 **출력**이 root에 `graphify-out/`으로 생긴다.
- **9 산출물 저장 위치**: 프로젝트 루트 `graphify-out/` — `graph.html`, `GRAPH_REPORT.md`, `graph.json`, `cost.json`, 캐시 디렉토리.
- **6 유지보수(재인덱싱)**: 셋 다 지원 — 수동 `graphify ./docs --update`(변경분만) / 자동 `graphify hook install`(git commit 후 AST만 재구성, API 비용 없음) / 실시간 `--watch`.
- **5 팀 공유(연동)**: `graphify-out/`을 git 커밋하면 인덱스 공유 vs `.gitignore`하면 각자 생성 — 운영 결정. 자동 최신화는 팀원 각자 `hook install`.

### Round 6 — 왜 README에서 자꾸 막히나 (가이드의 동기 뿌리)
- 사용자: README는 원래 이렇게 쓰나? 내 이해가 다른 건가? 왜 이런 글에서 자꾸 의문이 생기나?
- 에이전트: 이분법 거부 — README도 정상, 사용자도 정상, **장르 불일치**가 원인.
  - README 장르 = *first-run 레시피*(설치→quickstart→기능). 운영 질문(1회/프로젝트별·팀공유·재인덱싱·출력위치)을 담는 칸이 장르에 없음.
  - 사용자 독해 = "운영으로 들이면?" 모드 → 설치를 **시스템 도입 결정**으로 봄. 결핍 아니라 운영자/시스템 사고(MPA 만드는 사람의 사고와 일치).
  - 근본: 사용자는 **멘탈 모델**(뭐가 어디 사는지의 그림)을 원하는데 README는 **레시피**(명령)를 줌 → 모델이 안 세워져 "정리 안 됨"으로 느껴짐.
  - 도구 성숙도 ∝ 운영 문서 존재: 성숙한 도구(Terraform/Docker)는 별도 ops docs 보유, 신생 OSS(graphify)는 누락. **→ 가이드 대상 선정 기준**: 운영 문서 없는 미성숙 도구일수록 가이드 가치 큼.

### Round 7 — serena로 9칸 stress test (실측: 이 프로젝트의 `.serena/`)
실측: serena 바이너리는 PATH에 없음, `uvx`로 실행, `~/.claude.json`에 MCP 등록(`serena@...`). 프로젝트 루트에 `.serena/` 생성 — `project.yml`(진짜 config: 언어·인코딩, **versioned**), `project.local.yml`(로컬 오버라이드, ignore), `.serena/.gitignore`(serena가 자체 동봉), `cache/`, `memories/`.

| 칸 | graphify (생성·인덱싱형) | serena (질의·상주형) | 발견 |
|----|----|----|----|
| 4 프로젝트별 | config 없음, 출력만 `graphify-out/` | **진짜 config** `.serena/project.yml`(언어 지정 필수) | 칸 의미가 갈림 → "설정"보다 "프로젝트마다 생기는 것/할 것"으로 일반화 |
| 9 산출물 | `graphify-out/`(핵심) | 거의 없음(cache/memories만) | 9는 **생성형 전용 조건부 칸** 확인됨 |
| 5 팀공유 | git 커밋/ignore 수동 판단 | serena가 **이미 분리 내장**(.serena/.gitignore) | 성숙한 도구는 운영정보 자체 내장 |
| 6 유지보수 | 재인덱싱 수동/자동/watch | LSP 실시간 → 재인덱싱 개념 약함 | 칸 유효(답이 "해당 없음"일 수 있음) |

**핵심 발견 — 빠진 칸: ⑩ 실행/생명주기 모델.**
- graphify=one-shot(돌리고 끝), serena=상주 서버(세션 동안 떠 있음). 이 차이가 9칸 어디에도 명시 안 됨. 3번(연결방식)에 *묻혀* 있으나 별개 축(같은 graphify도 CLI=one-shot / MCP=상주). [[mcp_server_and_operations]] 토론의 "연결상태 소유권" 핵심과 직결.
- 부수: serena는 1번(언제 유용)에 "**언어/스택 지원 여부**"가 설치 전 결정 요인(LSP 한계). graphify는 36개 언어 자동.

**Round 6 성숙도 가설 지지 증거:** serena는 graphify가 누락한 공유/비공유 분리를 도구 자체에 내장(.gitignore 동봉, project.yml "versioned" 주석) → 더 성숙해서 운영정보를 담음.

### Round 8 — ③·⑩을 묶는 상위 축: "도구 사상"
- 사용자: 툴마다 다르다. graphify는 **데이터를 다루는 사상이 CLI로 충분**해서 one-shot. serena는 **Claude 열면 plugin이 자동으로 서버를 연결·실행**(상주). (실측 일치: `~/.claude.json` plugin 등록 → ⑩의 serena 답 확정)
- 에이전트: 이 관찰이 "③·⑩ 별개 축" 주장을 정밀화함.
  - ③(노출)·⑩(생명주기)은 **기술적으론 독립**(graphify를 MCP로 띄우는 것도 가능)이나, **도구의 "사상"이 자연스러운 조합을 묶는다**: 산출물 생성형 → CLI+one-shot, 실시간 질의형 → 서버+상주.
  - **함의(구조적 단순화):** 칸이 평면 리스트가 아니라, **상위 칸 "도구 사상"이 ③·⑩(+6·9 일부)을 절반 도출**한다. → 사용자가 칸마다 따로 고민할 필요가 줆 = "고민 줄이기" 목적에 정확히 부합.
  - 검증: graphify(산출물형)→CLI·oneshot·`graphify-out/`·수동/hook재인덱싱 ✓ / serena(질의형)→plugin·상주·config·실시간 ✓ — 둘 다 사상에서 파생됨.
- 자기검증(미해결):
  - **이분법 경계**: 사상이 "생성 vs 질의" 2종으로 깔끔히 갈리나? 후보 더: 변환형(린터/포매터), 인덱스형(생성+반복질의=graphify), 오케스트레이션형. 최소 3~4종 가능 → 2분법으로 못박지 말 것.
  - **하이브리드**: 한 도구가 복수 사상 가능(graphify=그래프 생성 + 그 그래프를 MCP 질의). "주 사상 + 부 모드"일 수 있음.

### Round 9 — "동작 방식" 칸 추가 → 멘탈 모델 묶음 형성
- 사용자: 동작 방식에 대한 간단한 설명도 필요할 것 같다.
- 에이전트: Round 6("레시피 아닌 멘탈 모델을 원함")의 귀결. 두 정밀화:
  - **동작 방식 vs 사상**: 동작 방식=서사("스캔→AST+LLM→파일 산출"), 사상=그 서사의 라벨(생성형). 사상은 별도 큰 칸이 아니라 동작 방식에서 *읽히는 태그*. → 동작 방식이 상위, 사상은 그 요약.
  - **가이드의 동작 방식 ≠ README의 기능 설명**: README는 "무엇을 할 수 있나(능력·자랑)", 가이드는 "어떻게 도나(운영에 영향 주는 메커니즘만)". 예: README="36개 언어·비디오 전사", 가이드="스캔해 파일 산출, 코드 바뀌면 stale". **간단히** = 운영 결정에 필요한 만큼만.
- **도출된 2단 구조:**
  - 【멘탈 모델】 동작 방식(서사) → 사상(라벨) → ① 언제 유용(맞는 케이스)
  - 【운영 절차】 ② 설치 · ③ 연결 · ④ 프로젝트별 · ⑤ 팀공유 · ⑥ 유지보수 · ⑦ 비용·주의·보안 · ⑧ 되돌리기 · ⑨ 산출물위치 · ⑩ 생명주기
  - Round 6의 "멘탈모델 vs 레시피"가 여기서 문서 구조로 구현됨.

### Round 10 — 머릿말 배치 + 순서 확정
- 사용자: 머릿말에 "필요한 케이스 + 사상 + 동작 방식"을 넣자. → 2단 구조의 멘탈 모델 묶음을 **문서 헤더로 배치** 확정.
- 사용자가 순서를 Round 9와 **뒤집음**: (R9) 동작방식→사상→케이스 vs (사용자) 케이스→사상→동작방식.
- 에이전트(근거 대어 사용자 순서 지지):
  - R9 순서 = "설명/이해 흐름"(메커니즘부터 쌓아 올림, 구체→추상→적용).
  - 사용자 순서 = "스캔/필터 흐름"(관련성 먼저). **가이드는 정독용이 아니라 참조용** → 독자는 "이게 내게 필요한가?"를 먼저 걸러야 함. 관련성 필터가 앞에 와야 스캔 효율↑. (BLUF: 결론 먼저)
  - 정밀화: **케이스 = 한 줄 결론(TL;DR)** 으로 맨 앞, 사상·동작방식이 그 *근거*를 채움. (안 그러면 케이스가 근거 없는 주장으로 읽힘)

### 확정된 가이드 템플릿 뼈대

## 확립된 것 (최종 — 설계 수렴)
- **정체성**: README가 끝나는 곳에서 시작하는 *운영자 관점* 문서. (Round 4·6)
- **구조**: 2단 — 머릿말(멘탈 모델) + 본문(운영 절차). (Round 9)
- **머릿말 순서**: 언제 필요한가(한 줄 결론) → 사상(태그) → 동작 방식(운영 메커니즘만). (Round 10)
- **본문 칸**: ②설치 ③연결 ④프로젝트별 ⑤팀공유 ⑥유지보수 ⑦비용·주의·보안 ⑧되돌리기 ⑨산출물위치 ⑩생명주기.
- **위치**: repo 최상위 `tools/`. 이 repo = 지식 허브. (Round 11)
- **채움 원칙(Round 12 정정)**: ~~실측으로만~~ → **검증 가능한 출처 확인** 기반(공식 docs·소스·신뢰 사례). 직접 설치·실행은 가장 강한 경로일 뿐 **필수 아님**. 출처로 불확실/상충하는 항목은 단정 말고 **"미확인" 표시**.
  - ⚠️ 회고: 에이전트가 "실측=정확성" 혼동으로 "직접 설치 필수"를 과하게 주장 → 사용자 정정. (upgrade-candidates 후보)

- **대상 선정 힌트**: 운영 문서 없는 미성숙 도구일수록 가이드 가치 큼. (Round 6)
