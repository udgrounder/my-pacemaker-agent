# 에이전트 협업 워크플로우 하네스 — 설계 원칙

> **문서 성격:** AI agent와 함께하는 구조화된 작업 워크플로우 하네스의 핵심 원칙.  
> 하네스 = 워크플로우를 실현하는 메커니즘 (inject 파일, workspace/, 페르소나, 스킬).  
> **최종 업데이트:** 2026-05-29

---

## 적용 범위

**적용 기준: Task를 만들어야 할 규모의 작업**

사용자가 Task 생성을 요청하면 워크플로우가 시작된다.  
Task 문서의 내용이 작업 유형(개발/리서치/기획/검토 등)을 결정한다.  
Task 문서가 완성되기 전까지는 설계 단계다.

| 적용 O | 적용 X |
|-------|-------|
| Task가 필요한 규모의 작업 | 단순 질문 |
| 개발, 리서치, 기획, 검토 등 | 탐색적 확인 |
| | 즉시 처리 가능한 소규모 요청 |

적용 범위 밖의 요청은 하네스 절차 없이 자연스럽게 응답한다.

---

## 메타 원칙

> **신뢰할 수 없는 것에 의존하지 않고, 통제할 수 있는 것만으로 구조를 만든다.**

이것이 모든 설계 결정의 공통 근거다.

---

## 역할 경계

### 하네스가 담당하는 것
- 알려진 위험 관리
- 구조적 실패 예방
- 컨텍스트 연속성 (workspace/ 파일)
- 에이전트 행동 규칙 (inject)

### 사용자가 담당하는 것
- 검토 결과 판단 및 다음 행동 결정
- 가치 결정 ("무엇이 더 중요한가")
- 비즈니스 맥락 판단 (외부 제약, 이해관계자 의도)

> "사용자"는 역할이다. 실제 사람일 수도 있고, 오케스트레이터 에이전트일 수도 있다.  
> 단계마다, 워크플로우마다 사용자 역할을 맡는 주체가 달라질 수 있다.

**기본값: 사용자가 판단. 사용자가 명시적으로 위임한 경우: 에이전트가 판단.**

### 하네스 실패는 인간의 몫 (역할이 아닌 존재)
하네스 자체가 실패했을 때 그것을 감지하는 주체는 시스템 외부에 있어야 한다.  
이 역할은 자동화될 수 없다 — 시스템이 자기 실패를 감지할 수 없기 때문이다.

> 깨진 시스템은 자기 자신이 깨졌다는 것을 알 수 없다.  
> 관찰자는 관찰 대상 밖에 있어야 한다.

## 하네스 구조

### 파일 구조와 역할

| 디렉토리 | 역할 | 읽는 주체 | 시점 |
|---------|-----|---------|-----|
| `.mpa-workspace/` | 행동 규칙, 역할, 스킬 | 에이전트 | 매 세션 |
| `workspace/` | 프로젝트 지식, 태스크 상태 | 에이전트 | 매 세션 (on-demand) |
| `agent-configs/` | CLI/IDE 브리지 파일 | CLI/IDE | layer0 설치 시 1회 |

### 의존성 방향
```
CLI/IDE  →  harness 파일을 읽는다  ✅
harness  →  CLI/IDE 내부를 읽는다  ✗ (블랙박스)
```

하네스는 라이브러리다. 임포터(CLI/IDE)의 내부를 알 필요도, 알아서도 안 된다.

### 기준선
- 하네스의 실행 기준선: **파일 읽기/쓰기가 가능한 tool-using 에이전트** (Claude Code, Cursor 등)
- Raw LLM API는 지원 범위 밖 (파일 접근 불가)
- CLI/IDE 내장 기억은 블랙박스 — 의존하지 않는다

### .mpa-workspace/ = 프로젝트에 복사되는 템플릿

`install.py`가 `.mpa-workspace/` 전체를 프로젝트에 복사한다.

```
하네스 내부 관리 지침  →  harness 루트에 위치 (복사 안 됨)
프로젝트용 지침       →  .mpa-workspace/ 하위에 위치 (복사됨)
```

- **복사 안 됨**: 하네스 시스템 파일 수정, 배포, 유지보수 지침
- **복사됨 (프로젝트용)**: upgrade-candidates 기록·이전 등 "프로젝트 → 하네스" 방향 작업

### 설치/업데이트 = 지식 전달의 트리거

```
설치:   하네스 → 프로젝트  (단방향)
업데이트: 프로젝트 → 하네스  (upgrade-candidates 이전)
        + 하네스 → 프로젝트  (템플릿 최신화)
```

인스톨/업데이트 단계가 프로젝트의 발견을 하네스로 올리는 공식 경로다.  
`upgrade-candidates/` 내용은 설치/업데이트 시 비워진다 — 하네스 자체 후보는 프로젝트로 넘기지 않는다.

## 기록 원칙

### 검증 여부가 기준이다
- 기록해야 하는 것: **검증된 지식** (사용자 경험, 다른 프로젝트, 기존 코드)
- 기록하지 않는 것: **추정** (경험 전에 "아마 이럴 것"으로 채운 내용)
- 타이밍(지금 vs 나중)이 기준이 아니다. 검증 여부가 기준이다.

### 놀라움 필터
기록 전 자문: "다음에 이 역할이 몰랐다면 실수했을 내용인가?"

**기록 O:**
- 예상과 다르게 동작한 패턴
- 반복적으로 나타나는 실수 유형
- 코드/문서만 봐서는 알 수 없는 제약

**기록 X:**
- 태스크 완료 사실
- 코드나 문서에서 직접 읽을 수 있는 내용
- 일회성 결정

### 부정 정의 우선
일반적 패턴과 **다른 것**만 기록한다. 일반적인 것은 기록하지 않아도 에이전트가 안다.

## 운영 원칙

### plan.md는 살아있는 항법 도구다

plan.md는 처음부터 완성되어야 하는 계약서가 아니다.  
사용자와 에이전트가 최선의 길을 함께 찾아가도록 돕는 현재 인식 공유 도구다.

- 지금 알 수 있는 것에서 시작하고, 진행하면서 채워진다
- 발견이 생기거나 가정이 틀렸다는 걸 알게 되면 업데이트한다
- `구현 진행 상태` 테이블 = 현재 어디까지 왔는지 사용자와 에이전트가 함께 보는 위치 표시

### 기본 흐름 — 검토와 승인은 별개 단계다

```
계획서 작성
    → 사용자 검토 → (수정 필요 시) 수정 → 다시 검토 → ...
    → 확정
    → 사용자 승인
    → 진행
```

- **검토**: 피드백·수정 요청. 반복 가능.
- **승인**: 확정된 계획에 대한 진행 허락. 구현 시작의 공식 게이트.

승인 단계를 건너뛰는 것은 에이전트의 판단이 아니라 **사용자의 명시적 허락**으로만 가능하다.  
승인은 "전체 계획의 최종 서명"이 아니라 "지금 진행할 구간에 대한 허락"이다.  
진행 중 새로운 발견이 생기면 다음 구간에 대해 검토/승인 사이클이 다시 시작된다.

### 하네스는 학습 시스템이다
- 처음부터 완성된 하네스를 만들려 하지 않는다
- layer0: 구조(그릇과 트리거)를 완성하고, 내용은 운영하면서 채운다
- 성숙도는 사용 시간에 비례한다

### 하네스 시스템 업데이트는 사용자 주도다

upgrade-candidates를 하네스 시스템 파일에 반영할 때:
- **사용자가 명시적으로 요청했을 때만 시작**
- 전체 후보 목록을 먼저 제시한다
- **후보 하나씩, 사용자 확인 후 진행** — 일괄 자동 적용 금지
- 하네스 변경은 모든 프로젝트에 전파되므로 사용자가 내용을 검토하면서 진행해야 한다

### 세션 연속성
- plan.md = 하네스의 세션 메모리
- 세션이 끊겨도 plan.md + project_memory/ + inject 파일로 재개 가능
- CLI/IDE 대화 압축에 의존하지 않는다

### 파일 규모 관리
- **분리**: 자주 읽는 것(핵심)과 가끔 읽는 것(히스토리)을 나눈다
- **압축**: 불필요한 것을 제거해 순도를 높인다 (손실 감수가 아님)
- 무엇이 불필요한가 판단하기 어려운 경우 → 삭제 역설, 사용자 판단에 맡긴다

### 결과물은 다중 독자 문서다

plan.md, changelog.md는 인간과 여러 에이전트가 각각 다른 시점에 다른 목적으로 읽는 다중 독자 문서다.

```
설계 에이전트  → plan.md 작성
사용자        → plan.md 검토·승인
구현 에이전트  → plan.md 읽고 구현 + changelog.md 작성
검토 에이전트  → plan.md + changelog.md 정합성 확인 (독립 시행)
사용자        → 검토 결과 판단 및 다음 행동 결정 (위임 시 에이전트 판단)
```

좋은 결과물은 **구조화된 형식(에이전트용) + 맥락 설명(사용자용)**을 동시에 만족해야 한다.  
결과물이 읽히지 않으면 사용자 개입은 명목상으로만 존재한다.

### 결과물의 최종 책임은 사용자에게 있다

파이프라인이 얼마나 자동화되든, 결과물을 쓸지 말지 판단하고 그 결과에 책임지는 것은 사용자다.  
하네스는 사용자가 판단할 수 있는 재료를 만드는 것까지 담당한다.

```
하네스:   결과물을 만들고, 판단할 수 있도록 가시화한다
사용자:   결과물을 쓸지 말지 판단하고, 그 결과에 책임진다
```

결과물이 읽히지 않으면 사용자가 판단할 수 없고, 판단할 수 없으면 책임을 질 수도 없다.

### 실패는 과정에 있다

결과물이 사용되지 않는 것은 실패가 아니다. 실패는 그것을 만드는 과정에 있다.

- 과정 실패: 잘못된 이해, 미기록 가정, 아키텍처 규칙 위반
- 실패 아님: 결과물이 사용되지 않는 것, 방향이 바뀌어 필요 없어진 것

하네스의 성공/실패 기준은 **결과물의 사용 여부가 아니라 과정의 정합성**이다.  
학습(roles/, upgrade-candidates/) 기록 대상도 과정의 실수다.

### 구조가 행동을 만든다 — 쓰여진 것만 판단 가능하다

사용자는 오직 **쓰여진 것**만 판단할 수 있다. 에이전트가 내부적으로 고려했더라도 쓰지 않으면 사용자에게는 존재하지 않는다.

"구조가 행동을 만든다" = **구조가 에이전트를 쓰게 만든다.**

- plan.md "에이전트 보고" → 가정을 써야 사용자가 검토 가능
- 가치 결정 영향도 → 써야 사용자가 선택 가능
- 구현 진행 상태 → 써야 현재 위치 공유 가능

기억에 맡기지 않고 구조가 강제한다.

## 하네스의 한계 (알고 운영해야 하는 것)

### 세 역설
모두 동일한 구조: **진짜 위험한 것은 보이지 않는다. 보이는 것은 덜 위험하다.**

- **탐지 역설**: 에이전트가 잘못 이해한 것은 잘못 이해했다는 신호를 보내지 않는다
- **측정 역설**: 실패 비용이 높은 곳일수록 측정하기 어렵다
- **보고 역설**: 에이전트가 놓친 것은 보고되지 않는다

하네스는 알려진 위험만 관리할 수 있다. 알 수 없는 위험은 사용자의 영역이다.

### 삭제 역설
침묵하는 규칙은 "잘 지켜지는 것"과 "불필요한 것"을 구분할 수 없다.  
기원(어떤 실수에서 나왔는가)과 빈도(실제 발동 횟수)를 신호로 삼고, 최종 판단은 사용자가 한다.
