# MCP 서버와 관리 운영 방식

> 상태: 진행 중
> 최초: 2026-06-19 · 최종: 2026-06-19

## 테제 / 물음
MCP 서버를 어떻게 나누고, 어떤 운영 규칙으로 관리해야 실제 팀 생산성과 안정성이 같이 올라가는가?

## 오간 의견
### Round 1
- 사용자: MCP 서버와 관리 운영 방식에 대해 논의하고 싶다고 요청했다.
- 에이전트: 논의의 초점을 "서버 구조"와 "운영 방식"의 결합 문제로 잡았다.

### Round 2
- 사용자: 운영 방식이 중심이라고 좁혔다.
- 사용자: 별도로, 서버가 정확히 언제 떠 있고 언제 내려가는지 가시성이 부족하다고 지적했다.
- 사용자: 스킬, CLI, MCP 서버의 관계도 함께 알고 싶다고 요청했다.
- 에이전트: 현재 세션 기준으로 관찰 가능한 구성과, 그 위에 얹힌 개념적 관계를 분리해서 설명하기로 했다.

### Round 3
- 사용자: 서버가 항상 필요한 경우가 아니라면, CLI를 통해 필요할 때만 동작시키는 편이 유리하지 않느냐고 물었다.
- 에이전트: 기본값으로는 온디맨드 실행이 유리하다고 보되, 상태 유지·초기화 비용·공유 연결 자원 여부에 따라 상시형이 더 나은 경우가 있다고 봤다.

### Round 4
- 사용자: 일반적으로는 CLI가 있으면 CLI를 쓰고, 없으면 MCP를 쓰는 구조가 자연스럽지 않느냐고 제안했다.
- 에이전트: 그 운영 원칙은 실무적으로 타당하지만, "스킬 = CLI 사용"은 아니며 스킬은 도구 선택 규칙이라는 점을 분리해야 한다고 봤다.

### Round 5
- 사용자: 예외 조건이라면 애초에 CLI로 처리하면 안 되는 것 아니냐고 반문했다.
- 에이전트: 그 지적에 동의하며, 일부 예외는 "CLI 우선의 예외"가 아니라 처음부터 MCP나 세션형 실행 모델을 선택해야 하는 별도 분류라고 정리했다.

### Round 6
- 사용자: 쟁점은 MCP 대 CLI 차이보다, 서버 또는 CLI 동작 프로그램이 연결 상태를 관리해야 한다면 애초에 CLI를 만들지 말아야 하는 것 아니냐고 물었다.
- 에이전트: 핵심 쟁점을 "연결 상태 소유권"으로 재정의하고, 연결 상태를 인터페이스 경계 밖으로 감춘 CLI는 대체로 잘못된 추상화라고 보는 쪽으로 입장을 세웠다.

### Round 7
- 사용자: 지금까지의 이해를 다섯 문장으로 정리했다.
- 에이전트: 정리 방향은 대체로 맞지만, "초기 기동 비용"만으로 서버 여부를 결정하면 부족하고, "상태 관리 필요성"과 "운영 통제 요구"를 함께 봐야 한다고 정리했다.

## 확립된 것
- 이번 세션은 구현 태스크가 아니라 토론 모드다.
- 주제는 MCP 서버 자체보다, 그것을 팀에서 어떻게 운영할지까지 포함한다.

## 열린 질문
- 서버를 도메인별로 쪼갤지, 공통 플랫폼으로 통합할지
- 권한, 배포, 버전 관리, 장애 대응을 어느 계층에서 맡길지
- 개발자 자율성과 중앙 통제를 어디서 균형 잡을지
