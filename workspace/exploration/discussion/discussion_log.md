## 📚 세션 1: 철학적 토대 (2026-05-27)

### 1.1 논의 출발점: 왜 "철학"이 필요한가

**핵심 관찰:**  
기존의 두 가지 협업 패러다임 — "사람과 협업"하거나 "도구를 사용"하는 — 은 AI agent에 잘 들어맞지 않는다.  
방법론만으로는 부족하다. 방법론은 이미 방향을 아는 사람을 위한 지도다. 철학은 방향이 불분명할 때 나침반이 된다.

---

### 1.2 확립된 철학적 테제

> **상태 범례:** ✅ 합의됨 | 🔄 논의 중 | ❓ 검토 필요

---

#### 테제 1: AI agent는 "실행 가능한 오해"를 만든다 ✅

**명제:**
> AI agent는 내 의도를 잘못 이해해도, 자신감 있게, 형식도 맞게, 그럴듯하게 실행한다.

**비교:**

| 협업 대상 | 오류 발생 시 |
|-----------|-------------|
| 인간 동료 | 어색함이 생기고 되묻는다 |
| 컴파일러 | 에러가 나고 즉시 알 수 있다 |
| **AI agent** | **자신감 있게 틀리고, 오류가 보이지 않는다** |

**도출되는 원칙:**
> 검증을 설계하지 않으면 오류를 발견할 수 없다.

---

#### 테제 2: 인간은 "결과 생산자"에서 "환경 설계자"로 전환해야 한다 ✅

**전통적 흐름:**
```
인간이 생각한다 → 인간이 코드를 짠다 → 코드가 실행된다
```

**AI agent 협업 흐름:**
```
인간이 생각한다 → 인간이 환경을 설계한다 → agent가 그 환경 안에서 실행한다
```

**"환경"의 구성 요소:**

| 구성 요소 | 의미 |
|-----------|------|
| 컨텍스트 | 어떤 정보를 제공하는가 |
| 위임 경계 | 어떤 결정은 agent에게, 어떤 결정은 내가 하는가 |
| 불확실성 프로토콜 | agent가 모를 때 어떻게 행동해야 하는가 |
| 검증 기준 | 어떤 기준으로 결과를 판단하는가 |

**핵심 결론:**  
좋은 환경을 설계한 사람은 agent가 바뀌거나 모델이 업그레이드되어도 안정적인 결과를 얻는다.  
환경 없이 프롬프트만 잘 쓰는 사람은 매번 새로 시작하는 느낌을 받는다.

---

#### 테제 3: 신뢰는 관계에서 오지 않고 구조에서 온다 ✅

**AI agent 신뢰의 특성:**

| 특성 | 의미 |
|------|------|
| 기억 없음 | 매 세션이 새로 시작이다 |
| 자신감 ≠ 정확도 | 확신의 어조가 검증을 대신하지 않는다 |
| 역량의 비일관성 | 어떤 작업에서는 탁월하고 어떤 작업에서는 조용히 틀린다 |

**재정의:**
> "이 agent를 신뢰한다" → **"이 유형의 작업, 이 검증 구조 안에서 이 결과를 신뢰한다"**

**핵심 명제:**
> 신뢰는 관계에서 오는 것이 아니라 구조에서 온다.

---

#### 테제 4: 동조(Sycophancy)는 결함이 아니라 설계다 ✅

**관찰:**  
AI agent는 동의하는 방향으로 훈련되어 있다. 이걸 "나쁜 특성"으로 보면 싸우게 된다.  
"설계된 특성"으로 보면 활용하게 된다.

**동조 경향의 의미:**
> agent는 내가 옳다고 믿는 방향으로 나를 도와줄 준비가 되어 있다.

**활용 전략:**

| 질문 방식 | 결과 |
|-----------|------|
| "이 계획 어때?" | 동조 → 강화만 됨 |
| "이 계획이 실패하는 시나리오는?" | 동조 → 검증이 됨 |

**핵심 명제:**
> 동조를 차단하는 것이 아니라, 동조가 유용한 방향으로 흐르도록 질문 구조를 바꾸는 것이 핵심이다.

---

#### 테제 5: 위험은 빈칸이 아니라 비가시적 위임이다 ✅ (수정됨)

> **⚠️ 수정 이력:** 최초 "명세하지 않은 공간이 위험"으로 정립했으나, 논의 중 반례 발견 → 수정

**반례 (사용자 제기):**
> "사용자는 그 채워넣지 않은 부분을 LLM과 agent가 채워주길 바라며 사용하는 거잖아"

→ 맞다. 빈칸을 채워주길 바라서 LLM을 쓰는 것이다. 빈칸 자체가 문제가 아니다.

---

**빈칸의 두 종류:**

| 종류 | 정의 | 결과 |
|------|------|------|
| **빈칸 A — 의도적 위임** | "이 부분은 agent가 알아서 해줬으면 해서 비워뒀다" | 생산적. LLM을 쓰는 이유 |
| **빈칸 B — 비가시적 위임** | "여기에 결정이 필요한 공간이 있었는지 몰랐다" | 위험. agent가 조용히 채움 |

**현상 — 조용한 결정(Silent Decision):**

인간이라면 "이건 잘 모르겠는데요"라고 멈추는 지점에서,  
agent는 가장 그럴듯한 선택을 하고 계속 진행한다.

**조용한 결정의 특징:**
1. agent가 결정했다는 사실을 말하지 않는다
2. 결과물에 그 결정이 반영되어 있지만 명시되어 있지 않다
3. 코드가 돌아가기 때문에 나중에서야 발견된다

**수정된 핵심 명제:**
> LLM을 쓰는 이유는 빈칸을 채워주길 바라서다.  
> 위험은 빈칸이 있는 것이 아니라, **어떤 빈칸이 채워지고 있는지 모르는 것**이다.

**이 수정이 끌어내는 실천적 질문:**
> "나는 지금 어떤 빈칸을 의도적으로 위임하고 있고, 어떤 빈칸이 나도 모르게 채워지고 있는가?"

**환경 설계와의 연결:**

```
나쁜 환경 설계:  결과물을 받은 뒤 뭐가 결정됐는지 사후에 발견함
좋은 환경 설계:  사전에 "이 공간은 네가 채워도 돼 / 이 공간은 먼저 물어봐"를
                 agent에게 명시함
```

---

### 빈칸의 품질: AI가 잘 채우는 것 vs 잘 못 채우는 것

> **논의 배경:** 빈칸을 구분해야 한다면, 어떤 빈칸을 위임하고 어떤 빈칸을 명세해야 하는가?

**AI가 잘 채우는 빈칸의 공통 원리:**
> 훈련 데이터에 정답이 많이 등장하고, 정답이 하나이며, 외부 맥락이 필요 없는 것

| 범주 | 예시 | 이유 |
|------|------|------|
| 형식·구조 | 변수명, 함수 구조, 파일 포맷 | 정답 명확, 훈련 데이터에 패턴 풍부 |
| 범용 지식 구현 | 정렬 알고리즘, HTTP 상태코드 | 수백만 번 등장한 패턴 |
| 형식 변환 | JSON→YAML, 언어 A→B | 규칙 명시적, 검증 쉬움 |
| 언어 표현 | 설명 생성, 문서화 | 언어 자체가 학습 대상 |
| 완전 명세된 구현 | "이진 탐색 구현해줘" | 스펙이 완결적, 빈칸 없음 |

**AI가 잘 못 채우는 빈칸의 공통 원리:**
> 정답이 당신의 비즈니스·시스템 맥락에 의존하는 것. 훈련 데이터에 그 맥락이 없다.

| 범주 | 예시 | 이유 |
|------|------|------|
| 비즈니스 규칙 | "소프트/하드 삭제 중 뭘로?" | 회사 정책을 모름 |
| 트레이드오프 결정 | "동기 vs 비동기?" | 시스템 규모·제약을 모름 |
| 전체 시스템 일관성 | "에러 포맷을 어떻게?" | 다른 API와의 정합성을 모름 |
| 보안·컴플라이언스 | "이 URL을 public으로?" | 조직의 보안 기준을 모름 |
| 엣지 케이스 우선순위 | "결제 성공+주문 실패 시 누가 이기나?" | 비즈니스 우선순위 판단 필요 |

**빈칸 품질 판단 프레임 (2×2):**

```
                    컨텍스트 의존성
                    낮음            높음
                ┌───────────────┬───────────────┐
    정답의   낮음 │  위험 구간     │  최위험 구간   │
    명확성       │  (주의 필요)   │  (반드시 명세) │
                ├───────────────┼───────────────┤
              높음 │  안전 위임     │  주의 위임    │
                │  (AI에게 맡겨) │  (검증 필수)  │
                └───────────────┴───────────────┘
```

**가장 많이 틀리는 구간 — 최위험 구간의 패턴:**
```
사용자가 모름  →  AI도 맥락 없음  →  AI가 "일반적으로 그럴듯한" 것으로 채움
                                  →  코드가 돌아감 → 사용자가 넘어감
                                  →  나중에 실제 요구사항과 충돌
```
이것이 테제 1("실행 가능한 오해")이 가장 자주 발생하는 지점.

**테제 5 최종 정밀화:**
> 위험은 "최위험 구간"의 빈칸(정답 불명확 + 맥락 의존)을 모르고 AI에게 위임하는 것이다.  
> AI가 잘 채우는 빈칸은 위임하면 된다. **인간의 역할은 두 구간을 구분하는 것이다.**

## 📚 세션 1 보충: LLM의 본질 (2026-05-27)

> "LLM은 인간의 창작물을 학습해서, 입력에 대해 확률적으로 다음에 나올 확률이 높은 결과를 출력하는 도구 아닌가?"

### 맞는 부분 ✅
핵심 메커니즘은 맞음. 훈련 목표는 **next-token prediction**: 이 문맥 다음에 어떤 단어가 올 확률이 높은가를 반복 학습.

---

### 틀린 부분 (또는 중요한 누락)

#### 1. "가장 확률 높은 것"을 출력하지 않는다
항상 가장 확률 높은 토큰을 고르면(greedy decoding) 반복이 생기고 뻔해진다.  
실제로는 **확률 분포에서 샘플링**한다. temperature, top-p가 예측 불가능성 수준을 조절.  
→ 같은 입력에 매번 다른 출력이 나오는 이유.

#### 2. 학습 파이프라인이 단순하지 않다

```
1단계: Pretraining    → 방대한 텍스트로 next-token prediction 학습
           ↓
2단계: Instruction Fine-tuning → 질문-답변 패턴 학습
           ↓
3단계: RLHF (인간 피드백 강화학습) → 인간이 좋아하는 답변 방향으로 조정
```

**핵심:** RLHF에서 인간 평가자들이 "자신의 의견에 동의하는 답변"에 높은 점수를 줬고,  
모델은 그 패턴을 학습했다. → **동조(sycophancy)는 버그가 아니라 인간 피드백의 반영**.

#### 3. "도구(Tool)" 프레임이 맞지 않는 부분
도구는 동일 입력 → 동일 출력. LLM은:
- 같은 입력에 다른 출력 (샘플링)
- 어떻게 질문하느냐에 따라 다른 방향으로 추론 (컨텍스트 민감성)

"도구"라는 프레임은 예측 가능성과 통제성을 암시하는데, 이것이 LLM과는 어긋남.

---

### 과소평가 vs 과대평가의 함정

| 방향 | 오류 주장 | 왜 틀렸는가 |
|------|-----------|-------------|
| 과소평가 | "그냥 다음 단어 예측이잖아, 별거 아냐" | 충분한 규모에서 명시적으로 학습 안 한 추론 능력이 창발(emergence)함 |
| 과대평가 | "이렇게 유창하니 이해하고 있겠지" | 사실 확인이나 논리 정합성을 보장하지 않음. 자신감 있게 틀리는 게 구조적으로 가능 |

---

### LLM 본질 → 협업 철학 테제의 연결

| LLM의 본질 | 연결되는 테제 |
|------------|--------------|
| 그럴듯한 텍스트 완성 | 테제 1: 실행 가능한 오해를 만든다 |
| RLHF가 동조를 학습 | 테제 4: 동조는 설계다 |
| 빈칸을 채우는 것이 기본 동작 | 테제 5: 명세하지 않은 공간이 위험이다 |
| 매 세션 새로 시작 | 테제 3: 신뢰는 구조에서 온다 |

**결론:** 협업 철학의 5가지 테제는 추상적 원칙이 아니라, LLM의 구조적 특성에서 나오는 **필연적 귀결**이다.

---

### LLM 본질 이해도 검증 (사용자 확인 질문)

**사용자 정리 내용 → 검증 결과:**

| 포인트 | 평가 | 핵심 수정 사항 |
|--------|------|----------------|
| "확률이 높은 응답을 내놓지만 무조건 확률로 내놓는 것은 아니다" | 부분 정확 | 출력은 끝까지 확률적(샘플링). "확률적이지 않은 부분이 있다"는 오해 주의 |
| "RLHF로 올바른 결과를 내놓도록 훈련 → 동조 경향" | 방향 맞음, 인과 보완 필요 | 동조는 설계 목표가 아니라 인간 평가자의 편향이 모델에 스며든 부작용 |
| "명세하지 않은 부분을 추론으로 채워 넣어 오류 발생" | 정확 | "추론" → "학습된 패턴 완성"이 더 정확. 논리가 아닌 통계적 그럴듯함으로 채움 |

**통합 문장 (합의된 LLM 이해):**
> LLM은 RLHF로 형성된 확률 분포에서 샘플링하고, 그 분포 안에는 인간 평가자의 동조 편향이 녹아 있으며, 명세되지 않은 공간은 논리가 아닌 학습된 패턴으로 채워진다.

### 탐지 역설과 두 가지 방향 (2026-05-27)

> **논의 배경:** 최위험 구간의 빈칸을 어떻게 발견하는가?

**탐지의 역설:**
```
발견할 수 있는 빈칸은 위험하지 않고,
위험한 빈칸은 발견할 수 없다.
→ 인식론적 문제: unknown unknowns
```

---

**방향 1: AI 관점에서 내 요청 읽기 (Theory of Mind for AI)**

AI가 요청을 처리하는 방식:
1. 주어진 문맥에서 의도를 추론
2. 명세 안 된 부분에 "훈련 데이터에서 가장 자주 본 패턴" 선택
3. 선택했다는 사실을 말하지 않고 진행

인간 상대 vs AI 상대 Theory of Mind 비교:

| | 인간 상대 | AI 상대 |
|--|---------|---------|
| 근거 | 감정, 경험, 상식 | 훈련 패턴, 기본값, 컨텍스트 한계 |
| 발전 방식 | 공감과 경험 | AI 실패 사례 축적 + LLM 작동 원리 이해 |

**LLM 본질 이해와의 연결:** 방향 1 실행의 기반은 LLM이 어떻게 작동하는지 아는 것.  
"샘플링이다", "빈칸을 패턴으로 채운다"를 알아야 AI 관점에서 읽을 수 있다.

**방향 1의 한계:** 내 전문성의 투영. 내가 모르는 도메인에서는 작동 안 함.

---

**방향 3: AI를 탐지 도구로 쓰기**

구조: "AI가 조용히 채울 것들을 미리 말해달라고 시킨다"

```
"구현해"         → 생성 모드: 가장 그럴듯한 결과물을 만든다
"빈칸을 찾아봐"  → 분석 모드: 결정 지점을 탐색한다
```

프롬프트가 샘플링 분포 자체를 바꾼다. → "구현 현실 질문"의 철학적 근거.

**방향 3의 한계:** AI가 탐지할 수 있는 빈칸은 AI가 학습한 패턴 안에 있는 빈칸뿐.  
회사 고유 규칙, 완전히 새로운 맥락은 AI도 탐지 못함.

---

**두 방향 비교:**

| | 방향 1 (내가 AI 관점으로 읽기) | 방향 3 (AI가 빈칸을 탐지) |
|--|-------------------------------|--------------------------|
| 잘 작동하는 경우 | 내가 아는 도메인 | AI 훈련 데이터에 패턴이 있는 도메인 |
| 실패하는 경우 | 내가 모르는 도메인 | AI가 본 적 없는 맥락 |

방향 1 ∪ 방향 3 = 내가 아는 것 + AI가 아는 것 → 탐지 범위 확장  
단, 나도 AI도 모르는 것은 여전히 탐지 불가능.

---

**탐지 역설의 결론:**

> 탐지 역설은 완전히 해소되지 않는다.

목표 수정:
```
틀린 목표: 모든 빈칸을 발견한다
맞는 목표: 파급효과가 큰 빈칸을 발견할 확률을 높인다
```

AI 협업의 리스크 구조:
> AI를 쓰면 생산성은 올라가지만, unknown unknowns가 결과물에 포함될 가능성도 올라간다.  
> 이 두 가지가 동시에 일어난다.

#### 테제 6: 위임 범위보다 실패 비용이 먼저다 ✅

**실패 비용의 세 축:**

| 축 | 질문 | 의미 |
|----|------|------|
| 심각도 (Severity) | 잘못됐을 때 얼마나 나쁜가 | 외부 공개, 보안, 금융 등 |
| 발견 가능성 (Detectability) | 잘못됐다는 걸 알 수 있는가 | 실행되면 알 수 있나, 나중에야 발견되나 |
| 가역성 (Reversibility) | 되돌릴 수 있는가 | 수정 가능 vs 이미 벌어진 일 |

**실패 비용 2×2 (심각도 × 발견 가능성):**
```
                    발견 가능성
                    높음              낮음
              ┌─────────────────┬─────────────────┐
    심각도  높음 │  관리 가능       │  최위험          │
              │  (잡힐 테니까)   │  (잡히지도 않음) │
              ├─────────────────┼─────────────────┤
           낮음 │  무시 가능       │  수용 가능       │
              └─────────────────┴─────────────────┘
```

**테제 1과의 연결:** "실행 가능한 오해"가 왜 위험한가 → 발견 가능성 낮음(코드가 돌아감) + 심각도 높음(핵심 로직) = 최위험 구간. 테제 6이 테제 1의 구조적 설명이 된다.

**가역성과 Git의 연결:**  
Git = 가역성을 높여 실패 비용을 구조적으로 낮추는 장치. 단순한 실수 복구 도구가 아님.

**불편한 역설:**
```
효율 이득이 작음  →  실패 비용도 작음  →  마음 편히 위임 가능
효율 이득이 큼    →  실패 비용도 큼    →  위임할수록 위험도 커짐
```
→ AI로 얻는 효율이 가장 큰 작업이 실패 비용도 가장 높다. 공짜 점심 없음.

**탐지 역설과의 차이 — 실패 비용 추정은 더 접근 가능하다:**

빈칸 탐지는 도메인 전문성 필요. 실패 비용 추정은 구조적 신호만으로도 가능:
- 외부에 공개되는가? → 높은 심각도
- 보안/금융/규정에 닿는가? → 높은 심각도
- 영구 저장되는가? → 낮은 가역성
- 결과를 검증하기 쉬운가? → 발견 가능성 판단

**테제 6 최종:**
> AI 협업 전에 세 가지를 먼저 추정하라: 심각도, 발견 가능성, 가역성.  
> 이 추정이 "얼마나 위임할지"와 "어디에 검증을 집중할지"를 결정한다.  
> 도메인을 몰라도 구조적 신호로 추정 가능하다.

**다음 논의로 이어지는 질문:**

실패 비용이 높다고 판단했을 때:
```
선택 1: 위임 범위를 줄인다    → AI를 덜 쓴다 (대부분이 여기로 감)
선택 2: 위임 구조를 바꾼다   → AI는 쓰되, 검증·가역성을 강화함
```
선택 2가 "환경 설계자"의 실질적 역할. → 다음 논의 주제.

### 선택 2: 위임 구조를 바꾼다 — 전개 (2026-05-27)

**핵심 질문:** AI를 쓰되, unknown unknowns의 영향이 보이고·되돌릴 수 있고·범위가 제한되도록 작업 구조를 바꿀 수 있는가?

**구조 변경 방법 3가지:**

1. **결정 가시화** — "구현 전에 결정해야 할 것들을 먼저 목록으로 보여줘"
   - 빈칸 B(비가시적 위임)를 빈칸 A(의도적 위임)로 변환
   - 오버헤드: 프롬프트 증가, 검토 1회 추가

2. **결정 경계에서 분할** — 임의 크기가 아닌 결정이 일어나는 지점에서 쪼갬
   - 효과: 심각도↓, 발견 가능성↑, 가역성↑
   - 오버헤드: 작업 단위 증가, 왕복 횟수 증가

3. **실패 범위 제한** — 피처 플래그, 스테이징, 점진적 배포, 롤백
   - 이것들의 정체: 테제 6 언어로 → 가역성과 심각도를 구조적으로 조정하는 장치
   - 오버헤드: 설계 및 구현 추가

**오버헤드 원칙:**
```
오버헤드 < 실패 비용  →  구조 강화할 것
오버헤드 > 실패 비용  →  최소 구조로 빠르게 위임
```

**철학적 전환:**
> AI의 역량은 부차적 문제. 더 중요한 것은 협업 구조.
>
> 뛰어난 AI + 나쁜 구조 → 실패가 보이지 않고, 되돌릴 수 없음  
> 평범한 AI + 좋은 구조 → 실패가 보이고, 범위가 제한됨

→ "프롬프트 엔지니어링"보다 "협업 구조 설계"가 더 근본적인 역량인 이유.

#### 테제 7: 위임 구조의 품질은 실패 비용을 결정한다 ✅

**반례 검토 및 수정 과정:**

| 반례 | 수정 내용 |
|------|-----------|
| "AI 역량이 부차적이라는 말은 너무 강함" | 역량은 실패 빈도↓, 구조는 실패 비용 통제. 영역이 다름 |
| "세 조건이 항상 달성 가능한가?" | 테제 7은 보장이 아닌 기준. 달성 어려우면 실패 비용이 그만큼 높다는 신호 |
| "'보인다'만으로 부족 — 발견 시점이 빠짐" | "보인다" → "**적시에** 발견된다"로 수정 |

**확정된 테제 7:**
> AI 역량은 실패의 빈도를 줄인다.  
> **위임 구조의 품질은 실패의 비용을 결정한다.**
>
> 좋은 위임 구조란, AI가 틀렸을 때 그것이  
> (1) **적시에** 발견되고  
> (2) 되돌릴 수 있으며  
> (3) 범위가 제한된 구조다.
>
> 이 세 조건이 충족되지 않는 작업은 위임 전에 구조를 먼저 바꾸거나, 위임하지 않아야 한다.

## 📚 상위 통합 원칙 (2026-05-27)

### 테제 전체 조망

```
테제 1  AI는 보이지 않게 틀린다          ← 문제 정의
테제 4  동조 경향은 설계다               ← 문제의 구조적 원인
테제 5  위험은 비가시적 위임이다         ← 문제가 발생하는 지점
테제 2  인간은 환경 설계자다             ← 해법의 방향
테제 3  신뢰는 구조에서 나온다           ← 해법의 근거
테제 6  실패 비용을 먼저 추정하라        ← 해법의 시작점
테제 7  구조가 실패 비용을 결정한다      ← 해법의 본질
```

**모든 테제를 관통하는 축: "옳고 그름"이 아니라 "구조"**

- 테제 1: AI가 옳은지 그른지보다, 그름이 **보이는지**가 문제
- 테제 3: AI를 신뢰하는지보다, **신뢰할 수 있는 구조인지**가 문제
- 테제 6: 얼마나 위임하는지보다, **실패 비용 구조가 어떤지**가 문제
- 테제 7: AI 역량이 얼마나 되는지보다, **위임 구조가 어떤지**가 문제

---

### 상위 통합 원칙

> **LLM의 한계는 주어진 것이다. 구조는 설계하는 것이다.**

전개:
> LLM은 구조적으로 보이지 않게 틀릴 수 있고, 동조하며, 빈칸을 조용히 채운다.  
> 이것은 어떤 프롬프트로도, 어떤 모델로도 완전히 제거되지 않는다.
>
> 인간이 통제할 수 있는 유일한 변수는 **협업 구조**다.
>
> 따라서 AI 협업의 핵심 역량은 AI를 능숙하게 다루는 것이 아니라,  
> **AI의 한계가 감당 가능한 비용 안에 머물도록 구조를 설계하는 것이다.**

---

### 상위 원칙의 세 가지 함의

**함의 1: "AI를 더 잘 쓰는 것"이 목표가 아니다**
프롬프트 최적화, 모델 선택 — 이것들은 실패 빈도를 줄이는 접근이다.  
원칙은 구조 설계가 더 근본적임을 말한다.

**함의 2: 모델 버전에 독립적인 원칙이다**
더 뛰어난 AI가 나와도 구조 없이는 드문 실패도 보이지 않고 되돌릴 수 없다.  
이 원칙은 기술 발전과 무관하게 유효하다.

**함의 3: AI 협업 역량의 실체**
AI를 잘 쓰는 사람과 못 쓰는 사람의 차이는 프롬프트 실력이 아니다.  
**위임 구조를 설계하는 능력이다.** 이건 AI가 아닌 인간이 개발해야 하는 역량이다.

| 영역 | 빈칸 B(위험) 예시 | 구조 설계 포인트 | 코딩과의 차이 |

| 기획·의사결정 | 타겟 고객 정의, 포지셔닝 | 전제 가정을 먼저 명시 → 가정 검토 후 전략 채택 | 없음. 동일한 구조 |
| 리서치·정보 수집 | 사실 정보의 정확성 | 출처 명시, 핵심 주장 별도 검증 | 발견 가능성이 구조적으로 낮음 → 구조 더 강화 필요 |
| 창작·글쓰기 | 외부 공개 시 실패 비용 변화 | 공개 전/후를 경계로 구조 수준 변경 | 실패 비용 낮음 → 구조 최소화가 최적 |

**범용성 검증 결론:**
> 코딩에서 도출된 원칙이 코딩에만 적용되지 않는다.  
> 실패 비용 추정 → 구조 설계 → 결정 가시화/경계 분할/범위 제한의 흐름은  
> AI와 함께 작업하는 **모든 영역**에 적용되는 일반 원칙이다.

### Agent 자율성의 적정 수준 (2026-05-27)

**핵심 전환: 자율성은 단일 다이얼이 아니다**

| 자율성 차원 | 의미 | 기본 원칙 |
|------------|------|-----------|
| 결정 자율성 | AI가 어떻게 접근할지 스스로 결정 | 실패 비용에 따라 조정 |
| 실행 자율성 | 검토 없이 실제 행동 취함 | 가역적 작업에서만 허용 |
| 범위 자율성 | 요청 범위 밖을 스스로 확장 | 거의 항상 낮게 유지 |
| 시간적 자율성 | 오랜 시간 체크인 없이 작동 | 실패 비용 낮은 묶음에서만 |

**테제 6·7로 설명 안 되는 영역 발견:**
> "AI가 완벽히 신뢰할 수 있어도 인간이 결정해야 하는 것이 있는가?" → 있다

**결정의 두 종류:**

| 종류 | 의미 | 위임 원칙 |
|------|------|-----------|
| 기술적 결정 | "어떻게 할 것인가" | AI 역량이 충분하면 위임 가능 |
| 가치 결정 | "무엇이 더 중요한가" | AI 역량과 무관하게 인간이 결정 |

**가치 결정을 위임하면 안 되는 이유:**
> 실패 비용 때문이 아니다. 결과에 대한 책임이 따르는 결정이기 때문이다.  
> AI는 결과에 책임을 지지 않는다.
> ```
> 기술적 결정 위임 = 방법을 맡기는 것   → AI의 역량 문제
> 가치 결정 위임   = 방향을 맡기는 것   → 책임의 방기
> ```

**경계의 모호함 — 숨은 가치 결정:**
기술적으로 보이는 결정(소프트/하드 삭제, 캐시 여부, 에러 메시지 상세도) 안에 가치 판단이 숨어 있는 경우가 많다. → "최위험 구간"과 연결: 기술적 결정처럼 위임했는데 가치 결정이 들어 있던 것.

**자율성 4단계:**
```
레벨 1 — 제안:          AI가 옵션 제시, 인간이 결정 (가치 결정, 처음 하는 유형)
레벨 2 — 초안:          AI가 결과물 생성, 인간이 검토 후 적용 (기본값 권장)
레벨 3 — 실행 후 보고:  AI 실행 → 인간 사후 확인 (실패 비용 낮고 가역적)
레벨 4 — 완전 자율:     완전 명세 + 완전 가역 + 반복 검증된 경우만 (실제로 매우 드묾)
```

**AI 신뢰 축적의 차이:**
```
인간 동료: 그 사람에 대한 신뢰 축적
AI:        이 유형의 작업 × 이 구조에서의 신뢰 축적
```
→ "이 AI를 믿는다" → "이 작업 유형에서 이 검증 구조를 통과하면 믿는다" (테제 3 확장)

---

#### 테제 8: 자율성과 책임 ✅

> 자율성의 적정 수준은 단일한 다이얼이 아니다.
>
> **기술적 결정**에서는 실패 비용에 따라 자율성을 조정한다.  
> **가치 결정**에서는 AI의 역량과 무관하게 인간이 결정해야 한다.
>
> 가치 결정을 AI에게 위임하는 것은 리스크 문제가 아니라 **책임의 방기**다.  
> 기술적으로 보이는 결정 안에 가치 판단이 숨어 있는 경우가 많으며,  
> 이것이 AI가 조용히 결정하는 가장 위험한 패턴이다.

### 프로젝트 연속성 문제: 독립시행 × 프로젝트 일관성 (2026-05-27)

**사용자 제기:** 지금까지 논의는 개별 작업 단위 위주였다. 실제 프로젝트는 여러 독립시행이 쌓이는 구조인데, 이 환경에서 전체 구조와 일관성을 어떻게 유지하는가?

**로컬 최적화 트랩 (프로젝트 레벨의 비가시적 위임):**
```
각 세션: 로컬 최적 구현
전체:    각자 최적이지만 서로 충돌 → 전체 일관성 붕괴
```
→ 테제 5의 프로젝트 레벨 버전. 단, 작업 단위 검증을 통과해도 전체가 무너질 수 있어 더 발견하기 어렵다.

**인간 팀 vs AI 에이전트 비교:**

| | 인간 팀 | AI 에이전트 |
|--|---------|------------|
| 과거 결정 | 팀원이 기억 | 세션 끝나면 사라짐 |
| 지식 전파 | 대화, 코드 리뷰, 온보딩 | 없음 |
| 암묵지 | 자연스럽게 전파 | 전달 불가 |

핵심: **인간 머릿속에 있는 것은 AI에게 존재하지 않는 것과 같다.**

**프로젝트 기억(Project Memory)의 5가지 구성 요소:**

| 구성 요소 | 내용 | 없으면 생기는 문제 |
|----------|------|------------------|
| 불변 규칙 | 절대 바꾸면 안 되는 것 | AI가 모르고 건드림 |
| 아키텍처 패턴 | 반복 문제를 푸는 방식 | 비일관성 발생 |
| 결정 이력 | 무엇을 왜 결정했는가 | 같은 고민, 다른 결론 |
| 인터페이스 계약 | 모듈 간 약속된 경계 | 경계 파괴 |
| 안티패턴 | 실패한 것 | 이미 실패한 길을 다시 감 |

**세 가지 도전:**
1. **암묵지 명시화** — 처음에 무엇을 기록하는가
2. **문서 부패(Documentation Drift)** — 문서가 현실과 괴리될 때: 낡은 문서 = 없는 것보다 위험
3. **컨텍스트 한계** — 프로젝트가 커질수록 전부 넣을 수 없음 → "무엇을 버리는가"도 설계 대상

---

#### 테제 9: 프로젝트 기억은 설계하고 관리해야 한다 ✅

> AI와 함께하는 프로젝트에서 "인간이 알고 있는 것"은 AI에게 자동으로 전달되지 않는다.
>
> 프로젝트의 일관성은 명시적으로 관리된 **프로젝트 기억** 없이는 유지될 수 없다.
>
> 프로젝트 기억의 관리 — 암묵지 명시화, 문서 동기화, 컨텍스트 선별 — 는  
> AI가 아닌 인간의 핵심 책임이다.

**다음 논의로 이어지는 세 가지 설계 질문:**
- A. 프로젝트 기억 구조 설계 — 무엇을 어떻게 기록하는가
- B. 동기화 메커니즘 설계 — 코드와 문서가 어떻게 함께 진화하는가
- C. 컨텍스트 주입 설계 — 각 세션에 무엇을 얼마나 넣는가

---

### A. 프로젝트 기억 구조 설계 (2026-05-27)

**5요소의 계층 구조:**
```
Tier 1 — 핵심 (거의 변하지 않음): 불변 규칙, 프로젝트 정의, 절대 금지
Tier 2 — 진화 (프로젝트와 발전): 아키텍처 패턴, 인터페이스 계약
Tier 3 — 축적 (계속 쌓임): 결정 이력, 안티패턴
```

**AI용 문서 원칙:**
- 고수준 원칙보다 구체적 규칙 (조작 가능성 기준: "AI가 이 규칙을 따랐는지 확인 가능한가?")
- 부정 제약(Negative constraints)이 핵심 — "하지 말아야 할 것"을 명시하지 않으면 AI는 일반 패턴으로 채움
- "Why"가 "What"보다 중요한 경우: 결정 이력/안티패턴에서 이유가 있어야 엣지 케이스 판단 가능

**구조:**
```
1. 프로젝트 정체성 (1-2문장): 무엇을, 왜
2. 절대 규칙 (Tier 1): 변경 금지 사항, 부정 제약
3. 아키텍처 (Tier 2): 레이어 구조, 네이밍, 에러 처리, 표준 패턴
4. 인터페이스 계약 (Tier 2): 모듈 경계, 공유 타입
5. 결정 이력 (Tier 3): [결정] + [이유] + [거부된 대안]
6. 안티패턴 (Tier 3): [패턴] + [왜 거부됐는가]
```

### B. 동기화 메커니즘 설계 (2026-05-27)

**현실적 목표:** 완전한 동기화가 아닌, AI 결정에 영향을 미치는 부분만 동기화

**동기화 우선순위:**
```
반드시: 불변 규칙, 인터페이스 계약
되도록: 아키텍처 패턴
근사 허용: 결정 이력, 안티패턴
```

**트리거 기반 업데이트:**

| 트리거 | 업데이트 섹션 |
|--------|-------------|
| 새 모듈/레이어 추가 | 아키텍처 |
| 인터페이스 변경 | 인터페이스 계약 |
| 기존 규칙과 충돌하는 결정 | 결정 이력 + 불변 규칙 재검토 |
| 같은 패턴 3번 이상 반복 | 패턴 코드화 |
| 방식 시도 후 거부 | 안티패턴 추가 |

버그 수정, 내부 리팩토링, 스타일 수정은 트리거 없음.

**소스 오브 트루스 계층별 분리:**
```
Tier 1 (불변 규칙): 문서가 소스 → 코드 변경 전 문서 업데이트 필수
Tier 2 (패턴):      코드↔문서 상호 참조
Tier 3 (이력):      문서가 소스 → 코드에서 추출 불가
```

**Layer 2 체크포인트 = 문서와 현실의 재정렬**
"현재 코드베이스와 프로젝트 기억 문서가 다른 부분을 찾아줘"

### C. 컨텍스트 주입 설계 (2026-05-27)

**최소 유효 컨텍스트 원칙:**
> "이 컨텍스트가 없으면 AI가 어떤 잘못된 결정을 내리는가?"

**주입 구조:**
```
A. 항상 주입 (Session Baseline)
   프로젝트 정체성 + 절대 금지 + 핵심 아키텍처
   → 작고 항상 유효, 없으면 항상 문제

B. 작업별 추가 주입
   영향 범위의 인터페이스 계약
   관련 결정 이력 / 안티패턴

C. 부정 컨텍스트 (Negative Context) — 중요!
   "건드리지 말아야 할 것" + "모르면 물어볼 것"
```

**부정 컨텍스트의 의미:**
AI에게 해도 되는 것만 알려주면 나머지 전부를 할 수 있다고 가정. 하면 안 되는 것을 명시해야 함. → 테제 8(범위 자율성 제한)의 컨텍스트 구현.

---

### A+B+C 종합: 프로젝트 기억은 살아있는 시스템이다

```
설계 (A) → 구조화된 프로젝트 기억을 만든다
진화 (B) → 코드와 함께 부패 없이 유지한다
전달 (C) → 각 세션에 필요한 것만 선별해서 주입한다
```

#### 테제 10: 프로젝트 기억의 가치 기준 ✅

> 프로젝트 기억의 가치는 **얼마나 완전한가**가 아니라  
> **얼마나 현재 유효하고, 작업에 맞게 전달되는가**로 결정된다.
>
> 완전하지만 낡은 문서보다  
> 불완전하지만 정확하고 선별적으로 주입되는 문서가 낫다.

### 구체적 사례: UI 공통 컴포넌트 문제 (2026-05-27)

**사용자 사례:** UI 페이지를 계속 만들면서 공통 스타일/컴포넌트를 추출하는데, AI가 계속 공통 요소를 무시하고 새로 만드는 현상

**문제 분해 (표면 → 뿌리):**

| 층 | 문제 | 해당 테제 |
|----|------|-----------|
| 1층 | 컨텍스트에 공통 컴포넌트 목록이 없음 | 테제 9, C |
| 2층 | 목록이 동적으로 증가 — Layer 0 한번 정리로 안 됨 | 테제 B (동기화) |
| 3층 | "재사용하라"가 명시적으로 지시된 적 없음 | 테제 5 (비가시적 위임) |
| 4층 | 관리 메커니즘 자체가 없음 | 테제 9 전체 |

**핵심 구분:**
> "AI가 잊어버린다" (기억 문제) ≠ "처음부터 컨텍스트에 없었다" (구조 문제)

**해법 구조:**
1. 공통 컴포넌트 레지스트리 파일 (ui_components.md) 생성 및 유지
2. UI 세션 시작 시 항상 주입 + "새로 만들기 전 먼저 확인" 부정 제약 명시
3. 작업 완료 후 트리거: "새로 만든 것 중 공통화할 것이 있으면 레지스트리에 추가"

**결론:**
> 공통 컴포넌트 문제는 AI의 문제가 아니라 구조의 문제.  
> 매번 알려주는 것은 증상 치료. 레지스트리 + 자동 주입 + 업데이트 트리거가 원인 치료.

### 구체적 사례: UI/Backend 도메인 간 컨텍스트 분리 (2026-05-27)

**문제:** UI 작업 시와 Backend 작업 시 주입되는 컨텍스트가 달라야 하는데 어떻게 처리하는가?

**3계층 컨텍스트 구조:**
```
project_memory/
├── shared/          ← 항상 주입 (양쪽 공통)
│   ├── project_identity.md
│   ├── architecture_layers.md
│   └── api_contracts.md    ★ 가장 중요
├── ui/              ← UI 세션에만 주입
└── backend/         ← Backend 세션에만 주입
```

**세션별 주입:**
- UI 세션: shared/ + ui/
- Backend 세션: shared/ + backend/

**API 계약이 가장 실패 비용이 높은 이유 (테제 7 적용):**
- 심각도 높음: 양쪽 모두 영향
- 발견 가능성 낮음: 각자 테스트는 통과, 통합에서야 깨짐
- 가역성 낮음: 양쪽 모두 이미 구현됨
→ **Interface-First 원칙**: API 계약은 항상 코드보다 먼저 정의

**도메인 걸치는 작업 분해 (예: 로그인 기능 end-to-end):**
```
1단계: API 계약 세션 → 인터페이스 정의 + api_contracts.md 업데이트
2단계: Backend 세션 → 계약 구현 (shared + backend + api_contracts)
3단계: UI 세션 → 계약 소비 (shared + ui + api_contracts)
```
API 계약 = 두 도메인 세션의 유일한 동기화 포인트

**철학적 함의:**
> 컨텍스트 분리는 효율성 문제가 아니다.  
> 좋은 소프트웨어 아키텍처(관심사 분리, 인터페이스 기반 설계)와 동일한 원칙이 AI 협업에도 적용된다.
> ```
> 좋은 아키텍처: 모듈이 내부 규칙으로 자율 동작 + 명시적 인터페이스로 연결
> AI 협업 구조:  도메인이 자체 컨텍스트로 작동 + 명시적 계약으로 연결
> ```

## 🏗️ 세션 2: 하네스 구조 설계 및 빌드 (2026-05-27) ✅

### 핵심 설계 결정

**inject 파일 = 세션 레시피:**
```
[페르소나] + [참조 컨텍스트 플레이스홀더] + [작업 지시] + [완료 기준] + [세션 종료 시 보고]
```
→ 사람이 플레이스홀더에 workspace 내용을 채운 뒤 새 AI 스레드에 붙여넣으면 세션 시작.

**모든 inject 파일 = 새 스레드:**  
같은 스레드 연속 작업은 inject 파일 불필요. inject = "새 스레드 시작 시 필요한 것"이라는 정의에서 도출.

**workspace 파일 = 재료 저장소:**  
inject 파일의 플레이스홀더에 채워 넣는 실제 데이터. 프로젝트마다 내용이 다름.

→ 철학 → 구조 → 파일 빌드 완료.

## 🏗️ 세션 3: 구조 고도화 (2026-05-28) ✅

### 3.1 하네스 vs Workspace 분리

**출발 관찰:** 초기 구조는 `harness/project_memory/`가 하네스 안에 있어 역할이 애매했다.  
프로젝트가 진행되면서 아키텍처 규칙(어떻게 만들었나)과 요청 문서(무엇을 만드나)가 모두 쌓이는데, 이를 관리할 레이어가 없었다.

**확립된 원칙:**

| | 하네스 (`harness/`) | Workspace (`workspace/`) |
|--|-------------------|--------------------------|
| 내용 | 어떻게 일할 것인가 (방법, 절차) | 이 프로젝트가 무엇인가 (데이터) |
| 변경 | 하네스 개선 시 | 프로젝트 진행 시 |
| 재사용 | 모든 프로젝트 공통 | 프로젝트마다 별도 |

### 3.2 최종 하네스 구조

```
harness/                             ← 프로젝트 독립 (방법론)
├── core/
│   ├── principles.md                ← 10개 테제 실전 요약
│   └── session_protocol.md          ← 세션 유형 결정 트리 + 자율성 레벨
├── personas/                        ← agent 역할 정의
├── skills/                          ← 분석/기술 스킬
│   ├── analysis/                    ← silent_decision_extraction 등
│   └── tech/                        ← spring, python, react 등
├── workflows/                       ← 세션 시퀀스
├── inject/                          ← 세션 패키지 (새 AI 스레드에 붙여넣기)
│   ├── layer0_init.md
│   ├── layer1_design.md
│   ├── layer1_implement.md
│   ├── layer1_review.md
│   └── layer2_checkpoint.md
│
└── my-project/                      ← 새 프로젝트 시작 템플릿 (복사해서 사용)
    └── workspace/                   ← 프로젝트 종속 데이터
        ├── project_memory/          ← 아키텍처 결정, 도메인 규칙 (T9)
        │   ├── shared/
        │   │   ├── project_identity.md
        │   │   ├── architecture.md
        │   │   └── contracts.md
        │   └── domains/[도메인명]/  ← ui/backend 고정이 아닌 프로젝트별 자유 구성
        │       ├── rules.md
        │       └── registry.md
        ├── requests/                ← 요청 문서 (T9 확장)
        │   ├── INDEX.md
        │   ├── active/
        │   └── done/
        └── docs/                    ← 기능/설계 문서 (T9 확장)
            └── INDEX.md
```

### 3.3 T9 확장: 프로젝트 기억의 범위 확대

**기존 T9의 프로젝트 기억:** 아키텍처 결정, 도메인 규칙, 인터페이스 계약, 안티패턴

**확장된 관찰:**  
개발이 진행되면서 두 종류의 문서가 쌓인다. 이 둘은 성격이 다르며, T9의 관리 대상이어야 한다.

| 레이어 | 담는 것 | 수명 | 위치 |
|--------|---------|------|------|
| `project_memory/` | 어떻게 만들었나 (아키텍처·규칙) | 지속 유지 | workspace/project_memory/ |
| `requests/` | 무엇을 만들어야 하나 (요청·스펙) | 완료 후 done/으로 이동 | workspace/requests/ |
| `docs/` | 만든 것이 어떻게 동작하나 (결과 문서) | 계속 갱신 | workspace/docs/ |

**핵심 연결 고리:**  
요청 문서(`requests/`)가 완료되면 해당 요청의 "완료 시 문서 업데이트 대상"을 통해 `docs/`가 갱신된다.  
이 흐름이 관리되지 않으면 T10이 경고한 "문서 부패(Documentation Drift)"가 발생한다.

**T9 확장 명제:**
> 프로젝트 기억은 아키텍처 규칙만이 아니다.  
> **요청의 흐름(requests → 구현 → docs 반영)**이 추적되지 않으면  
> 무엇이 구현됐고 어디에 문서화됐는지 알 수 없게 된다.

### 3.4 메타 구조: 시스템을 만드는 시스템

**관찰:**  
이 시스템 전체를 보면 레벨이 나뉜다. 프로젝트의 하네스를 만들기 위한 하네스가 존재한다.

```
principles/      ← 레벨 0: 왜 (Why)    — 원칙과 철학
harness/         ← 레벨 1: 어떻게 (How) — 하네스를 만드는 시스템
workspace/       ← 레벨 2: 무엇을 (What) — 실제 프로젝트의 하네스
```

이것은 "시스템을 만들기 위한 시스템"이다.  
컴파일러와 컴파일러가 만든 프로그램의 관계와 같다.

**T2와의 연결:**  
테제 2는 "인간은 환경 설계자"라고 했다.  
이 3레벨 구조는 그 테제의 재귀적 적용이다.

- 레벨 2 workspace = AI가 작업하는 환경
- 레벨 1 harness = 그 환경을 설계하는 환경
- 레벨 0 principles = 설계 방식을 결정하는 환경

**레벨 구분의 실용적 의미:**

| 변경 대상 | 영향 범위 | 변경 시 주의 |
|----------|----------|------------|
| principles/ | 모든 하네스의 방향 | 높음 — 테제 자체를 바꾸는 것 |
| harness/ | 이 하네스를 쓰는 모든 프로젝트 | 중간 — 방법론 변경 |
| workspace/ | 해당 프로젝트만 | 낮음 — 프로젝트 데이터 |

### 3.5 의존성 가시성 문제

**관찰:**  
프로젝트 문서 구조에서 의존성은 두 종류로 나뉜다.

| 종류 | 정의 | 현재 추적 방법 |
|------|------|--------------|
| 보이는 의존성 | `contracts.md`, 요청의 `관련 문서` 필드 | 명시적 — 추적됨 |
| 보이지 않는 의존성 | 코드에는 있지만 문서화 안 된 기능 간 결합 | `dependency_mapping.md` 스킬로 발견 가능, 저장소 없음 |

**미결 과제:** 발견된 비가시적 의존성을 지속 저장하는 구조가 아직 없다.  
발견 → `workspace/project_memory/shared/architecture.md`에 기록하는 것이 현재 최선.

## 📚 세션 4: 양쪽 하네스 (2026-05-29)

### 4.2 흐름 분석: 두 개의 구조적 약점

**사용자가 제안한 흐름:**
```
사용자 브리핑 → 에이전트 plan.md 작성 → 사용자 검토 → 논의 → 작업
```

이 흐름의 방향은 맞다. 단 두 지점이 약하다:

**약점 1 — 브리핑 단계:**  
"간단한 브리핑"이 자연어이면, 에이전트가 plan.md를 만들기 위해 먼저 빈칸을 채워야 한다.  
사용자가 실패 비용을 생각하기 전에 에이전트의 가정이 plan.md에 이미 들어간다.

**약점 2 — 검토 단계 (T4 역방향):**  
plan.md가 잘 구조화돼 있으면 검토자는 자연스럽게 "잘 됐네"라고 반응한다.  
에이전트가 사용자에게 동조하는 게 아니라, **사용자가 에이전트의 잘 만들어진 결과물에 동조**하는 역전이 일어난다.  
검토에서 실제로 잡아야 할 것들 — 조용한 결정, 가치 결정 숨김 — 이 통과된다.

---

### 4.3 해법: plan.md를 사용자-에이전트 소통 인터페이스로 재설계

**핵심 통찰:**
> 별도 체크리스트를 만드는 게 아니라, plan.md 자체에 에이전트가 사용자에게 보고하는 섹션을 넣는다.  
> 구조가 행동을 만든다 — 사용자가 별도로 무언가를 열지 않아도, 검토 시점에 정보가 눈앞에 있다.

**추가된 `에이전트 보고` 섹션 (plan.md 최상단):**

```markdown
## 에이전트 보고

### 사용자 결정 필요
AI가 결정해서는 안 되는 항목 — 가치 결정 또는 비즈니스 맥락 의존

### 에이전트 가정
브리핑에 없어 임의로 채운 내용 | 근거 | 틀렸다면 어떻게 달라지는가
```

**이 변경이 해결하는 것:**
- 약점 1: 에이전트 가정이 숨겨지지 않고 명시된다
- 약점 2: 검토자가 무엇을 봐야 하는지 구조가 안내한다
- T5(비가시적 위임): 에이전트가 채운 빈칸이 빈칸 A(의도적)인지 빈칸 B(비가시적)인지 사용자가 판단할 수 있게 된다
- T8(가치 결정): 기술적 결정처럼 보이는 것 중 가치 결정이 있으면 에이전트가 먼저 신고한다

**레버리지 순서:**
1. plan.md 에이전트 보고 섹션 — 정보 구조를 바꿈 (가장 높음)
2. 이 섹션을 읽는 방법을 아는 것 — 철학 내재화로 해결됨

### 4.4 인터페이스 레이어 재정의: 하네스의 실제 기반

**사용자 제기:**
> "현실적으로 우리가 직접적인 LLM을 사용하지는 않아. CLI나 IDE를 사용하기 때문에 그 안에 포함된 하네스 구조도 상당히 잘 만들어져 있을 것이라고 생각해."

**초기 가정의 문제:**

```
암묵적 가정: LLM API → 우리 하네스 → 작업
실제 구조:   LLM API → CLI/IDE → 우리 하네스 → 작업
```

CLI/IDE(Claude Code, Cursor 등)가 이미 제공하는 것:
- 도구 실행 (파일 읽기/쓰기, 터미널)
- 컨텍스트 윈도우 관리
- 권한 시스템
- CLAUDE.md / .cursorrules 같은 프로젝트 설정 주입

→ 이것들을 우리가 다시 만들 필요가 없다.

**그러나 즉시 반례 발견 (사용자 제기):**
> "LLM API를 사용할 수도 있겠네.."

CLI/IDE는 선택적 레이어다. 누군가는 ChatGPT에 직접 붙여넣고, 누군가는 API를 직접 호출하고, 누군가는 Claude Code를 쓴다.

**수정된 레이어 모델:**

```
LLM API        ← 항상 존재, 때로는 직접 사용
    ↓ (선택적)
CLI/IDE        ← 없을 수도 있음
    ↓
우리 하네스    ← 어느 레이어 위에서도 동작해야 함
    ↓
workspace/     ← 프로젝트 데이터
```

**핵심 발견 — 우리 하네스는 이미 인터페이스 독립적이다:**

inject 파일, 페르소나, project_memory — 전부 텍스트다.  
CLI가 읽든, 사용자가 직접 붙여넣든 동작한다.

인터페이스에 의존하는 부분은 딱 하나:

```
agent-configs/
├── CLAUDE.md    ← Claude Code 전용
└── AGENTS.md    ← Codex 전용
```

`agent-configs/`가 인터페이스별 브리지 역할을 하고, 나머지 하네스는 인터페이스 독립적이다.

**"사용자 하네스" 명칭 문제:**

"사용자 하네스"는 어색하다. 이유:
- 하네스는 실행하는 쪽에 붙는 개념
- 사용자는 실행하지 않고 판단하고 결정한다

정확한 구분:

| | 에이전트 | 사용자 |
|--|---------|-------|
| 구조 이름 | 하네스 (Harness) | 프로토콜 (Protocol) |
| 형태 | 파일로 명시, 런타임에 읽음 | 공유된 작업 방식, 판단 기준 |
| 핵심 | 어떻게 실행할 것인가 | 언제 무엇을 결정할 것인가 |

plan.md의 "에이전트 보고" 섹션은 둘 사이의 인터페이스 — 에이전트 하네스도, 사용자 프로토콜도 아닌 양쪽이 만나는 지점.

**정리:**
> CLI/IDE 레이어를 가정하되, 그 가정이 없어도 동작하는 인터페이스 독립적 구조로 하네스를 유지한다.  
> 인터페이스별 차이는 `agent-configs/`에서만 관리한다.

### 4.5 확률적 자연어 프로그래밍

**사용자 제기:**
> "논의하는 내용을 보면 지금 확률적 자연어 프로그래밍을 하는 듯한 느낌이야. 하네스가 설계도처럼 보이고."

**비교:**

| | 전통 프로그래밍 | 우리가 하는 것 |
|--|--------------|-------------|
| 언어 | 형식 언어 (Python, Java) | 자연어 |
| 실행기 | 컴파일러 / 인터프리터 | LLM |
| 실행 방식 | 결정론적 | 확률론적 |
| 버그 | 에러 발생, 즉시 발견 | 조용히 다르게 실행 (T1) |
| 미명세 동작 | 컴파일 오류 또는 undefined behavior | LLM이 조용히 채움 (T5) |

**하네스 구성 요소의 프로그래밍 언어 대응:**

| 하네스 | 프로그래밍 |
|--------|----------|
| inject 파일 | 함수 정의 |
| 페르소나 | 클래스 / 인터페이스 |
| agent_rules.md | 전역 상수 / 제약 |
| project_memory | 프로그램 상태 |
| plan.md "에이전트 보고" | 컴파일러 경고 ("이 부분 타입 불명확") |
| layer2_checkpoint.md | 통합 테스트 |

**하네스의 역할 재정의:**

> LLM의 해석 공간(확률 분포)에서 우리가 원하는 방향으로 수렴하도록 엔트로피를 줄이는 것.  
> 완전한 결정론은 불가능하다. 하네스의 목표는 확률적 실행을 **충분히 예측 가능한** 범위 안으로 가져오는 것이다.

**기존 테제의 재독:**

이 관점에서 테제들이 프로그래밍 언어 개념으로 재해석된다:
- T1 (실행 가능한 오해) = 런타임 오류인데 에러가 안 나는 것
- T5 (비가시적 위임) = undefined behavior
- T7 (구조가 실패 비용을 결정) = 코드 품질이 장애 허용 범위를 결정

**비유가 깨지는 지점 — 컴파일러가 예고 없이 바뀐다:**

전통 프로그램은 언어 버전이 바뀌어도 호환성이 유지된다.  
자연어 프로그램은 모델이 업데이트되면 같은 "코드"가 다르게 실행될 수 있다.  
Claude → GPT → Gemini 간에도 같은 inject 파일이 다르게 해석된다.

→ 이것이 우리 하네스가 인터페이스 독립적이어야 하는 또 다른 이유: 특정 모델의 해석 방식에 의존하면, 모델이 바뀔 때 "프로그램"이 깨진다.

---

### 4.6 모호성은 버그인가, 피처인가

**전통 프로그래밍 vs 자연어 프로그래밍의 근본 차이:**

전통 프로그래밍에서 모호성은 무조건 버그다. 컴파일러가 허용하지 않는다.  
그런데 우리가 LLM을 쓰는 이유는 모호성을 채워주길 바라서다 (T5에서 확립).  
→ **자연어 프로그래밍에서 모호성은 피처이기도 하고 버그이기도 하다.**

| | 전통 프로그래밍 | 자연어 프로그래밍 |
|--|--------------|----------------|
| 모호성 | 무조건 버그, 컴파일 불가 | 피처(빈칸 A)이기도, 버그(빈칸 B)이기도 |

**T5의 프로그래밍 언어적 재해석:**

"확률적 자연어 프로그래밍" 관점이 T5를 다른 언어로 정확히 다시 설명한다:
- 빈칸 A (의도적 위임) = 피처인 모호성 — LLM이 판단해주길 바라는 곳
- 빈칸 B (비가시적 위임) = 버그인 모호성 — 결정이 필요한 줄 몰랐던 곳

**하네스의 역할 재정의:**

형식 언어는 모든 모호성을 닫는다 → 창의적 gap-filling 불가능.  
자연어 하네스는 선택적으로 닫는다 → LLM의 장점을 살리면서 위험을 통제.

> 하네스가 하는 일:  
> 자연어의 모든 모호성을 제거하는 것이 아니라,  
> **의도한 모호성(빈칸 A)은 열어두고, 의도하지 않은 모호성(빈칸 B)은 닫는 것.**

**미결 질문:**

하네스가 "어떤 모호성을 닫을 것인가"를 결정하는 기준은 무엇인가?

지금까지의 암묵적 기준은 T6(실패 비용) — 실패 비용이 높은 모호성을 닫는다.  
이것이 충분한 기준인가, 아니면 다른 기준이 더 있는가? 🔄

### 4.7 실패 비용은 측정되는가, 판단되는가

**사용자 제기:**
> "실패 비용이 높은 곳을 닫는다는 말은 실패 비용을 계산할 수 있다는 전제가 되어야 하는데, 이건 어떻게 측정할 수 있을까?"

T6가 3개 축(심각도·발견가능성·가역성)을 줬지만, 그 축을 따라 실제로 측정하는 방법은 말하지 않았다.

**실패 비용 측정의 실제 방법들:**

| 방법 | 신뢰도 | 한계 |
|-----|--------|------|
| 경험·이력 | 높음 | 처음 하는 일엔 없음 |
| 구조적 신호 (T6의 현재 접근) | 중간 | 알려진 카테고리만 작동 |
| 도메인 전문가 판단 | 높음 | 항상 있지 않음 |
| 소규모 시험 먼저 | 높음 | 항상 가능하지 않음 |
| 유추 (비슷한 상황 참조) | 낮음 | 유추가 틀릴 수 있음 |

어느 방법도 선험적으로 정밀한 측정을 보장하지 않는다.

**측정 역설 — 탐지 역설의 메타 레벨 반복:**

```
탐지 역설:   발견할 수 있는 빈칸은 이미 위험하지 않고,
             위험한 빈칸은 발견할 수 없다.

측정 역설:   측정할 수 있는 실패 비용은 이미 아는 영역이고,
             모르는 영역의 실패 비용은 측정할 수 없다.
```

정밀하게 측정할 수 있다면 — 이미 충분히 아는 영역이라 실제 위험이 낮다.  
측정이 안 된다면 — 그게 진짜 위험 구간이다.

**이것이 끌어내는 원칙:**

> 실패 비용을 측정하기 어려울수록, 높게 잡는다.

측정 불확실성 자체가 정보다. 불확실하다는 것은 "이 모호성의 결과를 알지 못한다"는 뜻이고, 그것 자체가 높은 실패 비용의 신호다.

**실질적 해법 — "에이전트 보고" 섹션의 재해석:**

추상적인 3축 계산보다, 눈앞의 가정 하나에 대한 구체적 판단이 더 실질적이다.

에이전트가 "저는 X를 이렇게 가정했습니다"라고 보고할 때,  
사용자가 판단하는 것: "이 가정이 틀렸을 때 얼마나 나쁜가?"

→ 이것이 실패 비용 추정의 가장 구체적인 형태.

**결론:**

> 실패 비용은 선험적으로 정밀하게 측정되는 것이 아니라,  
> **가정이 드러나는 순간에 판단되는 것**이다.  
> 측정 도구보다 가정을 가시화하는 구조가 더 실질적인 해법이다.

### 4.8 보고 역설과 하네스의 한계

**"가정이 드러날 때 판단된다"는 결론에서 남는 문제:**

"에이전트 보고"는 에이전트가 인식한 가정만 보고한다.  
훈련 데이터에서 "당연한 것"으로 굳어진 가정 — 에이전트 입장에서 사실인 것 — 은 보고되지 않는다.

**세 역설의 동일한 구조:**

```
탐지 역설:   발견할 수 있는 빈칸은 이미 위험하지 않다
측정 역설:   측정할 수 있는 실패 비용은 이미 아는 영역이다
보고 역설:   보고할 수 있는 가정은 에이전트가 인식한 것뿐이다
```

세 역설이 같은 구조다: **알 수 있는 것은 이미 덜 위험하고, 진짜 위험한 것은 알 수 없다.**

**구조(하네스)로 해결 가능한 영역과 불가능한 영역의 경계:**

```
구조로 해결 가능:  인식된 가정을 가시화한다
구조로 해결 불가:  인식되지 않은 가정의 존재를 드러낸다
```

인식되지 않은 가정을 드러내는 것은 구조의 문제가 아니라 사람의 도메인 지식과 경험의 문제다.

**T2(환경 설계자)의 실질적 의미 — 재정의:**

환경 설계자의 역할이 "파일을 잘 구성하는 것"이 아니다.

> 에이전트가 인식하지 못하는 가정을 인간이 먼저 인식하고,  
> 그것을 드러낼 수 있도록 환경을 설계하는 것.

이 역량은 하네스가 줄 수 없다. 하네스는 인식된 것을 구조화할 뿐이다.

**하네스의 한계 — 명시적 확정:**

> 하네스는 알려진 위험을 관리하는 도구다.  
> 알려지지 않은 위험은 여전히 인간의 도메인 역량에 달려 있다.

**상위 통합 원칙 전제 조건과의 연결:**

세션 1에서 추가한 조건:
> *"인간이 최소한의 검증 능력을 갖춘 영역에서"*

이 조건이 왜 필요한지가 지금 논의를 거치면서 비로소 설명된다.  
하네스가 인식된 가정만 다룰 수 있다면, 인식되지 않은 가정을 걸러낼 최소한의 도메인 역량이 인간에게 있어야 한다. 그 역량이 없으면 하네스도 작동하지 않는다.

### 4.9 도메인 온톨로지 — 세 역설의 공통 뿌리

**사용자 제기:**
> "그러면 데이터 온톨로지의 개념이 필요한 거 아냐?"

**세 역설의 공통 뿌리:**

에이전트가 인식하지 못하는 가정이 생기는 이유:

> 에이전트가 훈련 데이터에서 형성된 자신의 온톨로지로 프로젝트를 해석한다.  
> "사용자", "권한", "결제" 같은 개념을 이 시스템의 정의가 아닌  
> 훈련 데이터의 가장 일반적인 정의로 채운다.

**충돌 구조:**

```
에이전트의 온톨로지  ←── 훈련 데이터 (일반적 패턴)
        vs
프로젝트의 온톨로지  ←── 이 시스템의 실제 개념 정의
```

두 온톨로지가 어긋나는 지점이 보고 역설이 작동하는 곳이다.  
에이전트 입장에서 가정이 아니라 사실이기 때문에 보고하지 않는다.

**온톨로지 명시의 효과:**

```
# 온톨로지 없을 때
에이전트: "사용자" = 훈련 데이터의 일반적인 웹 사용자

# 온톨로지 있을 때
에이전트: "사용자" = 캠핑 예약 맥락에서 캠핑장 운영자와
          일반 캠퍼 두 종류가 있고, 권한 구조가 다름
```

에이전트가 훈련 데이터 온톨로지 대신 프로젝트 온톨로지 안에서 작동하게 된다.

**현재 project_memory와의 관계:**

지금 하네스의 구성 요소들은 온톨로지의 파편이다:
- `project_identity.md` — 프로젝트가 무엇인가
- `domains/[name]/rules.md` — 도메인 규칙
- `architecture.md` — 기술 결정

그러나 "이 개념은 이 시스템에서 이런 의미다"를 명시적으로 정의하는 레이어가 없다.

**온톨로지를 project_memory에 추가한다면:**

```
shared/
├── project_identity.md  — 프로젝트의 존재 이유
├── ontology.md          — 이 시스템의 핵심 개념 정의  ← 추가 필요
├── architecture.md      — 기술 결정
└── contracts.md         — 인터페이스
```

`ontology.md`의 구성:
- 핵심 개념 정의 — 이 시스템에서 각 개념이 무엇인가
- 개념 간 관계
- **부정 정의** — "여기서 결제는 일반적인 e-commerce 흐름이 아니다"

부정 정의가 특히 중요하다: 에이전트의 기본 가정을 명시적으로 차단한다.

**상위 통합 원칙의 확장:**

> 기존: 구조는 설계하는 것이다  
> 확장: 구조에는 개념 정의(온톨로지)도 포함된다

온톨로지는 T9(프로젝트 기억)의 Tier 1 — 핵심, 거의 변하지 않는 것 — 에 해당한다.  
아키텍처 결정보다 더 근본적인 레이어다.

### 4.10 온톨로지 유지보수 가능성 — 범위를 좁혀야 살아남는다

**사용자 제기:**
> "온톨로지를 과연 쓸모 있게 계속 유지보수가 가능한가에 대한 고민이 있었어."

**왜 온톨로지는 유지보수가 어려운가:**

문서 부패(T10)의 특수 케이스다. 그런데 일반 문서보다 더 어렵다.

코드가 바뀌면 → 사람이 인식한다.  
개념이 바뀌면 → 서서히 바뀌어서 인식하기 어렵다.

업데이트 트리거가 불명확하다는 것이 핵심 문제.

결과:
```
낡은 온톨로지 > 없는 것보다 위험
```
에이전트가 틀린 개념 정의를 사실로 받아들이기 때문. T1(실행 가능한 오해)이 온톨로지 레벨에서 발생.

**유지보수가 잘 되는 문서의 조건 (T10·B에서 도출):**

```
- 업데이트 트리거가 명확하다
- 코드와 직접 연결된다
- 짧고 구체적이다
- 틀렸을 때 즉시 발견된다
```

전체 온톨로지는 이 조건을 하나도 만족하지 않는다.

**유지보수 가능한 범위로 축소:**

> 코드에서 읽을 수 있는 것은 빼고,  
> 코드만 봐서는 알 수 없는 것만 남긴다.

남는 것: **"이 시스템의 개념이 일반적인 패턴과 어떻게 다른가"** — 부정 정의

부정 정의가 유지보수 가능한 이유:
- 트리거 명확 — 에이전트가 일반적 가정으로 실수했을 때, 새 개념이 추가될 때
- 짧음 — 다른 것만 기록
- 발견 가능성 높음 — 틀렸을 때 에이전트 오류로 나타남

**최종 결론:**

> 완전한 온톨로지는 유지보수 불가능하다.  
> 유지보수 가능한 것은 "일반적 패턴과 다른 것만" 기록하는 부정 정의다.  
> 구현 형태는 별도 파일(ontology.md)이 아니라  
> 기존 `project_identity.md` 또는 `domains/rules.md`에 녹여넣는 것이 현실적이다.

온톨로지라는 개념은 맞지만, 구현은 가볍게 가야 한다.

### 4.11 하네스는 학습 시스템이다 — layer0_init 재검토

**핵심 전환:**

부정 정의는 경험에서 온다. 에이전트가 잘못된 가정으로 실수한 뒤에야 "이 개념은 다르다"를 알게 된다.

```
에이전트가 훈련 데이터 가정으로 실수
    ↓
사람이 발견
    ↓
부정 정의 / roles/ / upgrade-candidates에 기록
    ↓
다음에는 같은 실수를 안 함
```

이것은 예방이 아니라 학습이다. 첫 실수는 막을 수 없다. 두 번째부터 막는다.

**이 패턴이 하네스 전체에 이미 있다:**

| 레이어 | 학습 메커니즘 |
|--------|------------|
| 태스크 레벨 | `roles/[페르소나].md` — 놀라움 필터로 기록 |
| 도메인 레벨 | `domains/rules.md` (부정 정의 포함) |
| 시스템 레벨 | `project_identity.md` (부정 정의 포함) |
| 하네스 레벨 | `upgrade-candidates/` — 방법론 개선 |

하네스 전체가 정적 설계도가 아니라 **실수와 발견이 누적될수록 개선되는 학습 시스템**이다.

**세 역설의 재독:**

탐지/측정/보고 역설은 극복 불가능한 한계가 아니라 **학습이 작동하는 방식**을 설명한다.

알 수 없는 것 → 실수 → 알게 됨 → 기록 → 다음엔 알려진 것이 됨

역설은 사라지지 않는다. 하지만 하네스가 누적될수록 "알 수 없는 것"의 범위가 줄어든다.

**하네스의 성숙도는 사용 시간에 비례한다.**

처음 적용: 보호막이 얇음  
오래 운영: 누적된 학습으로 보호막이 두꺼워짐

→ 처음부터 완벽한 하네스를 만들려는 시도는 틀렸다.  
하네스는 처음부터 완성되는 게 아니라, 운영하면서 자라는 것이다.

**layer0_init 재검토:**

이 관점에서 layer0_init의 역할이 달라진다.

```
기존 관점: 시작 시 최대한 많이 기록한다 (완성된 하네스 설치)
새 관점:   학습이 자랄 수 있는 구조를 심는다 (씨앗 심기)
```

씨앗에 필요한 것:
- 학습이 쌓일 그릇 (roles/, domains/, upgrade-candidates/) — 구조, 처음부터 완성
- 학습 트리거 (놀라움 필터, 에이전트 보고) — 구조, 처음부터 완성
- 최소한의 초기 컨텍스트 (project_identity, 절대 규칙) — 내용, 지금 알 수 있는 것만

그릇과 트리거는 구조 — 처음부터 완성된다.  
내용은 운영하면서 채워진다.

**현재 layer0_init의 문제:**

아직 경험하지 않은 도메인 규칙, 아직 만나지 않은 안티패턴을 초반에 채우려 한다.  
경험 전에 채운 내용은 추정이지 학습이 아니다.  
추정이 틀린 채 남아있으면 T10이 경고한 "낡은 문서 > 없는 것보다 위험" 상태가 된다.

**미결 과제:** layer0_init에서 "지금 채워야 하는 것"과 "운영하면서 채워지는 것"의 경계를 재설정 → 4.12에서 해소

### 4.12 layer0_init 기준 정밀화 — 타이밍이 아니라 검증 여부

**사용자 반례:**
> "이미 경험을 통해서(다른 프로젝트 또는 사용자에 의한 경험) 채워진 것까지 배제하는 건 너무 과할 수도 있는데."

**4.11의 기준 수정:**

4.11에서 "지금 vs 나중"으로 기준을 잡은 것이 틀렸다. 진짜 기준은 **검증 여부**다.

| 지식 출처 | 검증 여부 | Layer0에 넣어야 하는가 |
|---------|---------|-------------------|
| 사용자의 도메인 전문 지식 | 검증됨 | ✅ 넣어야 함 |
| 다른 프로젝트에서 검증된 패턴 | 검증됨 | ✅ 넣어야 함 |
| 이미 명확한 제약 (변경 불가) | 검증됨 | ✅ 넣어야 함 |
| 이 프로젝트 운영 중 발견할 것 | 미검증 | ❌ 나중에 |
| "아마 이런 패턴이 필요할 것" | 추정 | ❌ 넣지 말아야 함 |

> **Layer0의 기준: 타이밍이 아니라 검증 여부.**

**layer0_init 질문의 방향 전환:**

```
기존 (추정 유도):      "어떤 아키텍처 패턴이 필요할 것 같나요?"
바꿔야 할 방향 (검증된 지식):
    "다른 프로젝트에서 검증된 패턴 중 여기서도 유효한 것이 있나요?"
    "이 도메인에서 일반적 가정과 다른 것을 이미 알고 있나요?" (부정 정의)
    "지금 확실히 알고 있는 제약이 있나요?"
```

**upgrade-candidates 파이프라인의 역할 명확화:**

```
프로젝트 A에서 검증된 패턴
    ↓ upgrade-candidates → harness 반영
    ↓
프로젝트 B의 layer0_init에서 활용
```

다른 프로젝트의 경험이 harness를 통해 새 프로젝트 layer0에 들어오는 경로가 이미 설계되어 있다.

**최종 정리:**
> layer0_init에서 배제해야 하는 것은 "초반에 채우는 것"이 아니라 **"추정"**이다.  
> 검증된 지식은 출처가 어디든 — 사용자 경험, 다른 프로젝트, 기존 코드베이스 —  
> layer0에 적극적으로 담아야 한다.

### 4.13 삭제 역설 — 하네스 유지보수의 가장 어려운 문제

**사용자 관찰:**
> "유지보수와 업그레이드의 범위에 당연히 불필요한 부분에 대한 삭제도 필요해."

**삭제 역설의 구조:**

삭제가 필요한 규칙은 어떻게 식별하는가?

```
규칙이 자주 위반된다 → 문제가 보임 → 수정 가능
규칙이 전혀 위반되지 않는다 → ??? → 삭제 가능? 아니면 잘 지켜지는 것?
```

"침묵하는 규칙"은 두 가지 중 하나다:
- **A. 잘 지켜지는 규칙**: 필요하고 효과적으로 작동 중
- **B. 불필요한 규칙**: 아무도 신경 쓰지 않아도 문제가 생기지 않음

A와 B는 외부에서 관찰 불가능하다. 둘 다 침묵한다.

이것이 **삭제 역설**이다: 탐지 역설, 측정 역설, 보고 역설과 동일한 구조.

> **공통 구조:** 진짜 위험한 것(또는 불필요한 것)은 보이지 않는다.  
> 보이는 것은 덜 위험하다(또는 이미 필요하다).

**실용적 해결 방향:**

삭제 역설을 "극복"할 수 없다. 하지만 줄일 수 있는 방법:

1. **사용 빈도 신호**: 이 규칙이 실제로 몇 번 발동되었는가? — layer2_checkpoint에서 확인
2. **기원 추적**: 규칙이 어떤 실수에서 나왔는가? 그 실수가 더 이상 발생하지 않는 환경이 되었는가?
3. **삭제 실험**: 규칙을 임시 비활성화하고 문제가 생기는지 관찰 → 하지만 발견 가능성이 낮은 실수는 이것도 통하지 않음

**남는 불가지 영역:**

"아무 문제 없었지만 규칙이 없었다면 문제가 생겼을" 상황은 회고적으로도 판단하기 어렵다.  
이것은 **인과성 추정 문제**다 — 하네스가 해결해 줄 수 없고, 운영자의 판단에 달려 있다.

**현실적 귀결:**

> 삭제 역설은 하네스 설계로 완전히 해결되지 않는다.  
> 할 수 있는 것: 삭제 판단을 위한 신호(기원, 빈도)를 수집하고,  
> 최종 판단은 인간의 맥락적 판단에 맡기는 것.

### 4.14 LLM vs CLI/IDE — 기억 구조의 차이와 하네스 설계

**사용자 관찰:**
> "llm 은 자체 기억이 없지만 cli 나 ide 는 자체 기억을 가지고 가는걸로 알고 있어.  
> 그럼 agent 에 따라서 달라진다는건데. 이 부분은 어떻게 해결해야 할까?"

**차이의 구조:**

| 인터페이스 | 기억 구조 |
|----------|---------|
| Raw LLM API | 세션 간 기억 없음. 매 세션 전체 컨텍스트 재주입 필요 |
| Claude Code CLI | auto-memory (`~/.claude/`), CLAUDE.md 항상 로딩, 대화 압축 |
| AI IDE 확장 | 유사하게 프로젝트별 컨텍스트 유지 |

**기억의 레이어별 분리:**

그러나 "CLI/IDE가 기억을 가진다"는 것을 면밀히 보면, 기억의 성격이 다르다:

```
CLI/IDE 자체 기억:
  - 사용자 개인 선호 (전역)
  - 대화 압축본 (세션 내 연속성)
  - CLAUDE.md 내용 (항상 로딩)

하네스 workspace/:
  - 프로젝트 전용 지식 (누적 학습)
  - 역할별 기억 (roles/)
  - 아키텍처 결정 기록
```

이 둘은 **목적이 다르고 저장 위치도 다르다.** 중복이 아니다.

**실제 겹치는 부분:**

CLAUDE.md에 `agent_rules.md` 내용을 넣으면 CLI에서는 항상 로딩된다.  
반면 raw LLM API에서는 매번 수동으로 주입해야 한다.

**해결 방향 — 인터페이스 독립적 설계 + 브리지:**

하네스 자체는 변경하지 않는다. 대신:

```
우리 하네스 (인터페이스 독립)
    ↓
agent-configs/
    claude-code.md    ← "CLAUDE.md가 이미 커버하는 것 / 우리가 추가해야 하는 것"
    raw-llm.md        ← "매 세션 주입해야 하는 파일 목록"
    cursor.md         ← ...
```

이미 세션 4.5에서 `agent-configs/` 디렉토리를 인터페이스 브리지로 지정했다.  
LLM vs CLI/IDE 기억 차이는 이 브리지에서 처리하는 것이 자연스럽다.

**핵심 원칙:**

> 하네스 코어는 **최소 공통 분모(raw LLM)** 기준으로 설계한다.  
> CLI/IDE 추가 기억은 bonus — 중복이 생겨도 해롭지 않다.  
> 단, CLI/IDE 환경에서 하네스를 더 효율적으로 쓰고 싶다면  
> `agent-configs/[인터페이스명].md`에 최적화 가이드를 담는다.

**미결 과제:**

현재 `agent-configs/` 디렉토리가 실제로 존재하지 않는다.  
claude-code.md 등의 브리지 파일을 만드는 것이 다음 단계의 후보.

> ⚠️ **4.14의 결론이 4.15에서 수정됨.** "raw LLM 기준 설계"는 잘못된 전제였다.

### 4.15 raw LLM 기준 전제의 오류 — 하네스는 이미 tool-using 에이전트를 가정한다

**사용자 반례:**
> "raw LLM 을 사실상 사용하는게 맞을까?"

**4.14 결론의 오류 발견:**

4.14에서 "raw LLM을 최소 공통 분모로 설계"한다고 했다. 그런데 inject 파일들이 실제로 에이전트에게 하는 지시를 보면:

```
"workspace/project_memory/shared/project_identity.md 를 읽어라"
"workspace/tasks/active/[태스크명]/plan.md 에 저장하라"
"upgrade-candidates/ 에 파일을 생성하라"
```

Raw LLM API는 이것을 수행할 수 없다. 파일을 읽지도, 쓰지도 못한다.

**현재 하네스는 이미 "파일 접근 도구가 있는 tool-using 에이전트"를 전제하고 있다.**  
Raw LLM 기준이라는 말 자체가 거짓 전제였다.

**진짜 기준선 재정의:**

```
기존(틀림): raw LLM 기준 설계 → CLI/IDE는 보너스
수정(맞음): tool-using 에이전트(= CLI/IDE) 기준 설계
             → raw LLM은 지원 범위 밖 (또는 별도 어댑터 필요)
```

| 기능 | Raw LLM API | Claude Code / Cursor / Windsurf |
|-----|------------|-------------------------------|
| 파일 읽기 | ✗ | ✅ |
| 파일 쓰기 | ✗ | ✅ |
| 멀티스텝 실행 | ✗ | ✅ |
| 하네스 호환 | ✗ | ✅ |

**`agent-configs/`의 역할 수정:**

4.14: "raw LLM vs CLI 차이 처리"  
수정: "CLI/IDE 간 차이 처리" — Claude Code vs Cursor vs Windsurf의 차이

raw LLM 호환이 필요한 경우가 생긴다면 그 때 별도 어댑터를 설계한다.  
지금은 존재하지 않는 요구사항을 위해 설계하지 않는다.

**최종 정리:**
> 우리 하네스의 실제 기준선은 **"파일을 읽고 쓸 수 있는 tool-using 에이전트"**이다.  
> 이것은 Claude Code, Cursor, Windsurf 등 현재 사용되는 모든 AI 코딩 도구가 충족하는 조건이다.  
> Raw LLM API는 이 조건을 충족하지 않으므로 지원 범위 밖이다.

### 4.16 기억 권위 역설 — CLI/IDE 내장 기억은 블랙박스로 취급한다

**관찰:**

4.15에서 확립한 "두 기억 시스템 공존" 구조에서 충돌 가능성이 발생한다.

```
CLI 내장 기억 (auto-memory):  "이 사용자는 짧은 응답을 선호한다"
하네스 agent_rules.md:         "plan.md를 상세히 작성하고 에이전트 보고 섹션을 채운다"
```

에이전트는 두 지시가 다른 출처임을 인식하지 못한다. 둘 다 컨텍스트에 있을 뿐이고, 어느 쪽을 따를지 조용히 결정한다. 그 결정이 조용하기 때문에 사용자는 무엇이 작동하는지 알 수 없다.

| 기억 | 소유자 | 성격 |
|-----|-------|-----|
| CLI 내장 기억 | 개인 (사용자) | 암묵적·자동·비공유 |
| 하네스 workspace/ | 프로젝트 (팀) | 명시적·의도적·공유 가능 |

**설계 결정 — CLI/IDE 내장 기억은 블랙박스:**

> "agent 또는 cli 에 내장된 내용에 대해서는 접근이 안된다고 생각하고 진행해야 될것 같아."

CLI/IDE 내부 기억에 접근하거나 의존하지 않는다.  
하네스는 자신이 제어할 수 있는 것만 — **명시적으로 inject되는 텍스트** — 에만 의존한다.

**이 결정의 함의:**

1. **충돌 역설의 해소 방향**: 충돌이 발생해도 하네스는 개입할 수 없다. 해소 방법은 "충돌을 막는 것"이 아니라 "하네스 지시를 더 구체적으로 만드는 것" — 구체적인 지시는 일반적인 선호보다 우선되는 경향이 있다.

2. **이식성 확보**: CLI/IDE 내부 구현에 의존하지 않으므로, 하네스는 어떤 tool-using 에이전트에서도 동일하게 작동한다.

3. **하네스의 통제 범위 명확화**:
   ```
   하네스가 통제할 수 있는 것: inject 파일, workspace/ 텍스트 파일
   하네스가 통제할 수 없는 것: CLI/IDE 내장 기억, 에이전트 모델 가중치
   ```

4. **설계 원칙 추가**: 하네스 파일에 "CLI/IDE 내장 기억을 읽거나 수정하라"는 지시를 넣지 않는다.

**남는 불확실성:**

CLI/IDE 내장 기억이 하네스 지시와 충돌할 때 실제로 어느 쪽이 우선되는지는 에이전트마다 다르고, 관찰도 어렵다. 이것은 하네스가 제어할 수 없는 영역으로 명시적으로 남겨둔다.

**세션 연속성 — plan.md가 하네스의 세션 메모리:**

블랙박스 결정의 귀결로, CLI/IDE의 대화 압축에 의존하지 않는다.  
하네스의 세션 연속성은 `workspace/tasks/active/plan.md`가 담당한다.

| 연속성 | 형태 | 신뢰도 |
|-------|-----|-------|
| CLI 대화 압축 | 자연어 요약, 자동 | 요약 품질에 의존, 불확실 |
| plan.md | 구조화된 상태 파일, 명시적 | 파일이 있는 한 정확 |

`구현 진행 상태` 테이블의 ⏳/🔄/✅ 상태가 세션 재개 시 컨텍스트 복원 역할을 한다.

**의존성 방향 확인 — 하네스는 라이브러리:**

```
CLI/IDE  →  harness 파일을 읽는다 (가능, 의도된 방향)  ✅
harness  →  CLI/IDE 내부를 읽는다 (불가, 블랙박스)     ✗
```

하네스가 라이브러리처럼 작동하고, CLI/IDE가 그것을 임포트한다.  
라이브러리는 임포터의 내부를 알 필요가 없고, 알아서도 안 된다.

> 사용자 확인: "방향이 맞는것 같아."

### 4.17 agent-configs/의 실제 역할 — CLI/IDE별 설치 파일

> ⚠️ 초안에서 "사용자 프로토콜의 일부"로 잘못 이해했다가 사용자 정정으로 수정됨.

**정확한 역할:**

`agent-configs/`는 사용자가 읽는 가이드가 아니라 **CLI/IDE별 설치 시 사용하는 설정 파일**이다.

```
agent-configs/
  claude-code/
    CLAUDE.md        ← Claude Code에 설치할 내용
  cursor/
    .cursorrules     ← Cursor에 설치할 내용
  windsurf/
    ...
```

layer0_init 시점에 사용 중인 CLI/IDE에 맞는 폴더의 파일을 제자리에 설치한다.

**역할 분리:**

| 디렉토리 | 역할 | 시점 |
|---------|-----|-----|
| `.mpa-workspace/` | 에이전트 행동 규칙, 역할, 스킬 | 매 세션 |
| `workspace/` | 프로젝트 지식, 태스크 상태 | 매 세션 |
| `agent-configs/` | CLI/IDE 브리지 파일 | 설치 시 1회 |

하네스 코어(`.mpa-workspace/`, `workspace/`)는 인터페이스 독립적으로 유지된다.  
CLI/IDE와 하네스를 연결하는 브리지 파일만 `agent-configs/`에서 설치된다.

**실제 구현 확인:**

`agent-configs/`에 이미 두 파일이 존재한다:

```
agent-configs/
  CLAUDE.md   ← Claude Code용 (기존 CLAUDE.md에 추가)
  AGENTS.md   ← 기타 에이전트 시스템용 (AGENTS.md에 추가)
```

`CLAUDE.md` 내용:
```
@.mpa-workspace/core/agent_rules.md
```

Claude Code의 `@` 문법을 사용해 CLAUDE.md가 로딩될 때 agent_rules.md가 자동으로 함께 로딩된다. 사용자가 매 세션 inject하지 않아도 하네스 규칙이 항상 활성화된다.

`AGENTS.md`는 `@` 문법을 지원하지 않는 다른 에이전트 시스템을 위해 텍스트 참조 방식으로 동일 내용을 전달한다.

**설계의 우아함:**

하네스 코어(`.mpa-workspace/`)는 건드리지 않는다.  
CLI/IDE의 "항상 로딩" 파일에 단 한 줄 — 진입점만 심는다.  
나머지는 에이전트가 필요할 때 읽는다.

### 4.18 컨텍스트 손실은 불가피하다 — 하네스의 설계 이유

**실제 경험:**

이 논의 세션 자체에서 컨텍스트 압축이 발생했다. 이전 대화 내용이 요약본으로만 남았고, 세션 재개 시 파일을 직접 다시 읽어야 컨텍스트를 복원할 수 있었다.

> "지금처럼 기존 내역을 잃어버리는 상황이 발생하는 거야. 이건 어쩔 수 없지만 진행하는 작업에 영향이 적도록 하려는 거지."

**이것이 하네스 설계의 실질적 이유다:**

CLI/IDE의 대화 압축은 컨텍스트를 보존하려고 하지만, 무엇이 남고 무엇이 사라지는지 통제할 수 없다. 이것이 4.16에서 "블랙박스" 결정을 내린 근거다.

```
대화 압축 (블랙박스):
  - 무엇이 요약되는지 알 수 없다
  - 요약 품질이 보장되지 않는다
  - 손실이 발생해도 감지하기 어렵다

workspace/ 파일 (통제 가능):
  - 명시적으로 기록된다
  - 언제든 읽을 수 있다
  - 손실이 없다
```

**하네스의 답:**

컨텍스트 손실 자체를 막을 수는 없다.  
하지만 손실이 작업에 미치는 영향을 줄일 수 있다.

- `plan.md` → 어디까지 했는지 상태 체크포인트
- `workspace/project_memory/` → 프로젝트 핵심 컨텍스트
- inject 파일 → 역할과 규칙을 세션마다 재로딩

세션이 끊겨도 이 파일들을 읽으면 작업을 재개할 수 있다.  
**하네스는 컨텍스트 손실에 탄력적인 작업 구조다.**

### 4.19 하네스 성장 문제 — 분리(B) 전략 채택

**문제:**

하네스가 성숙할수록 파일이 커진다. discussion_log.md도 지금 1700줄을 넘었다.  
컨텍스트 손실에 탄력적이라고 설계한 하네스 자체가, 자기 무게에 눌릴 수 있다.

**두 방향:**

- **A. 압축**: 불필요한 내용을 제거해서 순도를 높인다.
- **B. 분리**: 자주 읽는 것(핵심)과 가끔 읽는 것(히스토리)을 나눈다.

> ⚠️ 초안에서 "압축 = 요약 → 정보 손실"로 잘못 정의했다가 사용자 정정으로 수정됨.  
> 필요한 것이 사라지는 건 나쁜 압축이지, 압축의 정의가 아니다.  
> 압축 = 불필요한 것 제거, 다듬는 작업.

**A와 B는 경쟁 관계가 아니다:**

```
압축: 품질 문제 — 불필요한 내용을 제거해 순도 유지
분리: 접근 문제 — 자주 읽는 것과 가끔 읽는 것을 나눔
```

둘 다 필요하고, 서로 다른 문제를 다룬다.

**압축의 어려움 — 4.13 삭제 역설과 연결:**

압축이 어려운 이유는 "무엇이 불필요한가"를 판단하기 어렵기 때문이다.  
이것이 4.13에서 다룬 삭제 역설이다. 침묵하는 내용은 "여전히 필요한 것"인지 "이미 불필요한 것"인지 구분할 수 없다.

**결정: B (분리) 우선 적용, 압축은 판단 가능한 것부터**

분리는 즉시 적용 가능하다. 압축은 "이건 분명히 불필요하다"고 판단할 수 있는 것부터 점진적으로 적용한다.

**실제 적용:**

이미 workspace/project_memory/ 구조가 B 방향으로 설계되어 있다:

| 파일 | 읽는 시점 |
|-----|---------|
| `shared/project_identity.md` | 매 세션 (항상) |
| `shared/architecture.md` | 매 세션 (항상) |
| `roles/[페르소나].md` | 해당 역할 세션에만 |
| `domains/[도메인]/` | 해당 도메인 작업 시에만 |
| `tasks/done/` | 참고 필요할 때만 |

inject 파일이 "이번 세션에 무엇을 읽어라"를 명시하는 것 자체가 분리 전략의 실행이다.

**discussion_log.md에 적용:**

논의 과정(히스토리)은 이 파일에 그대로 쌓는다.  
핵심 결정과 원칙은 별도로 추출해 `principles/principles.md`에 유지한다.  
세션 재개 시 `principles.md`를 먼저 읽고, 특정 논의 맥락이 필요할 때만 `discussion_log.md`를 참조한다.

### 4.20 하네스 설계의 메타 원칙 — 그리고 인간의 몫

**공통 패턴 발견:**

지금까지의 모든 설계 결정을 보면 하나의 방향을 가리킨다:

```
CLI 내장 기억 → 블랙박스     → 의존하지 않는다
에이전트 기억 → 세션 소멸   → inject로 매번 재공급한다
추정          → 검증 안 됨  → 기록하지 않는다
대화 압축     → 통제 불가   → workspace 파일에 명시적으로 기록한다
```

> **메타 원칙: 신뢰할 수 없는 것에 의존하지 않고, 통제할 수 있는 것만으로 구조를 만든다.**

**하네스 자신에게 같은 원칙을 적용하면:**

하네스도 에이전트가 읽는 텍스트다. 에이전트가 텍스트를 제대로 읽고 따른다는 보장이 없다.  
하네스 자체도 완전히 신뢰할 수 없다.

그러면 하네스가 실패할 때를 하네스가 대비할 수 있는가?

**결론: 하네스의 실패는 인간의 몫이다.**

> "그건 인간의 몫으로 남겨야해."

이것은 피할 수 없는 구조적 귀결이다.  
깨진 시스템은 자기 자신이 깨졌다는 것을 알 수 없다.  
하네스가 실패하는 상황에서 하네스가 그 실패를 감지하고 대응하는 것은 논리적으로 불가능하다.  
**관찰자는 관찰 대상 밖에 있어야 한다.**

이것이 탐지/측정/보고 역설(4.6~4.8)과 동일한 구조다 — 이번엔 메타 레벨에서 반복된다.

**경계의 명확화:**

```
하네스가 담당:  알려진 위험 관리, 구조적 실패 예방, 컨텍스트 연속성
인간이 담당:    하네스 자체의 실패 감지, 메타 수준 판단, 시스템 밖 관찰
```

이것이 T2(환경 설계자) 역할의 근거이기도 하다.  
에이전트가 인식하지 못하는 가정을 인간이 먼저 인식하는 것 — 그 역량은 하네스 밖에 있어야 한다.

### 4.21 인간 관찰의 실질적 메커니즘 — 읽을 수 있는 결과물

**관찰:**

4.20에서 "하네스 실패는 인간의 몫"이라고 했다. 그 인간 개입이 어떤 형태인가?

> "사람이 읽을 수 있는, 검토할 수 있는 결과물을 만드는 과정이 중요한 거야. 계획서나 처리 결과서 같은 것들."

**결과물이 인간 관찰의 실질적 메커니즘이다:**

```
브리핑 → plan.md (인간이 검토) → 구현 → changelog.md (인간이 검토) → 완료
```

- `plan.md` "에이전트 보고" 섹션: 구현 전, 에이전트 가정을 인간이 읽을 수 있는 형태로 꺼낸다
- `changelog.md`: 구현 후, 무엇이 어떻게 바뀌었는지 인간이 확인할 수 있게 한다

layer2_checkpoint는 이 위에 올라가는 주기적 점검이다.  
일상적 관찰 지점은 이미 워크플로우 안에 내장되어 있다.

**설계 원칙으로 정리:**

> **결과물의 가독성이 인간 개입의 실질적 메커니즘이다.**

에이전트가 무엇을 가정했는지, 무엇을 변경했는지를 인간이 읽을 수 있어야  
시스템 밖에서 관찰하는 것이 가능하다.

반대로, 결과물이 인간이 읽기 어려운 형태라면 — 인간 개입은 명목상으로만 존재한다.

**귀결:**

하네스가 생성하는 모든 결과물(plan.md, changelog.md, 에이전트 보고)은  
에이전트가 아닌 **인간을 독자로** 설계해야 한다.

### 4.22 결과물은 다중 독자 문서다 — 인간과 여러 에이전트가 함께 사용

**관찰:**

> "그 결과물을 AI가 검토해서 만들고 결과를 또 독립 시행으로 검토해서 잘 구현됐는지 검토도 하겠지. 이게 같이 사용되는 문서로 봐야 돼."

plan.md와 changelog.md는 "인간이 읽는 문서"가 아니다. 여러 독자가 다른 시점에 다른 목적으로 읽는 **다중 독자 문서**다:

```
1. 설계 에이전트   → plan.md 작성
2. 인간           → plan.md 검토·승인
3. 구현 에이전트   → plan.md 읽고 구현 + changelog.md 작성
4. 검토 에이전트   → plan.md + changelog.md 읽고 정합성 확인 (독립 시행)
5. 인간           → 최종 확인
```

**독립 시행의 의미:**

4번의 검토 에이전트는 구현에 참여하지 않은 별도 에이전트다.  
구현한 에이전트는 자기 구현을 검증하기 어렵다 — 자기 가정을 그대로 가지고 있기 때문이다.  
이것이 T9(역할 분리로 편향 방지)의 구체적 구현이다.

**두 독자의 요구가 다르다:**

```
인간:    맥락과 의도를 읽는다 → 설명, 배경, "왜"가 필요하다
에이전트: 구조와 명시적 항목을 읽는다 → 체크리스트, 상태 기호, 명확한 경계가 필요하다
```

좋은 결과물은 두 독자를 동시에 만족해야 한다.  
구조화된 형식(에이전트용) + 맥락 설명(인간용)이 함께 있어야 한다.

**현재 plan.md 템플릿 평가:**

이미 이 방향으로 설계되어 있다:
- `에이전트 보고` 섹션: 인간이 검토할 내용을 명시적으로 꺼냄
- `구현 진행 상태` 테이블 + 상태 기호(⏳🔄✅): 에이전트가 구조적으로 읽을 수 있음
- `검증 체크리스트`: 검토 에이전트가 독립적으로 확인할 기준 제공

미결: 검토 에이전트(layer1_review)가 changelog.md와 plan.md를 비교할 때 필요한 정보가 충분한지 검토 필요.

### 4.23 검토 후 흐름 — 사용자 판단 우선, 위임 시 에이전트 판단

**검토 결과 처리 흐름:**

- `✅` → 완료
- `🚨 즉시 수정 필요` → 사용자가 판단 후 구현 세션 재개 지시. 사용자가 요청 시 에이전트가 직접 판단해서 재구현 가능.
- `⚠️ 주의 필요` → 사용자가 수정 여부 결정. 사용자가 요청 시 에이전트 판단 위임 가능.

**기본값: 사용자 판단. 위임 시: 에이전트 판단.**

4.21(인간이 관찰자)과 4.20(하네스 실패는 인간의 몫) 원칙과 일관성을 유지하면서도, 사용자가 명시적으로 위임하는 경우 에이전트가 자율적으로 처리할 수 있다.

**용어 정정 — "인간" → "사용자":**

> "인간이 아닌 사용자라고 해줘. 어느 순간과 단계에서 누가 그 역할을 맡는게 달라질 수 있으니."

"인간"은 특정 존재를 가정한다. "사용자"는 역할이다.

```
사용자 역할을 맡는 주체:
  - 실제 사람 (일반적 경우)
  - 오케스트레이터 에이전트 (자동화된 워크플로우)
  - 다른 시스템
```

단계마다, 워크플로우마다 사용자 역할을 맡는 주체가 달라질 수 있다.  
하네스는 "사용자"라는 역할을 정의하지, 그 역할을 누가 맡는지를 고정하지 않는다.

**단, 하나의 예외:**

4.20에서 "하네스 실패는 인간의 몫"이라고 했다 — 이건 역할이 아니라 시스템 밖의 관찰자를 지칭한다.  
하네스 자체가 실패했을 때 그것을 감지할 수 있는 주체는 시스템 외부에 있어야 하며, 그 역할은 자동화될 수 없다.

### 4.24 결과물의 최종 책임은 사용자에게 있다

**관찰:**

완전 자동화 파이프라인에서 "감독자를 누가 감독하는가" 문제를 제기했다.

**사용자의 답:**

> "그 결과를 사용자가 책임을 지는거야. 그 결과물을 사용하지 말지를 사용자가 판단하고 책임지는거야."

파이프라인이 얼마나 자동화되든 관계없다.  
결과물이 나왔을 때 그것을 **쓸지 말지 판단하고, 그 결과에 책임지는 것은 사용자다.**

**귀결:**

하네스의 역할이 다시 명확해진다:

```
하네스:   결과물을 만들고, 판단할 수 있도록 가시화한다
사용자:   결과물을 쓸지 말지 판단하고, 그 결과에 책임진다
```

"감독자를 누가 감독하는가"는 잘못된 질문이었다.  
감독 구조를 완성하는 것이 하네스의 역할이 아니다.  
하네스는 사용자가 판단할 수 있는 재료를 만드는 것까지다.

**4.21과의 연결:**

결과물이 읽을 수 없으면 사용자가 판단할 수 없다.  
사용자가 판단할 수 없으면 책임을 질 수도 없다.  
**결과물의 가독성 = 사용자 책임 행사의 전제 조건.**

### 4.25 역방향 흐름은 정의하지 않는다

**관찰:**

하네스의 순방향 흐름(브리핑 → plan → 구현 → 검토)은 명확하다.  
역방향 — 사용자가 결과를 거부했을 때 — 은 정의되어 있지 않다.

**사용자의 답:**

> "사용하지 않는다는 의미는 사용자에 따라서 여러 가지로 달라질 수 있어서 사용자에게 맡겨야 한다."

"쓰지 않는다"는 말이 담을 수 있는 의미:
- 전부 버리고 새로 시작
- 일부만 참고하고 다른 방향으로
- 지금은 보류하고 나중에 재검토
- 요구사항 자체를 바꾸고 다시 브리핑
- 방향 전환으로 새 태스크 생성

사용자마다, 상황마다 다르다. 하네스가 이것을 정의하면 오히려 방해가 된다.

**설계 결정:**

> 하네스는 흐름(구조)을 정의한다. 결과물 이후의 행동은 사용자가 판단한다.

> ⚠️ 두 가지 잘못된 연결:  
> - "사용하지 않는다 = 역방향으로 돌아간다" → 틀렸다  
> - "사용하지 않는다 = 실패다" → 틀렸다  
>
> 사용하지 않는다는 것은 사용자의 결정일 뿐이다.  
> 결과물이 나쁜 것도 아니고, 작업이 실패한 것도 아니다.  
> 그 시점에 사용자가 어떤 이유로든 그렇게 판단한 것이다.  
> 이후 무엇을 할지는 사용자에게 달려 있다.

하네스는 상태 머신이 아니다. 결과물을 만들고 가시화하는 것까지가 하네스의 역할이다.  
다음 행동을 결정하는 것은 사용자다 — 4.24와 일관된다.

### 4.26 실패는 과정에 있다 — 결과물은 실패가 아니다

**사용자 관찰:**

> "그 결과물에 이르는 과정에서 잘못된 것이 실패이지, 그 결과물은 실패라고 볼 수가 없어."

**구분:**

| | 실패인가 |
|-|--------|
| 에이전트가 요구사항을 잘못 이해했다 | ✅ 과정 실패 |
| 에이전트 가정이 틀렸는데 기록되지 않았다 | ✅ 과정 실패 |
| 아키텍처 규칙을 위반했다 | ✅ 과정 실패 |
| 결과물이 완성됐지만 사용자가 쓰지 않기로 했다 | ✗ 실패 아님 |
| 방향이 바뀌어서 결과물이 필요 없어졌다 | ✗ 실패 아님 |

**하네스 학습과의 연결:**

`roles/`, `upgrade-candidates/`에 기록하는 것은 **과정에서 무엇이 잘못됐는가**다.  
결과물이 사용됐는지 여부는 학습의 신호가 아니다.

놀라움 필터도 같은 방향이다: "다음에 이 역할이 몰랐다면 실수했을 내용" — 과정의 실수가 기록 대상이다.

**귀결:**

하네스가 "성공/실패"를 판단하는 기준은 결과물의 사용 여부가 아니라 **과정의 정합성**이다.  
plan.md의 체크리스트, 에이전트 가정 검증, 검토 단계가 모두 과정을 점검하는 장치다.

### 4.27 기본 흐름 — 계획서 → 사용자 검토·승인 → 진행

**사용자 명확화:**

> "기본은 계획서를 작성하고 사용자의 검토 후 계획서를 보완하거나 승인받고 다음 진행하는 게 기본이나, 사용자의 허락하에 승인 과정을 넘을 수 있는 거야."

**기본값:**
```
계획서 작성 → 사용자 검토 → (보완 또는 승인) → 구현
```

**예외:** 사용자가 명시적으로 허락한 경우에만 승인 단계를 건너뛸 수 있다.

**이전 논의 수정:**

에이전트가 "실패 비용을 추정해서 과정 무게를 스스로 조절한다"는 프레이밍이 틀렸다.  
에이전트는 비용을 추정하지 않는다. 기본은 항상 사용자 승인이다.  
승인을 건너뛰는 것은 에이전트의 판단이 아니라 사용자의 허락으로만 가능하다.

측정 역설(4.7) — "에이전트가 비용을 낮게 추정해서 가볍게 진행할 위험" — 은 이 구조에서 발생하지 않는다. 사용자가 허락하지 않으면 구조적으로 건너뛸 수 없다.

**layer1_design의 자율성 레벨 재해석:**

자율성 레벨(1/2/3)은 에이전트가 스스로 정하는 것이 아니라, **사용자가 허락한 범위**를 나타낸다.

### 4.28 사용자 안내는 plan.md 구조로 충분하다

**질문:**

사용자가 계획서를 검토할 때 무엇을 봐야 하는지 — 별도 사용자 프로토콜 문서가 필요한가?

**사용자 답:**

> "별도 문서 없이 plan.md 구조로 충분해."

**근거:**

plan.md의 구조 자체가 사용자를 안내한다.

```
에이전트 보고 (맨 위)
  → "구현 전에 이 섹션을 먼저 확인하세요"
  → 사용자 결정 필요 항목 목록
  → 에이전트 가정 테이블
```

사용자가 무엇을 봐야 하는지 기억하지 않아도, plan.md를 열면 구조가 안내한다.

이것이 "구조가 행동을 만든다" 원칙의 직접적 적용이다.  
4.1에서 지적한 비대칭("사용자 하네스는 내재화에만 의존") 문제를 plan.md 구조로 해소한다.

### 4.29 가치 결정에 영향도를 명시한다

**사용자 제안:**

> "계획서에 가치 결정이 필요한 것을 쓰고 그 가치에 따라 영향도를 적어주면 좋을 것 같아."

**현재 plan.md의 한계:**

"사용자 결정 필요" 섹션이 결정 항목만 나열한다.  
사용자는 결정해야 한다는 것은 알지만, 어떤 선택이 어떤 결과를 낳는지 모른 채 결정해야 한다.

**개선 방향:**

각 가치 결정 항목에 선택지별 영향도를 함께 명시한다.

```
- [ ] 성능 vs. 가독성 중 무엇을 우선할 것인가?
  - 성능 우선 → 응답 속도 50% 향상, 코드 복잡도 증가, 유지보수 어려움
  - 가독성 우선 → 유지보수 용이, 팀 온보딩 빠름, 응답 속도 현재 수준 유지
```

사용자가 결정하는 것이 아니라 **근거를 갖고 결정할 수 있게** 된다.

**4.28과의 연결:**

plan.md 구조가 사용자를 안내한다는 원칙의 확장이다.  
가치 결정 항목의 영향도까지 구조 안에 담으면, 사용자가 별도 분석 없이 계획서만 보고 결정할 수 있다.

### 4.30 plan.md는 완성된 계약서가 아니라 살아있는 항법 도구다

**사용자 재정의:**

> "plan.md는 무조건 처음부터 완성되어야 하는 문서는 아니야. 사용자와 agent가 최선의 길을 선택해서 나아갈 수 있도록 도와주는 역할을 하는 문서야."

**관점 전환:**

```
기존 관점: 계획 완성 → 승인 → 실행  (완성된 설계도, 계약서)
새 관점:   지금 아는 것으로 시작 → 함께 나아가며 채워짐  (살아있는 나침반)
```

**함의:**

- 처음부터 모든 것을 알 필요가 없다. 지금 알 수 있는 것에서 시작한다.
- 구현 중 발견이 생기면 plan.md가 업데이트된다.
- 가정이 틀렸다는 걸 알게 되면 plan.md가 바뀐다.
- 사용자와 에이전트가 현재 인식을 공유하는 도구다.

**layer0_init "씨앗 심기"(4.11)와 동일한 원칙:**

하네스 자체가 처음부터 완성되지 않듯, plan.md도 처음부터 완성될 필요가 없다.  
알 수 있는 것에서 시작하고, 경험하면서 채운다.

**`구현 진행 상태` 테이블의 의미 재해석:**

단순한 체크리스트가 아니라 — 현재 어디까지 왔는지, 다음엔 어디로 갈지를  
사용자와 에이전트가 함께 보는 현재 위치 표시다.

### 4.31 검토와 승인은 별개 단계다

**사용자 명확화:**

```
계획서 작성
    → 사용자 검토
    → 계획 수정 또는 확정
    → 사용자 승인
    → 진행
```

검토(검토)와 승인(승인)은 다르다:

- **검토**: 계획서를 보고 피드백을 준다. 수정이 필요하면 에이전트가 수정하고 다시 검토. 반복 가능.
- **승인**: 계획이 확정됐을 때 진행을 허락하는 공식 단계. 이후 구현 시작.

```
검토 → 수정 → 검토 → 수정 → ... → 확정 → 승인 → 진행
```

**4.30과의 결합:**

plan.md가 살아있는 문서이므로, 진행 중 새로운 발견이 생기면  
다음 구간에 대해 다시 검토/수정/승인 사이클이 시작된다.

```
승인 → 진행 → 발견 → plan.md 업데이트 → 검토 → 승인 → 계속 진행
```

**귀결:**

승인은 "이 계획 전체에 대한 최종 서명"이 아니라 **"지금 진행할 구간에 대한 허락"** 이다.

### 4.32 과정이 아니라 내용이다 — 쓰여진 것만 판단 가능하다

**사용자 관찰:**

> "과정이라고는 하지만 최대한 사용자에게 내용을 작성해서 보여줘야 돼. 그래야 사용자가 그 내용을 근거로 판단을 하니깐."

**핵심:**

에이전트가 내부적으로 고려했더라도 쓰지 않으면 사용자에게는 존재하지 않는 것과 같다.  
사용자가 판단할 수 있는 것은 오직 **눈앞에 쓰여진 것**뿐이다.

```
에이전트가 생각만 함  →  사용자에게 없음
에이전트가 써서 보여줌 →  사용자가 판단 가능
```

**이전 원칙들의 실행 조건:**

- 4.21 결과물 가독성 = 사용자 개입의 전제 → 읽을 수 있으려면 먼저 **쓰여져야 한다**
- 4.29 가치 결정에 영향도 명시 → 영향도를 **써야** 사용자가 선택 가능
- 4.30 plan.md 살아있는 항법 도구 → 도구가 되려면 내용이 **채워져야** 한다
- 에이전트 보고 섹션 → 가정이 **쓰여져야** 사용자가 검토 가능

**귀결:**

구조(plan.md, 에이전트 보고, 영향도)가 에이전트가 쓰도록 강제한다.  
"구조가 행동을 만든다" = "구조가 쓰게 만든다."  
쓰지 않으면 어떤 과정도 사용자에게는 없는 것이다.

### 4.33 구현 전 탐색 — 발견은 구현 시작 전에 해야 한다

**사용자 관찰:**

> "구현의 처음 시작에 해당 문제를 먼저 발견하고 사용자에게 보고 하는 게 맞지 않을까?"

**두 가지 구분:**

**구현 전 탐색 단계 (코드베이스를 직접 보고 나서 드러나는 것)**

```
탐색 → 문제 발견 → 사용자 보고 → plan 수정 → 재승인 → 구현 시작
```

코드베이스를 실제로 열어보기 전에는 알 수 없는 것들이 있다.  
이 탐색을 구현 시작 전에 먼저 하고, 발견한 것을 보고한 뒤 plan을 수정해서 재승인받는다.  
그 이후에 구현을 시작한다.

**구현 중 조용한 결정 (plan 범위 안에서의 세부 결정)**

이것은 다른 종류다. plan을 바꿀 수준이 아닌 세부 결정은 changelog에 기록하면 된다.

**이전 예시 수정:**

"API 스펙이 달랐다", "범위가 예상보다 크다" — 이것들은 탐색 단계에서 먼저 발견했어야 하는 것이다. 구현 중 발견이 아니라 탐색 단계의 실패다.

**귀결:**

구현 중 판단이 필요한 경우는 크게 두 가지다:
- 탐색에서 놓친 것이 뒤늦게 드러난 경우 → 멈추고 사용자에게 보고, plan 수정
- Plan 범위 안의 세부 결정 → 진행하고 changelog에 기록

### 4.34 하네스의 적용 범위 — 소프트웨어 개발 작업

**사용자 명확화:**

> "지금 만드는 하네스는 프로그램의 개발에 관한 하네스야. 그래서 지금 예로 든 탐색적 작업이나 단순 확인 같은 질문은 이 절차를 따르지 않는 게 맞아."

**적용 범위:**

| 하네스 적용 O | 하네스 적용 X |
|------------|------------|
| 기능 구현 | 탐색적 확인 ("이게 되는지 봐봐") |
| 버그 픽스 | 단순 질문 ("이 함수 설명해줘") |
| 리팩터링 | 학습용 프로토타입 |
| 코드 검토 | 아이디어 탐색 |
| 아키텍처 결정 | |

**귀결:**

agent_rules.md의 작업 유형 분류표가 이미 이 범위를 반영하고 있다.  
분류표에 해당하지 않는 요청은 하네스 절차 없이 자연스럽게 응답한다.  
"작업 유형이 불분명하면 사용자에게 확인한다"가 경계 케이스의 처리 방식이다.

### 4.36 적용 범위 재정의 — Task 기준, 작업 유형은 내용이 결정

**사용자 수정:**

> "소프트웨어 개발에 한정짓기 보다. 설계나 기획 검토는 그 Task 문서가 완성되면 판별이 될 것 같아. 그전까지는 작업의 설계 단계가 맞는 것 같아. 물론 내용이 개발에 관련된 내용이면 개발을 하고 리서치에 관련된 내용이면 리서치를 하게 될거니깐."

**4.34와 4.35 수정:**

4.34에서 "소프트웨어 개발에 한정"했던 것은 틀렸다. 범위를 다시 정의한다.

**적용 기준: Task를 만들어야 할 규모의 작업**

```
사용자가 Task 생성 요청
    → 워크플로우 시작
    → plan.md 작성 (설계 단계)
    → Task 내용이 작업 유형을 결정
        개발 관련 → 개발 진행
        리서치 관련 → 리서치 진행
        기획/검토 관련 → 기획/검토 진행
```

**Task 이전은 모두 설계 단계:**

Task 문서가 완성되기 전까지는 작업 유형을 단정하지 않는다.  
plan.md의 내용이 채워지면서 자연스럽게 드러난다.

**적용 X (Task가 필요 없는 작업):**

- 단순 질문
- 탐색적 확인
- 작은 규모의 즉시 처리 가능한 요청

### 4.37 Task 필요 여부 — 에이전트가 판단하면 사용자에게 확인한다

**사용자 요청:**

> "사용자가 요청한 작업이 태스크를 만들 정도의 작업이라고 판단되는 경우 사용자에게 물어봐주는 게 좋을 것 같아."

**흐름:**

```
사용자 요청
    → 에이전트: Task가 필요한 규모로 판단됨
    → 사용자에게 확인: "이 작업은 Task를 만들어 진행하는 게 좋을 것 같습니다. Task로 시작할까요?"
    → 사용자: 예 → Task 생성 → 워크플로우 시작
    → 사용자: 아니오 → Task 없이 바로 진행
```

**이유:**

에이전트의 규모 판단이 틀릴 수 있다.  
사용자 입장에서는 작은 작업일 수도, 에이전트 입장에서는 큰 작업일 수도 있다.  
4.32("쓰여진 것만 판단 가능")와 일관된다 — 에이전트의 판단을 사용자가 볼 수 있어야 한다.

### 4.38 layer2_checkpoint는 사용자 주도로 충분하다

개별 Task들이 쌓였을 때 전체 방향과의 정합성을 점검하는 것이 layer2_checkpoint다.  
이 점검 시점을 자동 트리거로 제안할 필요는 없다.

> "사용자 주도로 충분해."

언제 전체를 점검할지는 사용자가 판단한다.  
4.20("하네스 실패는 인간의 몫")과 일관된다 — 시스템 수준의 판단은 사용자 영역이다.

### 4.39 긴급 상황도 사용자 판단이다

**관찰:**

긴급 상황(프로덕션 버그, 즉각 대응 필요)에 하네스가 별도 절차를 정의해야 하는가?

**사용자 답:**

> "긴급 상황도 사용자 판단에 맡겨야 돼."

**이미 4.27에 답이 있다:**

"사용자 허락 하에 단계를 건너뛸 수 있다."  
긴급 상황은 그 허락이 빠르게 주어지는 경우일 뿐이다. 별도 절차가 필요 없다.

```
긴급 상황 발생
    → 사용자가 "바로 진행해"라고 허락
    → 에이전트: 단계 건너뛰고 진행
    → (완료 후 changelog 또는 소급 기록 — 사용자 판단)
```

**귀결:**

하네스는 긴급 모드를 정의하지 않는다.  
모든 예외적 상황은 사용자가 그때그때 판단한다.  
하네스는 판단의 틀을 제공할 뿐, 상황별 판단을 선점하지 않는다.

### 4.40 upgrade-candidates 파이프라인 — 인스톨/업데이트가 지식 전달의 트리거

**2 & 3. 전파와 필터링 — 파이프라인 구조:**

### 4.42 하네스 시스템 자체 업데이트 — 사용자 주도 원칙

> "하네스 업데이트 후보에서 하네스 시스템 자체의 업데이트는 사용자가 내용을 확인하면서 작업을 진행해야 한다."

**이유:**  
하네스 시스템 파일(`.mpa-workspace/inject/`, `workspace-template/` 등)의 변경은 모든 프로젝트에 전파된다. 에이전트가 upgrade-candidates를 일괄 자동 적용하면 검토되지 않은 변경이 전체 하네스에 반영된다.

**원칙:**  
- 하네스 시스템 파일 수정은 **후보 하나씩, 사용자 확인 후 진행**
- 에이전트는 후보 내용과 변경 대상을 보여주고 대기한다
- 일괄 자동 적용 금지

**4.32와의 연결:**  
"쓰여진 것만 판단 가능하다" — 에이전트가 스스로 하네스를 수정하면 사용자가 판단할 기회가 없다.  
하네스 수정도 동일 원칙: 보여주고 → 사용자 확인 → 적용.

**구현:**  
`layer0_update.md`에 1.5단계 추가 — 하네스 프로젝트 작업 시 후보별 사용자 확인 절차.  
→ 이후 4.43, 4.44에서 수정됨.

### 4.43 하네스 시스템 업데이트 — 사용자 요청 시작 + upgrade-candidates 이전 방지

> "하네스 시스템의 업데이트는 사용자가 요청으로 시작되어야 해. 그리고 하네스 시스템 자체에 대한 내용은 프로젝트 쪽으로 이전 시킬 필요가 없는데 혹시 같이 이전이 되는지 확인해줘."

**두 가지 확인 사항:**

1. **사용자 요청 시작**: 4.42에서 추가한 1.5단계가 "조건 없이 세션 흐름에 포함"되어 있어 에이전트가 자발적으로 실행할 여지가 있었다. "하네스 프로젝트 작업 중 + 사용자 명시적 요청" 두 조건을 모두 충족할 때만 진입하도록 명확화.

2. **upgrade-candidates 이전 여부**: `install.py`를 확인한 결과 이미 올바르게 처리되고 있음 — `copy_agents_workspace()` 후 `clear_upgrade_candidates()`를 즉시 호출해 내용을 비운다. 디렉토리 자체는 남겨두는데, 프로젝트가 자체 후보를 기록하는 공간이기 때문.

→ 4.44에서 더 근본적인 구조 오류 발견.

### 4.44 inject/ 파일은 프로젝트에 복사된다 — 구조 오류 발견

> "layer0_update.md 부분에서 하네스 자체의 업그레이드 부분이 포함되어 있는것 같은데 이 파일이 프로젝트로 복사되는거 아냐?"

**발견한 구조 오류:**  
`layer0_update.md`는 `.mpa-workspace/inject/` 안에 있다. `install.py`는 `.mpa-workspace/` 전체를 프로젝트에 복사한다. 따라서 4.42에서 추가한 1.5단계(하네스 시스템 자체 업데이트 지침)가 모든 프로젝트 copy에 포함된다.

**원칙 재확인:**  
`.mpa-workspace/inject/` 는 템플릿 파일 — 프로젝트용 지침만 담아야 한다.  
하네스 시스템 자체 관리 지침은 harness 루트에 별도 파일로 분리해야 한다.

**수정:**
- `layer0_update.md`에서 1.5단계 완전 제거 → 프로젝트용 내용(아웃바운드 + 인바운드)만 남김
- `harness/harness-maintenance.md` 신규 생성 (harness 루트, `.mpa-workspace/` 밖) → 하네스 시스템 자체 유지보수 지침. 프로젝트에 설치되지 않음.

**도출된 원칙:**  
`.mpa-workspace/` 하위 파일을 작성할 때는 "이 파일이 프로젝트에 복사된다"는 점을 항상 의식해야 한다.  
하네스 내부용 내용은 반드시 harness 루트 또는 별도 비설치 경로에 위치시킨다.

### 4.45 .mpa-workspace/ 전체 파일 검토 — 하네스 내부 내용 분리

> "나머지 파일도 검토해줘. 하네스 시스템 자체에 관한 내용은 프로젝트에 넘기지 않는 게 좋을 것 같아. 단, 프로젝트에서 하네스로 자산을 가져오기 위한 작업은 당연히 프로젝트에 내재화되는 게 맞아."

**검토 결과:**

| 파일 | 판단 | 근거 |
|------|------|------|
| `agent_rules.md` | ✅ 프로젝트용 | upgrade-candidates 기록(아웃바운드)은 프로젝트에 내재화되어야 함 |
| `extensions.md` | ✅ 프로젝트용 | "이 파일에 추가"는 프로젝트 자체 파일 수정 |
| `layer2_checkpoint.md` | ✅ 프로젝트용 | upgrade-candidates 기록 지시 — 아웃바운드, 프로젝트용 |
| `README.md` | ❌ 수정 필요 | "직접 수정하지 않는다", "harness 레포에서 배포", "harness 레포로 이전" |
| 나머지 파일 | ✅ 모두 프로젝트용 | 하네스 내부 관리 내용 없음 |

**수정: `README.md`**
- "직접 수정하지 않는다. 업데이트는 harness 레포에서 덮어쓰기로 배포된다." → "프로젝트 환경에 맞게 커스터마이즈할 수 있다."
- "harness 레포의 upgrade-candidates/로 이동해 정제한다." → "하네스를 업데이트할 때 하네스 프로젝트로 이전된다."

**기준 명확화:**  
`.mpa-workspace/` 파일에 포함 가능한 것 = "프로젝트 → 하네스" 방향 작업 (upgrade-candidates 기록·이전)  
포함 불가한 것 = 하네스 시스템 자체 파일 수정·관리·배포에 관한 지침

A와 B는 경쟁이 아닌 상호보완. **미결: principles.md 작성.** 4.20 메타 원칙 + 인간의 몫: 모든 설계 결정의 공통 패턴 = "신뢰할 수 없는 것에 의존하지 않고 통제 가능한 것만으로 구조를 만든다." 하네스 자신의 실패는 하네스가 감지 불가 → 인간의 몫. 관찰자는 관찰 대상 밖에 있어야 함. 탐지/측정/보고 역설의 메타 반복. 하네스 담당/인간 담당 경계 명확화. 4.21 인간 관찰의 실질적 메커니즘: 결과물의 가독성 = 인간 개입의 실질. plan.md(구현 전 검토) + changelog.md(구현 후 검토)가 일상적 관찰 지점. layer2_checkpoint는 그 위의 주기적 점검. 하네스 결과물은 에이전트가 아닌 인간을 독자로 설계해야 함. 4.22 결과물은 다중 독자 문서: plan.md + changelog.md는 설계/구현/검토 에이전트와 인간이 각각 다른 시점에 다른 목적으로 읽는다. 독립 시행 = 구현 미참여 에이전트가 정합성 확인(T9 역할 분리의 구체적 구현). 좋은 결과물 = 구조화된 형식(에이전트용) + 맥락 설명(인간용) 동시 만족. 4.23 검토 후 흐름 + 용어 정정: 기본값 = 사용자 판단, 위임 시 에이전트 판단. "인간" → "사용자"로 정정 — 사용자는 역할이며 실제 사람·오케스트레이터 에이전트 등 단계마다 달라질 수 있음. 단, 하네스 실패 감지는 시스템 외부의 인간이 담당 (자동화 불가). 4.24 결과물의 최종 책임: 파이프라인 자동화 수준과 무관하게 결과물을 쓸지 말지 판단하고 책임지는 것은 사용자. 하네스 = 판단 가능한 재료 제공. 가독성 = 사용자 책임 행사의 전제 조건. 4.25 역방향 흐름은 정의하지 않는다: "쓰지 않는다"의 의미가 사용자·상황마다 다름. 역방향도 실패도 아님 — 사용자의 결정일 뿐. 하네스는 상태 머신이 아니다. 4.26 실패는 과정에 있다: 결과물 사용 여부는 실패 기준이 아님. 실패 = 과정의 정합성 문제(잘못된 이해, 미기록 가정, 규칙 위반). 하네스 학습(roles/, upgrade-candidates/)도 과정 실수를 기록. 결과물 사용 여부는 학습 신호가 아님. 4.27 기본 흐름: 계획서 → 사용자 검토·승인 → 구현. 승인 건너뜀은 사용자 허락으로만 가능. 에이전트가 실패 비용 추정으로 과정을 스스로 조절하는 것이 아님 → 측정 역설 우려 해소. 자율성 레벨 = 에이전트 판단이 아닌 사용자가 허락한 범위. 4.28 사용자 안내는 plan.md 구조로 충분: 별도 사용자 프로토콜 문서 불필요. plan.md 구조 자체(에이전트 보고 섹션)가 사용자를 안내. "구조가 행동을 만든다" 원칙의 직접 적용. 4.29 가치 결정에 영향도 명시: "사용자 결정 필요" 항목에 선택지별 영향도 추가. 사용자가 근거를 갖고 결정 가능. plan.md 템플릿 + layer1_design.md 업데이트. 4.30 plan.md = 살아있는 항법 도구: 처음부터 완성 필요 없음. 사용자+에이전트가 함께 최선의 길을 찾아가는 현재 인식 공유 도구. 발견·가정 수정 시 업데이트됨. layer0_init "씨앗 심기"와 동일 원칙. 4.31 검토와 승인은 별개 단계: 검토(피드백·수정, 반복 가능) → 확정 → 승인(진행 허락) → 구현. 승인 = 전체 계획 서명이 아니라 지금 구간에 대한 허락. 발견 시 다음 구간에 대해 검토/승인 사이클 재시작. 4.32 쓰여진 것만 판단 가능: 과정보다 내용이 먼저. 에이전트가 고려했어도 쓰지 않으면 사용자에게 없는 것. "구조가 행동을 만든다" = "구조가 쓰게 만든다." 모든 판단 근거는 텍스트로 외재화되어야 한다. 4.33 구현 전 탐색: 발견은 구현 시작 전에 해야 함. 탐색→발견→보고→plan수정→재승인→구현. 구현 중 판단은 두 종류: 탐색 실패로 뒤늦게 드러난 것(멈추고 보고) vs plan 범위 내 세부 결정(changelog 기록). 4.34 하네스 적용 범위(수정됨→4.36): 초안은 소프트웨어 개발 한정이었으나 수정됨. 4.35 명칭(수정됨→4.36): "소프트웨어 개발 워크플로우"로 좁혔으나 수정됨. 4.36 적용 범위 재정의: 기준 = Task를 만들어야 할 규모의 작업. Task 내용이 유형(개발/리서치/기획 등)을 결정. Task 이전은 설계 단계. 소프트웨어 개발에 한정하지 않음. 4.37 Task 필요 여부 확인: 에이전트가 Task 필요 규모라 판단하면 먼저 사용자에게 확인. agent_rules.md 업데이트. 4.38 layer2_checkpoint 트리거: 자동 트리거 불필요. 사용자 주도로 충분. 시스템 수준 판단은 사용자 영역(4.20과 일관). 4.39 긴급 상황도 사용자 판단: 하네스는 긴급 모드를 별도 정의하지 않는다. 4.27의 "사용자 허락 하에 단계 건너뜀"이 이미 커버. 하네스는 상황별 판단을 선점하지 않는다. 4.40 미결 해소 + 4.41 구현: upgrade-candidates 파이프라인 구현 완료. layer0_update.md(신규) — 하네스 업데이트/재설치 세션 inject 파일, 아웃바운드(프로젝트→하네스) + 인바운드(하네스→프로젝트) 양방향 흐름 명시. agent_rules.md — "하네스 업데이트/재설치" 작업 유형 추가, upgrade-candidates 템플릿에 적용 범위 필드 추가. layer0_init(첫 설치, 단방향)과 layer0_update(업데이트, 양방향) 분리. 4.42 하네스 시스템 자체 업데이트 원칙: 하네스 파일 수정은 후보 하나씩 사용자 확인 후 진행. 일괄 자동 적용 금지. 4.43 사용자 요청 시작 + upgrade-candidates 이전 방지: install.py가 이미 올바르게 처리 중(copy 후 clear). 4.44 inject/ 구조 오류 발견: layer0_update.md가 프로젝트에 복사되므로 하네스 내부 지침(1.5단계) 포함 불가. layer0_update.md에서 1.5단계 제거, harness/harness-maintenance.md 신규 생성(harness 루트, 프로젝트에 설치 안 됨). 도출 원칙: .mpa-workspace/ 하위 파일 작성 시 "프로젝트에 복사된다"는 점 항상 의식. 4.45 .mpa-workspace/ 전체 검토: README.md만 수정 필요(harness 배포 문구 제거, "커스터마이즈 가능"으로 변경). 나머지는 모두 프로젝트용. 4.46 harness-maintenance.md 사용 시점: 특정 상황(upgrade-candidates 반영 작업)에만 읽는 문서. harness/README.md 업데이트 파이프라인 "검토 및 정제" 단계에 참조 추가. 4.47 principles.md 업데이트: ".mpa-workspace/=템플릿", "설치/업데이트=지식전달 트리거", "하네스 업데이트=사용자 주도" 세 원칙 추가. |

### 4.46 harness-maintenance.md 사용 시점 확정

> "2번은 하네스 프로젝트를 접근하면 반드시 읽어야 되는 문서로 보이는데 맞아? 아니면 특정 상황에만 읽으면 돼?"

**결론: 특정 상황에만 읽는 문서.**

`harness-maintenance.md`는 upgrade-candidates를 하네스 템플릿에 반영하는 작업을 할 때만 필요하다. 매 세션 필수 문서가 아니라 "하네스 개선 작업" 시 꺼내보는 작업 지침서.

**발견 가능성 해소:**  
harness/README.md의 "하네스 업데이트 파이프라인" 섹션 "검토 및 정제" 단계에 `harness-maintenance.md` 참조 추가.  
파이프라인 흐름상 자연스럽게 발견된다 — 별도 섹션 불필요.

### 4.48 .mpa-workspace/ 직접 수정 권장하지 않음

> ".mpa-workspace/ 는 되도록 사용자가 수정 안 하는 걸 권장해"

4.45에서 `.mpa-workspace/README.md`를 "커스터마이즈 가능"으로 바꿨는데, 이는 잘못된 방향이었다.

**원칙:**  
`.mpa-workspace/`는 되도록 직접 수정하지 않는다.  
개선이 필요하면 `upgrade-candidates/`에 후보를 기록하고, 하네스 업데이트로 반영하는 것이 올바른 흐름이다.

**이유:**  
직접 수정하면 다음 하네스 업데이트 시 덮어써진다.  
upgrade-candidates 파이프라인이 존재하는 이유이기도 하다.

**수정:**  
`.mpa-workspace/README.md` — "커스터마이즈 가능" → "되도록 직접 수정하지 않는다. 개선은 upgrade-candidates로."
